# ハンドオフ自動検出ルール

> このルールは常時有効。ファイル書き込み後に自動適用される。

## 検出タイミング

部署のAgent（サブエージェント）が成果物を作成・更新した後、秘書（メインエージェント）は成果物を確認する。

## 推奨フォーマット（構造化YAML）

部署は成果物の末尾に以下のYAMLブロックでハンドオフを記述する。
正規表現マッチより確実にパースでき、ネストや重複を防止できる。

```yaml
# handoff
handoff:
  - to: pm
    tasks:
      - "上記改善案をタスクチケットに分割"
      - "P0施策の期限設定"
  - to: ai-dev
    context: "上記設計に基づき API の仕様策定"
    tasks:
      - "APIの仕様策定"
```

## 後方互換: Markdownパターン（非推奨）

以下のパターンも引き続き検出する（既存成果物との互換性のため）:

```
→ {部署名}部への依頼
→ {部署名}への依頼
### {部署名}部への依頼事項
## ハンドオフ
```

**新規の成果物では必ずYAMLフォーマットを使用すること。**

## 検出後のアクション

1. 依頼内容を抽出する（YAML → パースで確実に、Markdown → 正規表現でベストエフォート）
2. 現在のパイプラインに含まれる部署 → そのまま次ステップとして Agent 起動
3. パイプラインに含まれない部署 → 社長に確認してから起動
4. 記録: Supabase `activity_log` に `dept_handoff` を INSERT（可能な場合）
5. **同一ハンドオフの重複実行を防止**: ファイル内のhandoffブロックが処理済みなら再起動しない

## 成果物の自動登録

ファイルが以下の条件を**全て**満たす場合、artifacts テーブルへの登録を検討する:

1. パスが登録対象（`.company/departments/`, `docs/final_output/`, `secretary/archive/` 等）
2. ファイルサイズが 5KB 以上
3. 拡張子が `.md`, `.html`, `.pptx`, `.pdf`
4. ファイル名が `temp-`, `wip-`, `draft-` で始まらない

条件を満たす場合、パイプライン完了報告時にまとめて登録する。
