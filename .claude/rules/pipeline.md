# パイプライン自動選択・実行ルール

> このルールは常時有効。社長からタスク指示を受けた時に自動適用される。

## パイプライン選択（自動判定）

社長の指示内容から適切なパイプラインを自動選択する:

| キーワード | パイプライン | フロー |
|-----------|------------|--------|
| 「作って」「実装して」「新機能」「追加して」 | A: 新機能開発 | リサーチ ∥ UXデザイン → AI開発 ∥ PM → システム開発 → QA |
| 「直して」「バグ」「修正」「エラー」 | B: バグ修正 | システム開発 → QA |
| 「資料」「プレゼン」「提案書」「まとめて」 | C: 資料作成 | リサーチ ∥ UXデザイン → 資料制作 |
| 「調べて」「調査」「比較」「分析」 | D: 調査 | リサーチ（並列可） → 統合報告 |
| 「セキュリティ」「脆弱性」「監査」 | E: セキュリティ | セキュリティ ∥ リサーチ → システム開発 → QA |

**判定できない場合**: 社長に確認する。

## 並列実行

**Agent tool を1メッセージ内で複数呼べば並列実行される。**

依存関係の判定:
- A の出力を B が必要 → 直列（A完了後にB）
- A と B が互いの出力を不要 → 並列（同時起動、`run_in_background=true`）

## 部署の起動方法

Agent tool で対応するサブエージェントを起動。**最小権限の原則**に従い、モデルとツール範囲を指定する:

| 部署 | subagent_type | モデル | ツール範囲 |
|------|--------------|--------|-----------|
| リサーチ | `リサーチ部` | opus（高精度） | Read, Web系のみ |
| UXデザイン | `UXデザイン部` | opus | Read, Web系のみ |
| AI開発 | `AI開発部` | opus | 全ツール |
| システム開発 | `システム開発部` | sonnet | 全ツール |
| PM | `PM部` | haiku（高速） | Read, Write |
| 資料制作 | `資料制作部` | sonnet | Read, Write |
| 情報収集 | `情報収集部` | haiku | Read, Web系のみ |
| セキュリティ | — | opus | 全ツール |
| リファクタリング | — | opus | 全ツール |

## 完了検証（必須）

部署が「完了」と報告した際、秘書は以下を確認する:

- **コード変更の場合**: テストが実行済みか（git log でテスト実行の痕跡を確認）
- **ドキュメントの場合**: 必須セクション（結論・ネクストアクション）が含まれるか
- **成果物の場合**: artifacts テーブルへの登録が完了しているか
- **ハンドオフの場合**: YAML形式でハンドオフが記述されているか

不足があれば部署に差し戻す。

## 実行モード

小さなタスク（1部署で完結）: そのまま実行、報告
中規模タスク（2-3部署）: checkpoint なしで自動実行、完了報告
大規模タスク（4部署以上）: 設計完了後に checkpoint → 社長承認 → 実装
