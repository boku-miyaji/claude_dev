# パイプライン計画・実行ルール

> このルールは常時有効。`/company` の起動有無に関わらず適用される。

## いつ部署計画を立てるか

**全ての指示で部署を使うわけではない。** タスクの規模で判断する:

| 規模 | 判断基準 | 動き方 |
|------|---------|--------|
| **小** | 1ファイルの修正、質問への回答、設定変更など | **そのまま実行**。部署計画は不要 |
| **中** | 2部署以上が必要、または原因調査+実装など複合作業 | **部署計画を立てて実行**。計画を簡潔に提示 |
| **大** | 4部署以上、新機能開発、大規模リファクタ | **部署計画を立てて提示 → 社長承認 → 実行** |

**迷ったら「中」として部署計画を立てる。** 計画なしに複数部署にまたがる作業を始めない。

## 部署計画の立て方

### Step 1: タスク分析

- **何を達成するか**（ゴール）
- **何を知る/作る/検証する必要があるか**
- **何かおかしい点はないか**（異常の兆候）

### Step 2: 部署選定

利用可能な全部署から、必要なものを選ぶ。1つでも複数でもよい。

| 部署 | subagent_type | 得意領域 | モデル |
|------|--------------|---------|--------|
| リサーチ | `リサーチ部` | 市場・技術・企業の外部調査 | opus |
| 調査部 | `general-purpose` (opus) | 内部システムの問題特定・根本原因分析・再発防止 | opus |
| UXデザイン | `UXデザイン部` | ユーザー体験・行動文脈・UI設計 | opus |
| AI開発 | `AI開発部` | LLM/AI の要件定義・設計・実装・評価 | opus |
| システム開発 | `システム開発部` | バックエンド・フロントエンド・QA | sonnet |
| PM | `PM部` | プロジェクト管理・チケット・マイルストーン | haiku |
| 資料制作 | `資料制作部` | 提案書・プレゼン・デモ資料 | sonnet |
| 情報収集 | `情報収集部` | 外部の最新ニュース・X監視・Web巡回（速報・監視） | opus |
| セキュリティ | — | 脆弱性診断・監査・RLS設計 | opus |
| マーケティング | — | プロダクトの商用化・ポジショニング | opus |
| 運営改善 | — | Hook/バッチ/ルールの仕組み改善 | opus |

### Step 3: 依存関係と実行順序

DAG（有向非巡回グラフ）を組む：
- A の出力を B が必要 → 直列（A → B）
- A と B が互いの出力を不要 → 並列（A ∥ B）

**Agent tool を1メッセージ内で複数呼べば並列実行される。**

### Step 4: 計画の提示（中規模以上）

```
📋 タスク: [要約]
🏢 部署: リサーチ ∥ UXデザイン → AI開発 → システム開発
⏱  見積もり: 約XX分
```

大規模タスクでは社長の承認を得てから実行。中規模はそのまま実行。

## 参考: よくある組み合わせパターン

これに縛られず、タスクに応じて自由に組み替える。

| パターン | よくある流れ |
|---------|------------|
| 新機能開発 | リサーチ ∥ UXデザイン → AI開発 → システム開発 → QA |
| バグ修正（原因不明） | 調査部 → システム開発 → QA |
| 軽微な修正 | システム開発のみ |
| 資料作成 | リサーチ → 資料制作 |
| システム障害 | 調査部 → システム開発 or 運営改善 |

## 異常検知（常時稼働）

タスクの規模に関わらず、作業中に以下を検知したら **調査部の投入を検討する**：

- 前提としていた設定・ファイルが存在しない / 変わっている
- hookが走ったはずなのに結果がない
- 前回のセッションで完了した作業が反映されていない
- 部署のアウトプットにエラーや矛盾がある
- コミット済みの状態とワーキングコピーの乖離が大きい

明らかな障害は即座に調査部を起動。判断に迷う場合は社長に確認。

## 完了検証（必須）

部署が「完了」と報告した際の確認:

- **コード変更**: テスト実行済みか
- **ドキュメント**: 結論・ネクストアクションが含まれるか
- **成果物**: artifacts テーブルへの登録が完了しているか
- **ハンドオフ**: YAML形式で記述されているか

不足があれば部署に差し戻す。

## Managed Agents パターン（セッション永続化）

> Anthropic Engineering の Managed Agents 設計に基づく。Session = append-only 外部ログ、Harness = ステートレス。

### Session Log（agent_sessions テーブル）

中〜大規模パイプライン実行時、以下のイベントを `agent_sessions` に記録する:

| event_type | タイミング | payload |
|---|---|---|
| `pipeline_start` | パイプライン開始 | `{name, dag, task_summary}` |
| `dept_dispatch` | 部署起動 | `{description, model}` — Hook が自動記録 |
| `dept_complete` | 部署完了 | `{artifacts, handoff}` |
| `dept_error` | 部署エラー | `{error, retry_count}` |
| `handoff` | 部署間ハンドオフ | `{from, to, context}` |
| `checkpoint` | 社長承認ポイント | `{decision, reason}` |
| `pipeline_complete` | パイプライン完了 | `{summary, artifacts}` |

**`session_id` は環境変数 `CLAUDE_SESSION_ID` を使う。なければ日付ベース。**

### Pipeline State（pipeline_state テーブル）

中〜大規模パイプラインでは `pipeline_state` に DAG と進捗を保存:

```json
{
  "session_id": "20260410",
  "pipeline_name": "narrator-implementation",
  "dag": [
    {"step": "research", "dept": "リサーチ部", "status": "completed"},
    {"step": "design", "dept": "UXデザイン部", "status": "completed"},
    {"step": "implement", "dept": "システム開発部", "status": "running"},
    {"step": "test", "dept": "QA", "status": "pending"}
  ],
  "current_step": "implement",
  "status": "running"
}
```

### セッション復旧

セッション開始時に `pipeline_state` で `status='running'` or `status='paused'` のレコードを検出したら:

1. `agent_sessions` から該当 session_id のイベントを取得
2. 最後に完了した step を特定
3. 社長に「前回のパイプラインが途中です。続行しますか？」と確認
4. 続行する場合、未完了の step から再開

### 部署 Input 標準化

部署を Agent で起動する際、以下の構造で prompt を渡す:

```
## Task
{具体的な作業指示}

## Context Events（前工程の成果）
{agent_sessions から取得した関連イベントの要約}

## Constraints
{制約条件: 時間、コスト、依存関係}
```

**Context Events は session_id でフィルタし、直近の dept_complete イベントの payload を渡す。**
