# パイプライン計画・実行ルール

> このルールは常時有効。社長からタスク指示を受けた時に自動適用される。

## 基本原則: 秘書が動的に計画する

**固定パイプラインではなく、秘書がタスクごとに最適な部署の組み合わせ・順序を計画する。**

秘書はタスクを受け取ったら：
1. タスクの目的・スコープ・制約を分析する
2. 利用可能な全部署から、必要な部署を選定する
3. 部署間の依存関係を判定し、実行順序（直列/並列）を決める
4. 計画を社長に提示してから実行する

## 利用可能な部署

| 部署 | subagent_type | 得意領域 | モデル |
|------|--------------|---------|--------|
| リサーチ | `リサーチ部` | 市場・技術・企業の外部調査 | opus |
| 調査部 | `general-purpose` (opus) | 内部システムの問題特定・根本原因分析・再発防止 | opus |
| UXデザイン | `UXデザイン部` | ユーザー体験・行動文脈・UI設計 | opus |
| AI開発 | `AI開発部` | LLM/AI の要件定義・設計・実装・評価 | opus |
| システム開発 | `システム開発部` | バックエンド・フロントエンド・QA | sonnet |
| PM | `PM部` | プロジェクト管理・チケット・マイルストーン | haiku |
| 資料制作 | `資料制作部` | 提案書・プレゼン・デモ資料 | sonnet |
| 情報収集 | `情報収集部` | 最新ニュース・X監視・Web巡回 | haiku |
| セキュリティ | — | 脆弱性診断・監査・RLS設計 | opus |
| マーケティング | — | プロダクトの商用化・ポジショニング | opus |
| 運営改善 | — | Hook/バッチ/ルールの仕組み改善 | opus |

## 計画の立て方

### Step 1: タスク分析

以下を判断する：
- **何を達成するか**（ゴール）
- **何を知る必要があるか**（情報ギャップ）
- **何を作る必要があるか**（成果物）
- **何を検証する必要があるか**（品質担保）
- **何かおかしい点はないか**（異常の兆候）

### Step 2: 部署選定

タスク分析の結果から、必要な部署を選ぶ。**1つでも複数でもよい。**

選定の考え方：
- 外部情報が必要 → リサーチ or 情報収集
- 内部システムに問題がある → 調査部
- UI/UX の判断が必要 → UXデザイン
- AI/LLM に関わる → AI開発
- コードを書く・直す → システム開発
- 進捗管理・チケット化 → PM
- 資料を作る → 資料制作
- セキュリティに関わる → セキュリティ
- 仕組み自体を改善する → 運営改善
- 商用化・マーケットに関わる → マーケティング

### Step 3: 依存関係と実行順序

部署間の依存関係を判定し、DAG（有向非巡回グラフ）を組む：
- A の出力を B が必要 → 直列（A → B）
- A と B が互いの出力を不要 → 並列（A ∥ B）

**Agent tool を1メッセージ内で複数呼べば並列実行される。**

### Step 4: 計画の提示

```
📋 タスク: [要約]

🏢 部署計画:
  1. リサーチ ∥ UXデザイン  （並列・情報収集フェーズ）
  2. AI開発                （↑の結果を受けて設計）
  3. システム開発 → QA      （実装・検証）

⏱  見積もり: 約XX分
🔧 実行モード: checkpoint（設計完了後に確認）
```

## 参考: よくある組み合わせパターン

計画の参考として。秘書はこれに縛られず、タスクに応じて自由に組み替える。

| パターン | よくある流れ | 使う場面 |
|---------|------------|---------|
| 新機能開発 | リサーチ ∥ UXデザイン → AI開発 → システム開発 → QA | 大きな機能追加 |
| バグ修正 | 調査部 → システム開発 → QA | 原因不明のバグ |
| 軽微な修正 | システム開発 | 原因が明らかな修正 |
| 資料作成 | リサーチ ∥ UXデザイン → 資料制作 | 提案書・プレゼン |
| 調査・分析 | リサーチ ∥ 情報収集 → 統合報告 | 市場調査・比較分析 |
| システム障害 | 調査部 → システム開発 or 運営改善 | 動かない・おかしい |
| 仕組み改善 | 調査部 → 運営改善 → QA | Hook/ルール/バッチの改善 |

## 秘書の異常検知（常時稼働）

計画・実行中に以下の違和感を検知したら、**計画に調査部を追加する**：

- 前提としていた設定・ファイルが存在しない / 変わっている
- hookが走ったはずなのに結果がない
- 前回のセッションで完了した作業が反映されていない
- 部署のアウトプットにエラーや矛盾がある
- コミット済みの状態とワーキングコピーの乖離が大きい

検知した場合は社長に報告し、調査部を起動するか判断を仰ぐ。
明らかな障害の場合は即座に調査部を起動してよい。

## 完了検証（必須）

部署が「完了」と報告した際、秘書は以下を確認する:

- **コード変更の場合**: テストが実行済みか
- **ドキュメントの場合**: 必須セクション（結論・ネクストアクション）が含まれるか
- **成果物の場合**: artifacts テーブルへの登録が完了しているか
- **ハンドオフの場合**: YAML形式でハンドオフが記述されているか

不足があれば部署に差し戻す。

## 実行モード

小さなタスク（1部署で完結）: そのまま実行、報告
中規模タスク（2-3部署）: checkpoint なしで自動実行、完了報告
大規模タスク（4部署以上）: 設計完了後に checkpoint → 社長承認 → 実装
