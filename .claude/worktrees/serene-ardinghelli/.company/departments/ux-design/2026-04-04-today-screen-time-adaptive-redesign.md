# Today画面 時間帯適応型リデザイン

## 1. シナリオ分析

### シナリオA: 朝の起動（6:00-11:00）

```
誰:    社長（複数PJ横断管理、フリーランス/AI開発者/個人事業主CEO）
いつ:  平日 7:00-8:30、起床後またはの通勤中
どこ:  スマートフォン（片手操作、移動中）
何を:  「今日は何があるのか」を30秒で把握したい
       - 今日のカレンダー予定（MTGの時間と相手）
       - 期限が今日のタスク（やらなきゃいけないこと）
       - AIからの一言（今日のフォーカスは何か）
感情:  頭がまだ完全に起きていない。
       「処理」ではなく「受け取り」のモード。
       安心して一日を始めたい。「大丈夫、今日はこれだけやればいい」と思いたい。
       日記を書く気分ではない（まだ何も起きていない）。
```

### シナリオB: 昼の確認（11:00-17:00）

```
誰:    同上
いつ:  日中の隙間時間（MTG合間、移動中、昼休み）
どこ:  スマートフォンまたはPC
何を:  断片メモを残したい / 残りタスクを確認したい
       - さっきのMTGで思ったことをメモ
       - 午後のスケジュール確認
       - 習慣チェック（昼の分）
感情:  仕事モード中。効率重視。
       長い入力はしたくないが、「忘れる前にメモしたい」瞬間がある。
       「あと何が残っているか」を確認して安心したい。
```

### シナリオC: 夜の振り返り（17:00-翌5:00）

```
誰:    同上
いつ:  帰宅後、寝る前
どこ:  スマートフォン（リラックスした姿勢）
何を:  今日を振り返り、感情を言語化したい
       - 今日はどんな一日だったか（日記を書く）
       - 達成したこと（完了タスク・習慣の進捗）
       - 感情の変化（AIの気づき）
       - 明日の予定をちらっと確認
感情:  一日の疲れがある。でも「書きたい」気持ちもある。
       達成感がほしい。「今日もちゃんとやった」と思いたい。
       心を落ち着けて一日を閉じたい。
       日記入力のハードルが高いと「今日はいいや」になる。
```

## 2. 感情曲線マップ

### 朝のフロー（現状）

```
[Today画面を開く]
  → 期待（今日を始めよう）
    → [挨拶テキスト] → 少し安心
      → [日記入力エリア] → 戸惑い「まだ書くことない」 <-- フリクション1
        → [連続記録・WBI] → 軽い焦り「昨日書いてない」
          → [AIの一言] → やっとここで情報を受け取る
            → [断片リスト] → 空で寂しい
              → [タスク] → ここでやっと今日やることがわかる <-- フリクション2
                → スクロール疲れ <-- フリクション3
```

**問題**: 朝に最も必要な「今日の予定」「今日のフォーカス」にたどり着くまでに4セクションのスクロールが必要。
日記入力が最上部にあるが、朝は書くことがない。認知負荷だけ発生させている。

### 夜のフロー（現状）

```
[Today画面を開く]
  → 疲れている
    → [挨拶テキスト] → OK
      → [日記入力エリア] → 書きたいけど、白いテキストエリアが怖い <-- フリクション4
        → [タスク] → 終わっていないものが見えて罪悪感 <-- フリクション5
          → [習慣] → チェック漏れが見えてさらに罪悪感 <-- フリクション6
```

**問題**: 夜は「振り返り」と「達成感」が必要だが、未完了タスクと未チェック習慣がネガティブ感情を誘発する。
日記入力エリアのブランクページ問題（白紙恐怖症）。

### 理想の感情曲線

```
朝:
[画面を開く] → 安心（今日の全体像が一目でわかる）
  → 小さなモチベーション（AIの一言「今日はこれに集中しましょう」）
    → 行動開始（カレンダーの最初のイベントを確認して出発）

夜:
[画面を開く] → 温かさ（「おつかれさまです」の言葉）
  → 達成感（今日やったことが可視化されている）
    → 書きたくなる（プロンプトが今日の文脈を反映している）
      → 安堵（書き終えて、穏やかに一日を閉じる）
```

## 3. フリクションポイントと設計判断

### FP1: 朝に日記入力が最上部にある

```
フリクション:  朝は何も書くことがない。空のテキストエリアが認知負荷になる
適用原則:     #1 意図の先読み + #5 フロー状態の保護
設計判断:     時間帯でセクション順序を動的に変える
              朝 → スケジュール・AIブリーフィングが先頭
              夜 → 日記入力・振り返りが先頭
根拠:         ビジョン文書の「日常の接点」定義と一致
              「朝=今日のフォーカス確認」「夕方=ジャーナリング」
```

### FP2: カレンダー予定がToday画面にない

```
フリクション:  「今日何があるか」を知るためにCalendarページに遷移する必要がある
適用原則:     #1 意図の先読み + #2 認知負荷の最小化
設計判断:     Google Calendar APIから今日の予定を取得し、Today画面に埋め込む
              最大3件を表示（時間順）。それ以上は「+N件」で折りたたむ
根拠:         朝に最も必要な情報は「今日のスケジュール」（社長フィードバック）
              ミラーの法則: 3件なら認知負荷を超えない
```

### FP3: スクロールが長い（8セクション全表示）

```
フリクション:  モバイルで8セクションを縦に並べるとスクロール疲れが発生
適用原則:     #2 認知負荷の最小化（系列位置効果）
設計判断:     時間帯によってセクションの表示/非表示・順序を変える
              ファーストビュー（スクロールなし）に最重要情報3つを収める
根拠:         モバイルファーストの原則。親指圏内で最重要操作が完結すべき
```

### FP4: 白紙テキストエリア問題（ブランクページ恐怖症）

```
フリクション:  「思ったこと、感じたことを自由に...」は自由すぎて書けない
適用原則:     #3 フィードバックループの設計（フィードフォワード）
設計判断:     時間帯と文脈に応じた動的プロンプトを表示する
              朝: 「今日やりたいことは?」
              昼: 「さっきのMTGどうだった?」（カレンダーから推測）
              夜: 「今日一番印象に残ったことは?」
              さらに、今日の予定・完了タスクから文脈を組み立てて
              「[PJ名]のMTGの後、何か思ったことはありますか?」のような具体的プロンプト
根拠:         Grid Diary・Reflectlyの知見: 構造化プロンプトは白紙より
              3-5倍の入力完了率。ブランクページではなく文脈を提示する
```

### FP5: 未完了タスク・習慣の罪悪感

```
フリクション:  夜に未完了のタスクや習慣を見ると罪悪感が生じ、日記を書く気が失せる
適用原則:     #4 エラー回復の人間化 + #3 フィードバックループ
設計判断:     夜は「完了したもの」を先に見せる（ポジティブファースト）
              未完了は「明日に持ち越す?」という穏やかな表現で最後に表示
              完了数/総数の比率で達成感を演出（「今日は5件中3件完了」）
根拠:         ピーク・エンドの法則: 一日の終わりをポジティブにすることで
              全体の体験評価が向上する。ツァイガルニク効果を逆手に取り、
              完了済みの可視化で「やった感」を先に提供する
```

### FP6: 時間帯ごとの情報の優先順位が固定

```
フリクション:  朝も夜も同じ画面構成。文脈が変わっているのにUIが変わらない
適用原則:     #1 意図の先読み（文脈的トリガー: 時間帯）
設計判断:     3つの時間帯モード（morning / afternoon / evening）を自動切替
              各モードでセクション構成・順序・プロンプト・トーンが変わる
              手動切替も可能（「夜だけど朝のモードで見たい」に対応）
根拠:         2026 UXトレンドのcontext-aware adaptive design
              Self-Focus Platformビジョンの「日常の接点」設計と完全に一致
```

## 5. 日記入力のフリクション低減設計

### 5.1 動的プロンプト（コンテキストアウェア）

プレースホルダーテキストを時間帯と文脈に応じて動的に生成する。

**プロンプト生成ロジック:**

```typescript
function getDiaryPrompt(timeMode: TimeMode, context: TodayContext): string {
  // 夜 + 今日の予定がある場合
  if (timeMode === 'evening' && context.todayEvents.length > 0) {
    const lastEvent = context.todayEvents[context.todayEvents.length - 1]
    return `今日一番印象に残ったことは何ですか?`
  }
  
  // 昼 + 直前のMTGがある場合
  if (timeMode === 'afternoon' && context.recentEvent) {
    return `${context.recentEvent.summary}の後、何か思ったことは?`
  }
  
  // 朝
  if (timeMode === 'morning') {
    return `今日やりたいことは?`
  }
  
  // デフォルト
  return `今、頭に浮かんでいることは?`
}
```

**プロンプトのバリエーション（夜用、ローテーション）:**

| # | プロンプト | 狙い |
|---|----------|------|
| 1 | 今日一番印象に残ったことは何ですか? | 具体的なエピソード想起 |
| 2 | 今日の自分に点数をつけるなら? | 数値化で入力ハードルを下げる |
| 3 | 今日「ありがとう」と思った瞬間は? | ポジティブ回想（感謝日記の効果） |
| 4 | 明日の自分に一言伝えるなら? | 未来志向で締めくくる |
| 5 | 今日学んだことは? | 成長の言語化 |

### 5.2 入力UIの段階的開示

```
Level 0: ワンタップ気分ログ
  [良い] [普通] [疲れた] [もやもや]
  → タップだけで最低限の記録。テキスト入力不要

Level 1: 一行メモ
  ┌─────────────────────────────────┐
  │ プロンプトテキスト...    [記録]   │
  └─────────────────────────────────┘
  → 1行のテキストフィールド。Enter で送信

Level 2: じっくり書く（展開後）
  ┌─────────────────────────────────┐
  │ プロンプトテキスト...             │
  │                                  │
  │                                  │
  │                                  │
  │                          [記録]   │
  └─────────────────────────────────┘
  → テキストエリアが展開。入力中は他のセクションがフェードアウト
```

**時間帯による初期表示:**
- morning: Level 1（一行メモ）がデフォルト
- afternoon: Level 1（一行メモ）がデフォルト
- evening: Level 2（じっくり書く）がデフォルト

### 5.3 ワンタップ気分ログ

夜に日記を書く気力がないときでも、気分だけは記録できるようにする。

```
今日の気分は?

[良い]  [普通]  [疲れた]  [もやもや]

→ タップすると小さなフィードバック:
  「記録しました。お疲れさまでした」
  
→ 追加で一言書きたくなったら:
  「もう少し書きますか?」[はい]
```

**根拠**: Daylioの知見。2タップで完了する入力は、テキスト入力の5倍の継続率。
最低限の入力でも「連続記録が途切れない」ことが習慣化に重要。

```
  1. キャッシュデータで即座にレンダリング
  2. バックグラウンドで鮮度チェック
  3. 差分があればスムーズに更新（スピナーなし）
```

## 8. インタラクション仕様

### 8.1 時間帯モードの切り替え

- **自動切替**: `new Date().getHours()` で判定、コンポーネントマウント時に1回
- **手動切替**: 画面右上に小さなトグル（morning/afternoon/evening アイコン）
- **遷移アニメーション**: セクションの並び替えは `layout animation`（framer-motion等）で滑らかに

### 8.2 日記保存のフィードバック

```
入力中: テキストエリアのボーダーがアクセントカラーに
保存クリック: ボタンがスピナーに変化（100ms以内に反応）
保存成功: テキストエリアが縮小 + 「記録しました」トースト + 断片リストに新しいカードがスライドイン
感情分析完了: 断片カードに感情バッジがフェードイン
保存失敗: テキストは保持。「再試行」ボタンを表示（入力内容は消えない）
```

### 8.3 習慣チェックのフィードバック

```
タップ: チェックボックスにチェックマークがバウンスイン
全完了時: 「今日の習慣を全て達成」のミニ祝福（confetti不要、テキストで十分）
```

### 8.4 スケジュールの時間インジケーター

```
afternoon モードの場合:
  終了済みイベント → グレーアウト + チェック
  現在進行中 → 左にアクセントバー + 「Now」バッジ
  次のイベント → 「あと30分」のカウントダウン表示
```

```typescript
function getTimeMode(): TimeMode {
  const hour = new Date().getHours()

  if (hour >= 11 && hour < 17) return 'afternoon'
  return 'evening'
}
```

## 10. 見送った代替案

### A案: タブ切り替え式（朝/昼/夜タブ）

```
[朝] [昼] [夜]  ← タブで明示的に切り替え
```

**不採用理由**: 3つのタブを見せると「全部確認しなきゃ」という認知負荷が発生する。
自動切替+小さなオーバーライドボタンの方がCalm Technology の原則に合う。

### B案: 全セクション表示 + セクションの折りたたみ

**不採用理由**: 折りたたまれたセクションは「隠れている情報がある」不安を生む。
時間帯に不要な情報はそもそも表示しないほうが認知負荷が低い。

### C案: AIが完全にキュレーション（セクション構成をAIが決める）

**不採用理由**: 予測不可能なUIは不安を生む。ユーザーは「いつも同じ場所にあるもの」に安心する（ヤコブの法則）。
時間帯ベースのルールは予測可能で、ユーザーが学習できる。
