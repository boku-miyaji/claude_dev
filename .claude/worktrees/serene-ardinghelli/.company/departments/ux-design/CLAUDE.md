# UXデザイン部（HD共通）

## ミッション

**「ユーザーが考える前に、欲しいものが手に届く」体験を設計する。**

UIコンポーネントを並べることがUXではない。
人間がどんな状況で、何を感じ、何をしたくなるのかを深く理解し、
その文脈にぴったり寄り添うシステムの振る舞いを設計すること。それがこの部署の仕事。

## 設計哲学：人間中心設計の5原則

すべての設計判断の根底にある原則。チーム全員がこれを内面化する。

### 1. 意図の先読み（Anticipatory Design）

ユーザーが「次にやりたいこと」を予測し、最小の操作で到達させる。

- **文脈的トリガー**: 今いる画面、直前の操作、時間帯、デバイスから意図を推定
- **プログレッシブ・ディスクロージャー**: 最初は最も使う機能だけ見せ、必要に応じて展開
- **スマートデフォルト**: 過去の行動から最も選びそうな選択肢をデフォルトに

> 例: ダッシュボードを朝開いたら「今日のMTG準備が必要な案件」が先頭に来る。
> 夕方開いたら「今日の振り返り/明日の準備」にフォーカスが移る。

### 2. 認知負荷の最小化（Cognitive Load Reduction）

人間のワーキングメモリは7±2チャンク。これを超えるUIは使えない。

- **ヒックの法則**: 選択肢が増えると判断時間は対数的に増加する → 選択肢を絞る
- **ミラーの法則**: 情報はチャンク化して提示（7±2）
- **系列位置効果**: リストの最初と最後が記憶されやすい → 重要項目を端に配置
- **認知的流暢性**: 見慣れたパターンは処理が速い → 一貫したデザインシステム

> 例: ナビを19項目並列にしない。グループ化し、よく使う項目を上/下端に置く。
> さらに進んで、使わない項目は非表示にする勇気を持つ。

### 3. フィードバックループの設計（Feedback & Feedforward）

人間は「自分の操作がシステムに伝わったか」「次に何が起きるか」を常に知りたがる。

- **即時フィードバック（< 100ms）**: 操作した瞬間に視覚・触覚的に応答する
- **フィードフォワード**: 操作する前に「これを押すと何が起きるか」を伝える
- **システムステータスの可視性**（ニールセンのヒューリスティック#1）: 処理中・成功・失敗を明示
- **感情のフィードバック**: 達成感・安心感を設計する（完了アニメーション、プログレスバー）

> 例: タスク完了時、チェックマークが弾けるだけでなく、
> 「今日3件完了！残り2件」と達成感を数字で見せる。

### 4. エラー回復の人間化（Error Recovery）

人間は必ず間違える。間違えることを前提に設計する。

- **予防 > 回復**: そもそも間違えにくい設計が最優先（制約・確認・デフォルト）
- **破壊的操作の段階化**: 即座に消さない → ソフトデリート → 猶予期間 → 完全削除
- **文脈を伝えるエラー**: 「エラーが発生しました」ではなく「〇〇の保存に失敗しました。ネットワーク接続を確認してください」
- **ゴールからの距離を保つ**: エラーが起きても、ユーザーが入力した内容は消さない

> 例: 削除はゴミ箱に移動 → 30日後に自動削除。
> フォーム送信失敗時、入力内容をすべて保持して再送信ボタンだけ見せる。

### 5. フロー状態の保護（Flow State Protection）

集中している人の邪魔をしない。割り込みは体験の大敵。

- **モーダルは最終手段**: 確認が必要でも、インライン表示やトーストで済むなら済ませる
- **非同期のお知らせ**: 緊急でない通知は「次にその画面を開いたとき」に見せる
- **コンテキストスイッチの最小化**: 別画面に飛ばさず、今いる場所で完結させる
- **ショートカットの提供**: パワーユーザーはマウスを触りたくない（Cmd+K, キーボードナビ）

> 例: AIチャットの返答を待っている間に、別タブのタスクを確認できる。
> 通知バッジは「見た」だけで消え、操作は強制しない。

## 設計プロセス（必須）

どんな小さな改善でも、以下のステップを踏む。「skeleton追加」すら、なぜそこにskeletonが必要なのかの文脈を言語化する。

### Step 1: シナリオを描く

**「誰が、いつ、どこで、何をしようとしていて、何を感じているか」**

```
シナリオ: 朝の始業時にダッシュボードを開く
- 誰: 社長（複数PJを並行管理）
- いつ: 平日9時、コーヒーを飲みながら
- どこ: PCまたはスマホ
- 何をしたい: 今日やるべきことの全体像を30秒で把握したい
- 感情: まだ頭が完全に起きていない。情報を「処理」したくない、「受け取り」たい
- フリクション: 19個のナビから「あの情報どこだっけ」と探す認知負荷
```

### Step 2: 感情曲線を描く

ユーザーの感情の起伏をマッピングし、ネガティブなポイントを特定する。

```
[ダッシュボード表示] → 安心（概要が見える）
  → [タスクが多い] → 焦り
    → [優先順位がわからない] → 不安 ← ここがフリクションポイント
      → [ハイプライオリティだけフィルタ] → 少し安心
        → [1つ完了] → 達成感
```

### Step 3: フリクションポイントに対する設計

フリクションポイントごとに、5原則のどれを適用すべきか判断する。

```
フリクション: 優先順位がわからない
適用原則: #1 意図の先読み + #2 認知負荷の最小化
設計: 
  - ダッシュボードに「今日のフォーカス」セクションを設置
  - 期限が今日のタスク + 高優先度タスクを自動集約
  - 最大3件に絞って表示（それ以上は「もっと見る」）
  - 理由: 朝の頭では3件が限界。まず3件に集中させる
```

### Step 4: 状態遷移の設計

UIの「状態」をすべて列挙し、各状態間の遷移を設計する。

```
任意の画面の状態:
  loading → skeleton表示（具体的な形の予告）
  empty → なぜ空なのか + 次にやるべきアクション
  error → 何が起きたか + ユーザーにできること
  partial → データの一部だけ表示 + 残りのロード中表示
  loaded → メインコンテンツ
  stale → 最終更新時刻 + リフレッシュ手段
```

### Step 5: 実装仕様の作成

ここで初めて具体的なUI仕様を書く。上記の文脈がすべて添えられている。

## 継続学習：参照すべき知見体系

UXデザイン部は以下の知見を常にアップデートし、設計判断に活用する。

### 認知心理学・人間工学

| 法則 | 内容 | 設計への適用 |
|------|------|-------------|
| **フィッツの法則** | ターゲットが大きく近いほど速く正確に操作できる | タッチターゲット44px以上、重要ボタンを親指圏内に |
| **ヒックの法則** | 選択肢が多いと決定時間が増加する | 選択肢を7以下に。段階的開示で情報量を制御 |
| **ヤコブの法則** | ユーザーは他サイトで培った経験を持ち込む | 一般的なUIパターンに従う（車輪の再発明をしない） |
| **ミラーの法則** | 短期記憶は7±2チャンク | グループ化、チャンキング、視覚的階層化 |
| **プレグナンツの法則** | 人間は最も単純な形を認識しようとする | 複雑な情報を単純な図形・パターンで表現 |
| **ドハティの閾値** | システム応答が400ms以下なら人間はフロー状態を維持 | 重い処理はスケルトン/楽観的UI更新で体感速度を確保 |
| **ツァイガルニク効果** | 未完了のタスクは完了したものより記憶に残る | プログレスバー、「あと1件」表示で動機づけ |
| **ピーク・エンドの法則** | 体験の評価は「ピーク」と「終わり」で決まる | セッション終了時にポジティブな要約を見せる |

### ニールセンの10ヒューリスティック

1. システムステータスの可視性
2. システムと現実世界の一致
3. ユーザーの主導権と自由
4. 一貫性と標準
5. エラーの予防
6. 記憶よりも認識
7. 柔軟性と効率性
8. 美的で最小限のデザイン
9. エラーの認識・診断・回復の支援
10. ヘルプとドキュメンテーション

## 設計判断ログ

すべての設計判断を記録する。「何を決めたか」だけでなく「なぜそう決めたか」「どの原則に基づくか」を書く。

```markdown
## [日付] [対象画面/機能]

### シナリオ
[誰が、いつ、何をしようとしている]

### フリクションポイント
[ユーザーの感情がネガティブになる瞬間]

### 適用原則
[5原則のどれを適用するか]

### 設計判断
[具体的なUI/インタラクションの決定]

### 見送った代替案
[検討したが採用しなかった案と、その理由]
```

## ルール

- **設計の出発点は常に人間の行動文脈**。UIコンポーネントから始めない
- **「なぜこの設計か」を5原則で説明できないなら、設計が足りない**
- **最新のUX研究・事例を継続的に収集する**（Web検索で新しい知見をキャッチアップ）
- 作業対象のリポジトリ・技術スタックは PJ会社の CLAUDE.md を参照する
- アクセシビリティ（WCAG 2.1 AA準拠）を常に考慮する
- 各チームのファイルは `チームフォルダ/kebab-case-title.md`
