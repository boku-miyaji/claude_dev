# /zenn-article — Zenn記事生成（boku_yaji スタイル）

## いつ使うか

- `/zenn-article` を明示呼び出し
- 「この論文をZenn記事にして」「Zennで書いて」「Zenn化して」と言われたとき
- 入力として渡されるもの:
  - 論文のURL（arXiv / Nature / 公式PDF など）
  - 論文PDF のローカルパス
  - 記事のURL
  - 手元のテキスト
  - これら複数の組み合わせ

## 第一原則: ハルシネーション禁止

**LLMの推測で書かない。必ず原文を取得して読む。**

- URLは WebFetch で取得する
- PDFは Read で取得する（10ページ超は `pages` パラメータで分割して全体を読む）
- 取得できない場合は社長に確認。憶測で書かない
- 数値・著者名・所属・実験条件・引用文は原文と完全一致させる
- 不明な点は「論文に明記なし」と本文に明示する

## boku_yaji スタイル — 章構成テンプレート

以下を厳守。論文内容に応じて 3 のサブセクション数、5 の有無は変動可。

```
リード（タイトル直下、見出しなし）
## 0. TL;DR
## 1. 研究の背景と動機
   ### {専門用語}とは何か（必要なら）
## 2. 実験デザイン
   ### 2-1. 基本フレームワーク
   ### 2-2. {データ・条件など}
## 3. 主な結果
   ### 3-1. {結果1}
   ### 3-2. {結果2}
   …
## 4. 著者が述べる制約
## 5. {含意 / 安全性への影響}（該当する場合のみ）
## ぼくおも（僕が思ったこと）
## 参考
```

- 主見出しは H2（`##`）。サブは H3（`###`）。番号は `3-1.` 形式。
- 「ぼくおも」「参考」は番号を付けない。

### リード（必須テンプレート）

タイトル直下、見出しなしでこのまま（**ここだけです・ます調**。記事の枕詞として柔らかく）:

```
{YYYY/M/D}に{出版社/媒体}に掲載された{著者所属}らの論文：
[{論文の英語タイトル原文}]({論文URL}) の詳説記事です。

執筆にはLLMの助けも借りていますが、著者はちゃんと論文を読んだ上で適宜修正して投稿しています。
```

- 日付は記事内表記の `YYYY/M/D`。frontmatter とは別物。
- **透明性宣言の一文は必ず入れる。** boku_yaji 様式の核。

### 0. TL;DR

5行前後の箇条書き。文末「。」で揃える。

```
- {結論を1行で}。
- {実験の枠組みを1行で}。
- {主な結果を1行で}。
- {限界・制約を1行で}。
- {実務的含意を1行で}。
```

### 1. 研究の背景と動機

- 「なぜこの研究が重要か」「何が新しいのか」300〜500字
- 必要なら H3 で「{専門用語}とは何か」を切る
- 専門用語は **英語名（和訳）** または「平たく言うと：」で説明

### 2. 実験デザイン

- マークダウン表で整理する
- 「2-1. 基本フレームワーク」「2-2. {データ・条件}」のように分割
- ASCII図（縦線 `│` `├` `└`）で実験フローを描いてもよい

### 3. 主な結果

- サブセクションを `### 3-1.` `### 3-2.` … と番号付き H3 で分ける
- 各サブセクション 200〜500字 + 必要なら表
- 効果の有無は「あり/なし」「効いた/効かなかった」のような **言葉で表現する**
- 数値は原文と完全一致

### 4. 著者が述べる制約

- 論文の Limitations セクションを忠実に列挙
- ここに筆者の意見を混ぜない（混ぜるなら「ぼくおも」へ）

### 5. 含意（該当する場合のみ）

- AI安全性 / 産業応用 / 政策など、論文が明示的に示唆する含意

### ぼくおも（僕が思ったこと）

**ここで一人称「僕」が初めて登場し、文体がカジュアルに切り替わる。**

- 「〜なと思った」「気をつけよう」「〜かもしれない」
- 視点は3段構成が定番:
  1. 論文の妥当性 / 直感的な感想
  2. 人間社会・自分の経験との類比
  3. リスク・実務への注意（具体的なシナリオで）

### 参考

- 箇条書き
- インラインリンク `[タイトル](URL)` 形式
- 論文・公式ブログ・関連リソースを列挙

## 文体ルール

**範囲ごとに文体を切り替える。** 混在ではなく、明確に分離する。

| 範囲 | 文体 |
|------|------|
| リード（文献情報 + 透明性宣言） | **「です・ます調」**。記事の枕詞として柔らかく |
| 0. TL;DR 〜 解説本体（最終番号セクション） | **「だ・である調」で統一**。論文/記事の解説本体 |
| ぼくおも | カジュアル。「〜なと思った」「〜だろう」「気をつけよう」 |
| 引用 | インラインリンク。脚注は基本使わない |
| 段落 | 3〜5文で改行。視認性優先 |
| 絵文字 | **本文中は使わない**（✅❌🍣🤖等を散りばめない） |

- リードは「〜です」「〜ます」「〜しています」で締める。これだけ例外
- 解説本体（0〜6など）の文末は「〜だ」「〜である」「〜する」「〜になる」で揃える。**「〜です」「〜ます」が混じったら校正で直す**
- 強調は **太字** で十分。絵文字での強調はしない
- 「ぼくおも」は 例外的にカジュアル に切替えるが、絵文字は使わない

## 構造化の道具（mermaid・表・ASCII図）

可読性のため積極的に使う。**ただし無理に挿入しない**。効果的な箇所だけ。

- **表**：比較・分類・指標の対応関係に強い。データセットの条件、旧→新の対応、複数手法の比較など
- **mermaid**：フロー・状態遷移・関係図に強い。`flowchart`, `sequenceDiagram`, `stateDiagram` など
  - 中核モデルが「層構造」「処理フロー」「相互関係」を持つときに使う
  - ノード文字列は短く（10〜20字）。`<br/>` で改行可
- **ASCII図**：シンプルな階層・木構造。mermaidまで要らないとき
- **箇条書き**：3項目以上の並列を文章で繋がない

判断基準: その図/表がなくなったら理解度が下がるか? 下がるなら入れる。装飾なら入れない。

## frontmatter 仕様

```yaml
---
title: "{選んだタイトル}"
emoji: "{1文字の絵文字}"
type: "tech"
topics: ["{topic1}", "{topic2}", "{topic3}", "{topic4}", "{topic5}"]
published: false
---
```

### title

複数案を提示する → 社長が選ぶ → frontmatter に確定（後述「ワークフロー Step 3」）。

### emoji（最小限）

- Zenn の仕様で frontmatter 必須。1文字だけ
- 記事テーマに緩く関連するものを **1案だけ** 提案。深く考えない
- 本文中には絵文字を一切使わない（frontmatter のみ）

### type

- 論文解説・技術調査 → `tech`
- ポエム・意見記事 → `idea`
- 迷ったら `tech`

### topics

- **最大5つ**（Zenn制約）
- 英語小文字推奨。日本語可
- 論文のキーワードから抽出する
- 例: `["ai", "llm", "anthropic", "claude", "machinelearning"]`

### slug（ファイル名）

- a-z, 0-9, -, _ のみ。**12〜50文字**
- 社長の過去記事はランダム英数字（`c0b933283b21c1` `2eba9ff05fc3e2`）
- **デフォルトはランダム英数字14文字**。指定があれば英単語スラッグも可
- 生成例:

```bash
# zenn-cli が入っていれば
npx zenn new:article
# なければ手動でランダム生成
openssl rand -hex 7
```

### published

- ドラフトは必ず `false`
- 「公開準備OK」と社長が言ったら `true` に変更する

## タイトル候補の出し方（必須・複数案）

**Step 3 でタイトルを決める際、必ず 5 案以上を訴求軸付きで提示する。**

「見たいと思える」タイトルの設計原則:
- **数値を入れる** — 反直感的な数字ほど強い（19% / 3倍 / 100% など）
- **対比/ギャップ** — 「軽い具体例」と「重い含意」を並べる（フクロウ好き × 危険な思想）
- **断定で省略を作る** — 「だけで」「でも」「しか」などで核心を絞る
- **疑問・驚き** — 「？！」 / 「本当に〜なのか」
- **対象を読者に寄せる** — 「あなたの開発」「私たちのコード」など読者を巻き込む
- **専門×日常** — 専門用語に日常的な比喩を1つ添える

提示フォーマット:

```
タイトル候補（5案）:

1. {タイトル案A}
   訴求軸: 反直感的な数値 + 対比
   想定読者の反応: 「えっ、そんなに？」

2. {タイトル案B}
   訴求軸: 強い断定 + ギャップ
   想定読者の反応: 「マジで？読みたい」

3. {タイトル案C}
   訴求軸: 専門用語の日常翻訳
   想定読者の反応: 「これは何だろう」

4. {タイトル案D}
   訴求軸: 疑問形 + 驚き
   想定読者の反応: 「答えが気になる」

5. {タイトル案E}
   訴求軸: 読者の関心領域に直接訴求
   想定読者の反応: 「自分に関係ある」
```

タイトルそのものに絵文字は **入れない**（frontmatter の emoji フィールドが別であるため）。

## 出力先（必須）

- **ドラフト**: `/workspace/scratch/zenn/{slug}.md`
- **公開用**: 社長の Zenn リポジトリ（パスは社長に確認）
- `scratch/` 経由を必ず守る。`scratch-workspace.md` ルール準拠
- いきなり articles/ に書き込まない

## ワークフロー（一気通貫・途中承認なし）

**社長の指示があるまで途中で確認を求めない。** 取得→生成→校正までを一息で完了させ、最後にまとめて報告する。

1. 入力（URL / PDF / テキスト）を受け取る
2. 原文を WebFetch / Read で取得して通読する
3. 章構成・emoji・topics・slug を **内部で決定**（社長承認は取らない）
   - 論文用の標準構成が合わない場合（エッセイ/記事など）は適切にアレンジしてよい
   - frontmatter の `title` は仮タイトル（候補1案）で埋めておく
4. 本文を生成し `/workspace/scratch/zenn/{slug}.md` に書き出す
5. **校正・ファクトチェックを実行**（後述。**省略禁止**）
6. 社長への最終報告:
   - 完成ドラフトのパス
   - **タイトル候補 5案以上**（訴求軸 + 想定読者の反応つき。社長が最後に決める）
   - 校正・ファクトチェック結果サマリ
   - frontmatter の確定値（title 以外）
7. 社長がタイトルを選ぶ → frontmatter の `title` を確定値に書き換える
8. 社長レビュー → 追加修正
9. 「公開準備OK」を受けたら:
   - Zenn リポジトリのパスを社長に確認
   - `published: true` に更新
   - articles/{slug}.md にコピー（または社長が手動移動）

**例外: 一気通貫を中断する条件**

以下に該当した場合のみ、途中で社長に確認する:
- 原文が取得できない（WebFetch失敗、PDF読込不可）
- 原文が論文・記事として明らかに不適切（広告ページ、ログイン要求のみ等）
- 入力が複数あって優先順位が判断できない

## Step 5: 校正・ファクトチェック工程（必須）

ドラフトを書いたら必ず以下を実行する。**スキップ禁止。**

### A. ファクトチェック（原文との照合）

原文を再度開きながら以下を一つずつ確認する:

- [ ] **数値**: 全ての数字（パーセント、サンプル数、p値、年など）が原文と一致するか
- [ ] **著者名・所属**: 表記揺れがないか
- [ ] **論文タイトル**: 原文タイトルと完全一致か
- [ ] **直接引用**: `「...」` で囲んだ箇所は原文と一字一句一致するか
- [ ] **専門用語の英語表記**: 原文での綴りと一致するか（例: Subliminal Learning）
- [ ] **実験条件**: モデル名、データセット名、ハイパーパラメータが正しいか
- [ ] **限界・制約**: 著者が述べた制約のみ列挙し、書き手の意見が混ざっていないか

不一致があれば原文に合わせて修正。判断つかない箇所は「論文に明記なし」と注記。

### B. 文体・表記チェック

- [ ] リード〜5 が **「だ・である調」で統一**されているか（「です」「ます」が混ざっていないか）
- [ ] 「ぼくおも」だけがカジュアル文体か
- [ ] 本文中に **絵文字が混入していないか**（✅❌🍣🤖等）
- [ ] 用語の表記揺れ（AI/ai、LLM/llm、データセット/データ・セット 等）がないか
- [ ] 過剰な太字・記号装飾がないか

### C. 構造チェック

- [ ] 透明性宣言の一文がリードに含まれているか
- [ ] H2/H3 の番号が抜けていないか（3-1, 3-2, 3-3 …の連番）
- [ ] frontmatter の必須フィールド（title, emoji, type, topics, published）が揃っているか
- [ ] topics が **5つ以下** か
- [ ] emoji が **1文字** か
- [ ] published が `false` か（ドラフトのため）
- [ ] slug が a-z/0-9/-/_ のみ・12〜50文字か

### D. リンク・引用チェック

- [ ] 本文中の全URLが論文・公式リソースとして妥当か
- [ ] 「参考」セクションのリンク先が実在するか（最小限 WebFetch で疎通確認）
- [ ] 内部矛盾（TL;DR と本文で結論が食い違う等）がないか

### E. 報告フォーマット

校正結果を以下の形式で社長に報告:

```
校正・ファクトチェック結果:

A. ファクトチェック: {OK / 修正X件}
  - {修正があった場合、項目ごとに「論文: X、ドラフト: Y → Yに修正」と列挙}

B. 文体・表記: {OK / 修正X件}
  - {混入していたカジュアル表現や絵文字を列挙}

C. 構造: {OK / 修正X件}
  - {番号抜けや frontmatter 不備}

D. リンク・引用: {OK / 要確認X件}
  - {疎通不可だったURL}

総合: {そのまま公開可 / 修正後に再レビュー必要}
```

判断つかない箇所は社長に質問してから修正する。

## やってはいけないこと

- 原文を読まずに本文を書く（**最大の禁忌**）
- 透明性宣言の一文を省略する
- 文体を混在させる（リード〜5は **だ・である調** で統一。です・ます調を混ぜない）
- 「ぼくおも」を学術口調で書く（カジュアル切替が様式の核）
- 本文中に絵文字を散りばめる
- frontmatter の topics を 6 つ以上にする
- emoji を2文字以上にする
- いきなり `published: true` で出力する
- scratch/ を経由せず articles/ に直接書く
- 校正・ファクトチェック工程をスキップする
- タイトルを1案だけ提示する（必ず5案以上）
