## 0. プロジェクトの位置づけ

### 0.1 本PoCの本質：ソリューションセールス（アジェンダ営業）伴奏AI

> **本質**: 単発の商材提案ではなく、営業活動全体に伴奏するAI。営業担当者と一緒に考え、アカウントプランを継続的にアップデートしていく存在。

```
┌─────────────────────────────────────────────────────────────────────┐
│       本PoCの本質：ソリューションセールス（アジェンダ営業）伴奏AI      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【従来の「提案AI」の考え方】❌ これではない                          │
│  ・入力 → AI → 商材提案（完了）                                      │
│  ・AIが「答えを出す」存在                                            │
│  ・一発で完成形を出そうとする                                        │
│                                                                     │
│  【「伴奏AI」の考え方】✅ 本PoCの主眼                                 │
│  ・AIは「営業と一緒に考える」存在                                    │
│  ・商材提案 + 次のアクション（深掘り・関係構築）を提示               │
│  ・情報が足りない時は「何を確認すべきか」を示す                      │
│  ・アカウントプラン（経営アジェンダ・キーパーソンマップ）の           │
│    継続的なアップデートを支援                                        │
│                                                                     │
│  【伴奏のサイクル】                                                  │
│  入力 → AI → 提案 + ネクストアクション → 実行 → 情報更新 → AI → ... │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 0.2 プロダクトセールスとソリューションセールスの違い

> **MTG方針**: 本PoCは「ソリューションセールス」を支援するものであり、プロダクトセールスとは異なる

```
┌─────────────────────────────────────────────────────────────────────┐
│            ソリューションセールス（アジェンダ営業）とは                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【プロダクトセールス】❌ 本PoCの主眼ではない                          │
│  ・商品起点で「この商品を売りたい」から始まる                        │
│  ・商品カタログの情報整理が中心                                      │
│                                                                     │
│  【ソリューションセールス（アジェンダ営業）】✅ 本PoCの主眼           │
│  ・お客さんの状況・ニーズ起点で「何を求めているか」から始まる        │
│  ・顧客が本来求めていること、どうなると嬉しいかを起点にする          │
│  ・ニーズを推定し、それにマッチするソリューションを提案              │
│  ・経営アジェンダ・キーパーソンとの関係を継続的に深掘り・深化        │
│  ・難易度は高いが、カスタマイズ・オーダーメイドの要素も含む          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**重要な観点**:
- ニーズの推定からセールスが始まる
- お客さんが「何を本来求めていて、どうなると嬉しいのか」を理解することが出発点
- 商材提案だけでなく、経営アジェンダシートの深掘り・キーパーソンマップの関係深化も含む
- 最終的にはプロダクト観点での評価も含まれるが、比重はソリューションセールス側に置く
- 難しすぎるソリューションはPoC期間を考慮して除外可能

> rikyu様のToBe像では、経営アジェンダ・キーパーソンマップ・アカウントプラン等の更新にもAIを活用予定。本PoCではこれらを入力として活用し、更新提案も出力する。

│  │ アカウントプラン管理                                          │   │
│  │ ├─ 経営アジェンダシート（顧客の経営課題・優先事項）           │   │
│  │ ├─ キーパーソンマップ（意思決定者・影響者の関係図）           │   │

│  │                                                               │   │
│  │ 【入力】経営アジェンダシート、キーパーソンマップ、面談記録等   │   │

│  └─────────────────────────────────────────────────────────────┘   │

**動的情報の重要性**:
- 日々の営業コミュニケーションからの情報
- 最新ニュース（政治・経済変動等）への対応
- お客さんが会話で伝える動的な情報
- これらを「インプット」として活用し、ソリューションセールスに繋げる

│  ★本PoCの独自価値                                                               │
│  1. 暗黙知の蓄積サイクル（FB→抽出→投入→精度向上）                              │
│  2. 知見ベースの評価観点（業界知見・顧客理解・提案戦略）                        │
│  3. パーソナライズ（ユーザーレベル別の出力最適化）                              │
│  4. 伴奏AI体験（対話的なサイクル、使えば使うほど賢くなる）                      │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

**差別化サマリ**:

| 観点 | NTTData PoC2 | 本PoC |
|------|-------------|-------|
| データ | 実データ（SFA+TSR） | 仮データ（仕組み重視） |
| アプローチ | 情報収集・分析の自動化 | **ナレッジ蓄積・活用の仕組み** |
| 評価 | 一般的な評価指標 | **知見ベースの評価観点** |
| 精度 | 高精度な企業分析 | 改善サイクルの実証 |
| 出力 | 静的なレポート | **対話的な伴奏体験** |

### 1.2 キーメッセージ: 本PoCで明らかにすること

本PoCで明らかにしたいことは以下の6点です：

#### 1. 現時点での精度レベルの測定

> **目的**: AIの出力が「新人レベル」「中堅レベル」「ベテランレベル」のどこに位置するかを明確にする

```
┌─────────────────────────────────────────────────────────────────────┐
│                    精度レベルの物差し                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【新人レベル】                                                      │
│  ・基本的なニーズ・商材は拾えるが、抜け漏れがある                    │
│  ・表面的な情報から推定するのみ                                      │
│  ・先輩のチェック・修正が必要                                        │
│  → 営業支援ツールとして「叩き台作成」には使える                       │
│                                                                     │
│  【中堅レベル】                                                      │
│  ・主要なニーズ・商材を網羅的に抽出できる                            │
│  ・背景・文脈を踏まえた推定ができる                                  │
│  ・大筋は正しく、軽微な修正で使える                                  │
│  → 営業担当の「壁打ち相手」「セカンドオピニオン」として活用可能       │
│                                                                     │
│  【ベテランレベル】★PoCでの目標ライン                                │
│  ・潜在ニーズや将来課題まで先読みできる                              │
│  ・業界特有の事情・暗黙知を踏まえた提案ができる                       │
│  ・そのまま使える品質                                                │
│  → 経験の浅い営業の「指導役」「品質担保」として機能                   │
│                                                                     │
│  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─   │
│                                                                     │
│  【Mgrレベル】★将来展望（PoC対象外）                                 │
│  ・組織全体の営業戦略・ポートフォリオを踏まえた提案ができる          │
│  ・顧客との長期的な関係構築を見据えた判断ができる                    │
│  ・複数案件の優先順位付け、リソース配分の示唆ができる                │
│  ・部下の提案内容をレビュー・指導できる品質                          │
│  → 営業マネージャーの「意思決定支援」「戦略立案補助」として機能       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**測定観点**:
- ニーズ推定AI: どのレベルでニーズを拾えているか
- ソリューション推薦AI: どのレベルでマッチングできているか
- どの企業属性・状況で精度が高い/低いか（得意/苦手の特定）

#### 2. 中長期的な改善の道筋

> **目的**: 現時点の精度から、どうすればより高い精度が目指せるかを明らかにする

```
┌─────────────────────────────────────────────────────────────────────┐
│                    精度向上の道筋                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【短期（PoC期間内）】                                               │
│  ├─ プロンプト改善: 営業知見者フィードバックの反映                   │
│  ├─ Few-shot例の充実: 良い事例・悪い事例の追加                       │
│  └─ 商材情報の整備: 説明の充実、構造化                               │
│                                                                     │
│  【中期（MVP〜初期運用）】                                           │
│  ├─ ナレッジ蓄積: 成功/失敗事例のDB化                                │
│  ├─ フィードバックループ: 提案結果（成約/失注）の学習                │
│  └─ 類似事例検索: RAG/Vector DBによる精度向上                        │
│                                                                     │
│  【長期（本格運用）】                                                │
│  ├─ ファインチューニング: 自社データでのモデル調整                   │
│  ├─ マルチモーダル: 決算資料・IR資料の画像解析                       │

│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**PoCで明らかにすること**:
- 各改善施策の効果見込み（どこに注力すべきか）
- 精度向上に必要なデータ・ナレッジの種類と量
- 投資対効果（費用をかければ改善するか）

#### 3. 実用化に向けた課題整理

> **目的**: 本番運用に向けて何が必要かを明確にする
>
> **PoCスコープ**: 精度・データ観点の課題整理を中心に実施。既存システム連携・セキュリティ・運用体制はPoC対象外（MVP以降で検討）

| 観点 | 明らかにしたいこと | PoCスコープ |
|------|-------------------|:-----------:|
| **精度** | 実用に耐える精度の閾値はどこか | ✅ |
| **データ構造化** | 現状のデータをどう構造化・整備すべきか | ✅ |
| **データ品質** | データの欠損・ばらつきにどう対処するか | ✅ |
| **ナレッジ** | どのような暗黙知・ルールが必要か | ✅ |
| **フィードバック活用** | 制約理由・失注理由をどう蓄積・活用するか | ✅ |
| **ユーザー要望分析** | ブラッシュアップ時のプロンプトをどう分析するか | ✅ |
| **システム連携** | 既存システムとの連携ポイントはどこか | ❌（MVP以降） |
| **セキュリティ** | 顧客情報の扱いをどうするか | ❌（MVP以降） |
| **運用体制** | 営業知見者による監修・チェック体制は必要か | ❌（MVP以降） |

├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【制約・失注情報の蓄積】                                            │
│  ├─ 制約理由: なぜその商材が提案できなかったか                       │
│  │   （例: 与信上限、既存取引との競合、規制要件など）                │
│  ├─ 失注理由: なぜ成約に至らなかったか                               │
│  │   （例: 価格、タイミング、競合優位、顧客内部事情など）            │
│  └─ 活用方法: 同様のケースでの注意喚起、提案精度向上                 │
│                                                                     │
│  【ユーザーのブラッシュアップ過程の蓄積】                             │
│  ├─ 初回提案方針 → ユーザーの修正指示 → 最終版                       │
│  ├─ ユーザーがどのようなプロンプト（指示）を出したか                 │
│  ├─ どのような観点で修正を求めたか（追加、削除、優先度変更など）     │
│  └─ 活用方法:                                                        │
│       ・よくある修正パターンの分析 → プロンプト改善                  │
│       ・ユーザー要望の傾向分析 → 機能改善の優先度決定                │
│       ・ベストプラクティスの抽出 → Few-shot例として活用              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**PoCで検証すること**:
- ブラッシュアップ過程（ユーザーとのやり取り）をどう記録するか
- 制約理由・失注理由をどのフォーマットで蓄積するか
- 蓄積したデータをどう分析・活用するか（ダッシュボード、レポート等）

#### 6. ナレッジの集め方・持ち方 ★新規

> **目的**: ベテランの暗黙知をどう抽出・言語化・蓄積するかを明らかにする
> **詳細**: [ナレッジ設計](14_knowledge_design.md)

```
┌─────────────────────────────────────────────────────────────────────┐
│                  ナレッジ設計で明らかにすること                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【データとナレッジの違い】                                          │
│  ├─ データ: 構造化された事実情報（企業情報、取引履歴）              │
│  └─ ナレッジ: 経験に基づく判断基準・暗黙知                          │
│       例:「社長65歳以上なら事業承継を考慮」                          │
│       例:「借入を増やしたくないと言っても本音は違う」                │
│                                                                     │
│  【ナレッジの3層構造】                                               │
│  ├─ Layer 3: 暗黙知（言語化されていない、ベテランの頭の中）         │
│  ├─ Layer 2: 形式知（言語化済み、散在している）                     │
│  └─ Layer 1: データ（構造化済み）                                   │
│                                                                     │
│  【動的/静的情報の分類】                                             │
│  ├─ 静的: 商品カタログ、業界ナレッジ（年1回以下の更新）             │
│  ├─ 準動的: 事例、アカウントプラン（月〜四半期更新）                │
│  └─ 動的: 面談記録、顧客発言（日〜週単位で更新）                    │
│                                                                     │
│  【蓄積サイクル】                                                    │
│  面談 → 情報収集 → AI活用 → 修正FB → ナレッジ蓄積 → 精度向上        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**PoCで明らかにすること**:
- 暗黙知をどう抽出するか（FBからの抽出、ベテランインタビュー、事例分析）
- 抽出した暗黙知をどう言語化・フォーマット化するか
- ナレッジをどこに持つか（システムプロンプト固定、Few-shot、RAG）
- 動的情報の更新サイクルをどう設計するか
- ナレッジ投入による精度向上効果はどの程度か

#### 精度レベルの目標

| レベル | 定義 | PoC目標 |
|--------|------|---------|
| **新人レベル** | 基本は拾えるが抜け漏れあり、チェック必須 | 最低ライン（必達） |
| **中堅レベル** | 主要項目を網羅、軽微な修正で使える | 目標ライン |
| **ベテランレベル** | 潜在ニーズまで先読み、そのまま使える | 挑戦ライン |
| **Mgrレベル** | 組織戦略・長期関係を踏まえた判断ができる | 将来展望（PoC対象外） |

**PoC終了時の達成目標**:
- **必達**: 「新人レベル」以上を安定的に達成（営業の叩き台として使える）
- **目標**: 70%以上のケースで「中堅レベル」を達成
- **挑戦**: 一部のケースで「ベテランレベル」の出力を達成
- **将来展望**: Mgrレベル（組織戦略・ポートフォリオ視点での提案支援）
