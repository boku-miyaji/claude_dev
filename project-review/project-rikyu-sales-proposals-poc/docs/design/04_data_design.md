## 5.0 データとナレッジの関係 ★新規

> **詳細**: [ナレッジ設計](14_knowledge_design.md)

本PoCでは「データ」と「ナレッジ」を明確に区別して設計します。

```
┌─────────────────────────────────────────────────────────────────────┐
│                  データとナレッジの違い                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【データ】本ドキュメントで扱う                                       │
│  ├─ 構造化された事実情報                                            │
│  ├─ 例: 企業情報、商品カタログ、取引履歴、面談記録                  │
│  └─ そのまま格納・検索できる                                        │
│                                                                     │
│  【ナレッジ】14_knowledge_design.md で扱う                           │
│  ├─ 経験に基づく判断基準・暗黙知                                    │
│  ├─ 例:「社長65歳以上なら事業承継を考慮」                           │
│  │      「借入を増やしたくないと言っても本音は違う」                │
│  └─ ベテランの頭の中にあり、言語化が必要                            │
│                                                                     │
│  【関係】                                                            │
│  データ（入力）＋ ナレッジ（判断基準）→ AI出力                       │
│  ナレッジの質と量がAI精度を決定する                                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### データとナレッジの対応表

| 項目 | データ（本ドキュメント） | ナレッジ（14_knowledge_design.md） |
|------|------------------------|----------------------------------|
| 企業情報 | 業種、売上、従業員数 | 「製造業の設備投資サイクルは5-7年」 |
| 面談記録 | 顧客の発言内容 | 「この発言の裏にはこういう意図がある」 |
| 商品カタログ | 商品スペック、価格 | 「この規模の企業にはシンジケートローン」 |
| 過去事例 | 成約/失注の事実 | 「なぜ成約したか」の要因分析 |
| アカウントプラン | 経営アジェンダ、キーパーソン | 「経理部長は常務への影響力がある」 |

    経営アジェンダ: 「海外展開に伴う為替リスク管理」が真のニーズ
    キーパーソン: 社長・常務・経理部長との関係構築
    ニーズ把握度: 高

    提案商品: 為替予約、海外送金サービス
    成約/失注: 成約

  - 「コスト削減」から始まる企業への深掘りアプローチの参考

## 5.4 データ収集計画

### データ収集方針

```
┌─────────────────────────────────────────────────────────────────────┐
│                    データ収集方針                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【基本方針】                                                        │
│  ・お客さんからデータをもらうだけもらう                              │
│  ・不足分はこちら（弊社）で仮のデータを作成して補完                  │
│                                                                     │
│  【理由】                                                            │
│  ・顧客データの提供可否・提供時期に依存しすぎない進め方              │
│  ・仮データでも検証は可能（構造・形式の確認が主目的）                │
│  ・実データが来たら差し替え、精度検証はそこから本格化                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 収集スケジュール

| 時期 | 収集データ | 顧客提供 | 不足時の対応 |
|------|------------|:--------:|--------------|
| W1-W2 | **商材データ**（対象領域の全量） | 依頼 | 公開情報から仮データ作成 |
| W2 | **初回シナリオ**（L2成約ケース1件） | 依頼 | 公開情報から仮シナリオ作成 |
| W3以降 | **追加シナリオ**（課題に応じて） | 随時 | 課題に応じてLLM生成で補完 |

### データ構造化の課題

PoCで整理すべきデータ関連の課題:

| 課題 | 内容 | PoCでの対応 |
|------|------|-------------|
| **データ形式の統一** | 商材データがPDF/Excel/Word等で分散 | 共通フォーマットに変換、構造化ルール定義 |
| **データ欠損への対処** | 必須項目が空欄、記載が曖昧 | 欠損パターンの分類、補完ルール策定 |
| **データ品質のばらつき** | 記載粒度・表現が担当者により異なる | 正規化ルール定義、品質チェック基準策定 |
| **匿名化ルール** | 企業名・個人名のマスキング方法 | 匿名化ルール策定、自動化可否検討 |

※開発環境は弊社環境を使用（顧客環境への依存なし）
※商材データ・過去提案事例は社内資料をそのまま使用可能か要確認
※過去提案事例は匿名化が必要（企業名→「A社」等）
※大量の顧客データ収集はPoCスコープ外（MVP以降で検討）

## 5.5 評価用テストデータセット（アジャイル型）

### 設計方針と目的

```
【設計方針】
まず1件のシナリオでAI出力を作成し、FBをもらいながら改善。
課題に応じてシナリオを段階的に追加（最大50件目安）。

【進め方】
1. 初回シナリオ1件（L2成約ケース）で動作確認
2. FB収集 → 改善サイクル
3. 「この軸が弱い」と分かったらその軸のシナリオを追加
4. 必要十分な件数で止める（シナリオ件数は固定しない）

【PJ出口で示したいこと】
1. ニーズ推定AIの精度（どの程度正しくニーズを抽出できるか）
2. ソリューション提案AIの精度（ニーズ充足度 + ソリューション妥当性）
3. 自動評価AIの信頼性（概ね人の評価と一致するか）
4. 改善の方向性（どこを強化すれば精度が上がるか）
5. 継続利用による学習効果（使い続けることで精度が向上する仕組み）
   - ナレッジ蓄積（主軸）: 成功事例・失敗事例をナレッジとして追加
   - 実績との紐付け: 提案結果（成約/失注）をフィードバックとして蓄積
   - 業務連携: システムを通した修正・やり取りで継続的に精度向上
```

### シナリオの多様性設計（段階的拡張時の軸）

```
【多様性の4軸】※課題に応じて必要な軸を優先的に追加

軸1: 商品カテゴリ（最終目安: 各10件 × 5カテゴリ = 50件）
  1. 融資（長期・短期・シンジケートローン等）
  2. M&A・事業承継
  3. 外国為替・金利リスクヘッジ
  4. ビジネスマッチング
  5. その他（補助金支援等）

軸2: 顧客属性（各カテゴリ内でバランス）
  A. 製造業（売上50-100億 / 100-500億 / 500億超）
  B. 卸売・小売業（同上）
  C. サービス業（同上）
  D. 建設・不動産業（同上）
  E. その他業種

軸3: ニーズの明確さ（各カテゴリ内で半々）
  α. 明確なニーズ（顧客が課題を認識・発言している）
  β. 潜在ニーズ（顧客が気づいていない、行間から読み取る）

軸4: 複雑度（難易度のグラデーション）
  L1. 単一ニーズ・明確なマッチ（簡単）
  L2. 単一ニーズ・複数候補（中程度）
  L3. 複数ニーズ・優先度判断必要（難しい）
  L4. 矛盾情報あり・深い推論必要（最難）

```

### 段階的拡張時の目安件数

| 商品カテゴリ | 目安件数 | 内訳 |
|------------|----------|------|
| 融資 | 10件 | 業種×規模×難易度のバリエーション |
| M&A・事業承継 | 10件 | 後継者不在、事業拡大等 |
| 為替・金利 | 10件 | 海外取引、金利変動リスク等 |
| ビジネスマッチング | 10件 | 販路拡大、仕入先、提携等 |
| その他 | 10件 | 補助金、DX、SDGs対応等 |

**※件数は固定ではなく、課題に応じて調整**

### シナリオ追加プロセス（アジャイル型）

```
┌─────────────────────────────────────────────────────────────────────┐
│                シナリオ段階的拡張プロセス                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Step 1: 初回シナリオで動作確認（W2-W3）                            │
│  ├─ お客様から実データ1件（L2成約ケース推奨）を取得                │
│  ├─ AIの出力を作成、お客様に見せてFB収集                          │
│  └─ 「何が足りないか」「どこがズレているか」を把握                 │
│                              ↓                                      │
│  Step 2: 改善サイクル（W3以降、毎週）                               │
│  ├─ FBをもとにプロンプト・ナレッジを改善                          │
│  ├─ 同じシナリオで再実行 → 改善効果を確認                         │
│  └─ 課題が見えてきたら次のシナリオ追加を検討                       │
│                              ↓                                      │
│  Step 3: シナリオ追加（課題に応じて）                               │
│  ├─ 「この軸が弱い」→ その軸のシナリオを追加                      │
│  ├─ お客様から追加実データ or LLMで類似シナリオ生成               │
│  ├─ 自動品質チェック（ハルシネーション、論理整合性）              │
│  └─ 必要十分な件数で止める（最大50件目安）                        │
│                              ↓                                      │
│  Step 4: LLM自動評価AIの育成（W4以降、並行）                        │
│  ├─ 人の評価結果をもとにルーブリックを調整                        │
│  ├─ 概ね人と一致するようになったら残りに適用                      │
│  └─ W6までに確定、改善しなければ人評価中心に切り替え              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 人評価の進め方

```
【アジャイル型での人評価】

・シナリオ追加のたびに営業知見者がレビュー
・評価結果をLLM自動評価AIの教師データとして活用
・段階的にLLM自動評価AIの精度を向上

【人評価で確認すること】
- 入力データは現実的か（業界・規模・状況の整合性）
- AIの出力は妥当か（営業知見者観点での確認）
- 抜けている重要なニーズはないか
- 的外れなニーズはないか
- 推薦された商材は妥当か
```

### シナリオ難易度ラベルによるAIレベル判定

AIの精度を「新人レベル」「中堅レベル」「ベテランレベル」で表現するため、シナリオ自体に難易度ラベル（L1〜L4）を付与し、各難易度での精度からAIのレベルを判定します。

```
【目的】
AIの精度を分かりやすく表現する
  例: 「ニーズ推定AIは中堅レベル（L1-L2を高精度で対応）」
      「新人が使えば中堅並みの提案が可能に」

【アプローチ】
┌─────────────────────────────────────────────────────────────────────┐
│  シナリオ難易度ラベル方式                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  シナリオに難易度ラベルを付与（最終目安: 最大50件）:                 │
│                                                                     │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐  ┌───────────┐       │
│  │ L1(10件)  │  │ L2(20件)  │  │ L3(15件)  │  │ L4(5件)   │       │
│  │ 新人向け  │  │ 標準      │  │ 応用      │  │ 高度      │       │
│  └─────┬─────┘  └─────┬─────┘  └─────┬─────┘  └─────┬─────┘       │
│        ↓              ↓              ↓              ↓              │
│  ┌─────────────────────────────────────────────────────────────┐  │
│  │          AIが各難易度のシナリオに回答                         │  │
│  └─────────────────────────────────────────────────────────────┘  │
│        ↓              ↓              ↓              ↓              │
│  ┌─────────────────────────────────────────────────────────────┐  │
│  │          難易度別◯率からAIレベルを判定                        │  │
│  └─────────────────────────────────────────────────────────────┘  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 難易度の詳細定義

#### L1: 新人向け（基本） - 経験1-2年目の新人が対応可能

| 観点 | 具体的な特徴 |
|------|-------------|
| ニーズの明確さ | 顧客が課題を明示的に発言（例:「資金繰りが厳しい」） |
| 商材マッチ | 1ニーズ → 1商材の明確なマッチ |
| 情報の完全性 | 判断に必要な情報が入力に全て含まれている |
| 業界知識 | 業界特有の知識は不要、一般的なビジネス理解で対応可 |
| 推論の深さ | 表面的な情報から直接導出可能 |
| 想定正答率 | 新人80%↑、中堅95%↑、ベテラン98%↑ |

#### L2: 新人〜中堅向け（標準） - 経験2-4年目で対応可能

| 観点 | 具体的な特徴 |
|------|-------------|
| ニーズの明確さ | ニーズは明示的だが複数解釈可能（例:「成長したい」） |
| 商材マッチ | 1ニーズ → 複数商材候補、最適解選定が必要 |
| 情報の完全性 | 大部分は含まれるが一部推測が必要 |
| 業界知識 | 基本的な業界慣行の理解が必要 |
| 推論の深さ | 1段階の推論が必要 |
| 想定正答率 | 新人50-60%、中堅80%↑、ベテラン90%↑ |

#### L3: 中堅〜ベテラン向け（応用） - 経験4-6年目で対応可能

| 観点 | 具体的な特徴 |
|------|-------------|
| ニーズの明確さ | 潜在ニーズが存在（顧客が気づいていない課題） |
| 商材マッチ | 複数ニーズ × 複数商材の組み合わせ最適化が必要 |
| 情報の完全性 | 行間を読む必要あり |
| 業界知識 | 業界固有の課題・トレンドの理解が必要 |
| 推論の深さ | 複数段階の推論が必要 |
| 想定正答率 | 新人20-40%、中堅60-70%、ベテラン85%↑ |

#### L4: ベテラン向け（高度） - 経験6年以上のベテランでも難しい

| 観点 | 具体的な特徴 |
|------|-------------|
| ニーズの明確さ | 顧客の発言と真のニーズが異なる |
| 商材マッチ | 標準商材では対応困難、カスタマイズや複合提案が必要 |
| 情報の完全性 | 矛盾する情報含む、または重要情報が欠落 |
| 業界知識 | 深い業界知識と最新トレンドの理解が必要 |
| 推論の深さ | 深い洞察と仮説構築が必要 |
| 想定正答率 | 新人10-20%、中堅40-50%、ベテラン70-80% |

### 難易度ラベルの配分と判定基準

| 難易度 | 件数 | 比率 | 15件抽出時 | 目的 |
|--------|------|------|-----------|------|
| L1（新人向け） | 25件 | 25% | 3-4件 | 基本動作の確認 |
| L2（標準） | 40件 | 40% | 5-6件 | 標準ユースケースの精度測定 |
| L3（応用） | 25件 | 25% | 4-5件 | 中堅レベルの判断力評価 |
| L4（高度） | 10件 | 10% | 1-2件 | 限界性能の確認 |

### AIレベル判定基準

| AI判定 | 判定基準 | 解釈 |
|--------|----------|------|
| 新人レベル未満 | L1◯率 < 70% | 基本でも精度不足、実用困難 |
| 新人レベル | L1◯率 ≥ 80%、L2◯率 < 60% | 定型的な提案は可能 |
| 中堅レベル | L1◯率 ≥ 90%、L2◯率 ≥ 70%、L3◯率 ≥ 50% | 標準ケースを高精度で対応 |
| ベテランレベル | L1◯率 ≥ 95%、L2◯率 ≥ 85%、L3◯率 ≥ 70%、L4◯率 ≥ 50% | 複雑ケースにも対応 |

**PoCの目標**:
- **中堅レベル**の達成（L1◯率90%以上、L2◯率70%以上、L3◯率50%以上）
- 「中堅レベル」達成 → 新人支援ツールとして十分実用的
- 「ベテランレベル」到達 → ベテラン補助・代替として活用可
