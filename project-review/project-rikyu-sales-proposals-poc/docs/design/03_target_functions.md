## 4.1 ソリューションセールス伴奏AIの機能設計

> **本質**: 単発の商材提案ではなく、営業活動全体に伴奏するAI

```
┌─────────────────────────────────────────────────────────────────────┐
│           ソリューションセールス伴奏AIの全体像                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【入力】                                                            │
│  ├─ 企業情報・面談記録                                              │
│  ├─ 経営アジェンダシート（現状）★必須入力                          │
│  ├─ キーパーソンマップ（現状）★必須入力                            │
│  └─ 外部環境変化（ニュース、政策変更、市況等）                       │
│            ↓                                                        │
│  【Step1: ニーズ推定】                                              │
│  経営アジェンダを踏まえて「何を本来求めているか」を理解              │
│  情報の充足度を判定し、不足があれば深掘りポイントを特定              │
│            ↓                                                        │
│  【Step2: ソリューション提案】                                      │
│  ニーズにマッチするソリューションを推薦                              │
│  キーパーソンとの関係性を踏まえたアプローチを提案                    │
│            ↓                                                        │
│  【出力】                                                           │
│  ├─ 提案方針・推薦商材                                              │
│  ├─ ネクストアクション ★伴奏AIの核心                               │
│  │   ├─ 経営アジェンダシート深掘りアクション                       │
│  │   ├─ キーパーソンマップ関係深化アクション                       │
│  │   └─ 追加情報収集アクション                                     │
│  └─ アカウントプラン更新提案                                        │
│                                                                     │
│  【伴奏のサイクル】                                                  │
│  出力を踏まえて営業が行動 → 新情報を得る → 再度AIに入力 → ...       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**機能設計の方針**:
- 商材提案だけでなく「次に何をすべきか」を提示することが伴奏AIの核心
- 経営アジェンダシート・キーパーソンマップを入力として活用し、更新提案も出力
- 難しすぎるソリューションはPoC期間を考慮して除外可能
- ただし「カードから始める」ようなプロダクト起点のアプローチはしない
- ソリューションの内容は複雑・カスタマイズされる可能性があることを前提とする

## 4.2 機能① 顧客ニーズ推定AI

| 項目 | 内容 |
|------|------|
| **目的** | 企業情報・アカウントプランから顧客の潜在ニーズを推定し、情報の充足度を判定する |
| **入力** | 企業情報 + 面談記録 + **経営アジェンダシート（現状）** + **キーパーソンマップ（現状）** + 類似企業の過去事例 |
| **出力** | 推定ニーズリスト（カテゴリ、優先度、根拠） + **情報充足度判定** + **深掘りポイント** |
| **技術アプローチ** | LLM + プロンプトエンジニアリング |
| **重視する観点** | 「お客さんが何を本来求めていて、どうなると嬉しいか」の理解 |

**入力データの詳細**:
| データ | 必須/任意 | 内容 |
|--------|:--------:|------|
| 企業情報 | 必須 | 業種、規模、財務状況等 |
| 面談記録 | 必須 | 直近の面談内容、顧客発言 |
| **経営アジェンダシート** | **必須** | 顧客の経営課題、中期計画、優先事項（現状把握分） |
| **キーパーソンマップ** | **必須** | 意思決定者、影響者、関係性（現状把握分） |
| 類似企業の過去事例 | 任意 | 参考情報として提示 |

**過去事例の活用**:
- 類似企業（業種・規模・状況が近い）の過去ニーズを参考情報として提示
- 「この業種・規模の企業では〇〇のニーズが多い」という傾向をLLMに伝える
- **経営アジェンダ深掘りの成功事例**も参考として活用
- 過去事例はシステムプロンプトまたはFew-shot例として投入

## 4.3 機能② 提案方針策定AI

> **重要**: 商品起点ではなく、**ニーズ・経営アジェンダ起点**の提案方針を策定する

| 項目 | 内容 |
|------|------|
| **目的** | 推定されたニーズに対して**経営アジェンダ起点**の提案方針を策定し、次のアクションを提示する |
| **入力** | 推定ニーズ（根拠付き） + 商材情報 + **経営アジェンダシート** + **キーパーソンマップ** + 企業情報 |
| **出力** | 提案方針（根拠付き） + 活用商材（組み合わせ） + **ネクストアクション** |
| **技術アプローチ** | LLM + 商材データをシステムプロンプトに投入（RAG不要） |
| **重視する観点** | **経営アジェンダ起点**での提案ストーリー + **営業活動全体への伴奏** |

**出力の考え方**:
- ❌ 単なる商材リストの推薦（商品起点）
- ✅ 経営アジェンダを起点とした一貫した提案ストーリー（ニーズ起点）
- 「なぜこの方針なのか」の根拠を必ず明示
- 複数商材を段階的に組み合わせたPhase設計
- **ネクストアクション（経営アジェンダ深掘り・キーパーソン関係深化）を必ず提示** ★伴奏AIの核心
- キーパーソンマップを踏まえた提案アプローチ（誰にどう提案するか）を含む

**過去事例の活用（拡張）**:
| 事例タイプ | 内容 | 活用方法 |
|------------|------|----------|
| 商材提案事例 | 類似企業への過去提案（成約/失注結果含む） | 提案ストーリーの参考、注意点の把握 |
| **経営アジェンダ深掘り事例** | 類似企業での経営課題の深掘りプロセス | 「どう深掘りしたか」の参考 |
| **キーパーソン関係構築事例** | 類似企業でのキーパーソンへのアプローチ | 「誰にどうアプローチしたか」の参考 |
| **アカウントプラン更新事例** | 類似企業でのアカウントプラン運用 | 継続的な関係構築の参考 |

## 4.4 出力設計

### 顧客ニーズ推定AIの出力

```
【総評】○・△・× + 全体コメント

【情報充足度判定】★新規
- 経営アジェンダシート充足度: 高/中/低
  - 不足項目: 中期経営計画の具体的KPI、投資優先順位
- キーパーソンマップ充足度: 高/中/低
  - 不足項目: 常務Aの意向、経理部長との関係性

【抽出ニーズ一覧】
1. ニーズ名: xxxxx
   - 根拠: 企業情報/ヒアリング/経営アジェンダのどの部分から推定したか
   - 確度: 高/中/低
   - 優先度: 高/中/低
   - 関連する経営アジェンダ項目: ★新規

【参考：類似企業の過去ニーズ】
- 類似企業A（業種: ○○、規模: ○○億）: △△のニーズがあった
- 類似企業B（業種: ○○、規模: ○○億）: □□のニーズがあった

【経営アジェンダ深掘りポイント】★新規
1. 深掘り項目: 中期経営計画の投資優先順位
   - 現状: 「設備投資を検討」という情報のみ
   - 確認すべきこと: 投資規模、時期、優先順位
   - 確認方法: 経営企画部長への面談
   - 期待効果: 提案の優先順位付けが明確になる

【追加ヒアリング推奨項目】
1. 質問項目: xxxxx
   - 理由: なぜこの質問が必要か
   - 期待される情報: 何が明らかになるか
```

### 提案方針策定AIの出力（PoCスコープ）

```
【総評】○・△・× + 全体コメント

【提案方針】
この顧客に対しては、○○を軸とした提案が有効と考えられる。
理由は△△であり、□□という状況を踏まえると、
まず××から着手し、段階的に◇◇へ展開するアプローチが適切。

【提案の理由】
- なぜこの方針が適切か
- 顧客の状況・ニーズとの整合性
- 経営アジェンダとの整合性 ★新規
- 期待される効果・メリット

【注意すべきポイント】
- 提案時に気をつけるべき点（顧客の懸念、過去の経緯等）
- 競合状況・他行動向
- タイミングや進め方の注意点
- 失注リスクと回避策

【推薦商材】
1. 商材名: xxxxx
   - 対応ニーズ: どのニーズに対応するか
   - 提案上のポイント: この商材を提案する際の訴求点
   - 類似事例: ○○業の企業で同様のニーズに対して提案→成約

2. 商材名: yyyyy（組み合わせ提案の場合）
   - 対応ニーズ: どのニーズに対応するか
   - 商材1との関係: なぜ組み合わせが有効か

【ネクストアクション】★新規（伴奏AIの核心）
┌─────────────────────────────────────────────────────────────────┐
│ 1. 経営アジェンダシート深掘り                                     │
│    - 深掘り項目: 中期経営計画の具体的なKPI、投資優先順位          │
│    - 現状の把握レベル: 「設備投資検討中」のみ                     │
│    - 確認方法: 次回面談で経営企画部長に確認                       │
│    - 質問例: 「中計の達成に向けて、投資の優先順位はどうお考えですか」│
│    - 期待効果: 提案の優先順位付けが明確になる                     │
│                                                                   │
│ 2. キーパーソンマップ関係深化                                     │
│    - 重要人物: 常務A（決裁権者）、部長B（現場推進者）             │
│    - 現状の関係: 常務Aとの接点が薄い（部長B経由のみ）             │
│    - 推奨アクション: 常務A向けの経営層セミナーを案内              │
│    - アプローチ方法: 部長Bから常務Aを紹介してもらう               │
│    - 期待効果: 決裁プロセスがスムーズになる                       │
│                                                                   │
│ 3. 追加情報収集                                                   │
│    - 競合動向: B行の動きを確認（担当者へのヒアリング）            │
│    - 社内調整: 与信枠の事前確認（審査部への相談）                 │
│    - 外部情報: 業界ニュースのウォッチ                             │
└─────────────────────────────────────────────────────────────────┘

【アカウントプラン更新提案】★新規
- 経営アジェンダシート更新案:
  - 追加項目: 「設備投資の優先順位」を追加
  - 変更項目: 「事業承継」の優先度を「中」→「高」に変更（社長年齢を踏まえ）
- キーパーソンマップ更新案:
  - 追加人物: 経理部長（今回の面談で存在を把握）
  - 関係性更新: 部長Bとの関係を「良好」に更新

【参考：類似企業への過去提案】
- 類似企業A: △△を提案 → 成約（成功要因: ○○）
- 類似企業B: □□を提案 → 失注（注意点: ××）

【参考：類似企業でのアカウントプラン運用事例】★新規
- 類似企業C: 経営アジェンダを3回の面談で深掘り
  - 初回: 「コスト削減」が課題と把握
  - 2回目: 「海外展開」が本質的な課題と判明
  - 3回目: 「海外展開に伴う為替リスク」というニーズを特定
  - 成功要因: 段階的な深掘りで真のニーズを発見
```

※将来拡張時に以下を追加予定:
- 成約可能性スコア（過去稟議の学習結果に基づく）
- より多くの類似事例の自動検索（RAG/Vector DB活用）
- アカウントプランの自動更新連携

## 4.5 提案方針のブラッシュアップ機能

> **目的**: ユーザー（営業担当者）が提案方針を対話的に改善し、その過程を蓄積・分析する

```
┌─────────────────────────────────────────────────────────────────────┐
│                提案方針ブラッシュアップの流れ                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Step1】AI初回出力                                                 │
│  └─ 提案方針（理由・注意点・推薦商材）を生成                        │
│                ↓                                                    │
│  【Step2】ユーザーによるフィードバック                               │
│  ├─「この商材は今回使えない（与信上限のため）」                      │
│  ├─「A社との関係を踏まえてもっと慎重に」                            │
│  ├─「事業承継の優先度をもっと上げて」                               │
│  └─「競合B行の動向を考慮して」                                      │
│                ↓                                                    │
│  【Step3】AIによる修正版出力                                         │
│  └─ ユーザーの指示を反映した提案方針v2を生成                        │
│                ↓                                                    │
│  【Step4】最終版確定                                                 │
│  └─ 複数回のやり取りを経て最終版を確定                              │
│                                                                     │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  【蓄積・分析】                                                      │
│  ├─ やり取り全体（初回出力→ユーザー指示→修正版...→最終版）を記録    │
│  ├─ ユーザーがどのような観点で修正を求めたかを分類・集計            │
│  │   （制約追加/優先度変更/観点追加/トーン変更/商材変更 等）         │
│  ├─ よくある修正パターンを分析 → プロンプト改善に反映               │
│  └─ 成功した提案パターンをFew-shot例として活用                       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 蓄積するデータ

| データ | 内容 | 活用方法 |
|--------|------|----------|
| **初回出力** | AIが最初に生成した提案方針 | 改善前後の比較 |
| **ユーザープロンプト** | ユーザーが入力した修正指示（全履歴） | 要望パターン分析、プロンプト改善 |
| **修正版出力** | 各修正指示に対するAIの出力 | 修正精度の評価 |
| **最終版** | ユーザーが承認した最終的な提案方針 | ベストプラクティス抽出 |
| **制約理由** | 提案できなかった理由（与信、規制、競合等） | 類似ケースでの事前注意喚起 |
| **提案結果** | 成約/失注とその理由 | フィードバックループ |

### 分析レポート（イメージ）

```
【ユーザー要望分析レポート】

■ 修正タイプ別の発生頻度
  1. 制約条件の追加（与信、規制等）: 35%
  2. 優先度の変更: 25%
  3. 観点・論点の追加: 20%
  4. 商材の変更・除外: 15%
  5. トーン・表現の調整: 5%

■ よくある修正パターン TOP5
  1. 「与信上限を考慮して商材Aは除外」
  2. 「競合B行の動向を踏まえて」
  3. 「事業承継は社長が消極的なので優先度を下げる」
  4. 「M&Aは時期尚早なので言及しない」
  5. 「過去に○○で失注しているので慎重に」

■ 改善提案
  - プロンプトに「与信上限確認」の観点を追加すべき
  - 競合動向を入力情報に含める検討
  - 社長の意向確認をヒアリング推奨項目に追加
```

## 4.5.1 ナレッジ活用設計 ★新規

> **関連**: [ナレッジ設計](14_knowledge_design.md)

### ナレッジの機能への組み込み

```
┌─────────────────────────────────────────────────────────────────────┐
│                ナレッジ活用の全体像                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【第3層：暗黙知】プロンプト/Few-shot例として組み込み                │
│  ├─ 顧客ニーズナレッジ: 「こう言ったら実はこう」の判断基準          │
│  ├─ 商品ナレッジ: 「この状況ならこの商品」の判断基準                │
│  ├─ ソリューションナレッジ: 成功パターン・失敗パターン              │
│  └─ コミュニケーションナレッジ: 効果的なアプローチ方法              │
│            ↓                                                        │
│  【第2層：形式知】ルール・テンプレートとして適用                     │
│  ├─ 業界別パターン: 「製造業×大企業→よくあるニーズ」               │
│  ├─ 商品組み合わせルール: 「設備投資+リース」の提案パターン         │
│  └─ アプローチテンプレート: 「経営層へのアプローチ手順」            │
│            ↓                                                        │
│  【第1層：データ】構造化データとして入力                             │
│  ├─ 企業情報: 業種、規模、財務状況                                  │
│  ├─ 面談記録: 顧客発言、確認事項                                    │
│  └─ 過去事例: 成約/失注の結果データ                                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 機能①（ニーズ推定AI）でのナレッジ活用

| ナレッジタイプ | 活用方法 | 具体例 |
|---------------|----------|--------|
| **顧客ニーズナレッジ** | プロンプト内のガイダンス | 「"借入増やしたくない"は必ずしも借入拒否ではなく条件次第」 |
| **業界別パターン** | Few-shot例として提示 | 「製造業×売上300億→設備資金・運転資金ニーズが多い」 |
| **情報解釈の暗黙知** | プロンプト内の注意事項 | 「社長の年齢65歳以上→事業承継ニーズを検討」 |

### 機能②（提案方針策定AI）でのナレッジ活用

| ナレッジタイプ | 活用方法 | 具体例 |
|---------------|----------|--------|
| **商品ナレッジ** | プロンプト内の判断基準 | 「大型設備投資→シンジケートローン向き」 |
| **ソリューションナレッジ** | Few-shot例として提示 | 「類似企業Aでシンジケート提案→成約。成功要因は複数行取引活用」 |
| **コミュニケーションナレッジ** | ネクストアクション生成の参考 | 「経営層接点が薄い→セミナー招待が効果的」 |
| **失敗パターンナレッジ** | 注意事項として提示 | 「借入依存度高い企業に単独大型融資→与信NGリスク」 |

### ナレッジの蓄積サイクル

