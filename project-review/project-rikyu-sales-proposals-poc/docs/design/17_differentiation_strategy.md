## 1. エグゼクティブサマリー

### 1.1 NTTData PoC2と本PoCの本質的な違い

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                     2つのPoCの本質的な違い                                        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  【NTTData PoC2】データ×AI = 情報収集・分析の自動化                             │
│  ─────────────────────────────────────────────────                              │
│  ・先方環境内で構築（実データへのフルアクセス）                                  │
│  ・SFA実データ + 外部有償データ（TSR）+ 公知情報                                │
│  ・AIエージェントによる情報収集・分析・レポート生成                             │
│  ・→ すでに定量評価4.0/5.0達成、実業務展開可能レベル                            │
│                                                                                 │
│  【本PoC】暗黙知×AI = ベテランの判断基準を形式知化する仕組み                    │
│  ─────────────────────────────────────────────────                              │
│  ・仮データ環境（実データとの整合性は厳密でなくてよい）                         │
│  ・データ分析より「どうナレッジを蓄積・活用するか」の仕組み検証                │
│  ・ベテランの暗黙知をAIに教え、継続的に精度向上させるサイクル                  │
│  ・→ 精度より「仕組み」を見せることが価値                                       │
│                                                                                 │
│  ★差別化の核心：「データ化されていない暗黙知・ナレッジ」をどう蓄積・活用するか│
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 本PoCでやるべきこと・やらなくてよいこと

| 観点 | やるべきこと | やらなくてよいこと |
|------|-------------|-------------------|
| **データ** | 仮データで仕組みを検証 | 実データとの厳密な整合性 |
| **精度** | 改善の仕組みを示す | 高精度の企業分析 |
| **アウトプット** | 全体的な動作・利用イメージ | 提案アイデアの精緻さ |
| **評価** | 知見を含んだ評価観点の設計 | 一般的な評価指標のスコアリング |
| **価値** | 暗黙知の蓄積・活用サイクル | 単発の企業分析レポート |

## 2. 差別化ポイント詳細

### 2.1 データ vs ナレッジ（暗黙知）

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                 NTTData PoC2が解決したこと vs 残った課題                          │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  【NTTData PoC2の成果】✅ 解決済み                                               │
│  ├─ 行内情報のみでは不足 → 外部有償データ（TSR）で解消                          │
│  ├─ 単調な出力 → AIエージェント構成で幅広い示唆出し                             │
│  └─ 若手・新規着任者への効果を確認                                              │
│                                                                                 │
│  【NTTData PoC2で残った課題】❌ 未解決                                           │
│  ├─ 「りそなとして何が提案できるか」不明確                                      │
│  ├─ 提案が一般論的で漠然（人が思いつく範囲に留まる）                            │
│  ├─ 巷のLLM（Gemini等）との差別化が見えない                                     │
│  └─ 行内情報（CRM・交渉履歴）の活用不足                                         │
│                                                                                 │
│  ★これらの課題の本質                                                            │
│  「データ化されていない暗黙知・ナレッジ」がAIに入っていない                     │
│                                                                                 │
│  → 本PoCの役割：「データ化されていない暗黙知」を蓄積・活用する仕組みを見せる  │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 2.1.1 「データ化されていない暗黙知」とは

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│              暗黙知の種類：データにならない知見は多い                             │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  【種類1】データの解釈・判断基準                                                 │
│  ────────────────────────────────────────────────                              │
│  データは存在するが、解釈の仕方がAIに入っていない                               │
│  例:                                                                            │
│    ・データ「社長65歳」→ 知見「65歳以上なら事業承継を必ず考慮」                │
│    ・データ「自己資本比率35%」→ 知見「35%以下は資本増強も視野に」              │
│    ・データ「前期減益」→ 知見「コスト削減より売上向上の話から入る」             │
│                                                                                 │
│  【種類2】そもそもデータ化されていない経験則                                     │
│  ────────────────────────────────────────────────                              │
│  ベテランの頭の中にしかない、データとして記録されていない知見                   │
│  例:                                                                            │
│    ・「"借入増やしたくない"と言っても本音とは限らない」                        │
│    ・「創業者には選択肢を提示して選んでもらう方が響く」                         │
│    ・「製造業の設備投資サイクルは5-7年」                                        │
│    ・「卸売業は金利感応度が高いので手数料の話から入る」                         │
│    ・「経理部長が常務への影響力を持っている場合が多い」                         │
│                                                                                 │
│  【種類3】成功/失敗パターン                                                      │
│  ────────────────────────────────────────────────                              │
│  過去の経験から得られたパターン認識                                             │
│  例:                                                                            │
│    ・「Phase1で信頼構築→Phase2で本丸」という段階的アプローチ                   │
│    ・「初回でいきなりM&Aを切り出すと失敗する」                                  │
│    ・「設備投資と事業承継は組み合わせると効果的」                               │
│                                                                                 │
│  ★これらの知見は「データベースに入っていない」                                   │
│  → 本PoCで蓄積・活用の仕組みを作る                                              │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 具体的な差別化領域

#### 領域①：暗黙知の言語化・蓄積サイクル

```
┌─────────────────────────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  【サイクル1】FB起点の暗黙知抽出                                                 │
│  ┌───────────────────────────────────────────────────────────────────────────┐ │
│  │ AI出力 → 営業が修正 → 「なぜ修正したか」を分析 → 暗黙知として言語化       │ │
│  │                                                                           │ │
│  │ 例:                                                                       │ │
│  │   AI: 「資金調達ニーズを推定」                                            │ │
│  │   営業: 「この会社は借入を増やしたくないと言っているからリース優先」      │ │
│  │   → 暗黙知: 「"借入増やしたくない"発言時はリース優先で検討」              │ │
│  └───────────────────────────────────────────────────────────────────────────┘ │
│                                                                                 │
│  【サイクル2】成功/失敗事例からのパターン抽出                                    │
│  ┌───────────────────────────────────────────────────────────────────────────┐ │
│  │ 成約事例を分析 → 成功要因を特定 → パターンとして言語化                     │ │
│  │                                                                           │ │
│  │ 例:                                                                       │ │
│  │   成功A: 社長65歳、事業承継提案 → 成約                                    │ │
│  │   成功B: 社長68歳、事業承継提案 → 成約                                    │ │
│  │   失敗C: 社長60歳、事業承継提案 → 失注（「まだ早い」）                    │ │
│  │   → 暗黙知: 「社長65歳以上で事業承継を積極提案」                          │ │
│  └───────────────────────────────────────────────────────────────────────────┘ │
│                                                                                 │
│  【サイクル3】ベテランへのインタビューからの抽出                                 │
│  ┌───────────────────────────────────────────────────────────────────────────┐ │
│  │ ベテランの判断を観察 → 判断基準を言語化 → プロンプト/ナレッジに投入       │ │
│  └───────────────────────────────────────────────────────────────────────────┘ │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

## 3. ナレッジの種類と蓄積方針

### 3.1 蓄積すべきナレッジの分類

| カテゴリ | 具体例 | 蓄積方法 | 優先度 |
|---------|-------|---------|-------|
| **判断基準ナレッジ** | 「社長65歳以上→事業承継考慮」 | FB抽出、インタビュー | 最高 |
| **発言解釈ナレッジ** | 「借入増やしたくない≠本音とは限らない」 | FB抽出、事例分析 | 高 |
| **業界ナレッジ** | 「製造業の設備投資は5-7年サイクル」 | 営業知見者定義 | 高 |
| **商材マッチナレッジ** | 「この財務状況ならリース優先」 | FB抽出、事例分析 | 中 |
| **アプローチナレッジ** | 「創業者には選択肢提示型で」 | インタビュー、事例分析 | 中 |
| **成功パターンナレッジ** | 「Phase1で信頼構築→Phase2で本丸」 | 成功事例分析 | 中 |

### 3.2 パーソナライズの設計

```yaml
# パーソナライズ設定
user_profile:
  experience_level: "junior" | "mid" | "senior"
  specialization: ["製造業", "卸売業", ...] # 得意業界

output_adjustment:
  junior:
    explanation_depth: "detailed"    # 判断根拠を詳細に
    knowledge_display: "visible"     # 適用したナレッジを明示
    checklist: true                  # チェックリスト形式

  mid:
    explanation_depth: "summary"     # 要点のみ
    knowledge_display: "on_demand"   # 必要時に表示
    challenge_questions: true        # 壁打ち質問を提示

  senior:
    explanation_depth: "minimal"     # 簡潔
    blind_spot_focus: true           # 見落とし・盲点を中心に
    second_opinion: true             # 異なる角度の示唆
```

## 5. プロダクトとしての動作イメージ

### 5.1 デモシナリオ

```
【デモ構成】全体で15-20分

1. シナリオ概要（2分）
   - 対象企業：製造業、売上300億、社長65歳
   - 現状：設備投資検討中、後継者未定

2. AIによるニーズ推定（3分）
   - 入力：企業情報 + 面談記録
   - 出力：推定ニーズ（根拠付き）
   - ★ポイント：「なぜそう判断したか」を明示

3. 営業による修正とナレッジ蓄積（5分）
   - 営業：「この会社は借入を増やしたくないと言っていた」
   - AI：修正を反映し、リース優先に変更
   - ★ポイント：この修正が暗黙知として蓄積される

4. 提案方針策定（3分）
   - 出力：段階的アプローチ + ネクストアクション
   - ★ポイント：パーソナライズ切り替えのデモ
     - 新人向け：詳細説明モード
     - ベテラン向け：セカンドオピニオンモード

5. 継続的改善の説明（3分）
   - ナレッジ蓄積のダッシュボード
   - 精度向上の推移グラフ
   - ★ポイント：「使えば使うほど賢くなる」を可視化

6. まとめ（2分）
   - NTTDとの差別化
   - 本PoCの独自価値
```

### 5.2 デモで見せるべき価値

| 観点 | 見せる価値 | 具体的な表現 |
|------|----------|------------|
| **差別化** | データ分析ではなくナレッジ活用 | 「AIが判断基準を学習していく」 |
| **継続性** | 使えば使うほど賢くなる | ナレッジ蓄積数の可視化 |
| **実用性** | 営業と一緒に考える伴奏 | 対話的なUIフロー |

## 7. 成功の定義

### 7.1 本PoCの成功基準

| 観点 | 成功基準 | 測定方法 |
|------|---------|---------|
| **差別化の明確化** | 「NTTDとの違い」が営業知見者に伝わる | 理解度確認 |
| **仕組みの検証** | ナレッジ蓄積→精度向上のサイクルが回る | サイクル実証 |
| **デモ価値** | 「使いたい」と思わせるデモができる | デモ後FB |
| **知見活用** | 知見ベースの評価観点が有効 | 評価精度 |

### 7.2 成功しない場合の対処

```
【リスク1】ナレッジ抽出が難しい
  → 対処：より簡易な抽出方法（テンプレート化）を検討

【リスク2】パーソナライズの効果が不明確
  → 対処：2レベル（新人/ベテラン）に絞ってシンプル化

【リスク3】デモで価値が伝わらない
  → 対処：動画ではなくライブデモ形式に変更
```
