# 提案仮説構築AI PoC 検証設計サマリー（社内共有版）

> **プロジェクト**: rikyu様 提案仮説構築AI PoC
> **期間**: 9週間（2026/1/27〜3/27）
> **体制**: 正社員50% + 業務委託60%（計22人日/月）
> **技術基盤**: MS Agent Framework（Azure AI Agent Service）
> **対象読者**: PM・社内メンバー

```
【基本的な問い】
「AIが営業担当者のニーズ推定・提案作成をどこまで支援できるか？」
```

| 4 | **継続的改善の仕組み** | 使い続けることでどう改善していくか？フィードバックをどう蓄積・活用するか？ |

## 3. 検証進め方

### 全体の流れ（アジャイル型）

```
┌─────────────────────────────────────────────────────────────────────┐
│  Step 1: 初回シナリオで動作確認（1件）                              │
│  ────────────────────────────────────────                          │
│  ・まず1件のシナリオでAIの出力を作成                               │
│  ・お客様に見せてフィードバックを収集                              │
│  ・「何が足りないか」「どこがズレているか」を把握                  │
│                              ↓                                      │
│  Step 2: 改善サイクル（毎週）                                       │
│  ────────────────────────────────────────                          │
│  ・フィードバックをもとにプロンプト・ナレッジを改善                │
│  ・同じシナリオで再実行 → 改善効果を確認                           │
│  ・課題が解消されたら次のシナリオを追加                            │
│                              ↓                                      │
│  Step 3: シナリオの段階的拡張                                       │
│  ────────────────────────────────────────                          │
│  ・課題・改善の方向性が見えてきたらシナリオを増やす                │
│  ・どの軸（商材/業界/難易度）を増やすかは課題に応じて判断          │
│  ・最終的に必要な件数まで拡張（目安: 最大50件）                    │
│                              ↓                                      │
│  Step 4: LLM自動評価AIの育成（並行）                                │
│  ────────────────────────────────────────                          │
│  ・人の評価結果をもとにルーブリック（評価基準）を調整               │
│  ・概ね人と一致するようになったら残りのシナリオに適用              │
│  ・W6までに確定、それまでに改善しなければ人評価中心に切り替え      │
└─────────────────────────────────────────────────────────────────────┘
```

### シナリオの増やし方（アジャイル型）

| フェーズ | 方法 | 件数 | ポイント |
|---------|------|------|----------|
| **初回** | お客様から実データ1件 | 1件 | まず動くものを見せる |
| **段階的拡張** | 課題に応じて追加 | 必要に応じて | 「この軸が弱い」→ その軸のシナリオを追加 |
| **最終目標** | 多様性4軸で網羅 | 最大50件 | 必要十分な件数で止める |

### LLM自動評価の仕組み

```
【なぜLLM自動評価を使うか】

1. 正解が一意に定義できない
   ・同じ顧客に対して複数の妥当なニーズ・提案がありえる
   ・「唯一の正解」ではなく「妥当かどうか」を評価する必要がある
   ・LLMは文脈を理解して「妥当性」を判断できる

2. 評価工数の削減
   ・人が100件全部を評価するのは現実的でない
   ・LLMに評価させることで、高速・低コストで精度測定可能

3. エキスパートの感覚を再現
   ・お客様の知見をルーブリック（評価基準）に反映
   ・営業知見者と同じ感覚でAI出力を評価できるようにする
```

```
【LLM自動評価の育成プロセス】

1. ルーブリック（評価基準）の初版を作成
   ・「良い提案とは何か」「ニーズ抽出のポイント」を言語化

2. 人が10件を評価（教師データ作成）
   ・営業知見者がルーブリックに基づいて評価
   ・「なぜこの評価にしたか」の理由も記録

3. LLMに同じ10件を評価させる
   ・ルーブリックをプロンプトに組み込んで評価実行

4. 人とLLMの評価を比較 → ズレを分析
   ・一致率を算出
   ・「人は◯だがLLMは△」のケースを詳細分析

5. お客様の知見をルーブリックに反映
   ・「この観点が足りない」「この表現の方がしっくりくる」
   ・営業知見者と一緒にルーブリックをブラッシュアップ

6. 概ね人と一致するようになったら残りのシナリオに適用
```

### LLM評価における原因分析・改善策探索

```
【評価LLMの出力内容】

評価LLMは単なるスコア付けだけでなく、以下の分析も行う:

■ 成功要因分析（うまくいっている場合）
  ・なぜこの出力が妥当と判断できるか
  ・どの入力情報が有効に活用されたか
  ・どのナレッジ・ルールが効いているか

■ 失敗原因分析（うまくいっていない場合）
  ・表層的な原因: データ不足、プロンプト不備 等
  ・本質的な原因:
    - 業界特有の文脈理解不足
    - 暗黙知（ベテランの勘所）の言語化不足
    - 顧客発言の真意把握が困難なケース
    - 複数商材の組み合わせ判断ロジックの欠如
    - 時間軸（短期/中期/長期）の考慮不足

■ 改善策の探索・提案
  ・即効性のある改善: プロンプト修正、ナレッジ追加
  ・構造的な改善:
    - 「〇〇の判断には△△の情報が必要」
    - 「□□のケースは追加シナリオで学習すべき」
    - 「この業界では××の観点が重要（ナレッジ拡充）」
  ・将来的な課題:
    - 「このケースはAI単独では困難、人との協働が必要」
    - 「リアルタイムの外部情報（市況等）が必要」
```

```
【原因分析の出力例】

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
シナリオ: 製造業の設備投資（L2・失敗ケース）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 評価結果: NG（スコア 2/5）

■ 表層的原因
  ・推薦商材が顧客の財務状況と不整合
  ・「借入を増やしたくない」発言を考慮できていない

■ 本質的原因
  ・顧客発言の「本音」と「建前」の区別ができていない
  ・財務バランスを考慮した複合判断ロジックが不足
  ・業界特有の設備投資サイクル（製造業は5-7年）の知識不足

■ 改善策
  【即効】プロンプトに「顧客発言の裏にある懸念を推測せよ」を追加
  【構造】財務指標と商材の相性ルールをナレッジに追加
  【将来】業界別の設備投資サイクル情報をRAGで参照可能に
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 4. 検証データ

### シナリオ選定の観点（多様性4軸）

| 軸 | 内容 | 配分 |
|----|------|------|
| **商材カテゴリ** | 融資/M&A・事業承継/為替・金利/ビジネスマッチング/その他 | 各10件 |
| **顧客属性** | 製造業/卸売・小売/サービス/建設・不動産/その他 | バランス良く |
| **ニーズ明確さ** | 明示的ニーズ / 潜在ニーズ | 25件 / 25件 |
| **難易度** | L1新人向け / L2標準 / L3応用 / L4高度 | 10/20/15/5件 |

### 難易度レベルの定義

| レベル | 対象 | 特徴 | 件数 |
|--------|------|------|------|
| **L1** | 新人向け | 明示的ニーズ、1対1の商材マッチ、情報完全 | 10件 |
| **L2** | 標準 | 複数解釈可能、最適解選定が必要 | 20件 |
| **L3** | 応用 | 潜在ニーズあり、複数商材の組み合わせ | 15件 |
| **L4** | 高度 | 発言と真のニーズが異なる、複合提案必要 | 5件 |

**AIレベル判定基準**:
- **新人レベル**: L1◯率≥80%、L2<60%
- **中堅レベル**: L1≥90%、L2≥70%、L3≥50% ← **目標**
- **ベテランレベル**: L1≥95%、L2≥85%、L3≥70%、L4≥50%

## 5. お客様に用意いただくシナリオの要件

### 初回シナリオ（1件）

まず **1件** のシナリオで初回出力を作成し、フィードバックをもらう。

| 推奨 | 理由 |
|------|------|
| **L2（標準）レベル** | 簡単すぎず難しすぎず、AIの基本性能が確認できる |
| **成約ケース** | 正解が明確で評価しやすい |

### 段階的拡張時の目安

課題に応じてシナリオを追加。最終的には以下のバランスを目指す:

| 難易度 | 目安件数 | 目的 |
|--------|----------|------|
| **L1（新人向け）** | 2件程度 | 基本動作の確認 |
| **L2（標準）** | 4件程度 | 標準ケースの精度確認 |
| **L3（応用）** | 3件程度 | 潜在ニーズ抽出力の確認 |
| **L4（高度）** | 1件程度 | 限界性能の確認 |

### 1件あたりの必要情報

```
【必須項目】

■ 顧客属性
・業種: （例: 製造業、卸売業、サービス業 等）
・規模: 売上規模、従業員数
・特徴: 経営者の年齢、創業/老舗、成長フェーズ 等

■ 面談時の情報（インプット相当）
・顧客が発言した内容（課題、要望、相談事項）
・営業が観察した情報（雰囲気、反応、懸念点）

■ 営業が認識したニーズ（正解データ相当）
・ニーズ1: カテゴリ、優先度、なぜそう判断したか
・ニーズ2: （複数ある場合）

■ 提案内容
・推薦した商材
・提案理由（なぜこの商材を選んだか）

■ 結果
・成約 or 失注
・理由: なぜ成約/失注したか
```

### シナリオ例①: L2（標準）・成約ケース

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
シナリオ例①: 製造業の設備投資資金ニーズ（L2・成約）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 顧客属性
・業種: 製造業（精密機器）
・規模: 売上300億円、従業員800名
・特徴: 社長65歳、創業家2代目、工場は築30年で老朽化

■ 面談時の情報
・社長発言: 「古い機械をそろそろ入れ替えたいと思っている」
           「借入は増やしたくないんだよね」
           「複数の銀行さんとお付き合いがある」
・営業観察: 中期経営計画に工場新設（2028年稼働予定）の記載あり
           自己資本比率35%、借入依存度やや高め

■ 営業が認識したニーズ
・ニーズ1: 設備投資資金（優先度: 高）
  → 工場新設計画、機械入替発言から
・ニーズ2: 財務バランス維持（優先度: 中）
  → 借入増やしたくない発言、借入依存度やや高めから

■ 提案内容
・推薦商材: シンジケートローン
・提案理由: 大型設備投資に適しており、複数行で分散することで
           借入集中を避けられる。既に複数行取引があるため組成しやすい

■ 結果
・成約
・理由: 複数行取引の実績を活かせる点、借入分散のメリットが評価された
```

### シナリオ例②: L3（応用）・失注ケース

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
シナリオ例②: 卸売業の事業承継ニーズ（L3・失注）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 顧客属性
・業種: 卸売業（食品）
・規模: 売上80億円、従業員150名
・特徴: 社長72歳、創業者、息子（専務45歳）が後継予定

■ 面談時の情報
・社長発言: 「最近、同業者がM&Aで大手に買われた話を聞いた」
           「うちは息子に継がせるつもりだけど」
           「でも業界全体が厳しくてね…」
・営業観察: 息子（専務）との会話で「父は引退する気がない」とこぼしていた
           直近3期は減収傾向、利益率も低下
           社長は事業承継の話題になると話を逸らす傾向

■ 営業が認識したニーズ
・ニーズ1: 事業承継支援（優先度: 高）【潜在ニーズ】
  → 社長年齢72歳、後継者は決まっているが承継プロセスが進んでいない
  → 社長は認めたくないが、専務は焦りを感じている
・ニーズ2: 事業基盤強化（優先度: 中）
  → 業界縮小傾向、減収続きで競争力強化が必要
・ニーズ3: M&A情報提供（優先度: 低）
  → 同業M&Aへの関心あり（売り手としてではなく情報として）

■ 提案内容
・推薦商材: 事業承継コンサルティング + 経営改善支援
・提案理由: 事業承継と経営改善を一体で進めることで、
           息子への承継を円滑に進めながら事業基盤を強化

■ 結果
・失注
・理由: 社長が事業承継の話を「まだ早い」と拒否。
       息子（専務）は前向きだったが、社長の意向で見送り。
       → 潜在ニーズの顕在化タイミングが早すぎた
```

## 6. 評価方法

### 評価の3層構造

| Layer | 内容 | タイミング | 工数 |
|-------|------|-----------|------|
| **L1** | LLM自動品質チェック | 毎回・全件 | ゼロ（自動） |
| **L2** | LLM比較評価（A/B比較） | 改善サイクル毎 | 極小（自動） |
| **L3** | 営業知見者の実用性評価 | 週次 | 週1-2時間 |

### Layer 1: LLM自動品質チェック

| チェック項目 | 内容 | NG例 |
|-------------|------|------|
| ハルシネーション | 存在しない商材名を出力していないか | 「rikyuローンプレミアム」（存在しない） |
| 論理整合性 | 根拠と結論が矛盾していないか | 根拠「売上好調」→ニーズ「資金繰り改善」 |
| 業界整合性 | その業界に妥当な提案か | IT企業に「農業機械リース」を推薦 |
| 情報カバレッジ | 入力の重要情報を拾えているか | 「社長70歳」なのに事業承継に言及なし |

### Layer 2: LLM比較評価

```
【目的】改善の方向性が正しいかを素早く確認
```

## 8. 開発スケジュール

### 週別マイルストーン（1/27開始、9週間）

| 週 | 日程 | 達成状態 |
|----|------|---------|
| W1 | 1/27-1/31 | 環境構築完了（MS Agent Framework） |
| W2 | 2/2-2/6 | 初回シナリオ1件作成（内部定義）、ベースライン動作 |
| W3 | 2/10-2/14 | **初回output共有（1件）** → FB収集 → 改善サイクル① |
| W4 | 2/17-2/21 | 改善サイクル②、シナリオ追加検討、**LLM自動評価AI初回結果共有** |
| W5 | 2/24-2/28 | 改善サイクル③、課題に応じてシナリオ拡張、LLM自動評価AI改善 |
| W6 | 3/3-3/7 | 改善サイクル④、LLM自動評価AI確定 |
| W7 | 3/10-3/14 | 改善サイクル⑤、プロトタイプ完成 |
| W8 | 3/17-3/21 | **最終評価** |
| W9 | 3/24-3/27 | バッファ、最終報告 |

**進め方のポイント（アジャイル型）**:
- **まず1件で動かす**: W3で初回output（1件）を見せ、フィードバックをもらう
- **課題ベースで拡張**: 「この軸が弱い」と分かったらその軸のシナリオを追加
- **シナリオ件数は固定しない**: 必要十分な件数で止める（最大50件目安）
- W6までLLM自動評価AIを改善し確定。改善しなければ人評価中心に切り替え

## 9. リスクと対応策

### 主要リスク

| リスク | 影響度 | 発生確率 | 対応策 |
|--------|--------|----------|--------|
| **データが少ない/提供されない** | 高 | 中 | 公開情報から仮データ作成、LLMで合成データ生成。実データ到着後に差し替え |
| **LLM自動評価が機能しない**<br>（人との評価が一致しない） | 高 | 中 | ルーブリックを繰り返し改善。W5までに改善しなければ、シナリオ件数を絞って人評価中心に切り替え |
| **精度が目標未達** | 高 | 中 | W3で早期に課題把握、スコープ縮小（業界・商材を絞る）を検討 |
| **営業知見者の協力が得られない** | 高 | 中 | 早期アサイン確定、評価負荷を限定 |

### コンティンジェンシープラン

| トリガー | 判断時期 | 対応策 |
|---------|---------|--------|
| 初回シナリオ作成が遅延 | W2 | 公開情報から仮シナリオを作成、後で調整 |
| LLM自動評価が人と一致しない | W6 | シナリオ拡張を中止、人評価中心に切り替え |
| W3時点でoutputの質が低い | W3 | スコープ縮小（1カテゴリに絞る）、改善サイクルを増やす |
