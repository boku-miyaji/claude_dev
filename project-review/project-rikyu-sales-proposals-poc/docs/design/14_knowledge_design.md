## 1. ナレッジとは何か

### 1.1 データとナレッジの違い

```
┌─────────────────────────────────────────────────────────────────────┐
│                  データ vs ナレッジ                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【データ】構造化された事実情報                                       │
│  ├─ 企業情報（業種、売上、従業員数）                                │
│  ├─ 商品カタログ（商品名、スペック、価格）                          │
│  └─ 取引履歴（日付、金額、商品）                                    │
│  → そのまま格納・検索できる                                         │
│                                                                     │
│  【ナレッジ】経験に基づく判断基準・暗黙知                             │
│  ├─ 「この業界の社長はこういう言い方をする」                        │
│  ├─ 「借入を増やしたくないと言っても本音は違う」                    │
│  ├─ 「製造業の設備投資サイクルは5-7年」                             │
│  └─ 「経理部長は常務への影響力がある場合が多い」                    │
│  → ベテランの頭の中にあり、言語化されていない                       │
│                                                                     │
│  【本PoCの目的】                                                     │
│  ナレッジを「どう集めるか」「どう言語化するか」「どう蓄積するか」    │
│  を明らかにする                                                      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 ナレッジの3層構造

```
┌─────────────────────────────────────────────────────────────────────┐
│                  ナレッジの3層構造                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Layer 3: 暗黙知（Tacit Knowledge）                                  │
│  ───────────────────────────────────────                            │
│  ベテランの頭の中にある、言語化されていない知識                      │
│  ├─ 判断基準（「なんとなくこの会社は危ない」）                      │
│  ├─ 勘所（「この発言の裏にはこういう意図がある」）                  │
│  └─ パターン認識（「このタイプの経営者にはこう話す」）              │
│  → 言語化・形式化が最も困難、PoCで抽出を試みる                      │
│                                                                     │
│  Layer 2: 形式知（Explicit Knowledge）                               │
│  ───────────────────────────────────────                            │
│  言語化されているが、散在している知識                                │
│  ├─ 成功/失敗事例とその要因                                         │
│  ├─ 業界別の課題パターン                                            │
│  └─ 提案時の注意点・ベストプラクティス                              │
│  → 収集・整理してシステムに投入可能                                 │
│                                                                     │
│  Layer 1: データ（Data）                                             │
│  ───────────────────────────────────────                            │
│  構造化された事実情報                                                │
│  ├─ 企業情報、取引履歴、商品マスタ                                  │
│  └─ 面談記録、アカウントプラン                                      │
│  → システムに格納済み or 格納可能                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 2. ナレッジの分類

### 2.1 ナレッジカテゴリ一覧

| カテゴリ | 内容 | 具体例 |
|---------|------|--------|
| **顧客ニーズナレッジ** | 顧客の潜在ニーズを見抜くための知識 | 「社長が65歳以上なら事業承継を必ず考慮」 |
| **商品ナレッジ** | 商品の特性と適用シーンの知識 | 「シンジケートローンは複数行取引がある企業向け」 |
| **ソリューションナレッジ** | ニーズと商品のマッチング知識 | 「海外展開ニーズには為替予約とセットで提案」 |
| **業界ナレッジ** | 業界固有の事情・慣行の知識 | 「製造業の設備投資サイクルは5-7年」 |
| **人間関係ナレッジ** | 組織内の力学・関係性の知識 | 「経理部長は常務への影響力がある」 |
| **コミュニケーションナレッジ** | 顧客との対話の知識 | 「本音と建前の見分け方」 |

### 2.2 各カテゴリの詳細

#### 顧客ニーズナレッジ

```yaml
# 顧客ニーズを見抜くための暗黙知

判断基準:
  社長の年齢:
    - "65歳以上 → 事業承継ニーズを必ず考慮"
    - "後継者の話題が出たら優先度を上げる"
    - "ただし本人は認めたがらない場合が多い"

  財務状況からの推測:
    - "借入依存度が高い → 財務改善ニーズ（ただし直接言及は避ける）"
    - "自己資本比率35%以下 → 資本増強の提案も視野に"
    - "前期減益 → コスト削減より売上向上の話から入る"

  発言の裏読み:
    - "「借入は増やしたくない」→ 本当に増やしたくないとは限らない"
    - "「まだ早い」→ 興味はあるがタイミングを見ている"
    - "「検討します」→ ほぼ断り文句、真の懸念を探るべき"

行間を読むパターン:
  - "設備の話題が出る → 資金ニーズだけでなく生産性・競争力の課題"
  - "海外の話題が出る → 為替リスク、現地法人設立、海外送金など複合ニーズ"
  - "人の話題が出る → 採用難、人件費、働き方改革など広がる可能性"
```

#### 業界ナレッジ

```yaml
# 業界固有の暗黙知

製造業:
  設備投資:
    - "設備投資サイクルは5-7年（更新タイミングの見極め）"
    - "工場新設は2-3年前から計画が始まる"
    - "「古い機械を入れ替えたい」は大型投資のサイン"
  業界特性:
    - "中堅製造業は複数行取引が基本"
    - "技術流出を恐れるためM&Aには慎重"
    - "職人気質の経営者が多い"

卸売・小売業:
  季節性:
    - "決算期前に在庫資金ニーズが高まる"
    - "新規出店は半年前から資金計画"
  業界特性:
    - "薄利多売のため金利感応度が高い"
    - "EC対応が遅れている企業は危機感あり"

建設・不動産業:
  サイクル:
    - "公共工事は年度末に集中"
    - "不動産は金利動向に敏感"
  業界特性:
    - "重層下請構造のため資金繰りが複雑"
    - "担保評価が重要"
```

#### コミュニケーションナレッジ

```yaml
# 顧客との対話における暗黙知

本音と建前の見分け方:
  建前パターン:
    - "「うちは大丈夫」→ 実は不安を抱えている"
    - "「今は必要ない」→ 提案の仕方が悪い or タイミングが早い"
    - "「他行と話している」→ 条件交渉のカード"

  本音を引き出す質問:
    - "「仮に○○だったらどうしますか」（仮定形で聞く）"
    - "「同業他社では△△が多いのですが」（一般論から入る）"
    - "「以前おっしゃっていた件は」（過去の発言を引用）"

経営者タイプ別アプローチ:
  創業者タイプ:
    - "自分で決めたい → 選択肢を提示して選んでもらう"
    - "過去の苦労話を聞くと心を開く"
  二代目タイプ:
    - "先代との違いを出したい → 新しい提案を好む"
    - "社内での立場を気にする"
  サラリーマン社長:
    - "リスク回避志向 → 実績・事例を重視"
    - "稟議が通りやすい説明を求める"
```

## 3. 情報の動的/静的分類

### 3.1 情報の更新頻度による分類

```
┌─────────────────────────────────────────────────────────────────────┐
│                  情報の動的/静的分類                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【静的情報】更新頻度: 低（年1回以下）                               │
│  ───────────────────────────────────────                            │
│  ├─ 商品カタログ（商品改定時のみ更新）                              │
│  ├─ 業界ナレッジ（業界構造は急には変わらない）                      │
│  ├─ 基本的な判断基準（ベテランの経験則）                            │
│  └─ 企業の基本情報（業種、設立年、本社所在地）                      │
│  → システムプロンプトに固定投入可能                                 │
│                                                                     │
│  【準動的情報】更新頻度: 中（月〜四半期）                            │
│  ───────────────────────────────────────                            │
│  ├─ 過去事例（案件クローズ時に追加）                                │
│  ├─ 企業の財務情報（決算期に更新）                                  │
│  ├─ 経営アジェンダシート（面談後に更新）                            │
│  └─ キーパーソンマップ（人事異動・関係変化時）                      │
│  → 定期的にナレッジDBを更新、都度最新を参照                         │
│                                                                     │
│  【動的情報】更新頻度: 高（日〜週）                                  │
│  ───────────────────────────────────────                            │
│  ├─ 面談記録（毎回の面談後）                                        │
│  ├─ 顧客の発言・反応（リアルタイム）                                │
│  ├─ ニュース・市況（日々変化）                                      │
│  └─ 競合の動き（随時）                                              │
│  → 入力として都度投入、蓄積して傾向分析                             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 動的情報の更新トリガーと蓄積先

| 情報タイプ | 更新トリガー | 蓄積先 | 活用方法 |
|-----------|-------------|--------|----------|
| **面談記録** | 面談実施後 | 面談履歴DB | 次回AI入力、傾向分析 |
| **顧客の発言** | 面談中・後 | 面談記録に含む | ニーズ推定の根拠 |
| **経営アジェンダ** | 新情報入手時 | アカウントプラン | AI入力の更新 |
| **キーパーソン情報** | 人事異動・関係変化時 | アカウントプラン | 提案アプローチの更新 |
| **提案結果** | 案件クローズ時 | 事例DB | 成功/失敗要因の学習 |
| **修正フィードバック** | AI出力修正時 | FB履歴DB | プロンプト改善の材料 |

### 3.3 情報フローと更新サイクル

```
┌─────────────────────────────────────────────────────────────────────┐
│                  情報フローと更新サイクル                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【日次サイクル】                                                    │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ 面談実施 → 面談記録作成 → AIに入力 → 提案方針出力            │   │
│  │     ↓                                        ↓                │   │
│  │ 顧客の反応を記録                    営業が修正・採用判断      │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              ↓                                      │
│  【週次サイクル】                                                    │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ 週の面談記録を振り返り → アカウントプラン更新                 │   │
│  │     ↓                                                        │   │
│  │ 経営アジェンダシート更新（新情報追加、優先度変更）            │   │
│  │ キーパーソンマップ更新（関係性変化、新規人物追加）            │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              ↓                                      │
│  【月次/四半期サイクル】                                             │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ 案件クローズ時 → 事例DB追加（成約/失注とその要因）            │   │
│  │     ↓                                                        │   │
│  │ 修正FBの傾向分析 → プロンプト・ナレッジの改善                 │   │
│  │     ↓                                                        │   │

└─────────────────────────────────────────────────────────────────────┘
```

## 4. 暗黙知の抽出・言語化

### 4.1 抽出方法

```
┌─────────────────────────────────────────────────────────────────────┐
│                  暗黙知の抽出方法                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【方法1】AI出力への修正フィードバックから抽出                       │
│  ─────────────────────────────────────────────                      │
│  AI: 「資金調達ニーズを推定」                                        │
│  営業: 「いや、この会社は借入を増やしたくないと言っているから        │
│        リースを優先すべき」                                          │
│  → 暗黙知: 「借入を増やしたくない発言がある場合はリース優先」        │
│                                                                     │
│  【方法2】成功/失敗事例からのパターン抽出                            │
│  ─────────────────────────────────────────────                      │
│  成功事例A: 社長65歳、事業承継を提案 → 成約                          │
│  成功事例B: 社長68歳、事業承継を提案 → 成約                          │
│  失敗事例C: 社長60歳、事業承継を提案 → 失注（「まだ早い」）          │
│  → 暗黙知: 「社長65歳以上で事業承継を積極提案」                      │
│                                                                     │
│  【方法3】ベテランへのインタビュー                                   │
│  ─────────────────────────────────────────────                      │
│  Q: 「この企業を見て最初に何を考えますか？」                         │
│  A: 「まず社長の年齢を見る。65歳超えてたら事業承継は必ず頭に入れる」 │
│  → 暗黙知: 「社長年齢が最初のチェックポイント」                      │
│                                                                     │
│  【方法4】判断の分岐点を深掘り                                       │
│  ─────────────────────────────────────────────                      │
│  Q: 「なぜAではなくBを選んだのですか？」                             │
│  A: 「この業界は金利に敏感だから、金利より手数料の話から入った方が   │
│       いいんだよ」                                                   │
│  → 暗黙知: 「卸売業は金利感応度が高いため手数料訴求から入る」        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 言語化のフォーマット

```yaml
# 暗黙知の言語化テンプレート

暗黙知ID: TK-001
カテゴリ: 顧客ニーズナレッジ
タイトル: 社長年齢と事業承継ニーズの関係

条件（When）:
  - "社長の年齢が65歳以上"
  - "後継者が明確に決まっていない"

判断（Then）:
  - "事業承継ニーズを高優先度で考慮する"
  - "ただし直接的に切り出さず、将来の話として聞く"

理由（Why）:
  - "65歳を超えると体力面・健康面の不安が出始める"
  - "株価対策は時間がかかるため早めの検討が必要"
  - "本人は認めたがらないので、間接的なアプローチが有効"

注意点（Caution）:
  - "創業者は「まだまだやれる」と思っている場合が多い"
  - "二代目は逆に「早く次に渡したい」場合もある"

出典:
  - "ベテランAへのインタビュー（2026/01/15）"
  - "成功事例SC-012、SC-025"

検証状態: 仮説（PoCで検証予定）
```

### 4.3 暗黙知の検証プロセス

```
┌─────────────────────────────────────────────────────────────────────┐
│                  暗黙知の検証プロセス                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Step1】仮説として言語化                                           │
│  ├─ ベテランへのインタビューで抽出                                  │
│  ├─ 事例からのパターン認識                                          │
│  └─ 状態: 「仮説」                                                  │
│                     ↓                                               │
│  【Step2】AIに投入して効果測定                                       │
│  ├─ ナレッジありの出力 vs なしの出力を比較                          │
│  ├─ 営業知見者が「どちらが良いか」を評価                            │
│  └─ 精度向上が確認できたら採用                                      │
│                     ↓                                               │
│  【Step3】運用で継続検証                                             │
│  ├─ 実際の提案結果（成約/失注）との相関を分析                       │
│  ├─ 例外ケース・反例の蓄積                                          │
│  └─ 状態: 「検証済み」 or 「条件付き採用」 or 「棄却」              │
│                     ↓                                               │
│  【Step4】ナレッジの精緻化                                           │
│  ├─ 条件の細分化（「65歳以上」→「65-70歳」と「70歳以上」で分ける）  │
│  ├─ 例外条件の追加（「ただし、後継者が既に決まっている場合は除く」）│
│  └─ 定期的な見直し                                                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 5. ナレッジの蓄積・活用設計

### 5.1 ナレッジDB構造

```yaml
# ナレッジDBの概念構造

ナレッジDB:
  暗黙知ナレッジ:
    - 顧客ニーズ判断基準
    - 業界別パターン
    - コミュニケーションルール
    - 本音/建前の見分け方

  事例ナレッジ:
    - 成功事例（成約要因付き）
    - 失敗事例（失注要因付き）
    - 経営アジェンダ深掘り事例
    - キーパーソン関係構築事例

  フィードバックナレッジ:
    - AI出力への修正履歴
    - よくある修正パターン
    - ユーザーの追加指示傾向

  テンプレートナレッジ:
    - 業種別課題テンプレート
    - 提案ストーリーテンプレート
    - 質問テンプレート
```

### 5.2 ナレッジの投入方法

| 投入方法 | 対象ナレッジ | タイミング | 特徴 |
|----------|-------------|-----------|------|
| **システムプロンプト固定** | 基本的な判断基準、業界ナレッジ | 常時 | 安定、全ケースに適用 |
| **Few-shot例として投入** | 成功事例、判断の具体例 | 常時 | 出力形式・品質を誘導 |
| **入力コンテキストとして** | 類似事例、直近の面談記録 | ケース毎 | 個別最適化 |
| **RAG検索結果として** | 大量の事例DB | ケース毎 | 将来拡張（PoC後） |

### 5.3 継続的なナレッジ蓄積フロー

```
┌─────────────────────────────────────────────────────────────────────┐
│                  継続的なナレッジ蓄積フロー                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【入口1】日常の営業活動から                                         │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ 面談実施 → 面談記録 → 経営アジェンダ/キーパーソン更新        │   │
│  │                 ↓                                            │   │
│  │ 「今日の面談で分かった新しいこと」を記録                      │   │
│  │   - 「社長は実は海外展開に興味があった」                      │   │
│  │   - 「経理部長が常務に影響力があると判明」                    │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              ↓                                      │
│  【入口2】AI出力の修正から                                           │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ AI出力 → 営業が修正 → 修正内容を記録                         │   │
```

## 5.4 パーソナライズ設計 ★新規

> **関連**: [差別化戦略](17_differentiation_strategy.md)
> **背景**: NTTData PoC2の課題「使用者（経験者・未経験者）によって出力内容をアレンジできる仕組み」への対応

### パーソナライズの必要性

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                 なぜパーソナライズが必要か                                        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  【課題】同じ出力では全員に最適にならない                                        │
│  ├─ 新人: 判断根拠を知りたい、なぜこの提案なのかを学びたい                      │
│  ├─ 中堅: 要点だけ欲しい、壁打ち相手として使いたい                              │
│  └─ ベテラン: 見落としだけ指摘してほしい、自分の判断を補完したい                │
│                                                                                 │
│  【解決】ユーザーレベル別に出力を最適化                                          │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### パーソナライズ設定

```yaml

# 経験レベル別の出力調整
output_modes:

    name: "詳細説明モード"
    description: "判断根拠を丁寧に説明、教育的な記載"

      explanation_depth: "detailed"
      knowledge_display: "visible"      # 適用したナレッジを明示
      checklist_format: true            # チェックリスト形式

      explanation_depth: "summary"

      blind_spots: true                 # 見落としがちな観点を補足

  senior:

    settings:
      explanation_depth: "minimal"
```

## 6. 新人・中堅・ベテランの違いとナレッジ

### 6.1 レベル別のナレッジ保有状況

```
┌─────────────────────────────────────────────────────────────────────┐
│                  レベル別ナレッジ保有状況                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【新人】（経験1-2年）                                               │
│  ├─ 持っているもの: 商品知識、基本的な業界知識                      │
│  ├─ 足りないもの:                                                   │
│  │   - 顧客の発言の裏を読む力                                       │
│  │   - 業界特有の慣行・暗黙のルール                                 │
│  │   - 「この場合はこう」という判断基準                             │
│  │   - 成功/失敗パターンの蓄積                                      │
│  └─ AIで補完可能: 判断基準の言語化、事例の提示                      │
│                                                                     │
│  【中堅】（経験3-5年）                                               │
│  ├─ 持っているもの: 一通りの経験、主要なパターンの理解              │
│  ├─ 足りないもの:                                                   │
│  │   - 複雑なケースの判断                                           │
│  │   - 例外ケースへの対応                                           │
│  │   - 深い業界知識                                                 │
│  └─ AIで補完可能: 複合判断のサポート、レアケースの事例提示          │
│                                                                     │
│  【ベテラン】（経験10年以上）                                        │
│  ├─ 持っているもの: 豊富な暗黙知、直感的な判断力                    │
│  ├─ 課題:                                                           │
│  │   - 知識が属人化している                                         │
│  │   - 言語化・共有が難しい                                         │
│  │   - 退職時に知識が失われる                                       │
│  └─ AIの役割: ベテランの知識を抽出・言語化・継承                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 6.2 ナレッジが精度に与える影響

| 難易度 | ナレッジなし | 基本ナレッジあり | 豊富なナレッジあり |
|--------|-------------|-----------------|-------------------|
| **L1（基本）** | 新人並み | 中堅並み | ベテラン並み |
| **L2（標準）** | 新人未満 | 新人〜中堅 | 中堅〜ベテラン |
| **L3（応用）** | 対応困難 | 新人並み | 中堅並み |
| **L4（高度）** | 対応不可 | 対応困難 | 新人〜中堅 |

→ **ナレッジの質と量がAI精度を決定する**

## 7. PoCで検証すべきこと

### 7.1 検証項目

| 検証項目 | 内容 | 成功基準 |
|---------|------|---------|
| **暗黙知の抽出可能性** | ベテランの暗黙知を言語化できるか | 10個以上の暗黙知を言語化 |
| **暗黙知の効果** | 暗黙知を投入すると精度が上がるか | 精度+10%以上 |
| **動的情報の活用** | 面談記録等の更新がAI精度に反映されるか | 情報更新後に出力品質向上 |
| **蓄積サイクル** | 修正FB→ナレッジ蓄積→精度向上のサイクルが回るか | 週次で改善が確認できる |
| **言語化の限界** | 言語化できない暗黙知は何か | 限界の明確化 |

### 7.2 明らかにすべき課題

```
【課題1】暗黙知の抽出コスト
  - ベテランへのインタビューにどれくらい時間がかかるか
  - 言語化の品質をどう担保するか
  - 誰が抽出作業を担当するか

【課題2】ナレッジの鮮度管理
  - 古くなったナレッジをどう検知するか
  - 更新の優先順位をどう決めるか
  - 矛盾するナレッジが出た場合どうするか

【課題3】ナレッジの汎用性
  - ある営業のナレッジが他の営業にも有効か
  - 地域・支店による違いをどう扱うか
  - 個人の「癖」と普遍的な知識の切り分け

【課題4】言語化の限界
  - どうしても言語化できない「勘」はあるか
  - その場合、人間との役割分担をどうするか
```
