### 9.3 顧客コミュニケーション設計

#### 基本方針（アジャイル型）

```
【進め方】
・「精度○%」の報告ではなく、出力サンプルを見せてFBをもらう
・FB → 修正 → また見せる のサイクルを高速に回す
・精度は内部で測定し、最終報告時にまとめて報告
・W3から毎週、出力を見せてFBをもらう改善サイクルを実施

【共有方法】
・検証UIにエクスポート機能（HTML/Excel）を実装
・結果サンプルを非同期で共有（Slack等）、FBを随時受領
```

#### 正式MTG

| 時期 | 日程 | MTG | 内容 |
|------|------|-----|------|
| W1 | 1/27週 | キックオフ | 計画共有、データ確認、協力者アサイン依頼 |
| W3 | 2/10週 | 初回output共有 | 1件の出力を共有、FB収集、改善サイクル開始 |
| W4 | 2/16週 | LLM自動評価AI初回結果共有 | 自動評価の状況を共有、FB収集 |
| W9 | 3/23週 | 最終報告会 | 精度報告、検証総括、ロードマップ提示、MVP要件確認 |

#### 非同期コミュニケーション（随時）

| 方法 | 内容 | 頻度 |
|------|------|------|
| **結果サンプル共有** | 検証UIからエクスポート（HTML/Excel）をSlack等で共有 | 週1-2回 |
| **FB受領** | 出力への課題・修正要望をテキストで受領 | 随時 |
| **修正報告** | 「○○を修正しました」の報告 + 修正後サンプル | 修正の都度 |

#### 改善サイクルの進め方

```
【W3〜W7: 毎週の改善サイクル】

1. output共有（週初め）
   ・改善版の出力を共有
   ・前回との比較ポイントを説明

2. FB収集（週中）
   ・営業知見者からFBを収集
   ・「ここがダメ」「こうしてほしい」を把握

3. 改善実装（週後半）
   ・FBをもとにプロンプト・ナレッジを改善
   ・次週の出力を準備

※W6までにLLM自動評価AIを確定
※改善しなければシナリオ件数を絞って人評価中心に切り替え
```

### 9.5 進め方のポイント（アジャイル型）

- **まず1件で動かす**: W3で初回output（1件）を見せ、フィードバックをもらう
- **課題ベースで拡張**: 「この軸が弱い」と分かったらその軸のシナリオを追加
- **シナリオ件数は固定しない**: 必要十分な件数で止める（最大50件目安）
- **少人数で回す**: コミュニケーションコストを削減
- **マルチターン不要**: 1入力→1出力でシンプルに（UX的な面倒くささを避ける）
- **W6までLLM自動評価AIを確定**: 改善しなければ人評価中心に切り替え

## 11. リスクと対策

### 11.1 リスク一覧

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|----------|------|
| データ提供の遅延 | 高 | 中 | 早期にサンプルデータで開発開始、合成データで検証 |
| データ品質の問題 | 高 | 中 | データ品質チェックを1月中に実施、品質基準を定義 |
| 精度が目標未達 | 高 | 中 | Week 4中間確認で早期に課題把握、スコープ調整を検討 |
| 設計が遅れる | 高 | 中 | Week 1で業務委託に先行タスク、正社員は設計に集中 |
| 評価基準の合意遅延 | 中 | 低 | 1月2週までに仮決定、後から微調整可能な設計 |
| お客さん協力不足 | 中 | 中 | Week 4, 6, 8でMTG設定、関係構築 |
| 業務委託の立ち上がり遅延 | 中 | 中 | Week 1でオンボーディング実施、正社員がフォロー |
| 商材情報の不足 | 中 | 低 | 不足部分は営業担当者へのヒアリングで補完 |
| 営業知見者の評価協力が得られない | 高 | 低 | 早期に営業知見者アサイン確定、評価負荷軽減 |
| 商材・ニーズの分類体系がない | 中 | 中 | 既存資料からの整理、営業知見者へのヒアリング |
| 他PJ（NTTデータ/テイラーワークス）との重複・競合 | 中 | 中 | 早期に役割分担を明確化 |
| 経営アジェンダ・アカウントプラン取り組みとの整合性 | 中 | 低 | 関連取り組みとの位置関係を明確化し、将来的な合流を見据えた設計 |

### 11.2 コンティンジェンシープラン

| リスク | トリガー | 対応策 |
|--------|---------|--------|
| データが存在しない/品質が低い | 1/24までにデータ確保の目処が立たない | 機密データを含まない匿名化データでの検証に切り替え、スコープ縮小 |
| 営業知見者がアサインされない | 2/7までに営業知見者がアサインされない | 社内メンバーによる代替評価、評価項目の簡素化 |
| 精度が目標に達しない | 2/21時点でBase精度が著しく低い | スコープ縮小（業界・商材を更に絞る）、手法変更 |
