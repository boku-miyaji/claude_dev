## アイデア帳 (Ideas Backlog)

> まだ開発に着手していない、今後作りたいアプリのネタ

### IDEA-001: バーチャル共同作業スペース

| 項目 | 内容 |
|------|------|
| **仮称** | もくもくメイト / Mokumoku Mate |
| **着想日** | 2026-03-08 |
| **課題** | 一人で休日作業しているとき、モチベーションが続かない。孤独感で集中が切れる。頑張りを誰かに認めてほしい |
| **コンセプト** | 「カフェの隣の席」-- 顔は見えないが、誰かが隣で頑張っているのがわかる。自分の頑張りも認めてもらえる |
| **ターゲット** | 個人開発者、フリーランス、休日の自主学習者（一人だと頑張れない人） |
| **優先度** | 未定 |
| **ステータス** | アイデア → 検討中 |

#### 設計メモ

**作業証明（"いるフリ"排除）**
- スマホのインカメラで人体検知（MediaPipe等）。映像は送らず「人がいる/いない」の1bit情報のみ送信
- PC作業者はキーボード/マウス活動検知も併用（デスクトップアプリ: Tauri or Electron）
- 読書・手書き作業の人もスマホカメラで在席検知可能。追加デバイス不要

**プロフィール（選べる自己開示）**
- ニックネーム + 自作アバター/アイコン（顔写真不要・本名不要）
- 作業内容は任意記入（「読書中」「開発中」「英語の勉強」等）
- 完全匿名ではない。「自分の頑張りが自分のもの」として見える
- SNSリンク任意掲載（GitHub / Twitter / 個人サイト）→ 見たい人だけ見に行く。アプリは何もしない

**頑張りの可視化（自己顕示欲を満たす）**
- 作業時間の累積表示（今日/今週/今月）
- 連続作業日数ストリーク
- 実績バッジ（「早起き作業」「深夜の努力家」「100時間達成」等）
- 作業時間ランキング（週間/月間）
- SNSシェア機能（「今日5時間やりました」）

**コミュニケーション設計（集中を妨げない）**
- DM・チャット機能は**入れない**（会話が目的になり作業が止まる。チャットしたいならDiscord/LINEを使えばいい）
- ワンタップスタンプ（「頑張ってるね」「お疲れ」等のリアクション）
- 作業宣言つぶやき（開始時「今日は○○やる」/ 終了時「○時間やりきった！」）→ 作業中は投稿不可
- ポモドーロ休憩中のみ投稿可 → 集中中は書けないルール

**「部屋」によるゆるいつながり設計**
- 部屋は作業タグ別ではなく「場所」（カフェA、カフェB等）。人数上限ありで程よい密度を保つ
- 作業タグ（React / 英語 / デザイン等）は自分のステータス表示。同じ部屋に色々なタグの人が混在 → カフェっぽい
- 部屋に入ると「今日この部屋で作業している人たち」がタグ付きで見える
- 部屋単位の質問ボード: 自分の質問はいつでも投稿可、他人の質問・回答は休憩中のみ閲覧可。タグでフィルタ可能
- 何度も同じ部屋にいる人は「よく見かける人」として表示（常連の可視化）
- 1対1の関係ではなく「同じ場所に通ううちに顔見知りになる」カフェの常連モデル
- アプリ内で無理につなげない。場を提供するだけ。プロフィールのSNSリンク経由で勝手につながる

**あえて入れないもの**
- 顔出しビデオ → 身だしなみが気になり集中できない
- フリーチャット・DM → コミュニケーションアプリではない。出会い目的排除
- フリータイムライン → Twitterになってしまう
- プロフィールに年齢・性別 → 属性ではなく「何を頑張っているか」でつながる

**初期戦略（コールドスタート対策）**
- 開放時間帯を限定: 朝 6:00-8:00 / 夜 21:00-24:00（密度確保のため。24時間にしない）
- Twitter/X で開発過程を公開しながらテスターを募集（目標10人程度）
- 自分自身が毎日使う最初のヘビーユーザー
- ダミー/Botは入れない（「いるフリ排除」のコンセプトに矛盾する）

**フェーズ設計**
- Phase 1（MVP）: 作業する → 誰かがいる → 頑張れる → 記録が残る
  - 検証仮説: 「人がいると頑張れるか」を10人規模で確認
  - 収益化はしない。ただしアカウント・作業ログのデータ構造は課金拡張を見越して設計
- Phase 2: 部屋で質問を投げる → 誰かが休憩中に答える
- Phase 3: 同じ部屋の常連同士が顔見知りになる → コラボや仕事が自然発生

**収益化仮説（Phase 1 では実装しない。ユーザー行動を見てから決定）**
- 仮説A: フリーミアム（無料: 1部屋+基本タイマー / 有料: 詳細統計・バッジカスタム・月間レポート）
- 仮説B: 応援経済（投げ銭、手数料10-30%）
- 仮説C: スポンサー部屋（企業がブランドルームを月額運営、作業者は無料）
- 仮説D: 買い切り ¥500-1000（Forest方式、最もシンプル）
- Phase 1 の段階で決めておくこと: アカウント設計、作業ログ蓄積、無料範囲の仮説
- Phase 1 の段階で決めないこと: 具体的な金額、課金ポイント、決済実装

**技術候補**
- Tauri / Electron（デスクトップ、OS入力検知）+ React Native（スマホ、カメラ検知）
- Supabase Realtime（プレゼンス共有）
- MediaPipe（デバイス上で人体検知→映像はサーバーに送らない）

## 新しいアイデアの追加方法

新しいアイデアを追加する際は、以下のテンプレートを使ってください：

```markdown
### IDEA-XXX: タイトル

| 項目 | 内容 |
|------|------|
| **仮称** | アプリ名 |
| **着想日** | YYYY-MM-DD |
| **課題** | どんな問題を解決するか |
| **コンセプト** | どうやって解決するか |
| **想定機能** | 主な機能リスト |
| **ターゲット** | 誰向けか |
| **差別化** | 既存サービスとの違い |
| **技術候補** | 使いそうな技術 |
| **優先度** | 高/中/低/未定 |
| **ステータス** | アイデア → 検討中 → 設計中 → 開発中 |
```
