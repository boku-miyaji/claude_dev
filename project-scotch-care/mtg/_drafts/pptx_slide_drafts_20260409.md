## P-3: 別紙A-5（Slide 24）差し替え

**スライドタイトル:** 本プロジェクトにおける各プラットフォームの評価

**サブタイトル:** §2.1 評価観点に基づく技術適合度

**3列評価（レイアウトは現行踏襲）:**

### Databricks — ◎ 適合度が高い
- PySpark 219本の互換性が最大の強み（要件2）
- Unity Catalog + MLflowのAI統合が充実（要件3, 4）
- 学習曲線がSnowflakeより急（要件3 運用面）

### Snowflake — ○ 運用自走性で優位
- SQL完結で2名体制の自走に最も適する（要件3）
- PySpark書き直しの工数が主要リスク（要件2）
- AI/ML統合は限定的（要件4）

### AWS Native — △ 運用負荷が高い
- 複数サービスの習得・運用が必要（要件3）
- Glue PySparkは一部互換だが制約あり（要件2）
- 統合UXがなく運用設計の工夫が必要

**結論（差し替え）:**

> ~~現時点の技術評価ではDatabricksが最有力候補~~

> §2.1の技術評価観点に照らすと、Databricks と Snowflake は拮抗する。
> 運用自走性ではSnowflakeが優位である一方、既存PySpark資産の移行工数が主要制約である場合、Databricksが優位となる。
> したがって本案件では、**代表的なPySparkジョブでの移行検証を前提条件として、Databricksを第一候補、Snowflakeを対抗案とする**整理が妥当である。

## P-4: 別紙A-9（Slide 28）差し替え

**タイトル変更:**

~~セマンティックレイヤー: Ontology代替の要否~~
→ **セマンティックレイヤー: 導入方針と段階的拡張**

**本文は既存を維持。末尾に以下を追記:**

> **Foundry Ontologyとの概念的な違い**
>
> Foundry Ontologyは objects / links / actions を含む業務オペレーションモデルであり、セマンティックレイヤー（metrics / dimensions / relationships のKPI定義層）とは別概念である。
>
> 将来拡張性は以下の3層で整理する:
> (1) セマンティックレイヤー — KPI定義。本RFPの基本スコープの拡張候補
> (2) カタログ/ガバナンス — メタデータ・リネージ。基本スコープ内
> (3) オントロジー/アクション連携 — 業務オペレーションモデル。将来オプション
>
> 本RFPでは(1)(2)を基本スコープに近い拡張候補、(3)を将来オプションとして提案依頼する。
