## 1. 発表前に押さえる「データ基盤 最小チートシート」

これだけ押さえれば、浅い質問にはすべて自然に答えられる。深い質問は「RFPで確認する論点です」で受け流せる。

### 1-1. 登場する3つのプラットフォーム（と一言印象）

| 基盤 | 何者か | 立ち位置 | 今回の第一候補理由 |
|------|--------|---------|------------------|
| **Databricks** | Apache Spark（並列処理エンジン）を作った会社が運営する Lakehouse SaaS | データ + ML + AI を1つの箱で扱う。Spark/PySpark との親和性が世界最強 | PySpark 219本が **ほぼそのまま動く**（=移行確実性）。Unity Catalog でガバナンス統合 |
| **Snowflake** | SQL に特化した DWH SaaS。Amazon Redshift の次世代版として登場 | SQL中心で学習コスト低。運用自走性が最も高い | 少人数運用が現実的。Cortex Analyst（自然言語分析）も強い |
| **AWS Native** | Glue / Redshift / Athena / S3 等を自分で組み合わせる | インフラ単価は最安だが、組み立てと運用が重い | コスト単体では有利だが、6サービス横断の運用負荷で TCO は逆転しがち |

**一言で言うなら**: 「PySpark 資産を守れるのが Databricks、SQL 一本で回せるのが Snowflake、レゴブロックで自作するのが AWS Native」

### 1-2. 必ず出てくる専門用語（その場で一言で意訳できるようにする）

| 用語 | 一言意訳 | もう一歩踏み込む場合 |
|------|---------|------------------|
| **Lakehouse（レイクハウス）** | DWHとデータレイクを1つにまとめた構成 | 従来は DWH（構造化データ用）+ データレイク（生データ用）の2重構造だったが、両方を1つでやれるようにしたもの。Databricks が提唱 |
| **Delta Lake / Iceberg** | S3 上のファイルを「テーブル」として扱うためのメタデータ仕様 | Delta は Databricks 由来、Iceberg は Netflix 由来。2024年以降、UniForm で相互運用可能になった |
| **UniForm** | Delta Lake で書いたデータを、Iceberg 形式としても読めるようにする仕組み | 2024/6 GA。これでフォーマット戦争は実質終結。ロックインの懸念が大きく軽減 |
| **Unity Catalog** | Databricks の統合ガバナンス機能。データ + ML + AI資産を一元管理 | カタログ・リネージ・権限・監査ログを1つで扱う。Governed Tags（2026 GA）で タグ統制、Lakehouse Federation で外部DBのメタデータも横断検索 |
| **Bronze / Silver / Gold** | 生データ→加工済み→ビジネス用 の3層設計 | Bronze=事実の記録（不変）、Silver=クレンジング済、Gold=KPI適用済。BIはGoldに、分析者はSilverに繋ぐ |
| **PySpark** | Python で書く Spark 処理。Foundry の現行パイプライン219本が全てこれ | DataFrame API で書かれていれば Databricks でほぼそのまま動く。Foundry 固有デコレーター（`@transform_df`等）を除去するだけ |
| **dbt** | SQLでデータ変換を定義するツール | Python 書けない人でもデータ変換が書ける。業界標準。dbt Core は OSS |
| **Lakeflow / Airflow** | パイプラインの実行スケジューラ | Lakeflow は Databricks 純正、Airflow は OSS の業界標準。Databricks 中心なら Lakeflow 単体で十分 |
| **DBU** | Databricks Unit。Databricks の従量課金の単位 | 秒単位でコンピュート使用量が課金される。ワークロード種別（Jobs / SQL Warehouse / All-Purpose 等）で単価が違う |
| **Time Travel** | 過去の任意時点のデータに巻き戻せる機能 | Delta Lake / Snowflake 両方にある。事故時の切り戻しや監査対応に使う |
| **RLS（Row Level Security）** | 行レベルのアクセス制御 | 「施設長Aは施設Aの行だけ見える」をポリシーで実現。全3基盤が標準機能で持つ |
| **Markings（Foundry 固有）** | タグを付けると下流に自動伝播するアクセス制御 | Foundry の価値ある機能だが、Databricks Unity Catalog では **自動伝播がない**。CI/CDで補完設計が必要 ← **ギャップとして正直に認める** |
| **セマンティックレイヤー** | KPI の定義を一箇所で管理し、BI・AI・レポートで同じ数値を返す層 | 「入居率 = 入居者数 / 定員」を唯一の場所で定義 |
| **オントロジー** | エンティティ（施設・入居者・ケア記録）と関係性とアクションを統合定義するモデル層 | Foundry の差別化機能。Databricks/Snowflake には直接対応なし。Action Types（「入居者を別施設に移動」のようなビジネス操作の定義）が本質 |
| **MLflow** | ML 実験・モデル管理のデファクト標準 OSS。Databricks が作った | モデルのバージョン管理、実験記録、デプロイまで一気通貫 |
| **AI/BI Genie** | Databricks の自然言語分析機能 | 「先月の未タップ率が高い施設は？」と日本語で聞ける。セマンティックレイヤー + LLM |
| **Cortex Analyst** | Snowflake の自然言語分析機能 | Semantic Views（スキーマオブジェクト）にKPI定義を書き、Snowflake 内の LLM で SQL を生成。データが Snowflake から出ない |
| **Vector Search** | テキストや画像を「意味の近さ」で検索できる仕組み | RAG（検索+生成）の基盤になる。Databricks は Unity Catalog 統合、Snowflake は Cortex Search |

### 1-3. よく混同される3層を整理（柴田・竹森・山本氏が気にしていた論点）

```
┌───────────────────────────────────────────┐
│ (3) オントロジー / アクション層            │
│  エンティティ + 関係性 + ビジネス操作の定義    │
│  例: 「入居者A を 施設B に移動」            │
│  ★Foundry の差別化領域。DB/SF には直接対応なし │
├───────────────────────────────────────────┤
│ (2) セマンティックレイヤー                  │
│  KPI定義の一元管理                          │
│  例: 入居率 = 入居者数 / 定員               │
│  ★2025-2026 に急速に製品化。dbt/AI-BI Genie │
├───────────────────────────────────────────┤
│ (1) カタログ / ガバナンス                   │
│  何がどこにあるか、誰が所有者か、権限・リネージ │
│  ★全ての基盤に必要。これなしに(2)(3)は無意味  │
└───────────────────────────────────────────┘
```

**SOMPOケアへの示唆（覚えておく一言）**:
今回の移行では(1)を必須、(2)を RFP で要件化、(3)は将来論点として次フェーズ。Foundry の Ontology は Object + Link + Action Types が統合されていたが、**Action Types まで必要になるのは当面先**（現状の活用度が限定的のため）。まず(1)(2)で基盤を整えるのが合理的。

### p3 — §1.2 AI時代のデータ基盤選定（変わるもの / 変わりにくいもの）

- **要旨**: AI 時代の基盤選定は「特定の LLM やツール」ではなく「データを適切な権限で蓄積・加工・探索できる基盤」を選ぶこと。
- **言うこと**:
  - 「今回の RFP の主目的は Foundry からの確実な移行です。ただ、移行先の基盤は今後のデータ活用・AI 活用の自由度を決めるので、単なる移行先比較ではなく、**将来の拡張余地をどこまで確保できるか** を見極める判断でもあります」
  - 変わるもの/変わりにくいもの の2列を視覚的に対比させる
  - 「**変わるもの**: LLM、AIエージェント、AI関連ツール、実装パターン」
  - 「**変わりにくいもの**: データの蓄積・加工・探索、権限制御、ガバナンス」
  - 「**技術では解決しないもの**: 運用体制の設計、ナレッジの属人化解消、利用者の教育・定着」
  - 締め: 「LLM はどんどん変わりますが、だからこそ **データを蓄積・管理する部分を標準仕様で作ること** が重要です。特定の AI やベンダーにロックインしない設計です」
- **知っておくこと**:
  - この構成は 04/01 の石郷岡氏「Foundry ライクに見える」という指摘への回答として作った。
  - 「変わらない領域を標準で作る」というメッセージは、2026年の業界トレンド（モデル層の急変化 vs データ層の安定化）と整合している。
- **根拠**:
  - 5つの観点: ACES整理
  - SOMPOケアの示唆（多数施設のデータ保有）: **DDA活用推進全体像 + DDA戦略PDF**（受領資料/03.業務のあり方/20260213）
- **想定質問**:

### p23 — §4.2 データの段階的な階層設計（Bronze / Silver / Gold）

### p26 — 別紙 A-1 アーキテクチャ：DWH+DL vs Lakehouse

- **知っておくこと**:

  - **Databricks が Lakehouse の提唱元**。「Lakehouse」という言葉自体が Databricks のマーケティング用語として始まったが、業界全体で標準化された。

### p27 — 別紙 A-2 テーブルフォーマットとは

### p28 — 別紙 A-3 UniForm によるフォーマットの収束

- **要旨**: Delta Lake で書き込み → 自動で Iceberg メタデータ生成 → 外部エンジンから Iceberg として読める。**フォーマット戦争は終結**。

  - 現在: UniForm（2024/6 GA）で Delta を書いても Iceberg として読める。**データ複製なし**（メタデータのみ変換）。
  - 結論: Databricks 利用時は **Delta Lake + UniForm（Iceberg互換）を推奨**。マルチプラットフォーム時は Iceberg 標準化。
- **想定質問**:

  - セマンティックレイヤーとは: ビジネス用語の定義・計算ロジックを **一箇所で管理する層**（例: 入居率 = 入居者数 / 定員）
  - レイヤーがないと: Tableau、AI チャット、Excel でそれぞれ定義が異なり、数値が合わず信頼を失う
  - レイヤーがあると: Single Source of Truth、全ツールで同じ数値
  - 主な選択肢: **Databricks Metric Views**（Unity Catalog 統合、2025後半 GA）、**dbt MetricFlow**（OSS）、**Cube.dev**（OLAP高速化）
  - 結論: Databricks 利用時は **Metric Views が最もシンプル**
- **想定質問**:
  - Q: 「セマンティックレイヤーは今すぐ必要か？」→ A: 「**初期構築では不要** です。ただし **AI チャット（自然言語分析）をスコープに入れる場合は初期から設計が必要**。今回は AI 活用は次フェーズなので、Phase 2 での検討事項として残します」

### p33 — 別紙 A-8 / B-1 データガバナンス：Foundry 機能の代替マッピング

  - 対応状況:
    - **◎ そのまま代替可能**: RLS、監査ログ
    - **○ 運用設計で対応**: Markings、リネージ
    - **△ 別途ツール導入**: PII 検出（Presidio or AWS Macie）
- **想定質問**: → p13 と同じ（Markings 自動伝播ギャップ、CI/CD で補完設計）

# 想定質問集（深掘り対応）

スライド個別の質問は各スライドのガイドに埋めた。ここでは **セクション横断の質問** と **深掘り質問** を整理する。「なんでも知っている人」に見せるためには、以下の質問に即答できるようにしておく。

## A. Databricks 第一候補の根拠系

### A-1. 「なぜ Databricks 推奨なのか？Deloitte に引きずられていないか？」

**答**: 「4要件ベースでの技術評価の結果です。決め手は **要件2 の PySpark 219本の再利用可能性**。Databricks は **デコレーター除去のみ** で PySpark DataFrame API がそのまま動きますが、Snowflake は Snowpark への書き直し、AWS Glue は Glue 固有制約があります。要件3 のガバナンス（Unity Catalog）、要件4 の AI 拡張（MLflow・AI-BI Genie・Vector Search）も統合されており、1つのプラットフォームでカバーできます。**これは Deloitte が提案したからではなく、技術比較の結果** です」

### A-2. 「PySpark がそのまま動くというのは本当か？」

**答**: 「厳密には3つの注意点があります。①Foundry 固有の **`@incremental` デコレーター** は Databricks Delta Live Tables で **ロジック再設計** が必要な場合があります。②**Foundry Ontology を参照しているパイプライン** は、データソースの指定が Ontology の RID で行われているため、**Delta Lake のテーブルパスに書き換え** が必要です。③**Foundry の暗黙的な型変換** は Databricks では **明示的なキャスト** が必要になるケースがあります。これらは RFP 後の詳細設計で、**219本を代表パターン別に棚卸し** して、パターン別の移行工数を見積もります」

### A-3. 「Snowflake を選ぶ余地は本当にないのか？」

**答**: 「あります。**運用自走性を最優先する場合は Snowflake が合理的** です。SQL + Snowsight 中心で学習曲線が最も緩やか。Cortex Analyst はデータが Snowflake から出ない設計で、**要配慮個人情報の扱い** ではコンプライアンス説明が容易です。ただし **要件2 の PySpark 219本の書き直し（Snowpark）** が大きなリスクとコストになります。**どちらを優先するかのトレードオフ** で、最終判断は SOMPOケアさんにお願いします」

### A-4. 「AWS Native で自作するのが最安ではないか？」

**答**: 「**インフラコスト単体では最安の可能性** があります（PwC の試算も AWS Native）。ただし **6 以上のサービスを横断管理する運用負荷** が高く、**内部チーム約10名で自走できるか** が大きな論点です。PwC の試算でも Step2 で初期 1.6〜2.3億円 + 年間2,400万円 + AWS 費用 と、運用を含めた TCO では優位性が薄まります。**運用人件費を含めた TCO 比較** を RFP で必須化しています」

## B. コスト系（石郷岡氏が最も突っ込んでくる領域）

### B-1. 「Foundry より安くなるのか？」

**答**: 「**Foundry の現行コストはクライアント側でご確認いただき、そこと比較する形で RFP 後の評価で判断します**。Deloitte の RFI 回答では Databricks構成 年3,000万〜6,000万円（ライセンス料のみ、構築費・運用保守別途）、Fujitsu は梅3,500万〜松8,000万円 という試算が出ていますが、**TCO 全体での比較が必要** です」（← Foundry より安い、とは言わない）

### B-2. 「コスト急増のポイントは？」

**答**: 「3点あります。①**DBU（コンピュート課金）** — クエリ数・実行時間が増えると線形に増える ②**ストレージ膨張** — VACUUM ポリシーと Time Travel 保持期間の設計が重要 ③**BI ツールライセンス** — Tableau の閲覧者数に応じたライセンス。**RFP では データ量が2倍・5倍になった場合のコスト変化の提示** を必須にしています。**同一条件での TCO 比較** も必須です」

### B-3. 「Databricks の二重課金構造とは？」

**答**: 「Databricks は **DBU 課金**（Databricks 社への支払い）と **クラウドインフラ費**（AWS EC2/S3 等への支払い）が **別請求** になります。Snowflake の場合は Snowflake 1社への支払いで完結（ストレージ含む）します。この構造の違いで、**見積もりを単純比較すると Snowflake の方が安く見える** ことがあります。TCO 比較時には Databricks + AWS インフラ を合算する必要があります」

### B-4. 「Snowflake は設定次第でコストが爆発するのでは？」

**答**: 「はい、業界では事例があります。**ウェアハウスサイズ**（S→M→L）でクレジット消費が倍増するため、**初期設定とResource Monitor によるアラート・停止ポリシー** が重要です。SOMPOケアの **ピーク同時接続 40〜80名** 規模であれば、Tableau Extract（定期キャッシュ）と組み合わせて **S + 自動スケーリングで十分** な可能性が高いです」

## C. 運用・ガバナンス系

### C-1. 「運用自走は本当に 2 名で回せるのか？」

**答**: 「**日常運用** を 2 名程度で回す想定です。担当範囲は:
- **KPI 定義変更**（SQL or dbt で実装）
- **権限管理**（Unity Catalog GUI でのポリシー設定）
- **BI 修正**（Tableau ダッシュボードの追加・修正）
- **パイプライン監視**（Lakeflow Jobs の実行状況確認、失敗時の再実行）

逆に以下は **ベンダー支援を使う前提**:
- 新規ソースシステムの追加開発
- 大規模パイプライン改修
- 障害時の原因深掘り（特に Spark レイヤー）

**最終的な人員計画は RFP 後の詳細設計で確定** します」

### C-2. 「Markings の自動伝播がないのは大きな問題ではないか？」

**答**: 「**Foundry 移行時の主要なガバナンスギャップとして正直に認識しています**。Foundry Markings はデータにタグを付けると下流全てに自動伝播する仕組みで、1,000施設 × 要配慮個人情報のガバナンスで価値が高い機能でした。Databricks Unity Catalog では自動伝播がないため、**CI/CD パイプラインでタグ伝播を自動化する補完設計** が必要です。これを **RFP の必須提案事項** にしています。2026年 GA の Governed Tags はタグ統制の機能であって伝播機能ではないことも認識しています」

### C-3. 「Unity Catalog と Horizon の違いは？」

**答**: 「名前は似ていますが設計思想が異なります。
- **Unity Catalog（Databricks）**: データ + ML + AI の全資産を1つのカタログで管理。MLflow のモデル、Feature Store、Vector Search インデックスもカタログ対象。**Lakehouse Federation** で外部 DB（Snowflake/Redshift 等）のメタデータも横断検索可能。
- **Horizon（Snowflake）**: Snowflake 内のデータ資産に特化。リネージ・品質管理・Data Sharing を統合。**ML 資産の管理は対象外**。

SOMPOケアが AI/ML 活用を将来目指すなら **Unity Catalog が長期的に有利**、当面 BI/SQL だけなら **Horizon がシンプル** という整理です」

## D. セキュリティ・コンプライアンス系

### D-1. 「要配慮個人情報のアクセス制御はどう実現するのか？」

**答**: 「**4次元のアクセス制御** で実現します。①**施設単位** — 施設長 A は施設 A のデータのみ閲覧可（Row-Level Security、標準機能）②**データ種別単位** — 健康情報などはケアスタッフのみ（列レベル制御）③**時間単位** — 退去した入居者のデータ参照制限（カスタム実装、将来要件）④**AI 経由** — AI が代理取得する際もユーザー権限を継承（PoC 検証、将来要件）。①②は 3 基盤とも標準機能、③④は将来論点です」

### D-2. 「PII 検出はどの基盤にあるのか？」

**答**: 「**3基盤とも標準では限定的** です。追加で **Presidio（Microsoft OSS）** や **AWS Macie** 等の PII 検出ツールを導入する必要があります。これは RFP で対応方針を確認します」

### D-3. 「Cortex Analyst の『データが Snowflake 外に出ない』設計はどういう意味か？」

**答**: 「自然言語分析の LLM として、Snowflake 内部で動く **Mistral や Llama** を使う設計です。ユーザーの質問と回答生成が **Snowflake の境界の中で完結** し、データが外部の OpenAI API 等に送信されません。介護データには要配慮個人情報があるため、**コンプライアンス説明が容易** になります。一方、Databricks の AI-BI Genie は外部 LLM（Claude/GPT）も使える柔軟性がありますが、**データ境界を明示的に設計** する必要があります」

## E. 非構造データ・AI 将来論点系

### E-1. 「非構造データ（介護記録テキスト・眠りSCAN・音声）はどう扱うのか？」

**答**: 「**Bronze 層への格納は今回のスコープ内** です。カタログ登録までは移行時に実施します。ただし **加工・AI 活用は次フェーズ** の位置づけです。理由は、①ユースケースごとに加工設計が必要で、一般的な設計では対応できない ②具体的なユースケースが決まらないと精度要件が定義できない から。**スコープと次フェーズの切り分けを明確にしている** 点がポイントです」

### E-2. 「眠りSCAN のデータ量は何 TB か？」

**答**: 「**現在の資料では 1.92TB と記載していますが、山口さんから前回 20TB との言及をいただいており、未解決論点として残しています**。SOMPOケアさん側で実際のデータ量を確認いただけると、RFP の移行計画とストレージ試算に反映できます。**確認方法 or 担当者を決めていただけないでしょうか**」（← 正直に認めて、確認を依頼する）

### E-3. 「介護記録テキスト 340GB はどう活用するのか？」

**答**: 「**Bronze 層に格納してカタログ登録するところまで** が今回のスコープです。将来 NLP / 生成AI で活用する場合の前提を確保しておく位置づけで、具体的な活用（要約・検索・RAG 等）は **次フェーズの AI ユースケース検討時に設計** します」

### E-4. 「オントロジー的な製品はあるのか？（柴田氏・竹森氏）」

**答**: 「**Timbr、Stardog、Palantir Foundry、RelationalAI 等** がありますが、いずれも Foundry と同種のオントロジー中心設計の製品で、SOMPOケアの現状の活用度（施設360 など限定的）を踏まえると **今回の移行スコープに含めるのは時期尚早** です。まず (1) カタログと (2) セマンティックレイヤーで基盤を整え、**(3) オントロジー/アクション層が必要になった段階で外部製品を検討** するのが合理的です。RFP にはオプション提案ページを追加して、ベンダー側の提案を受け付ける形にしています（森田さんが追加提案領域として整理）」

### E-5. 「Foundry Ontology の本質的な価値は何か？」

**答**: 「Foundry Ontology は **Object Types（エンティティ）+ Link Types（関係性）+ Action Types（ビジネス操作）** の3つが統合されていることに価値があります。Databricks / Snowflake のセマンティックレイヤーは Object と Link に相当する機能を提供しつつありますが、**Action Types —『入居者を別施設に移動』のようなビジネスオペレーションをデータモデル上で定義・実行する機能** は、どちらにもありません。これが Foundry の本質的な差別化ポイントです。SOMPOケアでは現状 Action Types まで活用できていないため、当面は代替可能と判断しています」

## F. 移行計画・スケジュール系

### F-2. 「2028年の CareBase 刷新との関係は？」

## G. JOY 連携・HD 統合系

### G-1. 「JOY との関係は？」

**答**: 「**L0-L1（インフラ・ストレージ）は HD 共通化の方向**（AI-Ready データ基盤 FY2026 内に HD にて構築、委託領域認識合わせ資料 P3）。**L2-L3（データアーキ・アプリ）は SOMPOケア独自の設計** です。JOY 側の基盤が未確定なため、**Databricks の Lakehouse Federation**（外部 DB のメタデータを統合検索できる機能）で、JOY 側が何になっても連携できる設計を想定しています」

### G-2. 「JOY が Snowflake になった場合、Databricks を選ぶと連携に苦労しないか？」

**答**: 「**Lakehouse Federation で Snowflake のメタデータを Unity Catalog 経由で検索可能** です。データ本体は Snowflake 側に置いたまま、カタログだけを統合できます。また UniForm（Delta + Iceberg 互換）で **データ本体を Iceberg 形式として Snowflake からも読める** ので、大きな連携障壁にはなりません」

### H-2. 「Fujitsu の Kozuchi 依存は回避できるのか？」

### H-4. 「Sky の Quick Suite 固定問題は致命的ではないか？」

**答**: 「**3,000名への影響を考えると慎重な検討が必要** です。Sky の AI 設計力は高く評価できますが、**既存 Tableau 資産からの切り替え** が発生します。RFP で **BI 移行計画と影響範囲** を確認事項にしています。Tableau 併用の余地があるかも論点です」

### H-5. 「PwC の Step 案は？」

**答**: 「PwC は 3 Step 方式で、**Step 2 推奨で 初期 1.6〜2.3億円、年間2,400万円 + AWS 費用** の試算です。**MDM（マスターデータ管理）を唯一体系的に提案** しており、ガバナンス・組織設計の伴走者としての価値は別途評価できます。**技術ベンダーとの組み合わせ** も選択肢として残せます」
