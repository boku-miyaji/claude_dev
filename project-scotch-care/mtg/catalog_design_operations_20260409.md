# 第1章: なぜカタログが必要か — 3つの層と全体像

## 1.1 データ基盤の3つの層

データ基盤で「データの意味を管理する」仕組みは、以下の3層に分かれる。

```
┌───────────────────────────────────────────────────────────┐
│  (3) オントロジー / アクション連携                          │
│  「施設」「入居者」「ケア記録」等のビジネスオブジェクト間の    │
│   関係性を定義し、データ上でビジネス操作を実行できる           │
│  例: 「入居者Aを施設Bに移動」→ 関連データが連動更新           │
│                                                           │
│  ※ Foundry Ontologyが担っていた領域。DB/SFには直接対応なし   │
├───────────────────────────────────────────────────────────┤
│  (2) セマンティックレイヤー                                 │
│  KPIの定義（入居率 = 入居者数÷定員）を一箇所で管理し、        │
│  BI・AIチャット・レポートの全てで同じ数値を返す               │
│  例: 施設長が「未タップ率が高い施設は？」と聞ける            │
│                                                           │
│  ※ 2025-2026年に急速に「製品化」された層                    │
├───────────────────────────────────────────────────────────┤
│  (1) カタログ / ガバナンス                                  │
│  何のデータがどこにあるか。誰が管理しているか。              │
│  アクセス権限は誰に。データの血統（リネージ）は。            │
│  例: 「入居率のデータはgold.kpi.occupancy_rate」             │
│                                                           │
│  ※ 全ての基盤に必要な基礎。ここなしに(2)(3)は成立しない     │
└───────────────────────────────────────────────────────────┘
      ↑ 土台                                    応用 ↑
```

**重要: (1)がなければ(2)も(3)も成立しない。** セマンティックレイヤーは「どのテーブルのどのカラムがKPIの分子・分母か」をカタログから参照する。オントロジーも「施設オブジェクトはどのテーブルに対応するか」をカタログに依存する。

## 1.2 Foundryは何をしていたか — 3層が統合されていた世界

Foundryでは、この3層が**1つのプラットフォームに統合**されていた:

```
Foundry Ontology = (1) + (2) + (3) が一体化
  ├ データカタログ（テーブルの所在・権限・Markingsタグ）
  ├ セマンティック定義（Object Types: 「施設」「入居者」の定義）
  ├ 関係性（Link Types: 入居者→施設、入居者→ケア記録）
  └ ビジネスアクション（Action Types: 「入居者を移動」等の操作定義）
```

**この統合がFoundryの価値だったが、同時にロックインの原因でもあった。** 3層すべてがPalantir独自形式で、外に持ち出せない。

## 1.3 新基盤では何が変わるか — 分離と段階的構築

Databricks/Snowflakeでは、3層を**別々の仕組みで段階的に構築**する:

| 層 | Foundryでの統合 | 新基盤での実現方法 | いつ構築するか |
|----|-------------|---------------|------------|
| **(1) カタログ** | Foundry Ontologyに内蔵 | Unity Catalog / Horizon | **移行時（必須）** |
| **(2) セマンティック** | Object Types / KPI定義 | dbt Semantic Layer / Semantic Views + AI-BI Genie / Cortex Analyst | **運用安定後（拡張）** |
| **(3) オントロジー** | Object + Link + Action Types | 直接対応なし。**Timbr等のオントロジーベースのセマンティックレイヤー製品で補完可**（後述） | **将来（オプション）** |

**つまり:**
- (1)は移行時に作らないとデータが「どこにあるかわからない」状態になる
- (2)は後からでも入れられるが、AIチャットを使いたいなら設計だけは先にしておく
- (3)はSOMPOケアの現状の活用度（限定的）を踏まえると当面不要。ただし**Foundryに戻らずにオープンな構成で実現する選択肢が2026年時点で存在する**（Timbr等）

## 1.4 2026年の動向 — 「カタログ」から「意味のあるデータ」への進化

2025-2026年にかけて、データ基盤の業界では**カタログの役割が大きく変わりつつある**:

### トレンド1: カタログが「受動的な台帳」から「能動的な基盤」になった

```
従来のカタログ:
  人間が登録 → 人間が検索 → 人間が判断
  = メタデータの倉庫。登録が面倒で、すぐ廃れる

2026年のカタログ:
  パイプラインが自動登録 → AIが意味を理解 → 人間に自然言語で回答
  = データとAIをつなぐインターフェース
```

- **Unity Catalog**: Lakehouse Federation（外部DBのメタデータも自動統合）、Governed Tags（タグの統制）、Sample Data Explorer（自然言語でカタログを探索）が2026年にGA
- **Snowflake Horizon**: リネージ・品質管理・データ共有をカタログに統合。Semantic Views Autopilot（カタログのメタデータからセマンティックモデルを自動生成）が2026年GA

### トレンド2: セマンティックレイヤーが「製品」になった

2024年までセマンティックレイヤーは「あると嬉しいが、構築が面倒なもの」だった。2025-2026年で一変:

- **dbt Semantic Layer (Metricflow)**: KPI定義をYAMLで書くだけ。BI/AIチャット/APIから同じ定義を参照可能に
- **Databricks Metric Views**: Unity Catalog内蔵のセマンティック定義。AI-BI Genieがこの定義を参照してSQLを生成
- **Snowflake Semantic Views**: スキーマオブジェクトとしてメトリクス・ディメンション・リレーションを定義。Cortex Analystの精度を支える

→ **SOMPOケアにとっての意味**: 「入居率 = 入居者数 ÷ 定員」をセマンティックレイヤーに一度定義すれば、Tableauでも、AIチャットでも、Excelレポートでも、同じ数値が返る。現状の「定義がバラバラ」問題が技術的に解決可能になった。

### トレンド3: カタログが「AI活用の入口」になった

AIチャット（AI-BI Genie / Cortex Analyst）が自然言語でデータに回答するとき、**カタログのメタデータが回答精度を直接左右する**:

```
メタデータが貧弱なカタログ:
  施設長: 「未タップ率が高い施設は？」
  AI: 「"未タップ率"に該当するテーブルが見つかりません」

メタデータが充実したカタログ + セマンティックレイヤー:
  施設長: 「未タップ率が高い施設は？」
  AI: 「gold.kpi.untapped_rate を参照します。
       未タップ率 = タップされなかったナースコール数 ÷ 総コール数。
       上位3施設: 施設A (32%), 施設B (28%), 施設C (25%)」
```

→ **カタログは「管理のための管理」ではなく、AIがデータを正しく読むための土台。** カタログの品質 = AI回答の品質。

### トレンド4: 介護情報基盤（2026年4月〜）との連携

2026年度から全国で「介護情報基盤」が順次開始。要介護認定情報・LIFEデータ・ケアプランが標準化され、事業所間・医療機関間で電子共有される。

→ SOMPOケアが1,000施設分のデータを外部と連携する場合、カタログで「どのデータが外部共有可能か」「どのデータに個人情報が含まれるか」を管理する必要がある。タグ + RLSによるアクセス制御がここで効いてくる。

# 第2章: SOMPOケアにとっての実践的価値

## 2.1 カタログがないとどうなるか — 具体的な問題シナリオ

### シナリオ1: 「どのテーブルが正しいかわからない」

```
IT企画: 「入居率のテーブル、gold_kpi_monthlyとgold_facility_metricsの
         2つあるんですけど、どっちが正しいですか？」
前任者:  退職済み
ドキュメント: なし
結果: 両方のテーブルで数値が微妙に違う。どちらを信頼すべきかわからない
```

→ **カタログがあれば**: 各テーブルに「オーナー」「定義」「ソース」が登録されている。片方が「非推奨」とタグ付けされている。

### シナリオ2: 「要配慮データが無保護で公開される」

```
開発者: 新しいGoldテーブルを作成。Silver層の入居者テーブルを参照
問題: Silver層に要配慮個人情報が含まれていたが、
      Goldテーブルにはタグもアクセス制御も設定されていない
結果: 施設長がTableauで他施設の入居者情報を閲覧可能に
```

→ **タグ + CI/CD自動チェックがあれば**: PRマージ時に「要配慮データを参照するテーブルにタグがない」ことを自動検出してブロック

### シナリオ3: 「AIチャットが嘘をつく」

```
施設長: AI-BI Genieに「先月の入居率は？」と質問
AI: gold.facility_metricsを参照して「85%です」と回答
問題: そのテーブルは先月更新されていなかった（定義が古い）
結果: 施設長が間違った数値で意思決定
```

→ **セマンティックレイヤー + メタデータ鮮度管理があれば**: AIは「入居率」の正式な定義（唯一のKPIテーブル）を参照し、最終更新日時も確認してから回答

## 2.2 セマンティックレイヤーで何が変わるか — 施設長の体験が変わる

### 今の体験（Foundry → 新基盤 移行直後）

```
施設長の1日:
  朝: Tableauにログイン → 施設モニターダッシュボードを確認
  → 「未タップ率が先月より上がっている」
  → 「なぜ上がったか」はダッシュボードからはわからない
  → IT企画部にメールで問い合わせ
  → 3日後に回答が来る
```

### セマンティックレイヤー + AIチャット導入後の体験

```
施設長の1日:
  朝: Tableauにログイン → 施設モニターダッシュボードを確認
  → 「未タップ率が先月より上がっている」
  → AIチャットに「未タップ率が上がった原因は？」と質問
  → AI: 「夜間帯（22-6時）のナースコール応答時間が平均3分→5分に増加。
         要因: 夜勤スタッフ配置が先月から1名減（3名→2名）。
         同様の傾向がある施設: 施設B, 施設D」
  → その場で対策を検討
```

**この体験を実現するために必要なもの:**
1. **(1) カタログ**: 「未タップ率」がどのテーブルにあるかをAIが知っている
2. **(2) セマンティックレイヤー**: 「未タップ率 = 未応答コール数 ÷ 総コール数」の定義をAIが理解している
3. **データ**: ナースコールデータ + スタッフ配置データがBronze→Silver→Goldで結合されている

→ (1)は移行時に構築。(2)は運用安定後に追加。データは既存パイプラインの移行で確保。

## 2.3 オントロジーは今必要か — 判断基準

| 条件 | 必要性 | 理由 |
|------|--------|------|
| 施設長がダッシュボードを見るだけ | **不要** | (1)カタログ + Tableauで十分 |
| 施設長がAIに質問したい | **(2)セマンティックレイヤーが必要** | カタログだけではAIの精度が出ない |
| データから直接ビジネスアクションを実行したい（例: 「この入居者のケアプランを自動更新」） | **(3)オントロジーが必要** | (1)(2)では「データを見る」だけ。「データで動かす」には(3)が必要 |
| 複数事業部のデータを横断してノーコードで探索・分析したい | **(3)に近い機能が必要** | Foundryの活用度3に相当。ただし現状は活用度2も限定的 |

**SOMPOケアの現状**: 活用度2（オントロジー活用）が限定的。まず(1)カタログを整備し、(2)セマンティックレイヤーでAIチャットを実現する段階。(3)は将来オプション。

### Timbr — 「Foundryオントロジーをオープンに再現する」製品

柴田氏の「オントロジー的な製品の情報が欲しい」への回答として、Timbrは最も直接的な候補。

**Timbrの思想:**

Foundryのオントロジーが「全てをFoundry内に取り込み、独自形式で管理する」のに対し、Timbrは**データを動かさずに、既存のDWH（Databricks/Snowflake等）の上にオントロジーを被せる**アプローチを取る。

```
Foundry:
  データ → Foundryに取り込み → Foundry内でオントロジー定義
  → Foundry内でBI・分析
  = 全てがFoundryの中。出られない（ロックイン）

Timbr:
  データ → Databricks/Snowflakeに置いたまま
  → Timbrがその上にオントロジー（概念モデル）を被せる
  → SQLで「施設」「入居者」「ケア記録」をビジネス概念として扱える
  = データは動かさない。意味の層だけ追加する（ゼロコピー）
```

**Foundryとの対応:**

| Foundry Ontology | Timbr | 違い |
|-----------------|-------|------|
| Object Types（施設、入居者等） | SQL Ontology Concepts（仮想テーブル） | Foundryは独自形式。TimbrはSQL標準 |
| Link Types（入居者→施設） | Semantic Relationships | 同上 |
| Action Types（業務操作） | **対応なし** | Timbrはデータモデルに特化。業務アクション機能はない |
| Markings（タグ自動伝播） | Role-based + Row-level access | 類似だが自動伝播の深さはFoundryが上 |

→ Timbrは3層でいうと**(1)カタログと(2)セマンティックレイヤーの間を埋め、(3)のオントロジー的な概念モデリングに最も近い位置**にいる。ただし(3)のAction Typesには対応していない。

**Databricks/Snowflakeとの接続:**

- **Databricks**: Unity Catalogにメタデータを露出。Databricksノートブック内からTimbrの概念モデルをSQLで直接クエリ可能。Databricksがクエリ実行エンジンになる
- **Snowflake**: Snowflakeのテーブルに仮想的にオントロジーを被せる。データ移動なし
- **デプロイ**: SaaS / クラウド（AWS/Azure/GCP）/ オンプレ / Kubernetes

**具体的に何ができるようになるか（SOMPOケアの場合）:**

```
Timbrなし（カタログ + セマンティックレイヤーのみ）:
  「未タップ率が高い施設は？」→ AI回答可能（セマンティックレイヤーで対応）
  「施設Aの入居者Bのケア記録と眠りSCANデータを並べて見たい」
    → 複数テーブルをJOINするSQLを自分で書く必要あり（22行のSQL）

Timbrあり（オントロジー的概念モデル）:
  「施設Aの入居者Bのケア記録と眠りSCANデータを並べて見たい」
    → 「入居者B」という概念から関連データが自動でつながる（6行のSQL）
    → ほのぼの/Axist/眠りSCAN/ナースコール間の名寄せ済みデータを
       概念レベルで横断検索・分析可能
```

**SOMPOケアへの推奨:**

Timbrは「今すぐ必要」ではないが、**将来Foundryのオントロジー的な体験が必要になった場合に、Foundryに戻らずにオープンな構成で実現できる選択肢**として認識しておく価値がある。

```
SOMPOケアの段階的アプローチ:

Phase 0（移行時）: Databricks + Unity Catalog + dbt
  → (1)カタログ整備

Phase 1（運用安定後）: + dbt Semantic Layer / AI-BI Genie
  → (2)セマンティックレイヤー。AIチャット

Phase 2（将来オプション）: + Timbr
  → Foundryオントロジーに近い概念モデルをDatabricksの上に構築
  → データは動かさない。Timbrはレイヤーとして追加するだけ
  → 既存のDatabricks + Unity Catalog投資は全て活かせる
```

**RFPへの組み込み:** 竹森氏が求めた「将来の拡張性について提案してください」のオプション提案として:

> ベンダーには以下を提案してください:
> - データ基盤の上にビジネス概念モデル（施設・入居者・ケア記録等の関係性定義）を構築する場合の方針
> - Timbr等のオントロジーベースのセマンティックレイヤー製品の活用可能性とコスト感
> - 段階的導入（Phase 0: カタログ → Phase 1: セマンティック → Phase 2: オントロジー）の実現性

# 第3章: 設計・運用の実務

## 3.1 名前空間の設計 — SOMPOケアの場合どうするか

### 名前空間とは

Unity Catalog（Databricks）の場合、データは3階層で管理される:

```
カタログ（Catalog）
  └ スキーマ（Schema / Database）
      └ テーブル / ビュー / モデル / 関数
```

### SOMPOケアでの推奨設計

```
sompocare                          ← カタログ（組織単位）
  ├ bronze                         ← スキーマ: 生データ層
  │   ├ honobono_residents         ← ほのぼの: 入居者マスタ
  │   ├ honobono_care_records      ← ほのぼの: 介護記録
  │   ├ axist_vitals               ← Axist: バイタルデータ
  │   ├ nemuri_scan_raw            ← 眠りSCAN: 生データ（時系列）
  │   ├ nursecall_events           ← ナースコール: イベントログ
  │   └ carebase_*                 ← NDS CareBase: 新システムデータ
  │
  ├ silver                         ← スキーマ: クレンジング済み
  │   ├ unified_residents          ← 名寄せ済み入居者マスタ（★最重要）
  │   ├ unified_facilities         ← 名寄せ済み施設マスタ
  │   ├ cleaned_care_records       ← 正規化済み介護記録
  │   ├ aggregated_vitals_hourly   ← バイタル（1時間粒度に集約）
  │   └ nursecall_with_resident    ← 入居者紐付け済みナースコール
  │
  ├ gold                           ← スキーマ: ビジネスロジック適用済み
  │   ├ kpi_occupancy_rate         ← 入居率（KPI定義: 入居者数÷定員）
  │   ├ kpi_untapped_rate          ← 未タップ率
  │   ├ kpi_multitask              ← マルチタスク指標
  │   ├ kpi_yutori_time            ← 業務ゆとり時間
  │   ├ dashboard_cmc              ← CMCダッシュボード用ビュー
  │   ├ dashboard_monitor          ← モニターダッシュボード用ビュー
  │   ├ dashboard_nursecall        ← ナースコール分析用ビュー
  │   └ dashboard_facility360      ← 施設360用ビュー
  │
  ├ ml                             ← スキーマ: ML/AI資産（将来）
  │   ├ feature_store              ← 特徴量テーブル
  │   └ models                     ← MLflowモデルレジストリ
  │
  └ sandbox                        ← スキーマ: アドホック分析用
      └ user_yamamoto_*            ← 個人の実験用テーブル
```

### 設計時に決めなければいけないこと

| 決めること | 選択肢 | SOMPOケアでの推奨 | 理由 |
|----------|--------|----------------|------|
| **カタログの粒度** | 組織全体で1つ / 事業部ごと / 環境ごと | 組織全体で1つ + dev/stg/prodは別カタログ | 1,000施設を横断分析する必要があるため、分割しない |
| **スキーマの分け方** | Bronze/Silver/Gold / ドメイン別 / 用途別 | Bronze/Silver/Gold（データ層） | Foundryの3層設計を引き継ぐ。ドメイン分割は施設数が多すぎて破綻する |
| **テーブル命名規則** | 自由 / プレフィックス必須 / ドメイン_エンティティ | `{ソース}_{エンティティ}` (bronze) / `{目的}_{エンティティ}` (gold) | 命名規則なしだと半年で「何のテーブルかわからない」が発生する |
| **sandboxの管理** | なし / ユーザーごとのスキーマ / 共有スキーマ | ユーザーごとのプレフィックス（`user_{name}_`） | 自由に実験できる場を残しつつ、gold層を汚さない |

### ハマりポイント: 名前空間は後から変えにくい

**一度作ったカタログ/スキーマ構造は、後から変更するとBI接続・パイプライン参照が全て壊れる。** 移行時に正しく設計する必要がある。

特にTableauのデータソース定義は `catalog.schema.table` の3階層で参照するため、名前を変えると3,000名のBI閲覧が止まる。

## 3.2 タグとアクセス制御の設計 — Foundry Markingsなしでどう管理するか

### Foundry Markingsとは何だったか

Foundryでは、データにセキュリティタグ（例: 「要配慮個人情報」「施設A限定」）を付けると:
1. そのデータを参照する下流テーブルに**自動的にタグが伝播**する
2. タグに基づくアクセス制御が**自動的に下流にも適用**される

→ 「Bronze層の入居者テーブルに"要配慮"タグを付ければ、それを参照するSilver/Gold/ダッシュボード全てに自動で適用される」

### 新基盤（Unity Catalog）で何が変わるか

**Unity Catalogにはタグの自動伝播がない。** これが最大のギャップ。

```
Foundry:
  bronze.residents に「要配慮」タグ
     ↓ 自動伝播
  silver.unified_residents にも「要配慮」タグ
     ↓ 自動伝播
  gold.kpi_occupancy_rate にも「要配慮」タグ

Unity Catalog:
  bronze.residents に「要配慮」タグ
     ↓ 伝播しない
  silver.unified_residents → タグなし（手動で付ける必要あり）
     ↓
  gold.kpi_occupancy_rate → タグなし（同上）
```

### 対応策: CI/CDによるタグ伝播の自動化

```yaml
# タグ定義ファイル（Git管理）
tags:
  - name: "要配慮個人情報"
    applies_to:
      - bronze.honobono_residents
      - bronze.honobono_care_records
      - silver.unified_residents      # ← bronzeの下流を明示的に列挙
      - gold.kpi_*                    # ← パターンマッチ
    access_policy:
      - role: facility_manager
        filter: "facility_id = current_user_facility()"
      - role: data_team
        filter: null  # 全施設アクセス可
```

```
運用フロー:
  1. 開発者がパイプラインを追加（例: 新しいGoldテーブル）
  2. タグ定義ファイルに下流テーブルを追記
  3. PRレビューで「タグ漏れ」をチェック
  4. CI/CDでUnity CatalogのAPIにタグを適用
```

### ハマりポイント

- **タグ定義ファイルの更新を忘れる**: 新テーブルを作ったがタグを付け忘れ → 要配慮データが無保護で公開される
- **対策**: PRテンプレートに「タグ確認チェックリスト」を必須項目として追加。CI/CDで「Goldテーブルにタグがないもの」を自動検出してアラート

### アクセス制御の設計（1,000施設 × 多層権限）

| ロール | 範囲 | アクセス可能なデータ | 実装方法 |
|--------|------|----------------|---------|
| **施設長** | 自施設のみ | Gold層のダッシュボード用ビュー（自施設フィルタ） | RLS: `facility_id = current_user_facility()` |
| **エリアマネージャー** | 管轄エリアの全施設 | Gold層（エリアフィルタ） | RLS: `area_id IN (current_user_areas())` |
| **本部管理者** | 全施設 | Gold層（フィルタなし） | ロールベース |
| **IT企画部** | 全施設 + Silver/Bronze | Silver/Gold層 + Bronze層の参照 | スキーマレベル権限 |
| **データチーム** | 全施設 + 全層 | Bronze/Silver/Gold + sandbox | 最高権限 |

**ハマりポイント:**
- **施設長のfacility_idをどう紐づけるか**: IdP（Azure AD等）のグループ属性に施設IDを持たせるのが一般的だが、1,000施設分のグループ管理が必要。異動時の更新漏れが最大のリスク
- **対策**: 施設-ユーザー紐付けテーブルをマスタとして管理し、RLSのフィルタ条件をこのテーブルに結合する設計にする。IdPの直接参照より柔軟

## 3.3 メタデータの鮮度管理 — カタログが「廃墟」になる最大の原因

### カタログが死ぬパターン

```
移行直後:  全テーブルに説明文を書いた！カタログ完璧！
3ヶ月後:   新テーブルが10個増えたけど説明文なし
6ヶ月後:   「あのテーブルの定義、カタログと違うんだけど…」
1年後:     「カタログ？ 誰も見てないよ」
```

**原因: メタデータの更新がパイプラインの一部として組み込まれていない。**

### 対策: メタデータをコードとして管理する（Metadata as Code）

```yaml
# dbt schema.yml（メタデータ定義 = コードの一部）
models:
  - name: kpi_occupancy_rate
    description: "入居率。定義: 入居者数 ÷ 定員。月次集計。"
    meta:
      owner: "data_team"
      domain: "施設運営"
      refresh: "daily 06:00 JST"
      source: "ほのぼの → bronze.honobono_residents"
      sensitivity: "要配慮個人情報を含む"
    columns:
      - name: facility_id
        description: "施設ID（unified_facilitiesと結合可）"
      - name: occupancy_rate
        description: "入居率（0.0〜1.0）"
        tests:
          - not_null
          - accepted_values:
              values: [">= 0", "<= 1"]
```

**ポイント:**
- メタデータはdbtのschema.ymlに書く → パイプラインのコードと同じリポジトリで管理される
- テーブルを変更したらメタデータも変更する → PRレビューで「説明文の更新」がチェック対象に
- CI/CDでUnity Catalogに自動反映 → 手動更新の必要がない

### ハマりポイント

- **初期にメタデータを「一括登録」しようとする**: 219テーブル分を一度に書くのは非現実的。移行するテーブルから順にメタデータを書く方が持続可能
- **カタログの「検索」が使われない**: 施設長がカタログを検索するわけではない。施設長はTableauを見るだけ。カタログの主な利用者はIT企画部とデータチーム。利用者を絞って設計する

## 3.4 リネージの活かし方と壊れるパターン

### リネージとは

```
bronze.honobono_residents
  → silver.unified_residents（名寄せ処理）
    → gold.kpi_occupancy_rate（KPI計算）
      → Tableauダッシュボード「施設モニター」
```

この**データの血統（どこから来て、どう加工されたか）**を自動追跡するのがリネージ。

### リネージが壊れるパターン

| パターン | 何が起こるか | 対策 |
|---------|-----------|------|
| **SQLでアドホッククエリを書いてテーブルを作る** | ノートブックから直接 `CREATE TABLE AS SELECT ...` するとリネージが追跡されない | sandboxスキーマでのみ許可。Gold層への直接書き込みを権限で禁止 |
| **外部ツールからデータを書き込む** | CSVアップロードやExcel経由でGold層にデータを追加すると、ソースが追跡不能に | Bronze層経由を徹底。外部データもBronze→Silver→Goldの経路で取り込む |
| **dbtモデルの外でPySparkで加工する** | dbtのリネージとPySparkのリネージが分断される | dbtとPySparkの役割を明確に分ける（SQL変換=dbt、Python変換=PySpark） |

### SOMPOケアでリネージが特に重要な理由

介護領域では**「この数値はどのデータから計算されたか」のトレーサビリティ**が監査で問われる可能性がある。例えば:

- 「施設Aの未タップ率が異常に低いが、算出根拠は？」
- → リネージで `gold.kpi_untapped_rate → silver.cleaned_care_records → bronze.carebase_tap_records` と辿れる
- → Bronzeの生データまで遡って検証可能

リネージが壊れていると、この検証ができない。

## 3.5 データオーナーシップの設計 — 「誰のデータか」問題

### なぜオーナーシップが必要か

```
問題: 「gold.kpi_occupancy_rateの定義を変えたいんだけど、誰に聞けばいい？」
回答: 「…」
```

**データにオーナーがいないと、変更の承認も、品質の責任も、問い合わせの窓口もない。**

### SOMPOケアでのオーナーシップ設計案

| 層 | オーナー候補 | 責任範囲 |
|---|-----------|---------|
| **Bronze** | IT企画部（インジェスト担当） | データソースとの接続維持。取り込み失敗のアラート対応。ソースシステム（ほのぼの/CareBase等）の変更時の影響分析 |
| **Silver** | データチーム | 名寄せロジックの正確性。クレンジングルールのメンテナンス。データ品質チェックの閾値設定 |
| **Gold（KPI）** | IT企画部 + 業務部門（共同） | KPI定義の正確性。ビジネスロジックの変更承認。利用部門からの問い合わせ対応 |
| **Gold（ダッシュボード）** | IT企画部 | Tableauビューのメンテナンス。表示項目の追加・変更 |

### ハマりポイント

- **「データチーム」に全部のオーナーシップを押し付ける**: 2名体制ではGold層のKPI定義の正確性まで責任を持てない。KPI定義は業務部門と共同オーナーにすべき
- **オーナーが異動して引き継がれない**: カタログのメタデータにオーナー名を書くだけでなく、**ロール（役割）**で指定する。「山口氏」ではなく「データチーム リード」

## 3.6 運用で踏みやすい落とし穴 5選

### 落とし穴1: 「カタログに全部登録しよう」症候群

**症状**: 移行時に「全テーブルにメタデータを付けなければ」と考え、大量の作業が発生。結果、移行スケジュールが遅延。

**処方**: Gold層のダッシュボード用テーブル（4モジュール分）から始める。Bronze/Silverのメタデータは「ソース名」と「更新頻度」だけでいい。完璧を目指さない。

### 落とし穴2: タグの増殖

**症状**: 「要配慮」「施設限定」「本部のみ」「テスト用」「一時的」…タグが無秩序に増え、管理不能に。

**処方**: Governed Tags（Unity Catalog 2026 GA）でタグの作成権限を制限する。タグは5個以内でスタート。増やす場合はデータチームの承認を必須にする。

### 落とし穴3: dev/stg/prodのメタデータ不整合

**症状**: dev環境で新テーブルを作り、メタデータも書いた。しかしprodに反映するときにメタデータだけ反映し忘れる。

**処方**: メタデータ（dbt schema.yml）をCI/CDパイプラインに含め、テーブル定義とメタデータを必ずセットでデプロイする。

### 落とし穴4: RLSのフィルタ条件が施設マスタと不整合

**症状**: 施設が統合・分割されたが、RLSのフィルタ条件に使うfacility_idが古いまま。施設長が別施設のデータを見れる / 自施設のデータが見れない。

**処方**: RLSのフィルタ条件を静的なリストではなく、施設-ユーザー紐付けテーブル（Silver層で管理）に結合する。施設マスタの変更がRLSに自動反映される設計にする。

### 落とし穴5: カタログの存在を誰も知らない

**症状**: カタログを構築したが、IT企画部以外の誰もアクセスしない。BI閲覧者3,000名はTableauしか見ない。

**処方**: これは正常。カタログの主な利用者はIT企画部・データチームの10名。施設長がカタログを直接触ることは期待しない。施設長のセルフサービスはTableau + 将来のAIチャット（AI-BI Genie / Cortex Analyst）で実現する。カタログはその裏側で品質を支える基盤。

## まとめ

### カタログ運用の3原則

1. **メタデータはコードとして管理する** — dbt schema.yml + CI/CD。手動更新は廃墟への第一歩
2. **最小限から始めて育てる** — 初日に全テーブルを完璧に登録しようとしない。Gold層の4モジュールから
3. **タグ伝播は自動化する** — Foundry Markingsの代替はCI/CDで。手動運用は漏れの温床

### カタログ → セマンティック → オントロジーの段階的拡張

カタログは「管理のための管理」ではない。データ活用の**土台**であり、その上にセマンティックレイヤー（AIが正しく回答する仕組み）、さらにオントロジー（データを概念レベルで扱う仕組み）が積み上がる。

```
今ここ → → → → → → → → → → → → → → → → → → → → 将来
   |              |                    |
Phase 0         Phase 1             Phase 2-3
カタログ整備     セマンティック       オントロジー
(Unity Catalog)  (dbt + AI-BI)       (Timbr等)
   |              |                    |
何がどこに       KPIの定義が         データを概念で
あるか分かる     統一される           横断的に扱える
   |              |                    |
施設長:          施設長:              施設長:
Tableauを見る    AIに質問できる       複雑な分析も直感的に
```

Foundryではこの3段階が一体化されていた。新基盤では段階的に構築するが、**各段階の投資は次の段階の土台になる。戻る必要はない。**
