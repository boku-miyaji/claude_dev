# 第2部: プラットフォーム技術評価（軸1: 機能の有無）

**本セクションの位置づけ:**
第1部で整理した通り、全ての評価観点に技術面（機能の有無）とベンダー面（実績・経験）の両方がある。本セクションでは**軸1（プラットフォームの機能・仕様として何ができるか）**に絞って比較する。軸2（ベンダーが同規模で実際に構築した経験があるか）はRFP回答で評価する。

**前提:**
- Snowflakeは標準テーブル中心で記載。Iceberg tables活用時はオープン性の評価が向上する
- PySpark 219本の互換性は技術仕様上の評価。代表ジョブでの実証は前提条件とする
- Foundry継続は現状維持シナリオとして比較の基準線に置く
- 各プラットフォームの評価は「機能としてできるか」であり、「ベンダーがやった実績があるか」は含まない

## 2.1 要件1: コスト構造

| 評価観点 | Databricks | Snowflake | AWS Native | Foundry継続 |
|---------|-----------|-----------|------------|------------|
| 課金モデル | DBU従量課金。コンピュート選択に知識が必要 | クレジット従量課金。構成次第でコスト増 | 各サービス従量課金合算。見通しが複雑 | 定額ライセンス。機能過多の構成が継続 |
| コスト透明性 | Cost Management + System Tables。DBU単価の開示が限定的 | Resource Monitor + Usage Views。アラート可 | Cost Explorer + 各メトリクス。横断把握に工夫必要 | Palantir社との交渉依存 |
| コスト急増リスク | DBU膨張・ストレージ膨張 | クレジット消費急増・周辺機能追加 | 各サービス個別膨張 | 定額のため急増なし。削減もできない |

> **技術的な補足: コストで見落としやすいポイント**
>
> - **Databricksの二重課金構造**: DBU（Databricks課金）とクラウドインフラ（AWS EC2/S3）が別請求になる。Snowflakeの場合はSnowflake社への1枚の請求で完結する（ストレージ含む）。この構造の違いは見積もり比較時に注意が必要。DeloitteのRFI回答（p.73-74）では、**Databricks構成: 年3,000万〜6,000万円、AWS構成: 年3,000万〜8,000万円**のライセンス料試算が記載されている。ただしこれは**ライセンス料のみで、構築費・運用保守は別途**。見積もり時に「DBU + インフラ + 構築費 + 運用保守込みのTCO」で確認すること
> - **Snowflakeのウェアハウスサイズの罠**: ウェアハウスはS→M→Lと上げるとクレジット消費が倍増する。SOMPOケアのBI閲覧3,000名・ピーク同時接続40〜80名の規模であれば、Tableau Extract（定期キャッシュ）と組み合わせればS×自動スケーリングで足りる可能性が高い。ウェアハウスサイズの初期設定を見積もりの前提条件として確認すべき
> - **眠りSCANのストレージコスト**: 現時点で1.92TB（山口氏から20TBとの言及あり、要確認）。秒単位のバイタルデータは保持期間の設計でストレージが10倍以上変動する。VACUUMポリシー（Delta Lake）やTime Travel保持期間の設計が必須。デフォルト7日保持で設計し、監査・AI学習用に必要な場合のみ延長する

## 2.2 要件2: 移行確実性

| 評価観点 | Databricks | Snowflake | AWS Native | Foundry継続 |
|---------|-----------|-----------|------------|------------|
| **資産継承性** | **◎ デコレーター除去で高い再利用可能性（実証前提）** | △ Snowpark書き直し。工数大 | ○ Glue PySpark。一部制約あり | — |
| Tableau継続利用 | ◎ SQL Warehouse経由 | ◎ Connector経由。成熟 | ○ Redshift/Athena経由 | ○ 独自BIからの移行が別途必要 |
| 切り戻し | ◎ Delta Lake Time Travel | ◎ Time Travel（最大90日） | ○ S3バージョニング | — |
| 環境分離 | ◎ ワークスペース + Repos | ○ アカウント/スキーマ分離 | ○ 各サービスで個別設計 | △ ブランチ機能（不完全） |

> **技術的な補足: PySpark移行の「高い再利用可能性」の意味と注意点**
>
> Foundryの219本のPySparkパイプラインは、Foundry固有のデコレーター（`@transform_df`, `@configure`等）を除去すれば、PySpark DataFrame APIの部分はDatabricksでそのまま動作する。ただし以下に注意が必要:
>
> - **Foundry固有のインクリメンタル処理**: Foundryの `@incremental` デコレーターはDatabricksのDelta Live Tables（DLT）で代替できるが、1対1の書き換えではなくロジックの再設計が必要なケースがある
> - **Foundry Ontologyを参照しているパイプライン**: データソースの指定がOntologyのRID（リソースID）で行われている場合、参照先をDelta Lakeのテーブルパスに書き換える必要がある
> - **Foundry固有のデータ型**: Foundryの `@transform` で暗黙的に型変換している場合、Databricksでは明示的なキャストが必要になることがある
>
> **→ 219本全てが「デコレーター除去のみ」で動くわけではない。代表的なパターン（単純ETL / 集計 / インクリメンタル / Ontology参照）ごとに検証し、パターン別の移行工数を見積もるべき。**
>
> **Snowparkへの書き直しが「工数大」の理由:**
> SnowparkはPySpark互換のDataFrame APIを提供しているが、以下の非互換がある:
> - `spark.sql()` によるSQL直接実行がSnowparkでは `session.sql()` に変更
> - UDF（ユーザー定義関数）の登録方法が異なる
> - PySparkのRDD操作はSnowparkに存在しない
> - ストリーミング処理（Structured Streaming）の対応がない
>
> → 単純な変換処理は機械的に書き換え可能だが、UDFやRDD操作を含むパイプラインは実質的な再実装になる。219本のうちどの割合がこれに該当するかの棚卸しが先決。

## 2.3 要件3: 運用・ガバナンス

| 評価観点 | Databricks | Snowflake | AWS Native | Foundry継続 |
|---------|-----------|-----------|------------|------------|
| **運用スキル** | SQL+Python+IaaC。学習曲線は急 | **◎ SQL中心。最も緩やか** | △ 複数サービスの知識必要 | × 専用知識への依存大 |
| 統合カタログ | **◎ Unity Catalog（Governed Tags GA, Lakehouse Federation）** | ○ Horizon | △ Glue DC + Lake Formation | ○ 独自仕様 |
| リネージ | ◎ PySpark列レベル対応 | ○ Horizon Lineage | △ 限定的 | △ 独自形式 |
| データ品質 | ◎ DLT Expectations | ○ Tasks + Dynamic Tables | ○ Glue Data Quality | ○ あり |
| 監査ログ | ◎ System Tables（SQL分析可） | ◎ Access History | ○ CloudTrail + 各ログ | ○ あり |
| 再実行・変更トレース | ◎ Lakeflow Jobs + Repos | ○ Tasks + Git Integration | △ Step Functions/MWAA + 各ログ | ○ 独自 |
| RLS・列マスク | ◎ Unity Catalog | ◎ RLS + Dynamic Data Masking | ○ Lake Formation FGAC（read系中心） | ◎ Markings（独自） |

> **技術的な補足: Foundry Markingsの代替は本当にできるか**
>
> Foundryの**Markings（タグ自動伝播）**は、データにセキュリティタグ（例:「要配慮個人情報」「施設限定」）を付けると、そのデータを参照する全ての下流テーブル・ダッシュボードに自動的にタグとアクセス制御が伝播する仕組み。SOMPOケアのように1,000施設×多層アクセス権限の環境では価値が高い機能。
>
> **Databricks Unity Catalogでは自動伝播がない。** タグは手動またはAPI経由で付与し、CI/CDパイプラインで伝播を自動化する設計が必要。2026年にGAになったGoverned Tagsはタグの統制（誰が何のタグを作れるか）を管理する機能であり、伝播の仕組みではない。
>
> → **これはDatabricks移行時の主要なガバナンスギャップ**。CI/CDでの補完設計をRFPの必須提案事項にすべき。
>
> **技術的な補足: Unity Catalog vs Horizon — 実質的な差**
>
> Unity Catalog（Databricks）とHorizon（Snowflake）は名前が似ているが設計思想が異なる:
> - **Unity Catalog**: データ・ML・AIの全資産を1つのカタログで管理。MLflowのモデル、Feature Store、Vector Searchインデックスもカタログ対象。2026年のLakehouse Federationで外部DB（Snowflake/Redshift等）のメタデータもUC経由で検索可能に
> - **Horizon**: Snowflake内のデータ資産に特化。リネージ・品質管理・Data Sharingを統合。ただしML資産の管理は対象外
>
> → SOMPOケアがAI/ML活用を将来的に目指すなら、ML資産もカタログ管理できるUnity Catalogの方が長期的には有利。逆に当面はBI/SQLだけなら、Horizonの方がシンプルで運用しやすい。

| 自然言語→SQL | ◎ AI-BI Genie（GA） | ◎ Cortex Analyst（GA）+ Semantic Views | △ Amazon Q in QuickSight | △ AIP（クローズド） |
| **オープン性** | **◎ Delta Lake + UniForm。OSS中心** | 標準テーブル: △ / Iceberg利用時: ◎ | ○ Glue: Iceberg v3 / Redshift: v1-v2 | × 完全独自 |
| **ML実験管理** | **◎ MLflow ネイティブ。追加ライセンス不要** | △ Snowpark ML（限定的） | △ SageMaker（別サービス） | △ なし |

> **技術的な補足: テーブルフォーマット戦争の収束と示唆**
>
> 2023-2024年は「Delta Lake vs Iceberg」の二者択一が業界の主要論点だった。2025-2026年にかけて、この対立は実質的に収束しつつある:
> - **Databricks**: UniForm（2024/6 GA）でDelta Lakeに書いたデータをIcebergメタデータ付きで読めるようにした
> - **Snowflake**: 従来の独自形式から、Iceberg tablesを正式サポートし、ネイティブテーブルとほぼ同等の性能を達成
> - **業界全体**: Apache Iceberg v3（2025末）でdeletion vectorsや行レベルリネージが追加され、Delta Lakeとの機能差が縮小
>
> → SOMPOケアにとっての示唆: **フォーマットの選択はもはや致命的な意思決定ではない**。DatabricksでDelta Lake + UniFormを選んでも、将来Snowflakeや他のエンジンからIcebergとして読める。「ロックイン回避」の観点では、2年前より状況が大きく改善している。
>
> **技術的な補足: AI-BI Genie vs Cortex Analyst — 何が違うか**
>
> 両者とも「自然言語でデータに質問する」機能だが、アーキテクチャが異なる:
> - **AI-BI Genie（Databricks）**: ユーザーがGenieスペースにテーブルを登録し、サンプルクエリやドメイン知識をテキストで与える。LLMがコンテキストを理解してSQLを生成。Databricks外のLLM（Claude/GPT等）も利用可能
> - **Cortex Analyst（Snowflake）**: Semantic Views（スキーマオブジェクト）としてメトリクス・ディメンション・リレーションを定義。LLMはSnowflake内のMistral/Llamaを使用し、データがSnowflakeの外に出ない
>
> → SOMPOケアにとっての示唆: **介護データには要配慮個人情報が含まれる**。Cortex Analystの「データが外に出ない」設計は、コンプライアンス面で説明しやすい。一方、AI-BI Genieは外部LLMも使えるためモデル選択の自由度が高い。RFPでは、AIチャット機能利用時のデータの境界（どこまでLLMに送られるか）を確認事項にすべき。

## 2.5 将来拡張性の補足: 3層の整理

| 層 | 扱う対象 | Databricks | Snowflake | Foundry |
|----|---------|-----------|-----------|---------|
| **(1) セマンティックレイヤー** | metrics / dimensions / relationships | dbt + AI-BI Genie | Semantic Views + Cortex Analyst | — |
| **(2) カタログ/ガバナンス** | メタデータ、タグ、リネージ、権限 | Unity Catalog（Governed Tags, Federation） | Horizon | Markings, 監査ログ |
| **(3) オントロジー/アクション** | objects / links / actions / workflows | 直接対応なし（外部製品で補完可） | 直接対応なし | **Palantir Ontology** |

→ (1)(2)ではDatabricks/Snowflakeが急速に強化中。(3)の業務オペレーションモデルではFoundry型がなお先行。単純な優劣比較は困難。RFPでは(1)(2)を基本スコープの拡張候補、(3)を将来オプションとして提案依頼する。

> **技術的な補足: Foundry Ontologyが「再現困難」な本当の理由**
>
> Foundryのオントロジーは単なるメタデータ定義ではない。以下の3つが統合されていることに価値がある:
> 1. **Object Types**: 「施設」「入居者」「ケア記録」等のエンティティ定義（≒セマンティックレイヤー）
> 2. **Link Types**: エンティティ間の関係性（入居者→施設、入居者→ケア記録）
> 3. **Action Types**: データに対する操作の定義（「入居者を別施設に移動」等のビジネスアクション）
>
> Databricks/Snowflakeのセマンティックレイヤーは(1)(2)に相当する機能を提供しつつあるが、(3)のAction Types — つまり「データモデルの上でビジネスオペレーションを定義・実行する」機能は、どちらにもない。これがFoundry Ontologyの本質的な差別化ポイント。
>
> → SOMPOケアの現状の活用度（活用度2が限定的）を踏まえると、(3)まで必要になる時期は当面先。まず(1)(2)で基盤を整え、(3)が必要になった段階で外部製品（Timbr等）を検討するのが合理的。
>
> **技術的な補足: 2026年の業界トレンドとSOMPOケアへの示唆**
>
> **A. 介護情報基盤の全国展開（2026年4月〜）**
> 2026年度から「介護情報基盤」が全国で順次開始される。要介護認定情報・LIFEデータ・ケアプランが標準化され、市町村・介護事業所・医療機関間で電子的に共有される。SOMPOケアは1,000施設以上を運営しており、この基盤との連携は不可避。
>
> → **基盤選定への示唆**: 外部の標準化データ（HL7 FHIR等）を取り込むインジェスト設計が必要になる。Databricks/Snowflakeともにコネクタは自前設計だが、Bronze層への格納とカタログ管理の設計を今回の移行時に織り込んでおくべき。特に「利用者の同意に基づくデータ共有」の権限制御は、現行Foundryでも新基盤でも設計が必要な新規要件。
>
> **B. SOMPOグループ内のデータ連携（JOY基盤）**
> SOMPOグループのHD共通データ基盤（JOY）との連携が将来的に必要。DatabricksのLakehouse Federationは「データを移動せず、カタログだけを統合する」設計のため、JOY基盤がDatabricks/Snowflake/Redshiftいずれであっても、UC経由でメタデータを横断検索できる。一方、Snowflake同士ならData Sharingでゼロコピー共有が可能だが、JOY側の基盤が確定していない現時点では判断材料にならない。
>
> **C. 介護データ固有の技術課題**
> SOMPOケアのデータには、一般的なDWHでは遭遇しにくい特性がある:
> - **眠りSCAN（1.92TB〜推定20TB）**: 秒〜分単位のバイタル時系列データ。Bronze層への格納はどの基盤でも可能だが、Silver層以降の集計設計（何分粒度で保持するか）でストレージコストが大きく変わる。VACUUMポリシーとの組み合わせ設計が必須
> - **介護記録テキスト（340GB・年間数百万件）**: フリーテキスト。将来のNLP/AI分析の入力になるが、現時点では「Bronze層に格納してカタログ登録する」だけ。基盤選定時にこのデータの存在を考慮する必要はあるが、加工処理の設計は次フェーズ
> - **エンティティの名寄せ問題**: ほのぼの・Axist・眠りSCAN・ナースコールで施設ID・入居者IDの体系が異なる。「施設Aの入居者Bのバイタルとナースコール記録を紐付ける」にはマスタ統合が必要。PwCがRFIで唯一MDMを提案した理由はここ
> - **NDS CareBase移行との並行**: 介護S/Wの統一（NDS CareBase製品移行）が同時進行。データ基盤の移行時に、旧システム（ほのぼの等）と新システム（CareBase）の両方からデータを受け入れる設計が必要
>
> **D. セマンティックレイヤーの「製品化」と施設長向けAI分析**
> dbt Semantic Layer、Databricks Metric Views、Snowflake Semantic Viewsが相次いでGAに。KPI定義を一箇所で管理し、BI・AIチャット・レポートの全てで同じ数値を返す仕組みが技術的に成熟した。
>
> → **SOMPOケア固有の示唆**: 現行の4モジュール（CMC・モニター・ナースコール・施設360）で使われているKPI（未タップ率・マルチタスク・業務ゆとり時間等）の定義をセマンティックレイヤーに登録すれば、施設長が自然言語で「先月の未タップ率が高い施設は？」と聞ける環境が技術的に実現可能。BI閲覧者3,000名のうち、SQLを書けない施設長・管理者にとってはこれが最大の体験向上になる。
>
> **E. テーブルフォーマットの収束**
> Delta Lake vs Icebergの二者択一は実質的に終わった。Databricks UniForm（2024 GA）でDelta→Iceberg互換読み取りが可能に。Snowflakeも標準テーブルに加えてIceberg tablesを正式サポート。「フォーマットの選択がロックインの決定要因」だった時代は過ぎた。SOMPOケアにとっては、フォーマット選択よりも、カタログ管理とアクセス制御の設計の方が重要な意思決定になる。

## 2.7 技術評価の結論

### Databricksが優位な領域

| 要件 | 優位な根拠 |
|------|----------|
| 要件2: 移行確実性 | PySpark 219本について高い再利用可能性が見込まれる（仕様上。代表ジョブでの実証が前提） |
| 要件3: ガバナンス | Unity Catalogがデータ+ML+AI資産を一元管理。Governed Tags（2026 GA）、Lakehouse Federation（外部DB横断検索）で機能面が最も充実 |
| 要件4: AI拡張性 | MLflow統合（追加ライセンス不要）、AI-BI Genie（自然言語分析）、Vector Search（非構造データ検索）がネイティブ統合 |

### Snowflakeが優位な領域

| 要件 | 優位な根拠 |
|------|----------|
| 要件3: 運用自走性 | **SQL中心で習熟ハードルが最も低い。** IaaC知識が不要で、2名体制での自走に最も適する。日本語コミュニティが活発で、二次・三次情報が豊富 |
| 要件3: セキュリティ | Cortex Analyst（自然言語→SQL）は**データがSnowflakeの外に出ない設計**。介護データの要配慮個人情報を扱う上で、コンプライアンス面の説明がしやすい |
| 要件1: コスト透明性 | Resource Monitorによるアラート設定が容易。クレジット消費の可視性がDatabricksのDBUより直感的 |
| **BI中心のユースケース** | **SOMPOケアの当面の利用がBI閲覧（3,000名）+ SQLアドホック分析（10名）であれば、Snowflakeの方がシンプルに運用できる**。Cortex Analyst + Semantic Views（2026 GA）でAIチャットも対応可能 |

### 引き分けの領域

| 要件 | 状況 |
|------|------|
| 要件1: コスト構造 | 両者とも従量課金。TCOの優劣はデータ量・利用パターン・ウェアハウスサイズ設計に依存。RFP段階での精査が必要 |
| 要件4: 自然言語→SQL | AI-BI Genie（Databricks）とCortex Analyst + Semantic Views（Snowflake）はいずれもGA。機能面では同等 |
| 要件4: セマンティックレイヤー | dbt Semantic Layer（Databricks）とSemantic Views（Snowflake）の両方が2025-2026年に製品化。どちらでもKPI定義の一元管理は実現可能 |

### 軸1の結論

| 要件 | 優位 | 軸2（ベンダー実績）で覆りうるか |
|------|------|--------------------------|
| 要件1: コスト | 引き分け | ○ コスト試算精度はベンダー経験に依存 |
| 要件2: 移行確実性 | **Databricks** | △ 仕様上の互換性は明確だが実証が前提 |
| 要件3: 運用 | **Snowflake** | ○ 支援体制はベンダー次第 |
| 要件3: ガバナンス | **Databricks** | ○ 構築実績はベンダー次第 |
| 要件4: AI拡張性 | **Databricks** | ○ 構築実績はベンダー次第 |

> **軸1（機能面）で評価すると、Databricks と Snowflake は拮抗する。**
>
> Databricksは移行確実性・ガバナンス・AI拡張性で優位。Snowflakeは運用自走性・セキュリティのコンプライアンス説明・BI中心ユースケースで優位。
>
> **最終判断は「PySpark 219本の移行工数を主要制約と置くかどうか」で決まる。**
> - 移行工数が主要制約 → Databricksが優位（PySpark再利用可能性）
> - 運用自走性が主要制約 → Snowflakeが優位（SQL完結・学習曲線最小）
>
> したがって本案件では、**代表的なPySparkジョブでの移行検証を前提条件として、Databricksを第一候補、Snowflakeを有力代替案とする**整理が妥当である。
>
> 最終判断にはRFP回答による軸2（ベンダーの実績・体制）の評価が不可欠であり、軸1のみで確定するものではない。

| 要件4: AI拡張性 | **Databricks** | MLflow・AI-BI Genie・Vector Searchがネイティブ統合 |

### 結論

**メインメッセージ:** AWS基盤を前提として、Databricks、Snowflake、AWS Nativeを比較。Databricksが最有力候補

| 基盤 | 評価 | コメント |
|------|------|--------|
| **Databricks** | **◎ 第一候補** | PySpark 219本の再利用可能性が決定的な強み。Unity Catalog・MLflow・AI-BI Genieによるガバナンス・AI統合も充実。代表ジョブでの移行検証が前提条件 |
| **Snowflake** | **○ 有力代替案** | SQL完結で1-2名体制の運用自走に最も現実的。SQL分析性能は最速クラス。Tableau連携も最も成熟。AIチャット利用時にデータが外に出ない設計はコンプライアンス面で有利。BI中心の利用であればシンプルに運用可能 |
| **AWS Native** | **○ コスト優位だが運用負荷が高い** | プラットフォームライセンスがなくインフラコスト単体では最安の可能性がある。ただし6つ以上のサービスを横断管理する必要があり、3-4名体制が必要。運用人件費を含めたTCOで比較する必要がある |
>
> **最終判断は「PySpark 219本の移行工数を主要制約と置くかどうか」で決まる。**
> - 移行工数が主要制約 → Databricksが優位（PySpark再利用可能性）
> - 運用自走性が主要制約 → Snowflakeが優位（SQL完結・学習曲線最小）
>
> したがって本案件では、**代表的なPySparkジョブでの移行検証を前提条件として、Databricksを第一候補、Snowflakeを有力代替案とする。**
> 最終判断には軸2（ベンダーの実績・体制）の評価が不可欠。以降§2.2でRFI回答をベンダー面で評価する。
