## P3: AIが「データに繋がった」— ChatGPTからの2年間で起きたこと

**キーメッセージ**: AIは「会話ツール」から「データと繋がった業務ツール」に変わった。この変化は抽象的な将来論ではなく、今日起きている現実である。

---

### 2023年: AIと「会話」できるようになった

ChatGPTが登場した。世界が驚いたのは「AIと自然言語で対話できる」こと。
だがこの時点のAIは**データに繋がっていなかった**。一般知識で会話するだけ。
業務の数字を聞いても、正確な答えは返せなかった。

### 2024-2025年: AIが「データ・コード・システム」に繋がった

Cursor、Claude Codeといったツールが登場した。
AIがコードベース、データベース、ファイルシステムに**直接アクセス**し、
自然言語の指示で「このテーブルの先月集計を出して」と言えばSQLを書き、実行し、結果を返す。

**同じことが分析基盤でも起きている**:

| 製品 | 何ができるか | ステータス |
|------|-------------|-----------|
| Databricks AI-BI Genie | 自然言語 → SQL生成 → データ取得 → 回答 | GA（一般提供中） |
| Snowflake Cortex Analyst | 自然言語 → セマンティックモデル参照 → SQL → 回答 | GA |
| Tableau Agent | 自然言語 → ビジュアル自動生成 | GA |
| Tableau MCP | 外部AI（Claude等）からTableauのデータにアクセス | OSS公開済 |

→ すべて「自然言語でデータに聞く」という**同じ方向**を向いている。

### これが意味すること

「ノーコード」「ローコード」が長年目指してきた
「専門知識がなくてもデータを扱える世界」が、GUIツールではなく**自然言語 × AI**で実現されつつある。

今はまだエンジニアが主な利用者だが、「自然言語でデータに聞く」は本来**現場の人こそ使える能力**。
施設長が「先月の転倒件数、前年比で多い施設は？」と聞けば答えが返る世界が、技術的にはもう来ている。

## P4: データの「読者」が変わる — 人間からAIへ

**キーメッセージ**: データ基盤は「人間が見るダッシュボード」のためだけではなく、「AIが読んで理解し、分析する」ためにも設計する必要がある。

---

### これまでのデータ基盤

```
データソース → ETL → DWH → BIダッシュボード → 人間が見て判断する
```

データの「お客様」は人間だった。だからUI（見た目）が重要だった。
ダッシュボードの色、レイアウト、ドリルダウン...すべて「人間の目」に最適化されている。

### これからのデータ基盤

```
データソース → ETL → DWH → AI が読む → 分析・要約・予測 → 人間が確認・判断する
                               ↑
                      ここに「AIが読める仕組み」が必要
```

AIがデータを「正しく」読むために必要なもの:

| 人間が読む場合 | AIが読む場合 |
|--------------|-------------|
| ダッシュボードの注釈を読んで「この定義だ」と判断 | **セマンティックレイヤー**: 「稼働率 = 入居者数 ÷ 定員」を**コード**として定義しないとAIは毎回違う計算をする |
| 権限がない画面は「見えない」 | **AI経由の権限制御**: AIがデータを代理取得する時、**ユーザーの権限を引き継ぐ**設計が必要 |
| 数字がおかしければ「直感」で気づく | **AI出力の検証**: AIの回答に信頼度レベルをつけ、生成SQLを可視化し、レビュー可能にする仕組みが必要 |

**データ基盤の「読者」にAIが加わったことで、基盤の設計要件そのものが変わった。**

# Part 2: Foundry移行の設計判断 — なぜケアが自ら設計すべきか

## P6: 「基盤」の4レイヤー分解 — どこを共通化し、どこを自前で持つか

**キーメッセージ**: 「データ基盤」は単一のシステムではない。4つのレイヤーに分解し、レイヤーごとに「グループ共通/ケア独自」の判断を行う必要がある。

---

### 基盤の4レイヤー

HD（ホールディングス）がグループ基盤を構想している中で、ケアの基盤設計をどう位置づけるか。「HD基盤に乗る」か「自前で作る」かは、レイヤーごとに判断が異なる。

```
Layer 0: インフラ
  AWS契約, VPC, Direct Connect, IdP (Azure AD)
  → グループ共通化の合理性が高い。規模の経済。

Layer 1: ストレージ・コンピュート
  S3, Databricks/Snowflake, Spark/SQL Engine
  → 条件付きで共有可能。マルチテナント設計（事業会社ごとに独立した
    ワークスペース/スキーマ/カタログを許容する設計）であれば共用できる。
  → 「統一スキーマで全事業会社のデータを管理」の設計だと、ケアのデータモデルが入らない。

Layer 2: データアーキテクチャ
  スキーマ設計, ETLパイプライン, セマンティックレイヤー, データコントラクト
  → ドメイン固有。グループ共通化は構造的に困難。ケアが自ら設計する必要がある。

Layer 3: アプリケーション
  BI定義, AI/ML実験環境, 評価基盤, エージェント管理, PoC環境
  → ユースケース直結。ケア独自に構築。
```

### レイヤーごとの判断

| レイヤー | HD基盤に乗るか | 判断根拠 |
|---------|---------------|---------|
| **L0: インフラ** | **乗る** | AWS契約、VPC、IdP。規模の経済が効く |
| **L1: ストレージ・コンピュート** | **条件付き** | Databricks/Snowflakeの**マルチテナント設計**が前提。HDに確認が必要 |
| **L2: データアーキテクチャ** | **乗れない** | スキーマ、セマンティックレイヤー、パイプラインはドメイン固有（次ページで詳述） |
| **L3: アプリケーション** | **乗れない** | BI定義、AI/ML環境はケアのユースケースに直結 |

**→ L0-L1はグループ共通化の対象。L2-L3はケアが自ら設計・所有する。本資料はL2-L3の設計方針を定めるもの。**

## P7: なぜL2-L3はケア独自で設計する必要があるか — 4つの構造的差異

**キーメッセージ**: ケアのデータ特性は、保険ドメインと構造的に異なる。グループ共通の汎用設計では、ケアのAI活用要件を満たせない。

---

### 差異①: データモデルの構造差

保険ドメインのデータは**トランザクション中心**。契約・請求・支払いという明確なイベントがあり、スタースキーマやスノーフレークスキーマで自然にモデリングできる。

ケアドメインのデータは**状態遷移 + テキストストリーム**。入居者の状態は連続的に変化し、その観察が介護記録という非定型テキストで記録される。さらにバイタルセンサー（眠りSCAN: 1.92TB）は時系列データとして秒〜分単位で発生する。

```
保険:  Event-driven   → Fact/Dimension設計が自然にフィット
ケア:  State + Stream → Temporal Table + Event Sourcing + 非構造パイプラインが必要
```

グループ基盤がFact/Dimensionベースで設計された場合、ケアのTemporal TableやEvent Sourcingパターンは設計前提の外になる。

### 差異②: 非構造データパイプラインの有無

保険のパイプラインは基本的に**構造化データのETL**。dbt/PySpark/SQLで完結する。
ケアでは加えて以下が必要:

- テキスト: 介護記録 → PII検出・マスキング → LLMで構造化抽出 → ベクトル化
- 時系列: バイタルデータ → ストリーム処理 → 異常検知 → アラート
- 将来: カメラ映像 → 物体検出 → イベント抽出 → 構造化

これらはdbt/PySparkとは異なるランタイム（LLMパイプライン、ストリーム処理エンジン、ベクトルDB）を必要とする。グループ基盤が構造化ETLを前提に設計されている場合、このレーンがアーキテクチャ上存在しない。

### 差異③: アクセス制御の粒度要件

保険データの制御は典型的には**部門 × 商品区分**のマトリクス。粒度は比較的粗い。

ケアデータの制御は最大4次元に及ぶ:

| 軸 | 粒度 | 例 | 実装難度 |
|---|---|---|---|
| 施設 | 1,000+ 施設ごと | 施設長Aは施設Aのデータのみ | **既存製品で対応可能**（行レベルセキュリティ） |
| データ種別 | 要配慮個人情報を含む | 健康情報はケアスタッフのみ | **既存製品で対応可能**（列レベルセキュリティ） |
| 時間 | 在籍期間 | 退去した入居者のデータは参照制限 | **カスタム実装が必要**（標準RBACでは未サポート） |
| AI経由 | LLMのクエリにも適用 | AI経由でも施設長の権限を継承 | **PoC検証が必要**（製品のAI連携成熟度に依存） |

施設 × データ種別の2次元はRFP必須要件。時間 × AI経由は将来要件として設計に織り込み、PoCで実装可能性を検証する。

さらに、AIが「この入居者は認知症の進行リスクが高い」と推論した場合、**推論結果自体が新たな要配慮個人情報**になる。このAI出力に対するデータ分類・制御は、保険ドメインでは発生しにくい論点。加えて、AI出力に基づく業務判断の**法的責任の所在**（AIが「転倒リスク高」と予測→対応しなかった場合の責任）も、介護事業特有のリスクとして検討が必要。

### 差異④: AI実験の速度要件

保険のAI開発はリスク直結（保険料算定、引受判断）→ 慎重な検証が合理的。

ケアのAI活用はまだ正解が見えていない段階 → 仮説検証を速く回す必要がある。グループ基盤のガバナンスが保険の開発サイクルを前提にした場合、PoC用データアクセスのたびにグループ承認プロセスを経ることになり、**実験速度が致命的に遅くなる**。

## P8: 今の設計に織り込むべきこと — 追加コストと後付けコストの比較

**キーメッセージ**: AI-Readyの設計を今入れる追加コストはわずか。後から入れる場合のやり直しコストは桁が違う。

---

### AI-Ready設計を今入れるべき3つの判断

| 設計判断 | 今の追加コスト | やらなかった場合の後付けコスト |
|---|---|---|
| **① 時点データの保持設計** | ほぼゼロ。Delta Lake / Icebergは標準機能として時系列管理(Time Travel)を持つ。**製品選定で手に入る** | AI導入時にデータの蓄え直し。**1-2年の遅延** |
| **② 非構造データの取り込みレーン設計** | パイプライン設計時にテキスト・画像用のレーンを確保する。初期実装は後回し可 | パイプライン全体の再設計。**構築費用の30-50%相当の追加** |
| **③ AI接続インターフェースの確保** | SQL API / MCP対応を製品選定の評価軸に入れるだけ。**追加コストゼロ** | AIツールごとにカスタム接続を構築。**ユースケース数 × 開発コスト** |

### 投資判断の整理

```
問われている判断:

  ✕ 「AIに投資するかどうか」
  ○ 「今のFoundry移行の設計にAI-Readyの思想を入れるかどうか」

  追加される作業:
    ・製品選定の評価軸に4原則を追加する
    ・パイプライン設計に非構造データレーンを入れる
    ・基盤構築と並行してPoCを1-2個走らせる

  追加されないもの:
    ・AI基盤の全面構築（それは段階的にやればいい）
    ・AIモデルの本格開発（それはユースケースが固まってから）
    ・大規模な追加予算（設計判断の話であり、別建てのシステム構築ではない）
```

## P10: 課題① — ユースケースから逆算したデータ設計がない

**キーメッセージ**: 「何のAIを作るか」が決まっていないから「何のデータが必要か」も設計されていない。だが、確実に必要なデータ設計パターンはある。

---

### 問題の構造

```
現在のデータ設計の考え方:
  「今のBIダッシュボードに必要なデータ」を集めて加工する
  → 「今月の稼働率」「今の人員配置」= 今の値があれば十分

AIが必要とするデータ:
  「3ヶ月前の時点で、あの時知り得た情報だけを使って、今月を予測できたか？」
  → 過去のあらゆる時点での値が必要（Point-in-Time Correctness）
  → 今のDWH設計では、過去のスナップショットが残らない
```

### 具体例: 「スタッフの離職を3ヶ月前に予測したい」

```
必要なデータ（3ヶ月前の時点の値）:
  ・その時点での残業時間（直近90日平均）
  ・その時点での有休消化率
  ・その時点での施設稼働率
  ・その時点での面談記録

問題:
  BIダッシュボードは「今」が見えればいいので、
  これらの「3ヶ月前の時点の値」は上書きされて消えている。
  → 今日から溜め始めても、予測に十分なデータが溜まるには1年以上かかる。
```

### ユースケースが未確定でも必要な設計パターン

介護AIのデファクトスタンダードはまだ存在しない。だが以下は共通して必要:

| 設計パターン | なぜ普遍的に必要か | 製品選定への影響 |
|---|---|---|
| 時点スナップショット保持 | どんな予測AIも「過去の時点の状態」で学習する | Delta Lake / Iceberg（Time Travel機能）を持つ基盤 |
| イベントログの蓄積 | 「何が起きたか」のログがAI学習の最も基本的な入力 | Append-only のログテーブル設計 |
| 特徴量計算の分離 | BI用の集計とAI用の特徴量計算は異なる | dbt等でBI用とAI用のtransformationを分離 |

---

## P11: 課題② — データの繋ぎ込みはツールだけでは解決しない

**キーメッセージ**: データを「運ぶ」ことはツールで解決できる。だが「何を運ぶか」「運んだ先でどう構造化するか」「ソースシステムとの契約」はドメイン知識と設計判断が必須。

---

### ツールで解決できること / できないこと

```
ツールで解決できる（RFPでベンダーに任せられる）:
  ・RDB/API → DWH のデータ転送（Fivetran, Airbyte, AWS DMS等）
  ・定期バッチのスケジューリング
  ・標準的なファイル連携

ツールでは解決できない（先方と一緒に設計する領域）:
  ・ソースシステムのインターフェース確認
  ・テーブル構造の設計判断
  ・マスタデータの名寄せ
  ・スキーマ変更への耐性設計
  ・Foundryパイプライン・オントロジーの移行
```

### ソースシステムのインターフェース問題

介護業界の業務システム（ほのぼの、Axist、眠りSCAN、ナースコール）は、Fivetran/Airbyte等のコネクタが前提とするAPI/DB直接接続が提供されていないケースがある。CSV出力のみ、独自プロトコルのみの可能性がある。

**→ 先方に確認すべき: 各ソースシステムのエクスポート方法、粒度、現在Foundryがどう取り込んでいるか**

### マスタデータの名寄せ

```
異なるソースシステム間で同じ入居者・施設が異なるIDで管理されている:
  ほのぼの: 施設ID = "H001", 入居者ID = "R12345"
  Axist:   施設ID = "AX-Tokyo-01", 入居者ID = "2024-0123"
  眠りSCAN: デバイスID = "SC-A3F2", 入居者紐付け不明
  ナースコール: 居室番号 = "301", 入居者はナースが把握

→ 「施設Aの入居者Bのバイタルとナースコール記録を紐付ける」が
  マスタ名寄せなしでは不可能。全AIユースケースの前提条件。
```

RFIではPwCのみがMDM（マスタデータ管理）を明示的に提案。他4社は言及なし。

### データコントラクト — 2028年CareBase刷新への備え

2028年にソースシステムのスキーマが変わる。今のまま繋ぎ込みを作ると:

```
ソースのスキーマ変更 → ETLが壊れる → DWHが壊れる → BIが壊れる
→ 「誰も気づかないまま数字がズレている」最悪のパターン
```

**データコントラクト**: ソースと基盤の間で「約束」を定義する仕組み（必須カラム、データ型、NULLルール、値の範囲）。約束が破られたら自動検知・アラート。dbt schema tests / Great Expectations / Databricks DLT Expectations等で実装可能。

### Foundryオントロジー移行

Foundryのオントロジー（データ間の関係性定義）は、移行先で再現が最も困難な資産。**RFI 5社すべてがこの問題を直接的に扱っていない**。富士通のみ「現行エンジニアが在籍」と言っているが、オントロジーの再設計方針には触れていない。

**→ RFP時にオントロジー移行方針を明確に問うべき**

## P12: 課題③ — 非構造データの受け入れ基盤がない

**キーメッセージ**: SOMPOケアの最も価値あるデータ資産（介護記録、将来のカメラ・音声）は非構造データ。RFI 5社の提案に非構造データの取り込み設計は入っていない。

---

### 構造化データ vs 非構造データ

構造化データ（数字）は「何が起きたか」を記録する。
非構造データ（テキスト）は「なぜ起きたか」「どう対応したか」を記録する。

```
構造化データだけでわかること:
  「施設Aの転倒件数: 今月12件（前月比+5件）」

非構造データを加えるとわかること:
  「施設Aで転倒が増加。介護記録を分析すると:
   ・8件が夜間帯（23時-5時）に集中
   ・共通パターン: トイレに行こうとしてベッドから降りた際に転倒
   ・類似パターンが施設B, Cでも発生（→ 設備改修の検討を推奨）」
```

### 非構造データの処理は構造化ETLとは異なるパイプライン

```
テキストデータの処理工程:
  介護記録テキスト → PII検出・マスキング → LLMで構造化抽出(JSON)
  → ベクトル化（意味検索用） → ベクトルDB + メタデータDB に格納

  ※ dbt/PySparkとは異なるランタイムが必要:
    ・LLMベース変換パイプライン
    ・ベクトルDB（Databricks Vector Search / pgvector / OpenSearch）
    ・ストリーム処理（バイタル、ナースコール向け）
```

グループ基盤が構造化ETLを前提に設計されている場合（P7 差異②）、このレーンがアーキテクチャ上存在しない。後から追加する場合はパイプライン全体の設計見直しが発生する。

今の段階で必要なのは「テキスト・画像・音声を受け入れるレーンの設計」。初期実装は後回しでよい。**設計に入っていること**が重要。

## P13: 課題④ — 検証ナレッジが社内に蓄積しない

**キーメッセージ**: AI活用の最大の隠れコストは「毎回ゼロから始めること」。ベンダーが去ると環境もコードも消える。これは人の問題ではなく、基盤の設計問題。

---

### 今起きていること

```
AI検証プロジェクトの典型的な流れ:

  1. ベンダーAに検証を依頼
  2. ベンダーAが環境を構築（自社ツール or ローカル環境）
  3. ベンダーAが検証実施、コード作成
  4. ベンダーAがレポートを納品
  5. ベンダーAが去る
     → 環境は撤去される
     → コードはzip で受領（動かし方は不明）
     → 途中の試行錯誤・失敗の記録は消える

  6. 半年後、ベンダーBに次の検証を依頼
  7. ベンダーBが環境を一から構築（ベンダーAの環境とは別物）
  8. ベンダーBが「前回何を試して、何がダメだったか」を知らない
  9. 同じ失敗を繰り返す可能性がある
  10. また最初から...
```

### 根本原因は「知識を残す場所がない」

```
  ・実験環境: ベンダーごとに異なる → 統一された実験基盤がない
  ・コード管理: zip、ローカル散在 → バージョン管理されたリポジトリがない
  ・プロンプト: 個人のメモ帳 → プロンプト管理基盤がない
  ・評価結果: レポートのPDF → 再実行可能な評価パイプラインがない
  ・意思決定の記録: 人の記憶 → 実験ログ・比較結果のDBがない
```

基盤が「ベンダーが変わっても学びが残る」構造を持つべき。これは原則②で詳述する。

## P14: 課題⑤⑥ — ガバナンスの都度構築 / SSOTとAI出力信頼性

**キーメッセージ**: ガバナンスの都度構築はAI活用のスピードを殺す。SSOTがないとAIは「もっともらしい嘘」を量産する。

---

### 課題⑤: ガバナンスの都度構築

```
新しいAIツール/サービスを導入するたびに:
  □ セキュリティチェック → 一から実施（数週間）
  □ 権限設計 → 一から設計
  □ 個人情報の取り扱い → 要配慮個人情報の範囲を一から確認
  □ 監査ログ → ツールごとにバラバラの形式

  → AIツールが増えるほど線形にコスト増加
  → 「新しいAIは試すのが大変」→ 活用が進まない
```

**ただし注意**: 安全性だけを最優先にすると「リスクがあるからAIは使わない」が最適解になる。
2040年に69万人の介護人材不足は構造的に解決しない。AIの業務価値はリスクを大幅に上回る可能性がある。**ガバナンスは「防壁」ではなく「安全に攻めるための装備」と位置づけるべき。**

### 課題⑥: SSOTの不在とAI出力信頼性

```
セマンティックレイヤーがない状態でAIが稼働すると:
  社長:   「全社の稼働率は？」→ AI: 「87.2%」（空室含む計算）
  施設長: 「うちの稼働率は？」→ AI: 「83.5%」（空室除く計算）
  → 経営会議で「AIの数字が信用できない」→ AI活用が頓挫する
```

Foundryのオントロジーはこの問題に部分的に対処していた。**移行時にこの資産を「捨てずに引き継ぐ」設計が不可欠**。

**SSOTのアーキテクチャ**: Gold Layer（公式データ = SSOT、READ ONLY）とSandbox Layer（個人探索領域）を明確に分離する。Gold Layerの定義変更はPRレビュー + 承認フロー。AI出力には信頼度レベル（公式KPI / 定型分析 / アドホック分析 / 推論・予測）を付与する。

## P16: 原則① — AIと対話できるインターフェースを持つ基盤

**キーメッセージ**: 基盤がAIと接続する標準インターフェースを持っているか。これがないと、AIユースケースのたびにカスタム接続が必要になる。

---

### 4つのインターフェース

| インターフェース | 役割 | 製品例 |
|---|---|---|
| **SQL API** | AIがプログラムからSQLを投げてデータを取得する口 | Databricks SQL Warehouse / Snowflake SQL API / Redshift Data API |
| **MCP (Model Context Protocol)** | 外部AIツールからデータ基盤にアクセスする新興規格（2024年末〜） | Tableau MCP, Databricks MCP, Data 360 MCP（全てOSS公開済。ただしエンタープライズ本番運用の実績はまだ限定的） |
| **セマンティックモデル** | AIが「稼働率とは何か」を参照できる定義集 | Databricks AI-BI Genie / Snowflake Cortex Analyst のセマンティックモデル |
| **マルチモデル対応** | 複数のAIプロバイダ（Claude, GPT, Gemini等）を切り替えて使える | AWS Bedrock（複数モデルを統一APIで提供）|

### マルチモデル対応の重要性

特定のAIプロバイダに依存すると、価格改定・サービス停止・性能変化のリスクを直接受ける。AWS上でもClaude/GPT/Geminiは全てBedrock経由で利用可能だが、**各モデルのSDK統合度、レイテンシ、Fine-tuning対応には差がある**。製品選定時に、複数モデルの切り替えが容易な構成かを確認する必要がある。

### RFP評価軸

| 評価項目 | 確認すべきこと |
|---------|-------------|
| SQL API（**最重要**） | 外部プログラムからSQL実行可能か。レイテンシは実用的か |
| MCP対応（**加点要素**） | MCP Server/Clientが提供されているか。SQL APIが第一インターフェースであり、MCPは将来の拡張性として評価 |
| セマンティックモデル | ビジネスメトリクスの定義を構造化して管理・参照できるか |
| 権限の引き継ぎ | AI経由のアクセスでもユーザー権限が適用されるか |
| マルチモデル | 複数LLMプロバイダの切り替えが容易か |

### セマンティックレイヤーの実装に関する率直な注意

セマンティックレイヤーは本資料で繰り返し重要性を述べているが、**実装の難度とコストを過小評価してはならない**。

```
技術的な実装（=比較的容易）:
  ・Databricks AI-BI GenieやSnowflake Cortex Analystのセマンティックモデルを定義する
  ・YAML/SQLでKPIの計算ロジックを記述する

本当に難しいのは（=組織的な合意形成）:
  ・「稼働率にショートステイを含めるか」の部門間合意
  ・既存BIダッシュボードの定義との整合（矛盾があると現場が混乱）
  ・定義変更のガバナンス（誰が承認し、いつ反映するか）
  ・継続的なメンテナンス（新KPIの追加、定義の改訂）
  → セマンティックモデルの初期構築と継続メンテが最大のボトルネックになる
```

**→ RFP評価時に「セマンティックモデルの初期構築支援と継続メンテの運用設計」をベンダーに問うべき**

## P17: 原則② — 組織の学習が蓄積する基盤

**キーメッセージ**: AI開発で最も高コストなのは「同じ失敗を繰り返すこと」。実験・コード・プロンプト・評価結果が蓄積・検索・再利用できる基盤を選ぶ。

---

### 蓄積すべき「学習資産」

| 資産 | 今の状態 | あるべき姿 |
|------|---------|-----------|
| **実験記録** | ベンダーのローカルに散在 | MLflow等で統一管理。パラメータ・結果・比較が一覧可能 |
| **コード** | zip受領、動かし方不明 | Gitリポジトリでバージョン管理。CI/CDつき |
| **プロンプト** | 個人のメモ帳 | プロンプト管理基盤でバージョン管理・A/Bテスト |
| **評価結果** | レポートのPDF | ゴールデンセット + 自動評価パイプライン。再実行可能 |
| **モデル** | 納品されたバイナリ | Model Registryで管理。デプロイ・ロールバック可能 |

### これが意味すること

```
プロジェクト1（ベンダーA）:
  実験: 5回 → 結果: モデル精度75% → 記録: MLflowに保存

プロジェクト2（ベンダーB、半年後）:
  ベンダーBが着手時にMLflowを見る
  → 「前回はアプローチXで精度75%。Yはうまくいかなかった」
  → ゼロからではなく、75%の地点から改善をスタートできる
```

### RFP評価軸

| 評価項目 | 確認すべきこと |
|---------|-------------|
| 実験管理 | MLflow等の実験トラッキングが統合されているか |
| モデル管理 | Model Registry でバージョン管理・デプロイが可能か |
| コード管理 | Git連携が標準機能か |
| 評価基盤 | 自動テスト・品質モニタリングの仕組みがあるか |

## P18: 原則③ — ガバナンスが拡張可能な基盤

**キーメッセージ**: 一度作った統制が新しいAIにも自動適用される設計を選ぶ。ガバナンスは「AIを止める仕組み」ではなく「AIを安全に増やせる仕組み」。

---

### ガバナンスの3レイヤー

```
Layer 1: データアクセス制御
  ・行レベルセキュリティ（施設長は自施設のデータのみ）
  ・列レベルセキュリティ（給与データは人事部のみ）
  ・AIがデータを代理取得する際も、利用者の権限を自動適用
  ・LLM経由のクエリはSELECTのみに制限

Layer 2: AI固有の統制
  ・入力チェック: プロンプトインジェクションの検知
  ・出力チェック: 個人情報・機密情報の漏出防止
  ・行動記録: AIがどのデータを見て何を出力したか全記録
  ・人間承認: 重要な判断はAI単独で実行させない

Layer 3: 監査・説明責任
  ・全AIアクションの監査ログ
  ・AI出力のバージョン管理と追跡可能性
  ・バイアス評価フレームワーク
```

### 要配慮個人情報への対応

介護データには要配慮個人情報（病歴、障害等）が含まれる。さらにAIが「この入居者は○○病の可能性が高い」と**推論で新たな要配慮個人情報を生成する**リスクがある。AI出力に対するデータ分類・制御の設計が必要。

### RFP評価軸

| 評価項目 | 確認すべきこと |
|---------|-------------|
| 統合カタログ | Unity Catalog等でアクセス制御・分類・リネージが一元管理されるか |
| 権限粒度 | 行・列レベルのセキュリティがAI経由アクセスにも適用されるか |
| 監査ログ | AIを含む全アクセスの監査ログが自動生成されるか |
| データ分類 | 要配慮個人情報のラベリングと、ラベルに基づくAI制御が可能か |

## P19: 原則④ — ナレッジが社内に残る基盤

**キーメッセージ**: 構築はベンダーと一緒にやる。だが運用のナレッジがベンダーだけに留まると、判断のたびにベンダー待ちになり基盤が止まる。段階的に社内でできることを増やしていく設計が必要。

---

### よくある落とし穴: 「ベンダーしか知らない」問題

多くの企業でデータ基盤導入後に起きる典型的な失敗パターン:

```
Year 1: ベンダーが構築。社内は「お任せ」
Year 2: ベンダーの保守契約で運用。社内は「使うだけ」
Year 3: KPI定義を変えたい → ベンダーに依頼 → 2週間待ち
        パイプラインが壊れた → ベンダーに連絡 → 原因調査に3日
        新しいAI検証をしたい → ベンダーにデータ準備依頼 → 1ヶ月待ち
Year 4: 「基盤が使いづらい」→ 現場がExcelに戻る → 基盤投資が無駄になる
```

**根本原因は「ナレッジがベンダー側にしかない」こと。** 設定の意味、パイプラインの依存関係、KPI定義の経緯——これらが社内に残らないと、ベンダーが変わるたびにゼロリセットになる。

### データ基盤の運用には日常的にビジネス判断が発生する

| 判断の種類 | 頻度 | 具体例 |
|---|---|---|
| データアクセス承認 | 週次 | 「DDA推進部がナースコールデータを新分析に使いたい」 |
| KPI定義変更 | 月次 | 「稼働率の計算にショートステイを含めるか」の方針変更 |
| パイプライン障害対応 | 不定期 | 「昨夜のバッチが失敗してダッシュボードが更新されていない」 |
| PoC用データ提供 | 月次 | 「AI検証のためにマスク済みデータセットを作りたい」 |
| AI出力の品質判断 | 日常 | 「AIの予測精度が先月から落ちている。モデル更新するか」 |

**これらの判断を毎回ベンダーに回していたら、何も動かない。**

### 段階的な自走ロードマップ

初日から全てを社内で回す必要はない。**ベンダーと一緒に始めて、段階的に社内でできることを増やす。**

```
Phase 1（構築〜半年）: ベンダー主導、社内は学習
  ベンダーが構築・運用。社内メンバーがペアで作業し、設計意図とノウハウを吸収
  → 重要: 「なぜこう設計したか」のドキュメントを社内に残す

Phase 2（半年〜1年）: 日常運用を社内に移管
  KPI定義変更、権限管理、パイプライン監視は社内で実施
  ベンダーは「聞けば答えてくれる」サポート体制に移行

Phase 3（1年〜）: 社内が主導、ベンダーは大規模改修のみ
  新規ユースケースの要件定義も社内主導で実施
  ベンダーには新規データソース接続やアーキテクチャ変更のみ依頼
```

### 製品選定で意識すべきこと

## P20: 4原則の根拠 — 陳腐化しないものに投資する

**キーメッセージ**: 技術は半年で世代交代する。だが4原則が守るものは技術世代を超えて残り続ける。

---

### 陳腐化するもの / しないもの

| 陳腐化する（技術に依存） | 陳腐化しない（4原則が守る） |
|---|---|
| 特定のLLMモデル（GPT-4 → GPT-5 → ...） | データの蓄積（1,000施設 × 年数 = 不変の資産） |
| 特定のGUIツール | データ定義・セマンティックレイヤー（「稼働率とは何か」は技術に依存しない） |
| 特定のライブラリ/SDK | ガバナンスの設計思想（権限・監査・SSOT） |
| 特定のクラウドサービス機能 | 組織のナレッジ蓄積（実験記録・評価結果・意思決定ログ） |
| | 社内に蓄積されたナレッジと運用能力 |
| | オープンフォーマット（Delta Lake / Iceberg / Parquet） |

### 製品が変わっても持ち出せるもの

技術スタックは5-10年後に入れ替わる可能性がある。その時に:
- **セマンティックレイヤーの定義**: YAML/SQLで書かれていれば、別の基盤に移行可能
- **dbt models**: SQL標準で書かれたtransformationは、実行エンジンが変わっても動く
- **MLflow実験記録**: オープンソース。特定ベンダーに紐付かない
- **Delta Lake / Parquet**: オープンフォーマット。どの基盤からも読める

→ **オープンフォーマット・オープンソースを優先する**ことで、データ層のベンダーロックインを回避。

### 注意: データフォーマットの可搬性 ≠ プラットフォームの可搬性

Delta Lake/Parquetで**データ自体**は持ち出せる。だがプラットフォーム移行時には以下も再構築が必要:
- Unity Catalog等のメタデータ・アクセス制御設定
- Databricks Workflows等のジョブ定義・スケジュール
- MLflow Modelのデプロイ設定

**データのポータビリティは確保できるが、「基盤丸ごとの移行」が容易になるわけではない。** オープンフォーマットの採用は「5-10年後の次の移行の選択肢を狭めない」ための保険であり、移行コストをゼロにするものではない。

### ACESとしての仮説

これらの事例と、SOMPOケアのデータ資産（1,000施設 × 構造+非構造データ）を踏まえ、我々は以下を仮説として持っている:

1. **最もインパクトが高いのは「施設横断のパターン検出」**。1施設では気づけない異常パターンを、1,000施設のデータで検出する。これはSOMPOケアの規模でしか実現できない
2. **Phase 1は構造化データでのKPI照会から始めるべき**。非構造データの活用は技術的・組織的ハードルが高い。まず「基盤がAIにとって使いやすいか」を構造化データで検証する
3. **非構造データ（介護記録テキスト）の活用がPhase 2以降の本命**。ただし340GBの介護記録の内訳・品質・構造の事前調査が不可欠（Next Stepsに記載）

```
だから:
  ・基盤を作ってから試すのでは遅い。基盤構築と並行して試す
  ・PoCは技術検証だけでなく、組織・プロセス・ドキュメント整備の検証でもある
```

### HD基盤の完成を待つリスク

```
「HDのAI-Readyデータプラットフォームを待てばよい」への反論:

待っている間に失われるもの:
  ① データの蓄積期間: 今日から溜め始めないと1年後もデータがない
  ② PoC経験: 組織としてAIを扱う経験値が貯まらない
  ③ 要件定義の主導権: 待つと「HD/SJの要件に合わせる」ことになる
  ④ ケア固有要件が埋もれる:
     介護記録テキスト分析、要配慮個人情報の扱い、カメラ映像・音声データ
     → これらはSJにはない要件。HD基盤で優先的に対応される保証がない

  ・ケア固有のデータモデル設計（L2）
  ・セマンティックレイヤーの定義（L2）
  ・PoC環境の確保とデータマスキング設計（L3）
  ・アクセス制御ポリシーの設計（L2）
  → これらはHD基盤がどうなろうと必要な作業であり、待つ理由がない
```

### PoCのスコープ — 技術だけでなく組織を検証する

```

  技術面:

    ・セマンティックレイヤーは正しく機能するか

  組織・プロセス面（これが見落とされがち）:

    ・結果をどうドキュメント化し、次のPoCに繋げるか

推奨:
  一気に本番化までを1ユースケースで走らせて課題を洗い出す方が、
```

### Phase 1 PoCの推奨

**「自然言語でのKPI照会」を最初のPoCに推奨する理由**:

3. **現場に見せやすい**: 「こういうことができるようになる」のデモとして最もわかりやすい
4. **組織面の検証も可能**: データアクセス承認、マスキング、運用フローを一通り試せる

→ 検証すべきは「AIの精度」ではなく「**基盤がAIにとって使いやすい設計になっているか**」と「**組織として運用が回るか**」。

|---|---|---|

| **組織プロセス** | データアクセス承認 → マスキング → PoC環境構築が実用的な速度で回ったか | 2週間以内 |

## P24: データ飛輪 — 使い続けることで競争優位が蓄積する

**キーメッセージ**: AIは使えば使うほど賢くなる。1,000施設 × 数年分のデータは、後発が追いつけない競争優位。だがそのデータ飛輪を回すのは基盤の設計。

---

### データ飛輪の構造

```
  基盤にデータが蓄積される
        │
        ▼
  AIがより正確な分析・予測をする ←── フィードバックが蓄積される
        │                                    ↑
        ▼                                    │
  現場がAIを使い続ける ──────────────────────┘
```

**介護ドメインで飛輪が回るシナリオ**:

```
例: 転倒予防パターンの全社展開

  施設Aで転倒が多い → 介護記録 + バイタル + ナースコールを分析
  → 「夜間23-5時のトイレ移動時に転倒リスクが高い入居者の特徴」をAIが抽出
  → 類似パターンの入居者が施設B-Zにも存在することを検出
  → 予防介入（巡回頻度変更、センサー設置）を全社展開
  → 全社の転倒率が低下 → その結果データがさらにAIを改善
  → 1施設だけでは不可能。1,000施設の規模でのみ成立する飛輪
```

### SOMPOケアのポテンシャル

1,000+ 施設 × 数年分の稼働率推移、スタッフの勤務パターン、介護記録の文脈、ケアプランとアウトカムの関係 → **SOMPOケアだけが持てるデータ資産**。後発が同じデータを集めるには「同じ年数」がかかる。

**だがこの飛輪は、2つの前提条件を満たさないと回り始めない**:

前提条件①: **基盤の設計**
- AIの出力ログの保存設計（予測・回答を「データ」として書き戻す）
- ユーザーフィードバックの記録（「役に立った/立たなかった」を蓄積）
- 再学習パイプライン（蓄積されたフィードバックをAI改善に使う仕組み）
- → 後から追加可能だが、**「ログを取っていなかった期間」のデータは永久に失われる**

前提条件②: **記録品質の標準化**
- 1,000施設のデータ量があっても、記録の品質（粒度、正確性、入力率）が施設間でバラバラなら飛輪は回らない
- データ量よりも**データの質**がボトルネックになることが多い
- → 記録品質の標準化・モニタリングも基盤設計の一部として考慮すべき
