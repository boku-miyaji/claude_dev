1. Palantir Foundry を理解する

Palantir Foundryは、Palantir Technologies社が提供するエンタープライズ向けデータ統合・分析プラットフォームです。元々は政府・国防向けだった技術を民間向けに展開したもので、「データを繋いで、分析して、意思決定に使う」を一気通貫で行える点が特徴です。

アーキテクチャの全体像

Foundryは大きく5つのレイヤーで構成されています。一つのプラットフォームで「取り込み → 加工 → モデリング → 分析 → アプリ化」を全てカバーするのが設計思想です。

── Foundry のレイヤー構造 ── Layer 1: Data Connection（データ取り込み） ・外部DB、API、ファイル、ストリームからデータを吸い上げる ・Magritte（コネクタ基盤）でRDB/CSV/API等に対応 ・コネクタは独自仕様 → 汎用コネクタ（Fivetran等）とは非互換 Layer 2: Data Pipeline（データ加工） ・Code Repositories: PySpark/SQLでパイプライン定義 ・Transforms: Foundry独自ラッパーでSpark処理を抽象化 ・DAG（有向非巡回グラフ）でパイプライン依存関係を自動管理 ・データセットは全て不変（Immutable）＝ 上書き不可、常にバージョン管理 Layer 3: Ontology（オントロジー） ← 最もFoundry固有で移行困難 ・データの「意味」と「関係性」を定義するセマンティックレイヤー ・Object Types（入居者、施設、ケアプラン等）を定義 ・Link Types（入居者→施設、入居者→ケアプラン等）で関係を定義 ・Action Types（入居者を移動、ケアプランを更新等）で操作を定義 ・「稼働率 = 入居者数 ÷ 定員」のようなビジネスロジックを一元管理 Layer 4: Analytics & Visualization（分析・可視化） ・Contour: ドラッグ&ドロップの分析ツール ・Quiver: グラフ・チャート作成 ・Notepad: ドキュメント + インラインデータ Layer 5: Operational Apps（業務アプリ） ・Workshop: ノーコードでオントロジーベースのアプリを構築 ・Slate: カスタムWebアプリ構築（より自由度が高い） ・SOMPOケアの4モジュール（CMC、モニター、ナースコール、施設360）はここで稼働

Foundry の強み・弱み

強み

  • オールインワン: 取り込み→加工→分析→アプリが1プラットフォーム
  • オントロジー: データの意味・関係を一元的に定義でき、全アプリで共有
  • データリネージ: 全データの来歴を自動追跡。「この数字はどこから来た？」に即答
  • セキュリティ: 政府機関向けに鍛えられた堅牢なアクセス制御
  • Branching: パイプラインのGitライクなブランチ管理

弱み

  • コスト: ライセンス費用が高額。ユーザー数に比例して増加
  • ベンダーロックイン: 独自概念が多く、他基盤への移行が困難
  • エコシステム: OSS/第三者ツールとの統合が限定的
  • 透明性: 内部処理がブラックボックス化しやすい
  • UIの頻繁な変更: SaaS型リリースで現場が混乱
  • AI/LLM対応: 2025時点ではLLM統合がDatabricks等に比べ後発

ロックイン要素（移行が困難なもの）

最重要: Foundry固有のロックイン要素 以下の要素はFoundry独自の概念であり、1対1で他プラットフォームにマッピングできない。移行時に「再設計」が必要。

補足: オントロジーのバックエンド — 裏側で何が動いているか

オントロジーは「概念の定義」だけでなく、裏側に 3つのバックエンドサービス が動いている。これを理解すると、なぜ移行が難しいかがよりクリアになる。

1. Ontology Metadata Service (OMS) — 「辞書」の役割
  「入居者とは何か」「施設とは何か」「入居者と施設はどう関連するか」の定義そのものを管理。
  RDBで言うとテーブル定義（DDL）に近いが、ビジネスの意味（「稼働率 = 入居者数÷定員」等）も含む。
  → 移行時: Unity Catalog のメタデータ + dbt のセマンティックレイヤーで部分再現

2. Object Data Funnel — 「データの入口」の役割
  CareBaseやほのぼの等のソースシステムからデータを吸い上げ、OMS の定義に従ってオントロジーに書き込む。
  「生データ→ビジネスオブジェクト」への変換もここで行われる。
  → 移行時: ETLパイプライン（PySpark + dbt）で再構築。ロジック自体は流用可能

3. Object Set Service (OSS) — 「データの出口」の役割
  Workshop アプリやContour分析ツールが「入居者一覧」「施設Aの全データ」を取得するときの読み取りAPI。
  権限制御もここで行われる（「この人は施設Aのデータしか見れない」等）。
  → 移行時: Unity Catalog の行/列レベルセキュリティ + SQL API で代替

つまり、オントロジーの移行とは「辞書（OMS）の再定義 + 入口（Funnel）のETL再構築 + 出口（OSS）の権限/API再設計」の3つをまとめてやる必要があるということ。だから最も困難。

※ 2026年6月にObject Storage V1 (Phonograph)が廃止予定。V2への移行がPalantir内部で進行中。移行タイミングと重なるため要注意。

理解度チェック Q1

Foundryの「オントロジー」とは何か？なぜ移行が最も困難なのか？

オントロジーは「データの意味と関係性の定義」。入居者・施設・ケアプランなどの「概念（Object Type）」と、それらの「関係（Link Type）」、さらに「操作（Action Type）」を定義する。全アプリがこの定義を参照するため、Foundryの最大の資産かつ最大のロックイン要素。移行先には1対1の互換概念がないため「再設計」が必要。Databricksの場合、Unity Catalog（メタデータ管理）+ セマンティックレイヤー（AI-BI Genie等）+ dbt models（変換ロジック）を組み合わせて部分的に再現する。

2. AWS + Databricks 基盤を理解する

Databricks の主要コンポーネント

Databricksは「レイクハウス」というアーキテクチャを提唱する企業です。データレイク（安い大容量ストレージ）とデータウェアハウス（高速なクエリエンジン）の長所を1つのプラットフォームで統合します。

Delta Lake

S3上のファイル（Parquet）にトランザクション機能を追加するストレージレイヤー。ACIDトランザクション、Time Travel（過去の時点のデータに遡れる）、スキーマ進化をサポート。

ストレージ

Unity Catalog

データ・AI資産の統合ガバナンス基盤。テーブル、ML モデル、ファイルを1つのカタログで管理。行/列レベルのアクセス制御、データリネージ、監査ログを提供。

ガバナンス

SQL Warehouse

SQL クエリ専用のサーバーレスコンピュート。BIツールやAIからのSQLアクセスに最適化。自動スケーリングで使った分だけ課金。

コンピュート

Databricks Workflows

ETL/ML パイプラインのジョブスケジューラ。ノートブック、Pythonスクリプト、dbt、JAR等のタスクをDAGで定義・実行。

オーケストレーション

MLflow

ML実験のライフサイクル管理（OSS）。実験トラッキング、モデルレジストリ、デプロイメントを統合管理。LLM評価にも対応。

AI/ML

AI-BI Genie

自然言語でデータに質問できるBI機能。セマンティックモデルを参照し、SQLを自動生成。「稼働率が低い施設は？」→即回答。

AI-BI

Delta Live Tables (DLT)

宣言的パイプラインフレームワーク。「このテーブルはこう定義」と書けば、依存関係・データ品質チェック・エラー処理を自動化。

ETL

Vector Search

ベクトル検索エンジン。テキストの意味検索（RAG）に使用。介護記録の類似検索などに活用可能。

AI/ML

Mosaic AI

LLMのファインチューニング、サービング、評価を統合。Compound AI Systems（複数AIの連携）の構築基盤。AI Gateway（複数LLMへの統一API）、Agent Framework、Agent Bricks（タスク記述だけでエージェント自動構築）も含む。

GenAI

AWS のデータ基盤関連サービス

AWSは「サービスを組み合わせて基盤を作る」思想。それぞれが独立したサービスで、自由に選択・組み合わせできる反面、全体の設計は自分で行う必要があります。

データ移行の技術的課題

フォーマット変換

Foundry内部データ → S3 (Parquet/Delta) への変換。Foundryのエクスポート機能でCSV/Parquetで取り出し、S3に配置してDelta Lakeとして登録する流れ。データ量が大きい場合（眠りSCAN 1.92TB等）はバッチ転送の設計が必要。

スキーマ管理

Foundryのスキーマ定義をDelta Lakeのスキーマに変換。Delta Lakeはスキーマ進化（Schema Evolution）をサポートするため、カラム追加は非破壊的に行える。2028年CareBase刷新に備え、データコントラクト設計が重要。

データ品質

移行時のデータ整合性検証が最大の工数。行数・集計値の一致確認、NULL/欠損値の扱い、日時フォーマット、文字コード。DLT Expectations や Great Expectations でデータ品質チェックを自動化。

マスタデータ名寄せ

ほのぼの・Axist・眠りSCAN・ナースコール間で同じ入居者/施設が異なるIDで管理されている。移行のタイミングでMDM（マスタデータ管理）を設計すべき。RFI ではPwCのみが明示的に提案。

セキュリティ・ガバナンスの移行

介護データ特有のセキュリティ要件 介護データには要配慮個人情報（病歴、障害等）が含まれる。さらにAI推論結果（「認知症リスク高」等）自体が新たな要配慮個人情報になる可能性がある。アクセス制御は最大4次元（施設×データ種別×時間×AI経由）で設計する必要がある。

クラスタの自動停止、Serverless SQL Warehouse の活用でコスト最適化。Foundryからの移行でコスト構造が透明化するのは大きなメリット。

学習コストは上がるが、汎用的なスキルが身につく。Foundry固有スキルは社外で使えないが、Databricks/AWSスキルは市場価値がある。

4. 議論ポイント・意思決定フレームワーク

Databricks vs Snowflake vs AWS Native — 3構成の全体像

移行先の基盤構成は大きく3パターンに分かれます。それぞれの構成図・特徴・向いている組織・SOMPOケアへの適合度を整理します。

パターン A: Databricks on AWS（推奨）

── 構成図 ── ユーザー層 施設長・本部スタッフ → Tableau（BIダッシュボード） データ分析者 → Databricks SQL / Notebooks AI/LLM → AI-BI Genie / Mosaic AI 統合管理層 Unity Catalog ── メタデータ / 権限(RLS/CLS) / リネージ / 監査ログ → テーブル、MLモデル、ベクトルIndex、LLMエンドポイントを一元管理 → 1つのカタログで全資産を統治 処理層 ETL: PySpark（Foundryから移植）+ dbt Core（SQL変換） ジョブ管理: Lakeflow Jobs（サーバーレス、DAG定義） 品質管理: DLT Expectations（データ品質チェック自動化） ML: MLflow（実験管理）+ Model Serving GenAI: Mosaic AI + Vector Search + Bedrock(LLM) ストレージ層 Delta Lake on Amazon S3 フォーマット: Delta + UniForm（Iceberg互換メタデータ自動生成） 階層: Bronze → Silver → Gold（メダリオン） インフラ層（AWS） VPC / Direct Connect / IAM / KMS / CloudTrail / CloudWatch

パターンAの特徴

  • 統合プラットフォーム: ETL→DWH→ML→AIが1つのワークスペース内で完結
  • PySpark互換: Foundryの@transform_dfからデコレーター除去だけでそのまま動作（PDF A-8）
  • ガバナンス統合: Unity Catalogでデータ〜AIまでを1つのカタログで管理
  • OSS基盤: Delta Lake, MLflow, dbt がコアでロックイン軽減

なぜSOMPOケアに最適か（根拠5つ）

  1. 移植コスト最小: 219本のPySparkパイプラインがほぼそのまま動く。Snowflakeだと全面書き換え（Snowpark）が必要
  2. Foundryに最も近い体験: 統合PFなのでFoundryユーザーの学習曲線が緩やか
  3. AI-Ready: MLflow+Vector Search+Mosaic AIが標準統合。将来のAI活用にそのまま拡張
  4. 運用人材2名で回る: 分散管理不要。Snowflakeは1-2名(SQL)、AWS Nativeは3-4名(インフラ+データ)
  5. RFI 2社が提案: Deloitte(有力候補)とFujitsuがDatabricks構成。実績ある構成パターン

パターン B: Snowflake on AWS

── 構成図 ── ユーザー層 施設長・本部 → Tableau / Power BI データ分析者 → Snowflake SQL（Snowsightエディタ） AI/LLM → Cortex Analyst（限定的） 管理層 Snowflake Access Control ── RBAC / RLS / マスキング + Horizon（ガバナンス、Data Clean Rooms） → ML モデル管理は別途必要（Snowflake単体では不十分） 処理層 ETL: Snowpark（Python） ← PySpark全面書き換えが必要 ジョブ: Snowflake Tasks（シンプルだが機能限定） ML: Cortex ML（Databricks MLflowに比べ限定的） GenAI: Cortex LLM Functions（Claude/Llama等、ただしモデル選択肢少） ストレージ層 Snowflake内部ストレージ（独自形式 + Apache Polaris経由でIceberg） → データエクスポート時にコスト発生（Egress課金） → S3にもデータを持つ場合は2箇所管理になる

パターンBの特徴

  • SQL中心: SQL操作だけで完結する範囲が広い。学習コスト低
  • 最速のSQL性能: OLAP（分析クエリ）でDatabricksより高速な場面あり
  • フルマネージド: インフラ管理ゼロ。運用負荷が最小
  • データシェアリング: 組織間のセキュアなデータ共有が強い

SOMPOケアで不利な理由（根拠4つ）

  1. PySpark全面書き換え: 219本のパイプラインをSnowpark(Python)に変換。RDD/UDF/ML全てが書き換え対象。コスト大
  2. AI/ML統合が弱い: Cortex MLは発展途上。MLflow相当の実験管理がない。Vector Search非提供
  3. 独自形式のロックイン: データエクスポートにコスト発生。Apache Polaris経由のIceberg対応は進行中だが未成熟
  4. 非構造データ: 介護記録テキスト340GBのLLMパイプラインをSnowflake単体で構築するのは現実的でない

パターン C: AWS Native（Glue + Redshift + Athena）

── 構成図 ── ユーザー層 施設長・本部 → QuickSight（AWS純正BI、Tableauより機能限定） データ分析者 → Athena（S3直接クエリ）/ Redshift（DWH） AI/LLM → Bedrock + SageMaker（別サービス） 管理層（分散） ← 統合カタログがない。各サービスで個別管理 データカタログ: Glue Data Catalog 権限管理: Lake Formation 監査: CloudTrail → 3つのサービスを横断的に設定。統合ビューがない 処理層（6-8サービスを組み合わせ） ETL: AWS Glue（PySpark）or EMR（大規模Spark） SQL: Redshift（DWH）+ Athena（アドホック） ジョブ: Step Functions + Lambda（複雑なDAG構築が必要） ML: SageMaker（独立サービス、データ取り出しが必要） GenAI: Bedrock（マルチLLM、これは強み） ストレージ層 S3 + Iceberg or Parquet → OSSベースでロックインは最小 → ただし統合カタログ不在のため管理が煩雑

パターンCの特徴

  • 最安構成が可能: 各サービスをサーバーレスで使えば従量課金のみ
  • OSSベース: S3+Iceberg+Parquetで完全にオープン。ロックイン最小
  • Bedrock: マルチLLM対応はAWSの強み。Claude/GPT/Llama等を統一APIで利用
  • AWSスキルの汎用性: AWS認定資格等、市場での人材確保がしやすい

SOMPOケアで不利な理由（根拠4つ）

  1. 統合管理の欠如: Glue Catalog+Lake Formation+CloudTrailを個別設定。Unity Catalogのような統合ビューがなく障害切り分けが困難
  2. 運用人材3-4名必要: 各サービスの専門知識が必要。「全サービス横断の人材」が最も希少で確保困難
  3. 設計負荷: パイプラインの設計（Step Functions + Lambda + Glue の組み合わせ）が複雑。Foundryの統合体験から大きく後退
  4. PwCのみが提案: RFIでAWS Native構成を提案したのはPwCのみ。PwCはOSSを明示的に拒否しており、設計思想が保守的

メダリオンアーキテクチャ（Bronze/Silver/Gold）

Databricksが提唱するデータレイクハウスの設計パターン。データを3つの品質レイヤーに分けて管理します。

Bronze（ブロンズ）= 生データ ・ソースシステムからのデータをそのまま保存 ・変換なし。「来たまま」を記録 ・目的: 監査証跡、リプレイ可能性 ・例: CareBaseから抽出したCSVをParquetに変換してそのまま格納 Silver（シルバー）= クレンジング済みデータ ・Bronze を洗浄・型変換・正規化したもの ・重複排除、NULL処理、マスタ結合 ・ここで名寄せも実施（ほのぼのID ↔ AxistID ↔ 眠りSCAN ID） ・データサイエンティストが直接分析に使えるレベル ・例: 入居者テーブル（全ソース統合済み、スキーマ統一済み） Gold（ゴールド）= ビジネスKPI = SSOT（信頼できる唯一の情報源） ・Silver からビジネスKPIを集計したもの ・「稼働率」「人員配置率」等の公式定義がここに集約 ・BIダッシュボード、経営レポートはGoldからのみ参照 ・READ ONLY。定義変更はPRレビュー + 承認フロー ・例: 月次施設別稼働率テーブル（計算ロジック = dbt model で定義）
なぜメダリオンが重要か Foundryの現状課題「処理のブラックボックス化」「過去データとの不整合」を構造的に解決する。Bronze に生データが残るため、加工ミスがあっても遡って修正可能。Gold がSSOTとなることで「数字の定義が人によって違う」問題を排除。AIはGoldを参照するため、回答の一貫性が保証される。

メダリオンアーキテクチャのBronze/Silver/Goldの違いを、SOMPOケアのデータ（入居者情報）を具体例にして説明してください。

Bronze: ほのぼのからエクスポートされた入居者CSVをそのままParquet化。「施設ID=H001, 入居者名=山田太郎, 入居日=2024/01/15」— ソースの表記揺れ、NULL、重複もそのまま保存。
Silver: Bronzeを洗浄。施設IDをAxist・眠りSCANと名寄せして統合入居者IDを付与。入居日をISO8601に統一。NULLは業務ルールに従い補完。バイタルデータ（眠りSCAN）、ナースコールデータも入居者IDで結合可能に。
Gold: 「施設A 2026年3月 稼働率 = 87.3%」のようなKPIテーブル。計算定義（稼働率 = 入居者数÷定員、ショートステイ含む/含まないの定義）はdbtモデルでコード化。BIダッシュボードもAI-BI Genieもここを参照するため、「稼働率」の定義が全社で統一される。

5. 技術スタック採用方針

4原則（RFP評価の柱）

個別製品の比較ではなく、製品に依存しない設計原則を定め、RFPの評価軸とする。SOMPOケアPJTで策定済みの4原則:

原則 1: AIと対話できるインターフェース

基盤がAIと接続する標準的な口を持っているか。
SQL API（最重要）、MCP（加点要素）、セマンティックモデル、マルチモデル対応の4つのインターフェースを評価。

例: 施設長が「先月の稼働率が低い施設は？」と聞く → AI がSQL APIでGoldレイヤーを照会 → セマンティックモデルで「稼働率」の定義を参照 → 正確な回答

原則 2: 組織の学習が蓄積する基盤

実験・コード・プロンプト・評価結果が蓄積・検索・再利用できるか。
MLflow（実験管理）、Git連携、Model Registry、評価パイプラインを評価。

例: ベンダーAの検証結果がMLflowに残る → 半年後ベンダーBが着手時に参照 → ゼロスタートを回避

原則 3: ガバナンスが拡張可能な基盤

一度作った統制が新しいAIにも自動適用される設計か。
データアクセス制御、AI固有の統制（入出力チェック）、監査・説明責任の3レイヤー。

例: 新AIツール導入時、Unity Catalogの権限設定がそのまま適用 → 都度セキュリティチェック不要

原則 4: ナレッジが社内に残る基盤

ベンダーが去ってもナレッジが残る設計か。
段階的自走（Phase1: 学習 → Phase2: 日常運用移管 → Phase3: 社内主導）。セルフサービス性、運用透明性、変更安全性を評価。

例: KPI定義変更をGUIで社内完結 / パイプライン障害を社内で一次判断可能

9. 運用設計の深掘り — 選定の先にあるもの

技術選定はゴールではなくスタート。本番運用で 実際に問題になるのは権限管理、加工コードの発散、変更管理。ここを設計しないと「Foundryの方がマシだった」になりかねない。

やりがちな失敗

  • 「後から権限設計」: データが入った後に設計すると、既存クエリが権限で壊れてカオスに
  • 「権限を増やしすぎて管理不能」: 施設700×データ種別×時間で組合せ爆発。タグベースACLで汎用ルールに
  • 「BIツール側で制御すればいい」: Tableauで隠しても、SQLで直接叩けば見える。基盤レイヤーで制御が鉄則
  • 「本番データで開発」: 開発環境にも要配慮データが流れる。サンプルデータ戦略が必須
権限設計は Phase 0（基盤構築と同時）で行う

データが入る前にIAM統合、ロール設計、マスキングルール、監査ポリシーを定義する。後付けは指数関数的にコストが増える。RFPの評価項目に「権限設計の具体性」を入れるべき。

補足: 個人情報保護法の2026年改正動向

2026年1月、個人情報保護委員会が「制度改正方針」を公表。要配慮個人情報（病歴・診療情報等）の取得について、統計作成目的に限定し本人同意不要とする条件の緩和を検討中。2026年通常国会で改正法案提出見込み。介護データ基盤の権限設計は現行ガイダンスに基づきつつ、改正の方向性も視野に入れておくべき。

ACESの立場を明確にする

── ACESのポジション ── ACESは「AI × データ基盤」の計画策定支援者 できること（強み） ・AI/LLM活用の設計力（プロンプト、RAG、エージェント） ・技術スタック選定の目利き（4原則→7観点のフレーム策定実績） ・RFP設計（ベンダーに何を問うべきかの設計） ・PoC設計・技術検証の並走 できないこと（明示すべき） ・大規模データ移行の実行（→ SIerの領域。デロイト/富士通が候補） ・Foundryのオントロジー棚卸しの実務（→ 現行ベンダーまたはSOMPOケア内部） ・インフラ構築・運用（→ AWS/Databricks + SIer） つまり ACESは「何を作るべきか」「どう選ぶべきか」「AIをどう載せるか」を設計する。 「作る」のは別の人。この境界をSOMPOケアと合意しておくことが最も重要。

結論: ACESとしての推奨事項

一言で言うと

Databricks on AWS を主軸に、RFPで「オントロジー移行計画」と「権限設計の具体性」を評価軸に入れ、Phase 1 PoCを並行で走らせるべき。

なぜ Databricks on AWS なのか（改めて、最新情報を踏まえて）

1. 技術的に最も合理的

  • 219本のPySparkパイプラインがほぼそのまま動く（Snowflakeだと全面書き換え）
  • Unity Catalog + 行/列レベルセキュリティで要配慮個人情報の権限設計が一元管理可能
  • MLflow 3.0 + Agent Bricks + Vector Search で Phase 1-4 の AI ロードマップに対応
  • Delta UniForm (GA) で Iceberg 互換メタデータを自動生成 → 将来のポータビリティ確保

2. リスクが最小

  • RFI で Deloitte（有力候補）・富士通・Sky の3社が Databricks 構成を提案済み → ベンダー候補が複数いる
  • SMBC の 20+ 基幹系移行事例 → 日本大企業での実績あり
  • Databricks-Palantir 提携 → 段階的移行パスも視野に入る
  • Serverless 全面 GA → インフラ管理の負荷が下がっている

3. 注意すべき点

  • Iceberg REST Catalog API はまだ Preview — 外部エンジンとの相互運用を本番で使うならGA待ち
  • Serverless は便利だが高い — 長時間稼働ワークロードではクラシックコンピュートの方が安い場合あり
  • 学習コスト — SOMPOケア社内のデータエンジニアリング経験が浅い場合、立ち上げ支援の設計が必要
  • コスト管理 — DBU + インフラの二重課金構造。FinOps 設計を RFP に組み込むべき

Snowflake / Fabric は「なぜ今回は選ばないのか」を説明できるようにする

Snowflake を選ばない理由

  • PySpark 219 本の全面書き換えコスト（数千万円規模の差）
  • モデル訓練/FT は Databricks に劣後（Snowflake は「消費型 AI」志向）
  • Cortex AI のコスト暴走リスク（単一クエリ$5K 事例）が介護データ量だと深刻
  • ただし Cortex Analyst はセマンティック分析で強い。将来の BI サービングレイヤーとして部分的に使う可能性は残る

Fabric を選ばない理由

  • Azure 限定 — SOMPOケアの既存環境は AWS（Direct Connect 接続済み）
  • OneLake Security はまだ GA 前 → 要配慮個人情報の本番運用には時期尚早
  • 日本の介護事例はヤマシタ（介護用品レンタル）のみ → 大規模介護データ基盤の実績なし
  • ただし SOMPOグループ HD が Azure 路線ならば将来的に検討対象になりうる。L0 インフラの議論次第

LLM周りの論点整理 — 「Snowflakeでもできますよね？」への備え

Appendixの技術判断の要点

最終成果物のAppendix（別紙A-1〜C-2）は、各技術判断の根拠を詳細に説明している。MTGで「なぜその判断か」を聞かれた時の武器。

A-1: DWH+DL vs Lakehouse

結論: Lakehouse推奨。従来の2層構成（Data Lake + DWH）はデータの2箇所管理、同期コスト、整合性問題がある。Lakehouse（S3上にDelta Lake）なら1箇所で統合管理。BI/ML/AIが同一基盤でアクセス。

想定質問: 「LakehouseだとBI性能が落ちないか？」→ 「2025年のSQL Warehouse性能改善で実用上十分。Tableau ExtractでDatabricksへの負荷もゼロにできる」

A-2/A-3: テーブルフォーマット

結論: Delta Lake + UniForm（Iceberg互換）推奨。Delta LakeはDatabricks上で最高性能。UniFormでIceberg互換メタデータを自動生成し、Snowflake/Athena/Trino等からも読取り可能。

想定質問: 「UniFormに制約はないか？」→ 「読み取り専用（外部エンジンからの書き込み不可）、メタデータ遅延あり、Snowflakeからの読み取りは約20%性能低下。要認識だが実用上許容範囲」

A-5: プラットフォーム評価

結論: Databricksが最も適合度が高い。PySpark互換（そのまま動く）、AI/ML統合（MLflow+Vector Search）、ベンダーロックイン（Unity Catalog OSS化で懸念大幅に軽減）、運用負荷（統合PFでFoundryに最も近い体験）。

想定質問: 「Snowflakeは？」→ 「SQL分析は最速だが、PySpark全面書き換え（Snowpark）が必要でコスト大。Cortex MLもDatabricksに比べ限定的。介護テキスト分析のような非構造データ処理に不向き」

A-6: BI層にSnowflake不要

結論: Databricks → Tableau直接接続を推奨。以前はTableau同時多数接続のためSnowflakeを挟むパターンがあったが、現在は性能差が少なく、Snowflake追加コスト（年1,500-3,000万円）に見合わない。

想定質問: 「3,000人が使って大丈夫か？」→ 「登録3,000人≠同時接続。ピーク40-80人程度。SQL Warehouse自動スケーリングで対応。さらにTableau Extractで定期キャッシュすれば負荷ゼロ」

A-7: オーケストレーション

結論: Lakeflow（Databricks）単体推奨。外部連携が複雑な場合のみAirflow追加。フルマネージドで運用負荷最小、Unity Catalogとネイティブ統合。

想定質問: 「Airflowの方が汎用的では？」→ 「汎用性は高いがMWAA環境構築・運用コストが追加。Databricks中心なら不要。外部API連携が増えた段階で追加しても遅くない」

A-8: PySpark移植戦略

結論: デコレーター除去のみでほぼ動作。@transform_df, @configureはFoundry固有だが、中のPySpark DataFrame APIはそのまま。Code Repositories→GitHub+Asset Bundles+Lakeflow Jobsに移行。ブランチベースのデータ管理は開発/本番分離で代替。

想定質問: 「全ビルド無期限保持は？」→ 「非現実的。スナップショット・保持ポリシーの設計が必要。Delta Lake Time Travelで代替だが保持期間は有限」

A-9: セマンティックレイヤー

結論: 初期構築では不要。AIチャット導入時に必要。ダッシュボード4つの維持だけならTableauの計算フィールドで十分。10+のダッシュボード追加時は推奨、AIチャット（自然言語分析）導入時は実質必須（SQL精度が16%→54%に向上）。

そもそも「KPI定義」「dbt model」「セマンティックレイヤー」とは何か

KPI定義 = 数字の「計算ルール」を全社で統一すること

例えば「稼働率」。一見単純だが、人によって計算方法が違う:

誰が計算方法結果
IT企画部入居者数 ÷ 定員87%
施設長入居者数 ÷ ベッド数（空室含む）83%
DDA推進部入居者数 ÷ 定員（ショートステイ除く）91%

同じ「稼働率」で3人が違う数字を出す → 経営会議で「数字が合わない」→「信用できない」→「使わない」。
KPI定義 = 「稼働率とはこの計算式」と全社で1箇所に書いて統一すること。

dbt model = SQLの「設計図管理」ツール

新しい言語ではない。普通のSQLをGitで管理する仕組み。

-- 普通のSQL（Excelやメモ帳にある。誰がいつ変えたか不明） SELECT 入居者数 / 定員 AS 稼働率 FROM ... -- dbt model（Git管理。変更履歴が残る。テスト付き） -- models/gold/kpi_occupancy.sql SELECT 入居者数 / 定員 AS 稼働率 FROM ...

やっていることは同じSQL。違いは:
1. Gitで管理 → 誰がいつ何を変えたか分かる
2. テスト付き → 「稼働率が0〜100%か」自動チェック
3. 依存関係追跡 → 元テーブルが変わったら自動再計算
SQLが書ければ使える。

AI活用でなぜセマンティックレイヤーが必須になるか

セマンティックレイヤーなし:

社長: 「稼働率が低い施設を教えて」 AI: SQLを自分で推測して生成 → 入居者数 / ベッド数 ... (勝手な計算式) → 「施設Aが72%」(正しいか不明)

セマンティックレイヤーあり:

社長: 「稼働率が低い施設を教えて」 AI: セマンティックレイヤーを参照 → 定義発見: 入居者数 ÷ 定員(ショートステイ含) → 定義通りのSQLを生成 → 「施設Aが85%」(全社統一の正しい数字)

AIは賢いが「御社の稼働率の定義」は知らない。セマンティックレイヤーがAIへの「辞書」になる。
学術研究: SQL精度が16%→54%に向上（約3倍）

移行時に自動的にやること

エンティティ関係（B）= Silver層のテーブル設計。入居者↔施設↔スタッフの関係はJOINキー（外部キー）で表現。これはデータ移行の一部であり、「オントロジー復元」として特別に意識する必要はない。

後からでOKだが、設計思想だけ入れておくこと

KPI定義（A）= dbt models / Metric Views。初期はTableau計算フィールドで十分。ただし「Gold層はdbt modelで定義する」という設計方針だけは初期に決めておく。後からGold層にKPI定義を集約する際の土台になる。

ただし1つだけ初期にやるべきこと: Foundryオントロジーの「棚卸し」

セマンティックレイヤーの構築は後でいい。しかしFoundryに今あるオントロジー定義の「一覧化・文書化」はFoundry廃止前にやる必要がある。廃止後は参照できなくなるため。

  • Object Types（何種類あるか、プロパティは何か）
  • Derived Properties（計算KPIの定義式）
  • Links（どのオブジェクト間にリンクがあるか）
  • Actions（Workshop/Slateで使われている業務アクション）

Q: 「HD（SOMPOグループ）基盤との整合性は？勝手に進めていいのか？」

4レイヤー分解で整理済み（勉強ガイド セクション5参照）。
L0（インフラ）: AWS契約・VPC・Direct Connect → グループ共通化OK。既に接続済み。

L2-L3は「HDがどうなろうと必要な作業」であり、待つ理由がない。

Q: 「3,000人のBI閲覧者がいるが、移行で業務に支障は出ないか？」

登録3,000人 ≠ 同時接続3,000人。実態は施設長700名+本部スタッフ。日次アクティブ率20-30%、ピーク同時接続40-80人。
対策: ①Tableau Extractで定期的にデータをキャッシュ → Databricksへの同時接続は実質ゼロ。②Databricks SQL Serverlessは自動スケーリングで80人同時接続は問題なし。③並行運用期間を設け、段階的に移行。
むしろ改善される点: Foundryの「UIの頻繁な変更で現場が混乱」という現行課題がTableau移行で解消。Tableauは安定したUIを持つ。

Q3: レイヤー分解

基盤の4レイヤー（L0〜L3）を説明し、SOMPOケアにおいてどのレイヤーはグループ共通化でき、どのレイヤーはケア独自で設計する必要があるか？

L0（インフラ: AWS契約、VPC、Direct Connect）→ グループ共通化が合理的。L1（ストレージ・コンピュート: S3、Databricks）→ 条件付き共有（マルチテナント設計が前提）。L2（データアーキテクチャ: スキーマ、ETL、セマンティックレイヤー）→ ケア独自（ドメイン固有）。L3（アプリケーション: BI、AI/ML）→ ケア独自。ケアのデータはTemporal Table + Event Sourcing + 非構造パイプラインが必要であり、保険ドメインのFact/Dimension設計とは構造的に異なるため、L2-L3は共通化できない。

Q4: データコントラクト

データコントラクトとは何か？なぜ今設計する必要があるか？2028年CareBase刷新との関係は？

データコントラクトは、ソースシステムとデータ基盤の間で「約束」を定義する仕組み。必須カラム、データ型、NULLルール、値の範囲などを明文化し、約束が破られたら自動検知・アラートする。dbt schema tests / Great Expectations / DLT Expectations 等で実装。2028年にCareBaseがリプレースされるとソースのスキーマが変わる。データコントラクトがあれば変更を自動検知→局所対応で済む。なければETL全面改修（基盤構築費用の30-50%追加リスク）。

Q5: なぜDatabricksか

SOMPOケアの移行先としてDatabricks on AWSが有力な理由を3つ挙げてください。

1. PySpark互換性: Foundry Transforms で書かれた既存PySpark コードの移行が最もスムーズ。2. AI/ML統合: MLflow、Mosaic AI、Vector Search、AI-BI Genie が統合されており、Phase 1〜3のAI活用ロードマップに対応。3. AWS上で動作: 既存のAWS環境（Direct Connect接続済み）をそのまま活かせる。加えて、Unity CatalogによるAI時代のガバナンス（行/列セキュリティ + AIアクセス制御 + リネージ）が統合されている点も重要。

Q6: セマンティックレイヤーの落とし穴

セマンティックレイヤーの技術的実装は比較的容易だが、本当に難しいのは何か？

組織的な合意形成が最も難しい。具体的には: 1.「稼働率にショートステイを含めるか」等の部門間合意、2. 既存BIダッシュボードとの定義整合、3. 定義変更のガバナンス（誰が承認し、いつ反映するか）、4. 継続的なメンテナンス（新KPIの追加、定義の改訂）。YAML/SQLでの技術実装より、「部門間で用語の定義を揃える政治的プロセス」がボトルネックになる。

Q7: Phase 1 PoC の推奨

Phase 1 PoCとして「自然言語KPI照会」を推奨する理由と、PoCで検証すべき「技術面以外」の要素は？

推奨理由: 1. データが既にある（構造化データは移行対象）、2. 基盤のAI-Readyさを直接検証（SQL API、セマンティックレイヤー、権限継承が全て試される）、3. 現場に見せやすいデモ、4. 組織面の検証も可能。
技術面以外の検証: データアクセス承認フローは実用的な速度で回るか、PoCで使えるデータの範囲は十分か（マスキング設計）、社内チームがPoC環境を自走運用できるか、結果をどうドキュメント化し次のPoCに繋げるか。

Q2: Foundryではオントロジーの辞書と各データの紐付けはどうやってるの？

回答: 「バッキングデータセット」方式。1 Object Type = 1 Foundryデータセット で 1:1 マッピング。

紐付けメカニズム

データソース (S3, DB, API...) ↓ コネクタで取り込み Foundry データセット (Parquet) ↓ ETLパイプラインで変換・統合 バッキングデータセット (整形済みテーブル) ↓ Ontology Managerで紐付け設定 Object Type (プロパティ = 列, オブジェクト = 行) ↓ Link定義で関連付け オントロジーグラフ (Object同士の関係性)
  • Ontology Manager (GUI): バッキングデータセットの列をObject Typeのプロパティにドラッグ&ドロップでマッピング
  • 主キー指定: データセットの一意列をObject TypeのRIDとして設定
  • Link (関連): 外部キー相当の列を指定し、Object Type間のリレーションを定義
  • 自動同期: パイプラインでバッキングデータセットが更新されると、Ontologyも自動反映

移行先での代替

Databricks Unity Catalog のテーブル + カラムコメント + タグが最も近い。ただしOntologyのようなグラフ構造の関係性表現は、別途 Business Semantics（2025年発表）やナレッジグラフツールで補完が必要。

Q3: セマンティックレイヤーはどう紐づいてる？表記揺れ・自然言語のブレへの対処は？

回答: スキーマ定義は人間が行い、LLMはユーザーの自然言語入力とOntologyの「橋渡し」を担う。表記揺れはETL正規化 + LLMクエリ拡張の2段構え。

Ontologyの3層構造

層役割例
Semantic層ビジネス概念の定義「入居者」「ケアプラン」「施設」
Kinetic層アクション・ワークフロー「ケアプラン更新」「請求処理」
Dynamic層リアルタイムイベント「バイタル異常検知」「転倒アラート」

表記揺れへの対処

  • (a) ETL段階での正規化: パイプライン内で表記統一ルールを適用。「東京都」「東京」「TOKYO」→統一コードに変換
  • (b) Ontologyプロパティのエイリアス: 1つのプロパティに複数の表示名・検索名を設定可能
  • (c) AIP (LLM) によるクエリ拡張: ユーザーが「入居者の体温」と入力 → LLMが「バイタルサイン.体温」プロパティにマッピング。同義語展開・略称解決をLLMが担当
重要: LLMがスキーマを自動生成するわけではない。スキーマ（Object Type, プロパティ, Link）は人間が設計・定義し、LLMはエンドユーザーの曖昧な入力を正しいスキーマ要素に解決する役割。

移行先での代替

Databricks Unity Catalog Business Semantics（2025年発表）が最も近い対応機能。表記揺れ対処は dbt のセマンティックレイヤー + LLM Text-to-SQL の組み合わせで実現可能。

Q4: Phonographとは？何に変わる予定？

回答: Object Storage V1 (OSv1) の内部名称。後継は Object Storage V2 (OSv2)。2026年6月30日に OSv1 は完全廃止予定。

詳細

  • Phonograph = OSv1: Foundry Ontologyのバックエンドストレージエンジン（旧世代）。Ontologyオブジェクトのインデックス構築・クエリ応答を担当
  • OSv2（後継）: インデックスとクエリの分離アーキテクチャ、増分インデックス更新、水平スケーリング対応。大規模Ontology（数十億オブジェクト）でもパフォーマンス維持
  • 移行タイムライン: Palantirは2026年6月30日にOSv1を完全廃止予定。SOMPOケアの移行スケジュールとの兼ね合いに注意
SOMPOケアへの影響: 現行FoundryがOSv1を使用している場合、AWS/Databricksへの移行前にOSv2へのアップグレードが必要になる可能性がある。移行プロジェクトのスケジュールで確認すべき事項。

任意モデル持ち込み、フルFine-tuning、マルチエージェント、A/Bテスト、MLflow。AI開発チームがいる前提なら最強。

裏返し: 自由度の高さ = 運用の複雑さ。専門人材が必要。

Snowflake = 「SQLだけで始められる」

SELECT AI_COMPLETE(...) で即日LLM活用開始。既存SQLチームがAIを使い始めるのに最適。

裏返し: カスタムモデル、エージェント、実験管理は不可。天井が低い。

Bedrock = 「ガードレール最強のマネージド」

PII検出、ハルシネーション検出が充実。医療・介護データのガバナンスに最適。

裏返し: AWSエコシステム依存。OpenAI非対応。

判断の分かれ目: 「SOMPOケアはどの程度本格的にAI開発をやるのか？」
  • SQLアナリストがLLMを使う程度 → Snowflake Cortex で十分
  • カスタムモデル訓練・エージェント構築まで → Databricks 一択
  • ガードレール重視 + マネージド志向 → Bedrock
  • 段階的に拡大したい → Snowflakeで開始 → 必要に応じてDatabricks追加

LLM制約比較 — 各プラットフォームの「できないこと」

LLM活用時に直面する具体的な制限・上限・落とし穴（2026-04-02 調査）

各プラットフォームの最も危険な制約

Snowflake Cortex AI

  • コスト暴発リスク: SQLでSELECT AI_COMPLETE(...) FROM large_tableをWHERE句なしで実行 → 全行にLLM呼び出し → $5,000超の請求事例あり
  • レート制限が非公開: TPM/RPMの具体値がドキュメントに明示されず、キャパシティプランニングが困難
  • Cross-region推論がAWS限定: Azure/GCPアカウントのSnowflakeでも、LLM推論がAWSリージョンに流れる場合がある

Databricks

  • 料金の不透明さ: レート制限・DBU単価の詳細が営業経由でしか確認できない（3社中最も不透明）
  • モデル自動リダイレクト: 旧モデルが廃止されると後継モデルに自動切替。本番で意図しない挙動変化のリスク
  • 東京リージョン制限: 一部モデルのServingがソウルリージョン経由の可能性（要確認）

AWS Bedrock

  • 出力トークンのバーンダウン: Claude 3.7以降で出力1トークン = クォータ5トークン消費。max_tokens未設定が予期せぬスロットリングの最大原因
  • クォータ増加申請の壁: 利用実績がないと拒否される。PoC開始時に即座にスケールできない
  • デフォルト制限が低い: モデルによっては数 req/min レベル。本番利用には必ず引き上げが必要

データカタログの現実 — 入れたら解決、ではない

カタログは「機能」ではなく「運用」。導入しても使われなければ無意味で、使われるためには組織的な仕組みが必要（2026-04-02）

Gartner予測: データガバナンスイニシアチブの80%が2027年までに失敗する。主因は技術ではなく、組織・プロセス・文化の問題。

カタログを入れても失敗する4パターン

1. 誰も使わない

カタログを導入したが、データチームが直接SQLを叩く方が早いので、カタログを経由しない。検索UIが使いにくい、情報が古い、BI閲覧者はカタログの存在すら知らない。

→ SOMPOケアの3,000名のBI閲覧者はカタログを直接使わない。 直接ユーザーはデータチーム（推定10-30名）のみ。

2. メタデータが腐る

初期登録は頑張るが、テーブル追加・カラム変更のたびにカタログを更新する運用が定着しない。半年後にはカタログの情報と実態が乖離し、「カタログの情報は信用できない」状態に。

→ 自動同期できるのはスキーマ情報まで。 ビジネス説明（「この列は何の意味か」）は人間が書くしかなく、ここが腐る。

3. 責任が曖昧

「誰がカタログを更新するか」「誰がデータ品質を担保するか」が決まっていない。データスチュワードを任命しても、本業と兼務で後回しにされる。

→ 専任のデータスチュワードがいない組織では、カタログは形骸化する。

4. スコープが広すぎる

「全テーブル・全カラムにメタデータを付ける」から始めて、膨大な初期コストで疲弊。価値が出る前にプロジェクトが頓挫。

→ Gold層のBIテーブル（15画面分）から始め、価値を実感してから拡大すべき。

権限統制の現実 — 300施設 × 3,000名をどう制御するか

権限管理は「設定」ではなく「運用」。設定した瞬間から陳腐化が始まる（2026-04-02）

300施設の権限管理で起きる問題

ロール爆発

「施設A管理者」「施設Bの看護師」「エリアマネージャー（東京）」…
施設 × 職種 × 権限レベルの組み合わせでロールが数百〜数千に膨張。RBAC（ロールベース）だけでは管理不能。

→ ABAC（属性ベース）とのハイブリッドが必要。「所属施設=Aかつ職種=管理者」のようにユーザー属性で動的に権限を解決。

Permission Sprawl（権限の蓄積膨張）

異動しても前の施設の権限が残る。兼務で権限が増える一方。退職者のアカウントが残る。
半年後には「誰が何を見れるか」を誰も把握できない。

→ 人事マスタとの自動連携（異動→権限自動更新）が前提条件。手動管理は破綻する。

「見えてはいけないデータが見えていた」

最もよくあるパターン:
① Tableauの抽出データにRLSが効かず、ダウンロードしたCSVに全施設データが含まれていた
② テスト用に全権限を付与した管理者アカウントが本番に残った
③ BI上の「Web Edit」権限からRLSを回避するクエリが実行できた

権限設定の属人化

「権限のことは〇〇さんに聞いて」。設定のルール・意図がドキュメント化されず、担当者の退職で誰も触れなくなる。

→ 権限設定をコード管理（IaC）し、変更はPRレビュー必須にする。

Foundryの轍を踏まないために

Foundryのロックイン構造（これを繰り返さない）:
  • 独自データフォーマット → エクスポートに数ヶ月
  • 独自オントロジー → 移行先に同等概念がない
  • 独自UI/ワークフロー → ユーザー再教育コスト
  • Foundry専門人材 → 市場に少なく単価高い

Databricks: 最もオープン

OSS戦略が徹底。Unity Catalog OSS化、Delta UniFormでIceberg互換、MLflowが業界標準。「いつでも出られる」設計。マルチクラウドでコミットメント共有もできるため、クラウドベンダー自体のロックインも軽減。

Foundryと真逆のアプローチ

Snowflake: 条件付きでオープン

Icebergテーブルを積極採用すればリスク低減可能。 独自テーブルに全面依存するとFoundryほどではないが中リスク。Cortex AI関数（AI_COMPLETE等）への依存は、移行時にSQL書き換えが必要。Polaris Catalog (OSS) の成熟度に注意。

使い方次第

AWS全面依存: 別の種類のロックイン

Bedrock単体のロックインは中程度だが、Glue + S3 + Lambda + Step Functions + Bedrock と全面依存すると「Foundry→AWSに置き換わっただけ」になるリスク。dbt等のOSSレイヤーを挟むことで軽減可能。

要注意

MTGでの推奨論点:
  • データフォーマットはIcebergに統一する方針か？ → Databricks/Snowflake両方で読み書き可能になる
  • ETLはdbt等のOSSレイヤーを挟むか？ → ベンダー交換可能性が大幅向上
  • LLM機能は標準API呼び出しで実装するか、プラットフォーム固有関数に依存するか？
  • 5年後に「別の基盤に移りたい」となった時の脱出コストを各ベンダーに確認すべき
