# AI開発のためのデータ整備・ライフサイクル全体像

> 作成日: 2026-03-03 | 更新日: 2026-03-04 | 目的: 「普通のDWH」では足りないAI固有のデータの持ち方・管理方法を、具体例付きでわかりやすく整理する

---

## 0. この文書の位置づけ

**誰に向けて**: AIを使ったデータ活用を推進するが、MLモデルの開発経験はまだ少ない方向け。

**何がわかるか**: 普通のBIダッシュボード用のデータ管理と、AI（機械学習・LLM）のためのデータ管理は**何がどう違うのか**を、SOMPOケアの具体例で理解できる。

### 全体マップ

```
AI開発で必要な5つのステップ:

  ① データの持ち方を変える
     → 「今の値」だけでなく「あの時点の値」を保存する
     → AIが食べやすい形に加工する

  ② 学習データを正しく作る
     → 「未来の情報を使ってカンニング」しない仕組み
     → 正解データ（ラベル）をどう集めるか

  ③ AIを育てるサイクルを回す
     → 実験 → 評価 → 本番投入 → 監視 → 再学習

  ④ AIの出力も管理する
     → AIが出した予測・分析を「データ」として保存
     → 人間のフィードバックも記録

  ⑤ 本番で使い続ける
     → 精度が落ちていないか監視
     → データの傾向が変わったら再学習
```

## 1. 構造化データ: 普通のDWHとAI用データの違い

### 1-1. 最大の違い: 「今の値」ではなく「あの時点での値」が必要

**身近な例: 天気予報の検証**

天気予報の精度を検証したいとする:
- ❌ 「今日の天気」と「今日の予報」を比べる → これは簡単
- ✅ 「3日前の時点での予報」と「今日の実際の天気」を比べる → 3日前のデータが必要

AIの学習も同じ。「3ヶ月前のデータだけを使って、今月を予測できたか？」を検証するには、**3ヶ月前の時点のデータ**が必要。

```
SOMPOケアの例: 「スタッフの離職を3ヶ月前に予測したい」

❌ よくある間違い（カンニングしている状態）:
   「今日のデータ」を使って「3ヶ月前の離職」を予測するモデルを作る
   → AIは「あ、この人は既に退職してるな」という情報が見えてしまう
   → テストでは精度95%！ でも実際に使うと全然当たらない
   → これを「データリーケージ（情報漏洩）」と呼ぶ

✅ 正しい方法（カンニングしない）:
   「3ヶ月前のその時点で知りえた情報だけ」を使う
   → 3ヶ月前の残業時間、有休消化率、施設の稼働率...
   → テストでは精度75%。でも本番でも75%出る
   → これを「Point-in-Time Correctness（時点整合性）」と呼ぶ
```

**これがなぜ「DWH設計」に影響するのか?**

普通のBIダッシュボードは「今の値」があれば十分。でもAIの学習には「過去のあらゆる時点の値」が必要。つまり:
- 「2025年6月1日時点の施設Aの稼働率」
- 「2025年6月1日時点のスタッフBの残業時間（直近90日平均）」
をいつでも取り出せるようにデータを保存しておく必要がある。

**他社の事例**:
- **Capital One（銀行）**: クレジットカードの不正検知AIで、「申請時点で知りえた情報だけ」で判断するよう時点整合性を厳密に管理。これを怠ると不正検知の精度が実運用で30%以上落ちたという報告
- **Uber**: 到着時間予測AIで、「注文時点の交通状況」だけを使うよう設計。注文後の情報（実際のルート等）が混入すると予測精度が急落

### 1-2. Feature Store — AIのための「食材棚」

**身近な例: レストランの仕込み**

レストランでは、注文が来てからゼロから料理しない。事前に「仕込み」をしておく:
- 野菜はカット済み
- ソースは作り置き
- 肉は下味付き

AIでも同じ。予測に使う「特徴量（Feature）」を事前に計算・保存しておく仕組みが**Feature Store**。

```
普通のDWH:
  Bronze: 生データ → Silver: クレンジング済み → Gold: KPI集計・BI向け

AI用に追加:
  Feature Store: ML学習・推論向け ← AIのための「仕込み棚」

Feature Storeがやってくれること:

  ① 「仕込みレシピ」の統一管理
     「残業時間_直近90日平均」の計算ロジックが1箇所に定義
     → 離職予測AIも、健康リスクAIも、同じ「残業時間」を使える
     → 定義がバラバラにならない

  ② 「あの時点の仕込み」を取り出せる
     「2025年6月1日時点の仕込み済み食材セット」がすぐ取れる
     → カンニング防止が自動的に保証される

  ③ 「仕込み」と「盛り付け」で同じ食材を使う
     学習時: 過去データから特徴量をバッチ計算
     本番時: 現在のデータからリアルタイムで特徴量を計算
     → Feature Storeが両方を同じレシピから作る
     → 学習時と本番時で計算式がずれない（これ重要！）
```

**具体例: SOMPOケアの離職予測モデル**
```
Feature Storeに登録する特徴量:
  ├─ 残業時間_直近90日平均: sum(overtime) / 90
  ├─ 有休消化率_年度累計: used_leaves / total_leaves * 100
  ├─ 施設稼働率_当月: residents / capacity * 100
  ├─ 夜勤回数_直近30日: count(night_shifts)
  └─ 在籍年数: (today - hire_date) / 365

→ これらを「2025年6月1日時点で計算した値」として保存
→ 離職予測にも、メンタルヘルス予測にも、同じ特徴量を再利用
```

**技術選択肢**:

| Feature Store | どのプラットフォームで使えるか | 特徴 |
|---|---|---|
| **Databricks Feature Store** | Databricks | 一番統合が楽。Unity Catalogと一体 |
| **Feast** | どこでも（OSS） | ベンダーに依存しない |
| **SageMaker Feature Store** | AWS | S3ベース。Bedrockとの連携が楽 |

**他社の事例**:
- **Uber**: Feature Store「Michelangelo」を構築。社内の全MLモデル（数百個）が同じ特徴量を共有。新しいモデルを作るとき、既存の特徴量を再利用することで開発期間を50%短縮
- **Spotify**: 推薦アルゴリズムで使う特徴量（ユーザーの聴取パターン、楽曲の特性等）をFeature Storeで一元管理。100以上のモデルが同じ特徴量を共有
- **LinkedIn**: 求人推薦AIで、ユーザーのスキル・経験・行動をFeature Storeに格納。リアルタイム推薦と分析の両方で同じ特徴量を利用

### 1-3. 正解データ（ラベル）の管理

**身近な例: テストの採点**

学校のテストには「模範解答」がある。AIの学習にも「模範解答」が必要。これを**ラベル**と呼ぶ。

```
SOMPOケアの予測タスクと「模範解答」:

離職予測:
  AIへの問い: 「このスタッフは3ヶ月以内に辞めますか？」
  模範解答（ラベル）: 退職日（人事システムから取得）
  → でも「自己都合退職」と「契約満了」を区別する？
  → ラベルの定義がAIの性能を決める

転倒予測:
  AIへの問い: 「この入居者は30日以内に転倒しますか？」
  模範解答（ラベル）: 転倒発生日時（事故報告書から取得）
  → でも報告されなかった「ヒヤリハット」は？
  → ラベルの網羅性が問題になる

状態悪化予測:
  AIへの問い: 「この入居者の要介護度は6ヶ月後に上がりますか？」
  模範解答（ラベル）: 要介護度の変更日（介護保険更新時）
  → でも更新タイミングが3-6ヶ月ごとでバラバラ
  → ラベルの「遅延」問題
```

**他社の事例**:
- **Google Health**: 糖尿病性網膜症の診断AIで、5人の眼科医がそれぞれ画像にラベル（診断結果）を付け、多数決で「正解」を決定。1人の医師の判断ではラベルの品質が不安定だった
- **Tesla**: 自動運転で「この状況でブレーキすべきか」のラベル付けに、専門チーム200人以上が従事。ラベルの品質管理がAIの性能に直結

---

### 1-4. データの分割 — 「テスト」でカンニングしない方法

```
❌ よくある間違い: ランダムにデータを分ける
   ML入門書: 「データの80%で学習、20%でテスト」
   → 時系列データでこれをやると、2024年のデータで学習して
     2023年のデータでテスト、みたいなことが起きる
   → 未来の情報で過去を当てるのは簡単（カンニング）

✅ 正しい方法: 時間で区切る（時系列の鉄則）

  ┌──────────────┬────────────┬──────────┐
  │   Training   │ Validation │   Test   │
  │  〜2025/06   │ 2025/07-09 │ 2025/10〜│
  │  (学習用)    │ (調整用)   │ (最終テスト)|
  └──────────────┴────────────┴──────────┘
  → 常に「過去で学習 → 未来を予測」の形を維持
```

**SOMPOケアでの注意点**: 介護データは**季節性**がある。冬は転倒が増える。年度末に退職が増える。テスト期間が「冬だけ」だと、夏の精度がわからない。

**他社の事例**:
- **Kaggle（データ分析コンペ）**: 時系列コンペでランダム分割を使ったチームは、リーダーボード（テスト結果）と本番結果が大きく乖離。時間ベース分割のチームだけが安定した成績

### 1-5. 2028年の介護システム刷新への備え

```
2028年にCareBaseが刷新される → テーブル構造が変わる可能性

例:
  旧システム: staff_table.overtime_hours（残業時間、月次集計）
  新システム: work_log.actual_hours - work_log.scheduled_hours（日次の差分計算）

→ Feature Storeの定義「残業時間_直近90日平均」の裏側のSQLが変わる
→ でもAIモデルは「残業時間_直近90日平均」という名前で参照しているだけ
→ Feature Storeの定義だけ書き換えれば、AIモデルはそのまま動く

これが「抽象化」の威力。直接SQLを書いていたら全モデルを書き直し。
```

## 2. 非構造データ: テキスト・ファイルのAI活用

### 2-1. テキストデータのAI検索 — 「ベクトル化」とは

**身近な例: 本の索引 vs Google検索**

本の索引: 「転倒」で引くと、「転倒」という言葉が書いてあるページだけ見つかる
Google検索: 「転倒」で検索すると、「つまずいて倒れた」「バランスを崩した」も見つかる

この**「意味で検索する」**仕組みがベクトル化。

```
普通のテキスト検索（キーワード一致）:
  検索: 「転倒」
  結果: 「転倒」という文字が入った記録だけヒット
  → 「廊下でバランスを崩して倒れた」はヒットしない

ベクトル検索（意味の近さで検索）:
  検索: 「転倒」
  結果: 「転倒した」「バランスを崩して倒れた」「足がもつれて」も全部ヒット
  → AIが文章の「意味」を数値化（ベクトル化）しているから

ベクトル化のイメージ:
  「転倒した」→ [0.8, 0.2, 0.5, ...]（数百個の数値に変換）
  「バランスを崩した」→ [0.7, 0.3, 0.5, ...]（似た数値になる）
  「天気が良い」→ [0.1, 0.9, 0.2, ...]（全然違う数値になる）

→ 数値が近い文章 = 意味が近い文章
→ これを高速に検索できるのが「ベクトルDB」
```

**管理上の注意点**:
- ベクトル化するAIモデルを変更すると、全テキストを再計算する必要がある（「辞書が変わる」ようなもの）
- 新しい介護記録は日々追加される → 毎日ベクトル化するバッチ処理が必要
- テキストを分割する方法（1記録=1単位？500文字ごと？）で検索精度が変わる

**他社の事例**:
- **Notion AI**: 社内ドキュメントをベクトル化して、「あのミーティングで話した件」のような曖昧な検索にも対応。キーワード検索では見つからなかった関連ドキュメントが見つかるように
- **Elicit（学術論文検索）**: 2億本の論文をベクトル化。「がんの免疫療法の副作用」と検索すると、副作用に言及した論文が意味で見つかる

---

### 2-2. テキストからデータを取り出す — LLMで構造化抽出

```
介護記録テキスト:
  「本日14:30、入居者Aさんが食堂で転倒。右膝を打撲。
   看護師が確認し湿布対応。歩行器の使用を再度推奨。家族に連絡済み。」

↓ LLMが自動的にデータ化

{
  "日時": "14:30",
  "イベント": "転倒",
  "場所": "食堂",
  "怪我": "右膝打撲",
  "対応": "湿布",
  "フォローアップ": ["歩行器使用推奨", "家族連絡"],
  "重症度": "軽度"
}

↓ テーブルに格納

→ 「過去6ヶ月の転倒回数」「転倒場所の統計」がSQLで分析可能に
→ これが「構造化された特徴量」としてML学習にも使える
```

**精度管理が重要**:
- LLMの抽出精度を定期的にチェック（人間の手作業と比較）
- 「確信度」が低い抽出結果は人間がレビュー
- 抽出ルール（プロンプト）もバージョン管理

**他社の事例**:
- **Flatiron Health（がん研究）**: がん患者の診療記録から「腫瘍の大きさ」「ステージ」「治療法」をLLMで自動抽出。手作業では1レコード30分かかっていたのが30秒に短縮。抽出精度は人間の作業者と同等（95%以上）
- **Lemonade（保険テック）**: 保険金請求のフリーテキストから「事故の種類」「損害額の見積もり」を自動抽出。処理時間を90%削減

## 3. 学習サイクル: AIを育てる8つのステップ

### 身近な例で全体像を理解する: 「料理コンテスト」

AIの学習サイクルは、料理コンテストに参加するプロセスに似ている:

```
① 仮説: 「和風カレーなら勝てるのでは？」（何を予測するか決める）
② 食材調達: レシピに必要な食材を集める（データセットの作成）
③ 試作: いろんな作り方を試す（複数のモデル・パラメータで実験）
④ 味見: 美味しいか審査（テストデータで精度を評価）
⑤ メニュー登録: 「このレシピで行く」と決定（モデルをレジストリに登録）
⑥ 提供: お客さんに出す（本番で予測を実行）
⑦ お客さんの反応: 「美味しい？」「まずい？」（予測精度のモニタリング）
⑧ レシピ改良: 不評だったらレシピを見直す（再学習）
```

### Step ①: 仮説を立てる

**ここは人間の仕事**。でも「何をどう記録するか」が大事。

```
仮説の記録例（SOMPOケア）:

仮説: 「施設の稼働率は、3ヶ月前のスタッフ離職率で予測できる」
ビジネスの問い: 稼働率が80%を下回る施設を3ヶ月前に特定したい
成功基準: 実際に低下する施設の70%以上を事前にキャッチ
使う材料（特徴量の仮説）:
  - 現在の稼働率
  - 直近3ヶ月のスタッフ離職率
  - 施設の築年数
  - 地域の高齢者人口

→ これを記録しておくことで、
  「なぜこのモデルを作ったか」が後から説明できる（監査対応）
```

### Step ②: データセットを作る

Feature Store（仕込み棚）とラベル（模範解答）を組み合わせて、学習用データセットを作る。

```
イメージ:

施設ID | 時点      | 稼働率 | 離職率3M | 築年数 | ... | 3ヶ月後に80%以下？
A       | 2024-01  | 88%    | 5%       | 10年   | ... | NO（実際は85%）
A       | 2024-04  | 82%    | 12%      | 10年   | ... | YES（実際は76%）
B       | 2024-01  | 92%    | 3%       | 5年    | ... | NO（実際は90%）
...

→ 「施設ID×月」の組み合わせで数千～数万行
→ これが「カンニングしていない」正しい学習データ
```

**記録すべきメタデータ**:
```
このデータセットの「パスポート」:
  バージョン: v1.0
  作成日: 2026-04-15
  期間: 2023年1月～2025年6月
  レコード数: 45,000行
  正解の割合: 稼働率低下 = 15%（不均衡に注意）
  分割方法: 学習(〜2024/12) | 調整(2025/01-03) | テスト(2025/04-06)
```

### Step ③: いろんな方法を試す（実験管理）

```
実験管理ツール（MLflow等）で記録するもの:

  実験1: LightGBMで予測
    パラメータ: 木の深さ=6, 学習率=0.1
    結果: 精度78%, 適合率65%
    学習時間: 45秒

  実験2: XGBoostで予測
    パラメータ: 木の深さ=8, 学習率=0.05
    結果: 精度81%, 適合率68%  ← こっちの方がいい！
    学習時間: 120秒

→ 全実験が比較可能な形で記録
→ 「なぜXGBoostを選んだか」の根拠になる
```

**他社の事例**:
- **Airbnb**: 年間数千の実験をMLflowで管理。「この特徴量を追加したら精度が2%上がった」のような知見が蓄積され、新しいモデル開発が効率化

### Step ④: 精度を評価する — 数字をビジネス言語に翻訳

**ここが一番大事**。「AUC=0.81」だけでは経営層に伝わらない。

```
AI指標:
  精度(AUC) = 0.81, 適合率 = 0.68, 再現率 = 0.70

ビジネス言語に翻訳:
  「稼働率が下がる施設を3ヶ月前に予測します」

  1,000施設 × 月次予測の場合:
    実際に稼働率が下がる施設: 約50施設/月
    AIが正しく見つける: 35施設（70%）→ 事前に対策が打てる
    AIが見逃す: 15施設（30%）→ 従来通り事後対応
    AIが「下がる」と言って外れ: 16施設 → 不要な対策コスト

  → 35施設で事前対策が取れることのビジネス価値は？
  → 16施設の空振りコストは許容できるか？
  → この判断は経営層と合意が必要
```

**他社の事例**:
- **Waymo（自動運転）**: AIの精度を「100万マイルあたりの事故回避率」でビジネス翻訳。「精度99.9%」より「人間ドライバーより6倍安全」の方が伝わる

### Step ⑤: モデルを「本番投入」する

```
本番投入のステージ管理（料理コンテストの例え）:

  Development: まだ試作中。キッチンでだけ食べる
       ↓
  Staging: 身内に試食してもらう。本番の材料で作ってみる
       ↓
  Production: お客さんに提供開始。問題が起きたらすぐ差し替え
       ↓
  Archived: 引退。記録として残す

本番昇格の条件:
  ✓ テストデータで精度基準を満たしている
  ✓ 特定の施設に偏った予測をしていない（バイアスチェック）
  ✓ 責任者が承認した（データサイエンスリード + 事業責任者）
```

### Step ⑥: 予測を本番で使う

```
2つの使い方:

バッチ推論（ほとんどのケース）:
  毎月1日に全1,000施設の3ヶ月後稼働率を予測
  結果をテーブルに書き込み
  ダッシュボードやレポートで表示
  → 「来月のレポートに予測結果が載っている」

リアルタイム推論（特殊なケース）:
  入居者の状態が変わったその瞬間にリスクを再計算
  APIで呼び出して即座に結果を返す
  → 「今この瞬間の転倒リスクを知りたい」
```

### Step ⑦: 精度を監視する — AIは「静かに劣化する」

**身近な例**: エアコンのフィルター。詰まっていても動くけど、徐々に効きが悪くなる。AIも同じで、精度が「静かに」落ちていく。

```
AIの精度が落ちる4つの原因:

① データの傾向が変わった（データドリフト）
   例: コロナ禍で入居者の行動パターンが変化
   → AIが学習した「正常」の基準が古くなった

② 予測結果の傾向が変わった（予測ドリフト）
   例: 先月までアラートが月50件だったのに、今月は200件
   → 何かがおかしい。入力データの変化か、モデルの劣化か

③ 因果関係が変わった（コンセプトドリフト）
   例: 「離職率が高い→稼働率が下がる」が、
       人材派遣で補填するようになったら成り立たなくなった
   → モデルの前提が崩れた。再学習が必要

④ 純粋に精度が落ちた（パフォーマンスドリフト）
   例: 予測と実績を突き合わせたら、精度が80%→65%に低下
   → 再学習のタイミング
```

**他社の事例**:
- **DoorDash（フードデリバリー）**: 配達時間予測AIが、新しい道路開通や交通パターンの変化で精度低下。データドリフト検知を導入し、週次で自動チェック
- **Zillow（不動産）**: 住宅価格予測AIが市場変動（コロナ後の急騰）に対応できず、5億ドル以上の損失。ドリフト検知があれば被害を軽減できた可能性

### Step ⑧: 再学習する

```
再学習のきっかけ:

自動型:
  月次で新データを追加して自動再学習
  → 常に最新のデータを反映

アラート型:
  精度が基準を下回ったらアラート → 人間が判断 → 再学習実行

イベント型:
  2028年の介護システム刷新 → データ構造が変わる → 再学習必須

再学習の流れ:
  精度低下を検知
       ↓
  新データでデータセット作成（Step ②に戻る）
       ↓
  学習 → 評価
       ↓
  精度が上がった → Staging → 人間レビュー → 本番投入
  精度が下がった → アラート → 原因調査（データ？モデル？環境？）
```

## 4. LLM × 構造化データ: AIに正しくSQLを書かせるコツ

### AIが構造化データを「正しく」使うために

**身近な例**: 新入社員にExcelを渡して「売上を集計して」と頼んだ場面を想像する。
- カラム名が「A」「B」「C」→ 何のデータかわからない。間違える
- カラム名が「売上金額」「商品名」「店舗名」→ 正しく集計できる

AIも同じ。テーブルのカラム名だけでは情報が足りない。

```
LLMが正しいSQLを書くために追加で教えてあげること:

  ① カラムの意味
     「occ_rate」→ 「稼働率 = 入居者数 ÷ 定員 × 100」
     （専門用語の略称はAIにはわからない）

  ② テーブルの繋ぎ方
     facility_metrics と facility_master は facility_id で繋ぐ
     （外部キーだけでなく、ビジネス上のルールも伝える）

  ③ 暗黙のフィルタ
     「有効なデータだけ使って」= is_deleted=false AND date >= '2023-01-01'
     （2023年以前はデータ品質が低いから除外、というルールは人間は知ってるがAIは知らない）

  ④ NULLの意味
     稼働率がNULL = 「0%」ではなく「データ未入力」
     （AIはNULLを0と扱ってしまうことがある）

  ⑤ 値のコード表
     status=1は「在籍」、status=2は「退職」
     （コードの意味がわからないとAIは正しく集計できない）
```

**これらをYAMLファイルに書いておくのが「セマンティックモデル」**。doc 08のPart 1で説明した「レシピブック」の中身そのもの。

## 5. AI出力の管理 — AIが出した結果も「データ」として保存する

### なぜAIの出力を保存する必要があるか

**身近な例**: 医者の診断。
- 「先月の診断で何と言われたか」を確認したいのに、記録がなかったら困る
- AIの予測や分析も同じ。「3ヶ月前にAIは何と予測したか」を後から確認できないと困る

```
従来のBI:
  データ → ダッシュボード → 人間が見る → おしまい
  （画面に表示しただけ。記録は残らない）

AI OS:
  データ → AI分析・予測 → 結果を保存 → 人間がレビュー → 意思決定
                 ↓                           ↓
          AI Output Store              Feedback Store
          「AIは何と言ったか」          「人間はどう判断したか」
```

### AI Output Store に保存するもの

| 保存するもの | SOMPOケアの例 | なぜ必要？ |
|---|---|---|
| **予測結果** | 施設の稼働率予測、転倒リスク評価 | 「3ヶ月前にAIは何と予測したか」を検証するため |
| **分析レポート** | 施設の改善レコメンド、異常検知アラート | 行政監査で「AIの判断根拠」を説明するため |
| **提案内容** | ケアプラン提案、シフト最適化案 | 「提案」と「実際に採用した内容」の差分を追跡するため |
| **LLMの質問と回答** | 「稼働率が低い施設は？」への回答 | 精度改善・監査・セマンティックモデル改善のため |

### フィードバックループ — AIを「育てる」仕組み

AIの出力に対する人間のフィードバックは、**次の改善の種**になる。

```
フィードバックの3種類:

① 直接フィードバック（人間が意図的に教える）
   「この予測は間違っていた。実際は○○だった」
   「このケアプラン提案は良かった。採用した」
   → 最もわかりやすい。UIにフィードバックボタンをつける

② 行動フィードバック（人間の行動から推測する）
   AIの分析を見たのに何もしなかった → 役に立たなかった？
   同じ質問を違う聞き方で聞き直した → 最初の回答が不十分だった
   AI出力を会議資料に引用した → 信頼されている
   → 暗黙的だが大量に集まる

③ 結果フィードバック（答え合わせ）
   3ヶ月前の稼働率予測 → 実際の稼働率と比較
   転倒リスク「高」と判定 → 実際に転倒したか確認
   → AIの精度を客観的に計測できる

これらのフィードバックが改善に繋がる:
  LLMの回答精度が低い → プロンプト（質問の仕方）を改善
  予測精度が落ちた → 特徴量を追加・修正 → 再学習
  転倒予測の見逃しが多い → ラベルの定義を見直し → 再学習
```

**他社の事例**:
- **ChatGPT（OpenAI）**: ユーザーの👍/👎フィードバックを集めてモデルを継続改善。人間のフィードバックによるAIの改善(RLHF)の代表例
- **GitHub Copilot**: プログラマーがAIの提案を「採用/却下」する行動データを収集。採用率が高いパターンを学習して提案精度を向上

## 6. チェックリスト: 普通のDWHでは見落としがちなAI固有要件

### データの持ち方

- [ ] 「あの時点のデータ」を取り出せる設計か（Point-in-Time Correctness）
- [ ] Feature Store（AIの仕込み棚）がアーキテクチャに含まれているか
- [ ] 正解データ（ラベル）の収集パイプラインが設計されているか
- [ ] データセットの「パスポート」（バージョン・メタデータ）を記録しているか
- [ ] テストデータの分割は時間ベースか（ランダムではなく）

### 学習サイクル

- [ ] 実験管理ツール（MLflow等）は選定されているか
- [ ] モデルの「Development→Staging→Production」の昇格フローがあるか
- [ ] 昇格には責任者の承認が必要になっているか
- [ ] 再学習のきっかけ（トリガー条件）が定義されているか

### 評価・モニタリング

- [ ] AIの精度が「静かに劣化」していないか定期チェックする仕組みがあるか
- [ ] AI指標をビジネス言語に翻訳して経営層に報告する仕組みがあるか
- [ ] 特定の施設や入居者に偏った予測をしていないかチェックしているか

### LLM連携

- [ ] セマンティックモデル（カラムの意味、結合条件、NULLの解釈）が定義されているか
- [ ] テキストのベクトル化・更新パイプラインが設計されているか
- [ ] ベクトルDB / 検索基盤が選定されているか

### AI出力・フィードバック

- [ ] AIの出力（予測・分析・提案）を保存・バージョン管理しているか
- [ ] フィードバック収集の仕組み（直接+行動+結果）があるか
- [ ] フィードバックからの改善サイクルが設計されているか
- [ ] AI出力の監査ログ保存期間が定義されているか

### ガバナンス

- [ ] AIの学習に使えるデータの分類ポリシーがあるか
- [ ] 「どのデータで学習したか」を後から追跡できるか
- [ ] 2028年の介護システム刷新時に、Feature Storeの定義書き換えだけで済む設計か
