## 1. 構造化データ: 普通のDWHとAI用データの違い

### 1-1. 最大の違い: 「今の値」ではなく「あの時点での値」が必要

**普通のDWH（BI向け）**:
```sql
-- 「今」の稼働率を表示する
SELECT facility_id, occupancy_rate
FROM gold.facility_metrics
WHERE date = CURRENT_DATE
```

**AI学習用**:
```sql
-- 「2025年6月1日時点で知りえた情報だけ」で
-- 「2025年7月に離職するか」を予測するモデルを学習したい
-- → 2025年6月1日時点のスナップショットが必要

SELECT
  f.facility_id,
  f.occupancy_rate_as_of_20250601,     -- 6月1日時点の稼働率
  f.staff_count_as_of_20250601,         -- 6月1日時点のスタッフ数
  f.avg_overtime_hours_last_90days,     -- 3-5月の平均残業（6/1時点で計算可能）
  -- ↓ これが「答え（ラベル）」
  CASE WHEN s.resignation_date BETWEEN '2025-06-01' AND '2025-07-31'
       THEN 1 ELSE 0 END AS did_resign_in_2months
FROM features_as_of('2025-06-01') f
JOIN staff_outcomes s ON f.staff_id = s.staff_id
```

**これが「Point-in-Time Correctness（時点整合性）」**。ML学習で最も重要かつ最もミスが多い部分。

```
❌ よくある間違い: 「今のデータ」で過去を予測するモデルを学習する
   → 未来の情報がリークして精度が異常に高く出る
   → 本番で使うと全く当たらない（データリーケージ）

✅ 正しい方法: 「予測時点で知りえた情報だけ」で特徴量を作る
   → 時点ごとのスナップショットが必要
   → 普通のDWHにはこの仕組みがない
```

**SOMPOケアでの具体例**:
- 離職予測: 「3ヶ月前の残業時間・有休消化率・施設の稼働率」→「離職するか」
- 転倒予測: 「直近1ヶ月の身体状態・投薬情報・活動量」→「30日以内に転倒するか」
- 稼働率予測: 「過去の入退居パターン・地域人口動態」→「3ヶ月後の稼働率」

**どれも「その時点で知りえた情報だけで特徴量を作る」ことが必須**。

### 1-2. Feature Store — AI用のデータレイヤー

普通のDWHには Bronze → Silver → Gold のメダリオンアーキテクチャがある。AI用にはもう1層必要:

```
Bronze: 生データ
Silver: クレンジング済み
Gold:   KPI集計・BI向け
────────────────────────────
Feature Store: ML学習・推論向け ← これがAI固有
```

**Feature Storeとは**: MLモデルに入力する「特徴量（Feature）」を管理する専用レイヤー。

```
Feature Storeの役割:
  ① 特徴量の定義を一元管理
     「残業時間_直近90日平均」の計算ロジックが1箇所に定義される
     → 学習時と推論時で同じ計算が保証される

  ② 時点指定での取得
     「2025年6月1日時点の特徴量セット」をAPIで取得できる
     → Point-in-Time Correctnessが自動的に保証される

  ③ 学習と推論で同じ特徴量を使う
     学習: バッチで過去データから特徴量を計算
     推論: リアルタイムで現在のデータから特徴量を計算
     → Feature Storeが両方を同じ定義から生成する（Training-Serving Skew防止）

  ④ 特徴量の共有・再利用
     離職予測と転倒予測で「施設の稼働率」を共通の特徴量として使える
     → チーム間で特徴量を再利用
```

**技術選択肢**:

| Feature Store | プラットフォーム | 特徴 |
|---|---|---|
| **Databricks Feature Store** | Databricks | Unity Catalogと統合。Delta Tableベース |
| **Feast** | OSS（どこでも） | ベンダー非依存。Snowflake/Redshift/BigQuery対応 |
| **SageMaker Feature Store** | AWS | S3バックエンド。Bedrockとの連携 |
| **Snowflake Feature Store** | Snowflake | Snowpark ML統合（2025 GA） |

**先方への論点**:
> 「予測モデルを開発する場合、Gold Layerとは別にFeature Storeが必要。これをDWH内に作るか（Databricks/Snowflake標準機能）、外部に作るか（Feast等）はアーキテクチャ判断が必要」

### 1-3. データの持ち方: Slowly Changing Dimensions（SCD）の再考

普通のDWHでは「マスタデータの変更履歴」をSCD Type 2（履歴テーブル）で管理する。AI用にはこれが**もっと細かい粒度で**必要。

```
普通のDWH（SCD Type 2）:
  staff_id | facility_id | start_date  | end_date    | is_current
  001      | A           | 2024-01-01  | 2025-03-31  | false
  001      | B           | 2025-04-01  | NULL         | true

AI学習に必要な粒度:
  staff_id | date       | facility_id | overtime_hours | absence_days | ...
  001      | 2025-01-01 | A           | 20.5           | 2            | ...
  001      | 2025-01-02 | A           | 22.0           | 2            | ...
  ...

  → 日次スナップショット（= Feature Storeに格納）
```

**なぜ日次粒度が必要か**: 「1月15日に異動した人の、1月14日時点での特徴量」と「1月16日時点での特徴量」は全く違う。月次集計では予測精度が落ちる。

### 1-4. ラベル（正解データ）の管理

**予測モデルには「正解」が必要**。これをどう集め、どう管理するかはDWH設計だけでは出てこない論点。

```
予測タスクとラベルの例:

離職予測:
  ラベル = 退職日（人事システムから取得）
  注意: 「自己都合退職」と「契約満了」を区別するか？
  → ラベル定義がモデルの性能を決める

転倒予測:
  ラベル = 転倒発生日時（事故報告から取得）
  注意: 報告されなかった転倒（ニアミス含む）は？
  → ラベルの網羅性問題

入居者の状態悪化予測:
  ラベル = 要介護度の変更日（介護保険更新から取得）
  注意: 更新タイミングにばらつきがある（3-6ヶ月ごと）
  → ラベルの遅延問題
```

**ラベル管理で必要なこと**:

| 項目 | 内容 |
|---|---|
| ラベル定義書 | 何をもって「正解」とするかの文書化。ビジネス部門との合意 |
| ラベルパイプライン | ソースシステムからラベルを自動抽出するETL |
| ラベル品質モニタリング | 欠損率、分布変化の監視 |
| アノテーション基盤 | 人間がラベル付けする場合のツール・ワークフロー（フリーテキスト分類等） |

### 1-5. データ分割（Train/Validation/Test）

```
❌ ランダム分割
   ML入門書では「データの80%で学習、20%でテスト」と書いてある
   → 時系列データでこれをやると未来の情報がリークする

✅ 時間ベース分割（時系列データの鉄則）

  ┌──────────────┬────────────┬──────────┐
  │   Training   │ Validation │   Test   │
  │  〜2025/06   │ 2025/07-09 │ 2025/10〜│
  └──────────────┴────────────┴──────────┘
  → 常に「過去で学習 → 未来を予測」の構造を維持
```

**SOMPOケアでの注意点**: 介護データは**季節性**がある（冬は転倒が増える、年度末に退職が増える等）。分割期間の選び方がモデル性能に直結する。

### 1-6. スキーマ進化（Schema Evolution）への対応

```
2028年に介護システムが刷新される（CareBase変更）
  → テーブル構造、カラム名、データ型が変わる可能性
  → 学習済みモデルが前提としている特徴量が消える

対策:
  ① Feature Storeの抽象化: モデルは「残業時間_90日平均」という名前で参照
     → 裏側のSQLが変わっても、Feature Storeの定義を更新すればモデルは動く
  ② データコントラクト: 上流テーブルの破壊的変更を検出してアラート
  ③ 特徴量の依存関係マップ: どの特徴量がどのソースに依存しているか可視化
```

## 2. 非構造データ: LLMとの相性 + 構造化以外にやること

（LLMとの相性が良いことは理解済みとのことなので、**それ以外**に焦点を当てる）

### 2-1. エンベディングの管理 — 「もう一つのFeature Store」

```
テキスト → エンベディング（ベクトル）に変換
          → ベクトルDBに格納
          → RAG検索で使う

ここで見落とされがちなこと:
  ① エンベディングモデルのバージョン管理
     モデルを変えるとベクトルの意味空間が変わる → 全再計算が必要
  ② エンベディングの鮮度
     新しい介護記録は日次で追加される → 増分エンベディングのパイプライン
  ③ チャンク戦略のバージョニング
     「1記録=1チャンク」か「500文字ごとに分割」かで検索精度が変わる
     → チャンク戦略を変えたら全再計算
```

### 2-2. 非構造→構造化のパイプライン（LLMで構造化抽出）

```
介護記録テキスト:
  「本日、入居者Aさんが食堂で転倒。右膝を打撲。看護師が確認し湿布対応。
   歩行器の使用を再度推奨。家族に連絡済み。」

↓ LLMで構造化抽出

{
  "event_type": "転倒",
  "location": "食堂",
  "injury": "右膝打撲",
  "response": "湿布対応",
  "follow_up": ["歩行器使用推奨", "家族連絡"],
  "severity": "軽度"
}

↓ 構造化テーブルに格納

→ これが「構造化された特徴量」としてML学習に使える
→ 「過去6ヶ月の転倒回数」「転倒の重症度推移」等のFeatureが作れる
```

**ポイント: LLMによる構造化抽出の精度管理**
- 抽出精度を定期的にベンチマーク（人間のアノテーションと比較）
- 抽出ルールのバージョン管理（プロンプトのバージョニング）
- 抽出結果に confidence score を付与（低スコアは人間レビュー）

### 2-3. マルチモーダルデータの将来

```
今は考えなくていいが、3-5年後に来るもの:
  ├─ 画像: 褥瘡（じょくそう）の写真 → 重症度自動判定
  ├─ 音声: 夜間の音声モニタリング → 異常検知
  ├─ センサー: IoTデバイス（バイタル、活動量）→ リアルタイム特徴量
  └─ 動画: 見守りカメラ → 転倒検知

データ基盤として「今」やるべきこと:
  → マルチモーダルデータの格納スキーマを拡張可能な設計にしておく
  → S3/GCS等のオブジェクトストレージとDWHの連携パスを確保
```

### 3-2. 各ステップの詳細

#### ① 仮説定義

ここは人間の仕事。だが**記録が重要**。

```yaml
# 仮説管理テンプレート
hypothesis:
  id: HYP-001
  title: "スタッフ離職率による稼働率予測"
  business_question: "3ヶ月後に稼働率が80%を下回る施設を事前に特定したい"
  target: "3ヶ月後の稼働率が80%未満か否か（二値分類）"
  features_hypothesis:
    - 現在の稼働率
    - 直近3ヶ月のスタッフ離職率
    - 施設の築年数
    - 地域の高齢者人口
  success_criteria:
    - "AUC > 0.75"
    - "適合率 > 0.6（偽陽性を減らしたい）"
  owner: "データサイエンスチーム"
  created: "2026-04-01"
```

#### ② データセット作成

```python
# Feature Storeからの学習データ作成（概念コード）

# 1. 予測対象期間を定義
prediction_dates = generate_monthly_dates("2023-01", "2025-06")

# 2. 各時点でのFeatureを取得（Point-in-Time）
features = feature_store.get_historical_features(
    entity_ids=all_facility_ids,
    feature_names=["occupancy_rate", "staff_turnover_90d", "building_age", ...],
    timestamps=prediction_dates  # ← この時点の値を取得
)

# 3. ラベル（正解）を結合
labels = get_labels(
    entity_ids=all_facility_ids,
    target="occupancy_below_80pct",
    prediction_horizon="3months",
    timestamps=prediction_dates
)

# 4. 結合してデータセットを作成
dataset = features.join(labels)

# 5. バージョニング
dataset.save(version="v1.0", path="s3://ml-datasets/occupancy-prediction/")
mlflow.log_artifact("dataset_manifest.json")  # メタデータ記録
```

**バージョニングで記録すべきもの**:
```yaml
dataset_manifest:
  version: "v1.0"
  created_at: "2026-04-15T10:30:00Z"
  feature_store_version: "fs-2026-04-15"
  features: ["occupancy_rate", "staff_turnover_90d", ...]
  label: "occupancy_below_80pct_3months"
  date_range: "2023-01 to 2025-06"
  row_count: 45000
  label_distribution: {0: 38000, 1: 7000}  # 不均衡の確認
  split:
    train: "2023-01 to 2024-12"
    validation: "2025-01 to 2025-03"
    test: "2025-04 to 2025-06"
```

#### ③ 実験管理

```
MLflow（or 同等ツール）で記録するもの:

  実験1:
    モデル: LightGBM
    パラメータ: {max_depth: 6, learning_rate: 0.1, ...}
    データセット: v1.0
    メトリクス: {AUC: 0.78, precision: 0.65, recall: 0.72}
    実行時間: 45秒
    特徴量重要度: {staff_turnover_90d: 0.35, occupancy_rate: 0.28, ...}

  実験2:
    モデル: XGBoost
    パラメータ: {max_depth: 8, learning_rate: 0.05, ...}
    データセット: v1.0
    メトリクス: {AUC: 0.81, precision: 0.68, recall: 0.70}
    ...

→ 全実験が比較可能な形で記録される
→ 「なぜこのモデルを選んだか」の根拠になる（監査対応）
```

#### ④ 評価

**ML指標とビジネス指標の紐付けが重要**。

```
ML指標:
  AUC = 0.81, Precision = 0.68, Recall = 0.70

ビジネス翻訳:
  「稼働率低下を3ヶ月前に予測」
  → 予測が当たった場合（Recall 70%）: 低下する施設の70%を事前にキャッチ
  → 予測が外れた場合（Precision 68%）: アラートの32%は空振り
  → 空振りコスト vs 見逃しコスト のバランスで閾値を決める

  施設1,000箇所 × 月次予測:
    稼働率低下施設（実際）: 約50施設/月
    正しく予測: 35施設 (70%)  → 事前対策が取れる
    見逃し: 15施設 (30%)     → 従来通り事後対応
    空振り: 16施設            → 不要な対策コスト
```

#### ⑤ モデル登録

```
Model Registry のステージ管理:

  Development → Staging → Production → Archived
       │            │           │
       │            │           └─ 本番稼働中。変更には承認が必要
       │            └─ テスト環境で検証中。本番データの一部で動作確認
       └─ 開発中。実験段階

承認フロー:
  Staging → Production の昇格には:
    ✓ テストデータでの精度が基準を満たす
    ✓ バイアスチェック（特定施設に偏っていないか）
    ✓ 責任者の承認（データサイエンスリード + ビジネスオーナー）
```

#### ⑥ サービング

```
2つのパターン:

バッチ推論（大半のユースケース）:
  ├─ 毎月1日に全施設の3ヶ月後稼働率を予測
  ├─ 結果をGold Layerのテーブルに書き込み
  └─ BI/ダッシュボードで閲覧可能に

  スケジューリング: Databricks Workflows / Airflow
  出力先: Delta Table / Snowflake Table

リアルタイム推論（限定的なユースケース）:
  ├─ 入居者の状態変化をリアルタイムでリスク判定
  ├─ APIエンドポイントとして公開
  └─ レイテンシ要件: <500ms

  基盤: SageMaker Endpoints / Databricks Model Serving
```

#### ⑦ モニタリング

```
監視すべき4つのドリフト:

① データドリフト（入力データの分布変化）
   「先月から施設の平均稼働率が10%下がった」
   → モデルが学習した分布と乖離 → 予測精度が落ちる可能性
   監視方法: KS検定、PSI（Population Stability Index）

② 予測ドリフト（出力の分布変化）
   「先月までアラートが月50件だったのに、今月は200件」
   → モデルの挙動が変わった。入力データの変化が原因か確認

③ コンセプトドリフト（入力と結果の関係性の変化）
   「離職率が高くても稼働率が下がらなくなった」（人材派遣で補填等）
   → モデルが前提としていた因果関係が崩れた → 再学習が必要

④ パフォーマンスドリフト（精度の劣化）
   予測と実績を突き合わせ、AUCが低下していないか定期チェック
   → 「予測は80%低下と言ったが、実際は85%だった」のズレを集計
```

#### ⑧ 再学習

```
再学習のトリガー:

定期型:
  ├─ 月次/四半期で自動再学習
  └─ 新データを追加して全期間で再学習

イベント型:
  ├─ データドリフト検知 → 自動再学習パイプライン起動
  ├─ 精度が閾値を下回った → アラート → 再学習判断
  └─ 上流データソース変更（2028年CareBase刷新）→ Feature定義見直し + 再学習

自動再学習パイプライン:
  データドリフト検知
       │
       ▼
  新データセット作成（Feature Store + 最新ラベル）
       │
       ▼
  学習（前回と同じモデル構成 + 新データ）
       │
       ▼
  自動評価（テストセットで精度チェック）
       │
       ├─ 精度向上 → Staging登録 → 人間レビュー → Production昇格
       │
       └─ 精度低下 → アラート → 人間が原因調査
```

## 4. LLM × 構造化データ: セマンティックレイヤーの追加要件

（doc 07-08で記述済みだが、構造化データ固有の観点を補足）

### 4-1. LLMが構造化データを「正しく」使うために必要なもの

```
普通のDWH:
  テーブル定義（DDL）があればSQLが書ける

LLMがSQLを書くために追加で必要なもの:
  ① カラムの意味（「occ_rate」= 稼働率 = 入居者数÷定員×100）
  ② テーブル間の結合条件（FKだけでなくビジネス上の結合ルール）
  ③ 暗黙のフィルタ条件（「有効なデータ」= is_deleted=false AND date >= '2023-01-01'）
  ④ 集計の粒度ルール（「施設の稼働率」は日次？月次？）
  ⑤ 値の解釈（status=1は「在籍」、status=2は「退職」）
  ⑥ NULL の扱い方（NULL = 未入力？データなし？0扱い？）
```

**これらを「セマンティックモデル」として定義しておかないと、LLMは毎回違うSQLを生成する**。

### 4-2. 構造化データのコンテキスト設計

```yaml
# LLMに渡すコンテキスト（セマンティックモデル定義例）
tables:
  - name: facility_daily_metrics
    description: "施設の日次KPI。1施設1日1レコード"
    primary_key: [facility_id, metric_date]
    columns:
      - name: occupancy_rate
        description: "稼働率(%) = 入居者数 / 定員 × 100"
        type: DECIMAL(5,2)
        range: "0-100"
        null_means: "その日のデータが未入力"
      - name: staff_count
        description: "当日の出勤スタッフ数（常勤換算）"
        note: "非常勤は0.5として換算"
    common_filters:
      - "metric_date >= '2023-01-01'（2023年以前はデータ品質が低い）"
      - "is_active = true（閉鎖施設を除外）"
    common_joins:
      - target: facility_master
        on: "facility_id = facility_master.id"
        note: "施設名・地域・定員を取得"
```

## 5. 技術基盤の選択肢まとめ

### 5-1. 各コンポーネントの選択肢

| レイヤー | Databricks構成 | Snowflake構成 | AWS Native構成 |
|---|---|---|---|
| **Feature Store** | Databricks Feature Store | Snowflake + Feast | SageMaker Feature Store |
| **実験管理** | MLflow (組込み) | MLflow (外部) | SageMaker Experiments |
| **モデルレジストリ** | Unity Catalog Models | MLflow Registry | SageMaker Model Registry |
| **バッチ推論** | Databricks Jobs | Snowpark ML | SageMaker Batch Transform |
| **リアルタイム推論** | Model Serving | Snowflake Cortex | SageMaker Endpoints |
| **モニタリング** | Lakehouse Monitoring | 外部ツール (Evidently等) | SageMaker Model Monitor |
| **LLM連携** | AI-BI Genie | Cortex Analyst | Bedrock |
| **ベクトルDB** | Vector Search | Cortex Search | OpenSearch / Bedrock KB |
| **セマンティックレイヤー** | Unity Catalog | Semantic Views | dbt MetricFlow |
| **自動再学習** | Databricks Workflows | Snowpark + Airflow | SageMaker Pipelines |

### 5-2. SOMPOケアでの推奨

```
Phase 1 (PoC): まずは「Databricks構成」か「Snowflake構成」を検証
  → Feature Store / 実験管理 / セマンティックレイヤーの3つをPoC範囲に含める
  → 1つのKPI予測（稼働率 or 離職率）をEnd-to-Endで回す

Phase 2 (要件定義): MLOps基盤の詳細設計
  → 再学習パイプライン、モニタリング、承認フローの設計
  → 非構造データパイプラインの設計（エンベディング基盤含む）

Phase 3 (構築): 段階的に機能追加
  → 最初はバッチ推論のみ → 運用安定後にリアルタイムを追加
  → 最初は手動再学習 → 運用安定後に自動化
```

## 6. AI出力の管理 — 出力もデータである

### 6-1. なぜ「AI出力」を保存・管理する必要があるか

従来のBIではダッシュボードは「画面表示」であり、データとして保存されない。AI OSでは**AIが生成した分析・予測・提案自体がデータ資産**になる。

```
従来のBI:
  データ → ダッシュボード → 人間が見る → 終わり（出力は揮発性）

AI OS:
  データ → AI分析/予測 → 構造化された出力 → 人間がレビュー → 意思決定
                │                                    │
                ▼                                    ▼
          AI Output Store                   Feedback Store
          (出力のバージョン管理)              (修正・採択・却下の記録)
```

### 6-2. AI Output Store の設計

| 保存対象 | SOMPOケアの例 | なぜ保存するか |
|---|---|---|
| **予測結果** | 施設の稼働率予測、転倒リスク評価 | 「3ヶ月前にAIは何と予測したか」を検証するため |
| **分析レポート** | 施設の改善レコメンド、異常検知アラート | 監査時にAIの判断根拠を説明するため |
| **生成コンテンツ** | ケアプラン提案、会議資料の自動生成 | 「提案内容」と「実際に採用された内容」の差分追跡 |
| **LLMクエリ+結果** | 自然言語質問→SQL→結果のセット | 精度評価・監査・セマンティックモデル改善のため |

```yaml
# AI Output Storeのスキーマ例
ai_output:
  output_id: "uuid"
  output_type: "prediction | analysis | recommendation | llm_query"
  model_id: "model-registry-ref"       # どのモデルが生成したか
  model_version: "v1.2.3"
  input_data_ref: "s3://..."           # 入力データへの参照
  output_content: {}                    # 構造化された出力
  confidence_score: 0.85
  created_at: "2026-04-01T10:00:00Z"
  created_for: "facility_A"            # 対象エンティティ
  status: "generated | reviewed | accepted | rejected"
  feedback: []                          # フィードバック配列
  ttl_days: 365                         # 保存期間
```

**基盤への要件**: Gold Layer（BI用）とは別に、AI Output Storeのテーブル群が必要。Delta Table上に構築し、Time Travelでバージョン追跡可能にする。

### 6-3. フィードバックループのデータパイプライン

AIの出力に対する人間のフィードバックは、**次の改善サイクルの入力データ**になる。

```
フィードバックの種類:

① 明示的フィードバック（人間が意図的に提供）
   ├─ 修正:  「稼働率予測85%→90%に修正。近隣新施設の影響は織り込み済み」
   ├─ 採択:  「このケアプラン提案を採用した」
   ├─ 却下:  「この分析は的外れ。理由: ...」
   └─ 評価:  👍/👎、5段階評価

② 暗黙的フィードバック（行動から推測）
   ├─ 閲覧: AI出力を見た/見なかった
   ├─ 再質問: 同じ質問を別の聞き方で聞き直した → 初回の回答が不十分
   ├─ 無視: 予測アラートを無視した → アラートの精度/価値が低い
   └─ 引用: AI出力を会議資料に貼り付けた → 信頼されている

③ 結果フィードバック（Ground Truth）
   ├─ 予測の答え合わせ: 「3ヶ月後の稼働率予測」→ 実際の稼働率
   ├─ レコメンドの効果: 「改善提案を実行」→ KPIが改善したか
   └─ リスクの実現: 「転倒リスク高」→ 実際に転倒したか
```

**フィードバック→改善のパイプライン**:

```
                    ┌─────────────────────────┐
                    │   Feedback Store         │
                    │   (Delta Table)          │
                    └───────┬─────────────────┘
                            │
              ┌─────────────┼─────────────────┐
              ▼             ▼                   ▼
     プロンプト改善    Feature改善         モデル再学習
     (LLM系)         (特徴量追加/修正)    (MLモデル系)
              │             │                   │
              ▼             ▼                   ▼
     プロンプト       Feature Store         Model Registry
     レジストリ       更新                  新バージョン
```

**基盤への要件**:
- Feedback Storeテーブル（AI Output Storeと紐付け）
- フィードバック集計パイプライン（週次/月次でフィードバックを集計→改善チケット化）
- 結果フィードバックの自動収集（予測→実績の突き合わせバッチジョブ）

## 7. チェックリスト: 普通のDWHで見落としがちなAI固有要件

```
データの持ち方:
  □ Point-in-Time Correctnessが保証されるデータ設計か
  □ Feature Storeのレイヤーがアーキテクチャに含まれているか
  □ 日次スナップショット（SCD強化版）の設計があるか
  □ ラベル（正解データ）のパイプラインが設計されているか
  □ データセットのバージョニング方法が決まっているか
  □ Train/Validation/Testの時間ベース分割の設計があるか

学習サイクル:
  □ 実験管理ツール（MLflow等）の選定があるか
  □ モデルレジストリとステージ管理の設計があるか
  □ モデル昇格の承認フローが定義されているか
  □ バッチ推論のスケジューリング設計があるか
  □ 再学習のトリガー条件が定義されているか

評価・モニタリング:
  □ データドリフト検知の仕組みがあるか
  □ 予測精度の定期ベンチマークが設計されているか
  □ ML指標とビジネスKPIの紐付けがあるか
  □ モデルのバイアスチェック手法が決まっているか

LLM連携:
  □ セマンティックモデル（カラムの意味、結合条件、フィルタ）が定義可能か
  □ エンベディング生成・更新パイプラインの設計があるか
  □ ベクトルDB / 検索基盤の選定があるか
  □ LLM生成SQLのバリデーション機構があるか

AI出力・フィードバック:
  □ AI出力（予測・分析・提案）の保存・バージョン管理設計があるか
  □ AI出力に信頼度スコア・根拠データ参照が付与されるか
  □ フィードバック収集の仕組み（明示的 + 暗黙的）があるか
  □ 予測→実績の自動突き合わせパイプラインがあるか
  □ フィードバック→改善のサイクル（プロンプト改善/Feature改善/再学習）が設計されているか
  □ AI出力の監査ログ保存期間が定義されているか

ガバナンス:
  □ AI学習に使えるデータの分類ポリシーがあるか
  □ モデルの監査証跡（どのデータで学習したか）が追跡可能か
  □ Feature定義のGit管理・変更履歴管理があるか
  □ スキーマ変更時のFeature破損検知があるか
```
