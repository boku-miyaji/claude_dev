# LLMネイティブ時代のデータ基盤設計

> 作成日: 2026-03-03 | 視点: 「LLMが分析の主役になる」前提で、従来の選定軸を再考する

---

## 1. パラダイムシフト: ノーコード/ローコードは「過渡期の発想」

### 従来の考え方
```
ビジネスユーザー → ノーコードETLツール → データ加工 → BI閲覧
                   (HULFT/Glue DataBrew)
```

### これから起きること
```
ビジネスユーザー → 自然言語で指示 → LLM → SQL/Python自動生成 → 実行 → 結果
                   「施設Aの先月の稼働率を前年比で」
```

Claude Code、Cursor、Databricks AI-BI Genie、Snowflake Cortex Analyst、Tableau Agent――これらは全部、**「ユーザーの自然言語をコードに変換して実行する」** という同じ方向を向いている。

つまり**ノーコード/ローコードツールは「GUIでポチポチ」を「自然言語で話す」に置き換えられる過渡期の産物**になりつつある。HULFTやGlue DataBrewのGUIより、LLMに「このCSVを前処理してSnowflakeに入れて」と言うほうが速い時代が来ている（というか、もう来ている）。

### これが本PJTに意味すること

RFI回答で「ノーコードETL」を売りにしている日立システムズ（HULFT Square）やSalesforce（Tableau Prep）の優位性は、**2-3年後には大幅に薄まる**。逆に、LLMが直接叩けるAPIやSQLインターフェースを持つ基盤（Databricks/Snowflake）のほうが、LLMネイティブ時代との相性が良い。

## 2. LLMネイティブ基盤で何が変わるか

### 2-1. 分析の民主化が「次の段階」に入る

**従来のデータ民主化**:
- IT部門がダッシュボードを作る → ビジネスユーザーが閲覧する
- 「セルフサービスBI」= Explorer権限でフィルター/ドリルダウンできる

**LLM時代のデータ民主化**:
- ビジネスユーザーが自然言語で質問する → LLMがSQL生成 → 結果を返す
- ダッシュボードを事前に作る必要がない。**質問駆動**で分析が動く
- 施設長が「今月の転倒リスクが高い入居者は？」と聞けば答えが返る

### 2-2. 各社のLLM対応状況（RFIから読み取れるもの）

| 製品 | LLMインターフェース | 成熟度 |
|---|---|---|
| **Databricks AI-BI Genie** | 自然言語 → SQL → 結果。Unity Catalogのメタデータを活用 | 実用段階 |
| **Snowflake Cortex Analyst** | 自然言語 → SQL。セマンティックモデル定義が必要 | 実用段階 |
| **Tableau Agent** | 自然言語 → ビジュアル生成。Tableau Pulse と統合 | GA |
| **Tableau MCP** | **外部LLM（Claude等）からTableauのデータにアクセス可能** | OSS公開済 |
| **Data 360 MCP Server** | Claude Desktop/Cursor等から自然言語→SQLでData 360クエリ | 公開済 |
| **Snowflake Copilot** | SQL Worksheet上でのSQL自動生成補助 | GA |
| **Amazon Q（QuickSight）** | 自然言語でダッシュボード生成 | GA |

### 2-3. Claude Code / Cursor のようなツールの位置づけ

Tableau MCPやData 360 MCPが示唆しているのは、**BIツール自体がLLMのバックエンドになる**未来。

```
施設長 → Claude Desktop → Tableau MCP → Tableau Cloud → データ取得 → 回答生成
```

この世界では:
- **ダッシュボードを「見に行く」必要がない**。チャットで聞けば返ってくる
- **Explorerライセンスが不要になる可能性がある**。LLMがExplorerの仕事をする
- **BIツールはAPIレイヤーに後退**し、UIはLLMチャットが担う

## 3. だからこそ危険なこと: ガバナンスの崩壊

LLMが分析を民主化するほど、**統制の問題が指数関数的に大きくなる**。以下は具体的な地雷。

### 地雷1: 「誰でもデータに触れる」= 誰でも壊せる

**問題**: LLMが自然言語からSQL/Pythonを生成し、個々人が自由にデータを加工・保存できるようになると、**「自分用に加工したデータ」が乱立**する。

```
施設長A: 「稼働率を計算して」 → LLMが独自集計 → 施設長Aの稼働率: 92%
施設長B: 「うちの稼働率は？」 → LLMが別ロジックで集計 → 施設長Bの稼働率: 87%
本部: 「全社稼働率」→ また別のロジック → 本部の稼働率: 89%
```

**全員が違う数字を「正しい」と思っている状態**。これはExcel地獄の再来であり、Foundryが解決しようとしていた問題そのもの。

**対策: Single Source of Truth（SSOT）の設計**

```
                      ┌──────────────────┐
                      │  Gold Layer      │ ← ここだけが「公式データ」
                      │  (認定済データ)    │    ・定義が文書化されている
                      │                  │    ・変更はPR/承認フロー
                      │  稼働率 = X/Y    │    ・バージョン管理されている
                      └────────┬─────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
         BI閲覧(Viewer)   LLMクエリ(読取専用)  API連携
         定型ダッシュボード  自然言語質問        外部システム
```

**ルール**:
- Gold Layerのデータ定義（KPI計算式等）は**コードとしてGit管理**
- Gold Layerの変更は**PRレビュー + 承認フロー必須**
- LLMがクエリできるのは**Gold Layer（読み取り専用）のみ**
- 個人が「自分用データマート」を作る場合は**Sandbox領域に限定**し、Goldとは分離

### 地雷2: LLMが「もっともらしい嘘」をつく

**問題**: LLMは統計的にもっともらしい回答を生成するが、**データの文脈を完全には理解しない**。

例:
- 「離職率が高い施設」→ LLMが「退職日」フィールドを使ったが、それは「有給消化開始日」だった
- 「入居者の平均年齢」→ 死亡退去者を含めて計算してしまった

**対策: セマンティックレイヤーの必須化**

LLMが直接テーブルを叩くのではなく、**ビジネス用語とデータの対応関係を定義したセマンティックレイヤー**を必ず挟む。

```yaml
# semantic_model.yaml（例）
metrics:
  - name: 稼働率
    description: "当月の入居者数 ÷ 定員数。退去日が当月内の者は入居者に含む"
    sql: "COUNT(CASE WHEN status='入居中' OR exit_date >= month_start THEN 1 END) / capacity"
    table: gold.facility_monthly
    filters:
      - "facility_type IN ('介護付', '住宅型', 'グループホーム')"
    owner: DDA戦略部
    last_reviewed: 2026-02-01

  - name: 離職率
    description: "当月の退職確定者数 ÷ 月初在籍者数。有給消化期間は在籍に含む"
    sql: "COUNT(CASE WHEN termination_confirmed_date BETWEEN ... END) / headcount_bom"
    table: gold.staff_monthly
    owner: 人事部
    last_reviewed: 2026-01-15
```

**Databricks AI-BI GenieやSnowflake Cortex Analystは、このセマンティックモデル定義を前提にしている**。LLMが勝手にSQLを書くのではなく、定義済みのメトリクスに基づいてクエリを生成する。

→ **これがないと「もっともらしいが間違った分析」が量産される**

### 地雷3: アクセス権限の粒度問題

**問題**: LLMが代理でデータにアクセスする場合、**「誰の権限で」実行されるかが曖昧になる**。

```
施設長A → Claude Desktop → MCP → Tableau/DWH → 全施設のデータが見えてしまう？
```

**対策: LLMアクセスに対する権限モデル**

```
┌─────────────────────────────────────────────────┐
│              権限管理の3レイヤー                   │
│                                                 │
│  Layer 1: 認証 (Authentication)                 │
│    誰がLLMを使っているか                          │
│    → SSO/IdP連携（Azure AD等）                   │
│                                                 │
│  Layer 2: 認可 (Authorization)                   │
│    その人がどのデータにアクセスできるか              │
│    → 行レベルセキュリティ（施設単位/部門単位）      │
│    → LLMのクエリにWHERE句を自動注入               │
│                                                 │
│  Layer 3: 操作制限 (Action Control)              │
│    読み取り専用か、書き込みも可能か                 │
│    → Gold Layer: 全員READ ONLY                   │
│    → Sandbox: 本人のみREAD/WRITE                 │
│    → LLMによるDDL/DML: 原則禁止                  │
│                                                 │
└─────────────────────────────────────────────────┘
```

**重要**: LLMがSELECT文だけでなく、CREATE TABLE / INSERT / UPDATE / DELETEを生成する可能性がある。**LLM経由のクエリはSELECTのみに制限する**のが原則。データを変更する操作は従来のパイプライン（dbt/DLT）経由のみ許可。

### 地雷4: マスターデータの汚染

**問題**: 複数人がLLMを使って「データ整備」を始めると、マスターデータが各自の解釈で書き換えられる。

例:
- 営業担当が「この施設名、旧名称になってるから直して」→ LLMがUPDATE実行
- 別の担当が「いや、法的にはまだ旧名称」→ 戻す
- 元のデータがどれだったかわからなくなる

**対策: Immutable Data + Change Logパターン**

```
マスタ更新は直接UPDATEしない。常にINSERTで履歴を積む。

facility_master:
  facility_id | name          | valid_from | valid_to   | changed_by | change_reason
  001         | ○○ケアホーム   | 2020-01-01 | 2026-02-28 | system     | 初期登録
  001         | ○○ケアハウス   | 2026-03-01 | 9999-12-31 | 山田太郎    | 名称変更（法人変更に伴う）
```

- 変更履歴が全て残る
- 「いつ時点のデータが正しかったか」を遡れる
- LLMにはcurrent view（valid_to = 9999-12-31のレコード）のみを見せる

### 地雷5: 「分析結果」の信頼性担保

**問題**: LLMが生成した分析結果を、**誰がどう検証するか**のプロセスがない。

施設長がLLMに「うちの施設の改善点は？」と聞いて得た回答を、そのまま本部への報告に使ってしまう → 実はデータの読み間違いだった → 経営判断を誤る

**対策: 分析結果の信頼度レベル**

```
Level 1: 公式KPI（Gold Layerから直接取得）
  → 定義済み、レビュー済み、自動更新
  → そのまま経営判断に使える
  → 例: 稼働率、離職率、事故発生率

Level 2: 定型分析（承認済みクエリテンプレートから生成）
  → セマンティックモデルに基づくLLM生成
  → 生成されたSQLが可視化され、検証可能
  → 例: 前年同月比、施設間比較

Level 3: アドホック分析（LLMによる自由なクエリ）
  → LLMが自力で生成したSQL/分析
  → 必ず「生成されたSQL」を表示し、ユーザーが確認
  → 「この分析は未検証です」のウォーターマーク付き
  → 意思決定に使う場合はIT/データチームのレビュー必須

Level 4: 推論・予測（MLモデル出力）
  → モデルの精度・バイアスに依存
  → 信頼区間や不確実性を必ず表示
  → 「参考値」として扱い、最終判断は人間
```

## 8. まとめ: 3年後に「正しかった」と言える選定基準

ノーコード/ローコードETLツールは**3年後には「レガシー」になる可能性が高い**。LLMがコードを書く世界では、重要なのは:

1. **基盤がLLMと対話できるインターフェースを持っているか**（SQL API、MCP、セマンティックモデル）
2. **「正しいデータ」を定義・維持する仕組みがあるか**（SSOT、セマンティックレイヤー、変更管理）
3. **誰が何にアクセスできるかを、LLM経由でも制御できるか**（権限継承、SELECT制限、監査ログ）
4. **LLMが出した答えの信頼性を検証できるか**（SQL可視化、信頼度レベル、レビューフロー）

ツールが変わっても、この4つの設計原則は変わらない。**ツール選定よりもデータガバナンスの設計に時間を使うべき**。

## 9. LLM利用のコスト爆発問題

### 問題の構造

全員がLLMで自由に分析できる = **全員がAPIコールを無限に叩ける**。

```
従来: ダッシュボード閲覧 → コスト≒固定（Viewerライセンス + DWHクエリ）
LLM時代: 自然言語質問 → LLM APIコール + DWHクエリ が質問のたびに発生
```

コストの発生箇所:
1. **LLM API利用料**: Claude/GPT等のトークン課金。1質問あたり数円〜数十円
2. **DWHコンピュート**: LLMが生成したSQLの実行コスト。複雑なクエリほど高い
3. **Databricksクラスター**: クエリ負荷に応じてスケールアップ → 従量課金が膨らむ
4. **Snowflakeクレジット**: クエリごとにクレジット消費。大量の非効率クエリで爆発

### 施設1,000箇所 × 数千人が使うと何が起きるか

```
仮に1人が1日5回LLMに質問すると:
  3,000人 × 5回/日 × 20営業日 = 月間30万クエリ

LLM API: 30万回 × ¥10/回 = ¥300万/月
DWHクエリ: 30万回 × ¥5/回(平均) = ¥150万/月
合計: 約¥450万/月 = ¥5,400万/年

→ これだけで現行Foundryのライセンス(1億超)の半分に迫る
```

しかもこれは**平均的な使い方**の試算。施設長が「全施設の全データを5年分クロス集計して」と聞くと1クエリで数百〜数千円になることもある。

### 対策: LLMコストのガバナンス設計

#### 対策1: ユーザー/部門別のクォータ制

```yaml
cost_quota:
  tier_1_executive:  # 経営層
    llm_queries_per_month: unlimited
    dwh_compute_budget: ¥500,000/月
    alert_at: 80%

  tier_2_manager:  # 施設長・部門長
    llm_queries_per_month: 200
    dwh_compute_budget: ¥50,000/月
    alert_at: 80%
    hard_limit: true

  tier_3_staff:  # 現場スタッフ
    llm_queries_per_month: 30
    dwh_compute_budget: ¥10,000/月
    alert_at: 80%
    hard_limit: true
```

#### 対策2: クエリのキャッシュと共有

```
同じ質問を100人が聞くなら、実行は1回でいい。

「先月の稼働率は？」
  → 最初の1人: LLM生成 + SQLクエリ実行 → コスト発生
  → 2人目以降: キャッシュされた結果を返す → コスト≒ゼロ

Tableau Pulseはまさにこのアプローチ:
  メトリクスを定義 → 自動更新 → 全員がフォロー → 個別クエリ不要
```

**Tableau Pulseが「LLMネイティブかつコスト効率的」な理由はここにある**。個々人がLLMに質問するのではなく、**定義済みメトリクスをAIが自動で監視し、変化があったら通知**する。質問駆動ではなく**通知駆動**。

#### 対策3: クエリ複雑度の制限

```
LLMが生成するSQLに以下のガードレールを設ける:

- JOINは最大3テーブルまで
- スキャン行数の上限（例: 1000万行）
- 実行時間の上限（例: 30秒。タイムアウトしたら集計済みGold Layerに誘導）
- CROSS JOINは禁止
- SELECT * は禁止（必要カラムのみ）
```

#### 対策4: LLMモデルの使い分け

```
全部の質問をClaude Opus/GPT-4oで処理する必要はない。

簡単な質問（定型KPI参照）: Haiku/GPT-4o-mini（低コスト）
中程度の質問（アドホック集計）: Sonnet/GPT-4o
複雑な分析（多テーブルJOIN + 解釈）: Opus/o3

→ ルーティングを入れるだけでLLMコストは1/3〜1/5に減る
```

#### 対策5: 定型化のサイクル

```
LLMへの質問ログを分析 → よくある質問を特定 → 定型ダッシュボード/メトリクスに昇格

月間30万クエリのうち:
  - 60%は同じ質問のバリエーション → 定型化
  - 30%は部門固有の分析 → テンプレート化
  - 10%が本当のアドホック → ここだけLLMフル活用

定型化するほどLLMコストが下がるフィードバックループ
```

### コスト最適化アーキテクチャ

```
ユーザーの質問
     │
     ▼
┌─────────────┐
│ ルーター     │ ← 質問の種類を判定
│             │
│ 定型質問？ ──▶ キャッシュ/メトリクス参照（コスト≒ゼロ）
│             │
│ テンプレ？ ──▶ 小モデル(Haiku) + テンプレSQL（低コスト）
│             │
│ アドホック? ─▶ 大モデル(Opus) + SQL生成（高コスト・クォータ消費）
│             │
│ 複雑すぎ？ ──▶ 「データチームに依頼してください」（人間にフォールバック）
└─────────────┘
```

### この視点がRFI回答に完全に欠けている

**5社すべてが「LLMで分析できます」とは言っているが、「LLMコストをどう制御するか」を提案している会社はゼロ**。PoCの評価項目に「LLM利用時の月間コストシミュレーション」を追加すべき。

## 10. Palantir AIPという「不都合な事実」

### 10-1. Foundryはすでに「LLMネイティブ基盤」だった

本ドキュメントでここまで「新プラットフォームに必要な要件」として挙げてきたものの多くが、**実はPalantir Foundryにすでに実装されている**。

| 本書の要件 | Foundry AIPの対応機能 | ステータス |
|---|---|---|
| セマンティックレイヤー | **オントロジー**（ビジネス概念のグラフ構造。KPI定義・関係性を保持） | GA（egakuで構築済み） |
| 自然言語→データ操作 | **AI FDE**（自然言語でFoundryを操作。パイプライン・コードリポ・オントロジー管理） | Beta (2025/11〜) |
| LLMのグラウンディング（幻覚防止） | **Ontology-Augmented Generation (OAG)**（LLMをオントロジーに固定。確定的ロジックと組合せ） | GA |
| ノーコードLLM関数 | **AIP Logic**（オントロジーオブジェクトを操作するLLM関数をノーコードで構築） | GA |
| 文書抽出 | **AIP Document Intelligence**（ローコードの文書抽出ワークフロー） | GA (2026/2/4〜) |
| ML学習・デプロイ | **Model Studio**（ノーコードでMLモデルをトレーニング・デプロイ） | GA (2026/2/9〜) |
| 権限制御 | オントロジーベースのアクセス制御（すでにegakuで運用中） | GA |
| データリネージ | Foundry標準機能（パイプライン全体の依存関係追跡） | GA |

### 10-2. オントロジー = セマンティックレイヤーそのもの

> "The Palantir Ontology is the core data model and semantic layer — the 'business-aware' representation of an organization's entire data universe, turning raw data into actionable, real-world concepts that both humans and AI systems can understand and reason over."

セクション3で「セマンティックレイヤーが必須」「KPI定義をYAMLでGit管理すべき」と書いたが、**egakuのオントロジーはすでにその役割を果たしている**。しかも、YAMLファイルではなくグラフ構造として、はるかにリッチな関係性を表現できる。

### 10-3. OAG: RAGを超えるグラウンディング

一般的なLLMデータ分析（Databricks Genie、Snowflake Cortex等）はRAG（Retrieval-Augmented Generation）ベース:
```
質問 → 関連データを検索 → LLMに渡す → 回答生成
```

Palantir AIPは**OAG（Ontology-Augmented Generation）**:
```
質問 → オントロジーの構造を理解 → 確定的ロジック（予測・最適化）も使う
     → LLMは「解釈と表現」に集中 → 回答生成 → ソースシステムに書き戻しも可能
```

この違いは大きい。RAGは「検索した範囲のデータ」しかLLMに渡せないが、OAGは**ビジネスの構造そのもの**をLLMが理解する。「施設Aの稼働率」と「施設Aのスタッフ配置」の**関係性**をオントロジーが保持しているから、LLMは両方を結びつけた回答ができる。

### 10-4. AI FDE: 自然言語でFoundryを操作

AI FDE（AI-powered Forward Deployed Engineer）は、本書セクション1で「これから起きること」として描いた世界を**すでにFoundry上で実現している**:

```
従来: エンジニアがPySpark書く → コードレビュー → デプロイ
AI FDE: 「このCSVを取り込んで、施設ごとに集計して、稼働率のオブジェクトタイプを作って」
       → AI FDEが自動でTransformコード生成 → オントロジー更新
```

対応モデル: Anthropic Claude、OpenAI GPT、Google Gemini、xAI — マルチモデル対応済み。

### 10-5. プロジェクトへの影響: 「移行」の前提を問い直す

#### 現在の前提
```
Foundryは高い（年間1億超）→ より安いプラットフォームに移行する
```

#### 再考すべきコスト比較

| 項目 | Foundry継続+AIP活用 | 新プラットフォーム構築 |
|---|---|---|
| ライセンスコスト | 高い（年間1億超。ただし交渉余地あり） | 低い（OSS中心なら数千万） |
| オントロジー（セマンティックレイヤー） | **すでに構築済み**（egakuで運用中） | **ゼロから作り直し**（数千万〜数億の工数） |
| LLMグラウンディング | OAGが標準装備 | 自前で設計・構築が必要 |
| 権限管理 | オントロジーベースで運用中 | 新規設計（行レベル、LLM経由含む） |
| データリネージ | 標準機能 | dbt lineage等で再構築 |
| 500+パイプライン移行 | **不要** | PySpark→dbt/SQL書き換え（巨大工数） |
| 学習コスト | 既存ユーザーは継続利用可 | 全ユーザーが新ツール学習 |
| LLMネイティブ機能 | AIP Logic, AI FDE, Model Studio | Databricks Genie, dbt等の組合せ |
| ベンダーロックイン | 強い（Palantir独自） | 弱い（OSSベース） |

**「ライセンスが高い」は事実だが、移行コスト（特にオントロジー再構築とパイプライン書き換え）を加算すると、Foundry継続のほうが3〜5年のTCOでは安い可能性がある**。

#### RFIベンダーの中での位置づけ

- **富士通**: Foundry継続オプションを提案した唯一のベンダー。DI Basic（Foundry+Databricks+dbtハイブリッド）案は、この観点で再評価に値する
- **他4社**: すべて「Foundryからの完全移行」を前提。オントロジー移行のコスト・リスクを過小評価している可能性

### 10-6. それでもFoundryから離れるべき理由はあるか

LLMネイティブ機能があるからといって、Foundry継続が自動的に正解になるわけではない。以下のリスクは依然として存在する:

1. **ベンダーロックイン**: Palantirへの依存が深まる。オントロジー・パイプラインすべてがPalantir独自形式
2. **コスト交渉力**: 競合不在になると値上げリスクが高い。特にAIP利用で追加課金が発生する可能性
3. **エコシステムの広さ**: Databricks/Snowflakeのほうがサードパーティツール・人材・ナレッジが豊富
4. **AI FDEの成熟度**: まだBeta。本番運用には時間がかかる
5. **オントロジーの「呪い」**: 一度オントロジーに深くコミットすると、将来の移行がさらに困難に

### 10-7. 推奨: 「Foundry継続」をPoCの評価対象に加える

```
現行方針: 5社のRFI回答からベンダー選定 → Foundryから移行

提案: Foundry+AIPを「6番目の選択肢」としてPoCに含める

PoC比較項目:
  1. 同一KPI（稼働率等）を各プラットフォームで定義 → どちらが速い・正確か
  2. 自然言語クエリの精度比較 → AI FDE vs Databricks Genie vs Snowflake Cortex
  3. 権限制御の確実性 → オントロジーベース vs 行レベルフィルタ
  4. 5年TCO比較 → ライセンス + 移行コスト + 運用コスト + LLMコスト
  5. ベンダーロックイン脱出コスト → 将来Foundryから離れる場合の移行コスト見積
```

**少なくとも「Foundry+AIPのままで何がどこまでできるか」を検証せずに、1億超のライセンス削減だけを理由に移行を決めるのはリスクが高い**。
