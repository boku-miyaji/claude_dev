# AI OS視点での論点整理 — 「普通のデータ基盤」では足りないこと

> 作成日: 2026-03-03 | 役割: 従来のデータ基盤設計では見落とされるAI開発・運用視点の論点を先方に提示する

---

## 0. 前提: 「データ基盤」と「AI OS」の違い

```
従来の「データ基盤」が考えること:
  データを集める → 加工する → 可視化する → 人間が見て判断する

「AI OS」が考えること:
  データを集める → AIが理解できる形にする → AIが分析・判断する
  → 人間が検証する → AIが学習する → より良い判断をする → ループ
```

**最大の違い: データの「消費者」が人間からAIに変わる**

BIダッシュボードは人間が読む。だからUIが重要だった。
AI OSでは**AIがデータを読む**。だから重要なのはUIではなく:
- AIが「正しく」データを理解するための**セマンティックレイヤー**
- AIが「安全に」データにアクセスするための**権限・ガードレール**
- AIの出力が「信頼できる」ことを保証する**検証・監査の仕組み**

RFIの5社は「データ基盤」を提案している。我々の役割は**「AI OS」として何が足りないかを指摘すること**。

## 2. 【ためる】— AIが消費するデータをどう準備するか

### 戦略視点

#### 論点A: 構造化データだけでは「AIの力」の半分しか使えない

```
SOMPOケアのデータ資産:
  構造化データ: 施設運営データ、入居者情報、勤怠、財務 → 今のegakuでカバー済み
  非構造データ: 介護記録（フリーテキスト）、事故報告書、ケアプラン、研修資料
             → ほぼ未活用

RFI 5社の回答: 構造化データのETL/DWH/BIしか語っていない
```

**先方への問い**:
> 「介護記録のフリーテキスト（年間数百万件）をAIで分析する計画はあるか？ あるなら、非構造データの取り込み・前処理・エンベディング生成の基盤設計が要件に入っていない。これは後付けできない」

#### 論点B: 「AIに食わせていいデータ」と「ダメなデータ」の線引き

介護データには**要配慮個人情報**が含まれる（病歴、障害、介護度）。

```
データ種別と分類例:
  ├─ AI学習OK:  集計済み施設KPI、匿名化済みトレンドデータ
  ├─ AI推論OK（学習NG）: 個人を特定できるが、リアルタイム判断に必要なデータ
  ├─ AI利用NG:  生の医療記録、マイナンバー、家族情報
  └─ 要検討:    フリーテキストの介護記録（匿名化すればOK？）
```

**先方への問い**:
> 「AIモデルの学習に使えるデータと、推論（質問応答）にのみ使えるデータの分類ポリシーは策定済みか？ 個人情報保護法・介護保険法の観点で、データ分類とAI利用可否のマトリクスが必要」

#### 論点C: 組織横断ナレッジの設計

1,000施設に共通する「汎用ナレッジ」と、施設固有の「ローカルナレッジ」の切り分け。

```
汎用ナレッジ（全施設共通）:
  ├─ 介護業界の標準指標・ベンチマーク
  ├─ 法規制・コンプライアンス要件
  ├─ SOMPOケア全体の方針・ガイドライン
  └─ ベストプラクティス事例集

ローカルナレッジ（施設固有）:
  ├─ 施設ごとの入居者特性
  ├─ 地域特性（人材市場、競合状況）
  └─ 施設固有のオペレーション手順
```

**先方への問い**:
> 「AIが参照するナレッジベースのスコープと管理主体は？ 本部が一元管理か、施設が個別に追加できるか。後者の場合、ナレッジの品質管理はどうするか」

### 設計視点

#### 論点D: 非構造データの処理パイプライン

```
介護記録テキスト                      AI利用可能な形
   │                                    │
   ▼                                    ▼
[取り込み] → [PII検出・マスキング] → [チャンク分割] → [エンベディング生成] → [ベクトルDB]
                                                                              │
                                                      ┌──────────────────────┘
                                                      ▼
                                              [RAG検索] → [LLM回答]
```

**RFI回答で欠けているもの**:
- エンベディング生成基盤（どのモデル？ どこで実行？ コストは？）
- ベクトルDB（Databricks Vector Search? OpenSearch? Pinecone?）
- PII検出の精度と運用（日本語の介護記録で固有名詞を正しくマスクできるか？）

#### 論点E: データのバージョニング（AI学習用）

AIモデルを学習する場合、**「どのデータで学習したか」の再現性**が必要。

```
普通のDWH:
  テーブルは常に最新状態。過去のスナップショットは限定的

AI学習に必要なDWH:
  「2025年12月時点のデータセット」を正確に再現できること
  → モデルの再学習・検証・監査に必須
  → Delta Lake Time Travel / Iceberg Snapshot で対応可能だが、設計が必要
```

#### 論点F: データリネージの拡張 — 「AI経由」の追跡

```
従来のリネージ:
  ソースA → ETL → テーブルB → ETL → テーブルC → ダッシュボード

AI OSのリネージ:
  ソースA → ETL → テーブルB → LLMがSQL生成 → 結果 → ユーザーの意思決定
                                ↑                      ↓
                          セマンティックモデル      フィードバック → モデル改善
```

**先方への問い**:
> 「LLMが生成したSQLとその結果を、データリネージの一部として追跡する設計はあるか？ 監査時に『このAIの回答はどのデータに基づいているか』を遡れる必要がある」

## 3. 【つくる】— AIモデル・エージェントをどう開発・管理するか

### 戦略視点

#### 論点G: 「BIの延長」か「AI開発基盤」か — アーキテクチャの分岐点

```
Path A: BIの延長（RFI 5社が提案しているもの）
  データ → DWH → BIツールのAI機能（Genie/Cortex/Tableau Agent）
  → 既製品のAI機能を使うだけ。カスタマイズ性は低い

Path B: AI開発基盤（AI OSとして必要なもの）
  データ → DWH → セマンティックレイヤー → 自社開発AIエージェント
  → 介護ドメイン特化のAIを自分たちで作れる基盤
  → 転倒予測、ケアプラン生成、シフト最適化 等
```

**これが最大の戦略的論点**。Path Aなら既製品選定で済む。Path Bなら**ML/AIの開発・実験・デプロイ基盤**が必要になる。

**先方への問い**:
> 「3年後にSOMPOケア独自のAIモデル（転倒予測、離職予測等）を開発・運用する計画はあるか？ あるなら、今回のデータ基盤にML開発基盤（実験管理、モデルレジストリ、デプロイパイプライン）を含めるべき。後から追加すると基盤設計をやり直すことになる」

#### 論点H: マルチクラウド・マルチモデル戦略

```
モデル選択の現状:
  ├─ AWS (Bedrock): Claude, Titan, Llama
  ├─ Azure (OpenAI Service): GPT-4o, o3
  ├─ Google (Vertex): Gemini
  └─ Databricks (DBRX, Mosaic AI): OSS models

RFIベンダーの前提:
  PwC/Deloitte → AWS Bedrock
  Fujitsu → 明記なし（Databricksから利用？）
  Hitachi → Snowflake Cortex
  Salesforce → Einstein（Salesforce内蔵）
```

**先方への問い**:
> 「LLMベンダーは1年で勢力図が変わる。特定のLLMプロバイダにロックインされない設計が必要。『モデルを差し替え可能なインターフェース設計』はRFP要件に入っているか？」

#### 論点I: AI開発における情報統制 — 部署間の壁

```
懸念シナリオ:
  経営企画部が開発した離職予測モデルが、
  人事部のデータにアクセスして学習している
  → 部署間のデータ共有ポリシーに違反していないか？

  施設Aの介護記録で学習したモデルの知識に、
  施設Bの入居者情報が混入していないか？
  → データの「学習汚染」問題
```

**先方への問い**:
> 「AIモデルの学習に使うデータの部署間・施設間の境界はどう設計するか？ 全社共通モデルと施設固有モデルの使い分けは？」

### 設計視点

#### 論点J: AI開発の具体的な基盤要件（参照PJで「未検討が多い」とされた領域）

参照PJでも「AIモデル・エージェントの開発・運用の詳細は未検討項目が多い」と注記されていた。具体的に何が必要か:

```yaml
ai_development_platform:
  experiment_management:  # 実験管理
    - 仮説 → データセット → モデル → 評価のトラッキング
    - MLflow / Weights & Biases / Databricks Experiment Tracking
    - 「どのデータ・どのパラメータで学習したか」の完全記録

  model_registry:  # モデルレジストリ
    - 学習済みモデルのバージョン管理
    - ステージング → プロダクション の昇格フロー
    - モデルの承認ワークフロー（誰が本番投入を承認するか）

  prompt_management:  # プロンプト管理（LLM時代の新要件）
    - システムプロンプトのバージョン管理
    - プロンプトテンプレートのA/Bテスト
    - 「稼働率を聞くときのプロンプト」の標準化と品質管理

  evaluation_framework:  # 評価基盤
    - LLM出力の品質評価（正確性、安全性、バイアス）
    - 自動評価パイプライン（回帰テスト的に定期実行）
    - 人間評価のワークフロー（RLHF的フィードバック）

  agent_framework:  # エージェント管理
    - エージェントの権限定義（何にアクセスできるか）
    - エージェントの行動ログ（何をしたか）
    - エージェント間の連携設計（マルチエージェント）
```

#### 論点K: フリーテキストの価値化

SOMPOケア固有の**最大の未活用資産**は介護記録のフリーテキスト。

```
活用段階:
  Level 1: 検索（「転倒に関する記録を探す」）
  Level 2: 要約（「入居者Aの3ヶ月の状態変化を要約」）
  Level 3: 構造化（テキストから「転倒リスクスコア」を自動算出）
  Level 4: 予測（テキストパターンから「3ヶ月後の状態悪化」を予測）
  Level 5: 生成（ケアプランの自動起案）
```

**先方への問い**:
> 「介護記録のフリーテキストをどのレベルまで活用する想定か？ Level 1-2なら RAG + ベクトルDBで対応可能。Level 3以上ならファインチューニングや専用モデル開発が必要になり、基盤要件が大きく変わる」

## 4. 【つかう】— AIをどう安全に展開・運用するか

### 戦略視点

#### 論点L: AIエージェントのログ・監査 — 「AIが何をしたか」の可視化

```
従来の監査:
  「誰がどのダッシュボードを見たか」→ アクセスログで十分

AI OS時代の監査:
  「誰がAIに何を聞いたか」
  「AIはどのデータを使って回答したか」
  「AIの回答は正しかったか」
  「AIがアクション（データ書き込み等）を実行した場合、その根拠は何か」
  → 全部記録して、後から追跡できる必要がある
```

介護領域では**行政監査**がある。AIの判断根拠を説明できないと問題になる。

**先方への問い**:
> 「AIが生成した分析結果が経営判断に使われた場合、その判断根拠を行政監査で説明できるか？ AI監査ログの保存期間・保存形式・検索性の要件定義が必要」

#### 論点M: AI利用の規制対応 — 介護×AI固有の論点

```
一般的なAIガバナンス:
  ├─ AI倫理ガイドライン
  ├─ 個人情報保護法対応
  └─ 透明性・説明責任

介護×AI固有の論点:
  ├─ 要配慮個人情報の取扱い（病歴、障害情報、介護度）
  ├─ AIによるケア判断の責任所在（AIが「転倒リスク高」と出した場合、対応しなかった責任は？）
  ├─ 介護保険法上のデータ取扱い制限
  ├─ 利用者・家族への説明義務（AIで分析していることの告知）
  └─ 自治体ごとの規制差異
```

**先方への問い**:
> 「AIによる入居者のリスク判定や、ケアプラン提案を行う場合、法的責任の整理は済んでいるか？ 技術基盤の前に、AIが出していい判断・出してはいけない判断の線引きが必要」

### 設計視点

#### 論点N: エージェントの実行基盤と権限管理

```
エージェントの種類（将来的に必要になるもの）:
  ├─ 分析エージェント: 自然言語→SQL→結果返却（読み取り専用）
  ├─ レポートエージェント: 定期レポートの自動生成・配信
  ├─ アラートエージェント: 異常値検知→関係者に通知
  ├─ 予測エージェント: MLモデルの推論結果を提供
  └─ 業務エージェント: ケアプラン起案、シフト提案（書き込みあり）

権限の粒度:
  ├─ データアクセス: どのテーブル/行にアクセスできるか
  ├─ アクション権限: 読み取りのみ？書き込みも？
  ├─ コスト権限: 1回のクエリでいくらまで使えるか
  └─ 承認フロー: 書き込みアクションは人間の承認が必要か
```

**先方への問い**:
> 「AIエージェントの権限モデルは、ユーザーの権限とは別に設計が必要。エージェントAは全施設のデータを読めるが、ユーザーBは自施設のみ、という場合、エージェントはユーザーBの権限で動くべきか、エージェント自身の権限で動くべきか？」

#### 論点O: モニタリング・評価 — AIの「品質管理」

```
従来のシステム監視:
  CPU使用率、メモリ、レイテンシ、エラー率

AI OSの品質監視:
  ├─ 回答精度: LLMの回答が正しい割合（定期的なベンチマーク）
  ├─ ハルシネーション率: 根拠なく生成された回答の割合
  ├─ バイアス監視: 特定施設・特定属性への偏り
  ├─ ドリフト検知: 時間経過でモデル精度が劣化していないか
  ├─ コスト効率: 1回答あたりのコスト推移
  └─ ユーザー満足度: フィードバック（👍/👎）の集約
```

#### 論点P: コスト管理 — AI利用の従量課金の可視化と制御

（doc 07 セクション9で詳述済み。ここでは論点として再掲）

```
可視化すべきもの:
  ├─ 部門別・施設別のAI利用コスト
  ├─ エージェント種別ごとのコスト
  ├─ クエリ複雑度別のコスト分布
  └─ 月次トレンド（コスト増加の早期検知）

制御すべきもの:
  ├─ ユーザー/部門別のコストクォータ
  ├─ クエリ複雑度の上限
  ├─ モデルルーティング（コスト最適化）
  └─ キャッシュ活用率
```

| 参照PJの論点 | SOMPOケアでの対応 | 差異・追加 |
|---|---|---|
