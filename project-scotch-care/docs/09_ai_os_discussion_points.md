# AI OS視点での論点整理 — 「普通のデータ基盤」では足りないこと

> 作成日: 2026-03-03 | 更新日: 2026-03-04 | 役割: 従来のデータ基盤設計では見落とされるAI開発・運用視点の論点を先方に提示する

---

## 0. 前提: 「データ基盤」と「AI OS」の違い

**身近な例で理解する: 図書館とGoogle**

「データ基盤」は**図書館**に似ている:
- 本（データ）をきれいに整理する
- 棚（テーブル）に分類して並べる
- 目録（カタログ）を作る
- 人間が来て、本を探して、読んで、判断する

「AI OS」は**Google検索**に似ている:
- データを集めるだけでなく、**AIが読んで理解する**
- 人間は質問するだけで、AIが分析・判断してくれる
- AIの回答が間違っていたら、フィードバックで改善される
- **AIが学習して、どんどん賢くなる**

```
従来の「データ基盤」が考えること:
  データを集める → 加工する → 可視化する → 人間が見て判断する

「AI OS」が考えること:
  データを集める → AIが理解できる形にする → AIが分析・判断する
  → 人間が検証する → AIが学習する → より良い判断をする → ループ
```

**最大の違い: データの「読む人」が人間からAIに変わる**

BIダッシュボードは人間が読む。だからUIが重要だった。
AI OSでは**AIがデータを読む**。だから重要なのはUIではなく:
- AIが「正しく」データを理解するための**セマンティックレイヤー**（→ 定義集）
- AIが「安全に」データにアクセスするための**権限・ガードレール**（→ 安全装置）
- AIの出力が「信頼できる」ことを保証する**検証・監査の仕組み**（→ 品質検査）

**RFIの5社は「図書館」を提案している。我々の役割は「Googleに必要なものは何か」を指摘すること**。

## 1. フレームワーク: ためる × つくる × つかう

論点を「ためる（データ準備）→ つくる（AI開発）→ つかう（現場展開）」の3ステップで整理する。

```
      ためる                    つくる                      つかう
  ① データ収集・整備      ③ モデル・アプリ構築        ⑤ 展開・運用・改善
  ② データ生成           ④ 現場検証・FB収集          ⑥ 安全性・ガバナンス
```

## 2. 【ためる】— AIが食べるデータをどう準備するか

### 論点A: テキストデータ（非構造データ）を活用しないのは宝の持ち腐れ

**何が問題か**: RFI 5社が提案しているのは構造化データ（数値、マスタ）のパイプラインだけ。でもSOMPOケアには**AIにとって最も価値が高い「テキストデータ」**がほぼ手付かずで眠っている。

**SOMPOケアの事例**:
```
構造化データ（今のegakuでカバー済み）:
  施設運営データ、入居者情報、勤怠、財務 → 表形式の数字

非構造データ（ほぼ未活用 = 宝の山）:
  介護記録（フリーテキスト）→ 年間数百万件
  事故報告書 → 転倒、誤薬等の詳細な状況記録
  ケアプラン → 入居者ごとのケア方針
  研修資料 → ノウハウの蓄積
```

**他業界の事例**:
- **メイヨークリニック（米国最大の医療機関）**: 電子カルテの自由記述をAIで分析し、見落とされがちな症状パターンを発見。構造化データだけでは見つからなかった「薬の副作用」を年間数百件追加で検出
- **損保ジャパン（保険）**: 事故報告書のフリーテキストをAIで分析し、不正請求の検知精度を40%向上。数値データ（金額、日付）だけでは見抜けないパターンをテキストから発見
- **トヨタ（製造業）**: 工場の作業日報をAIで分析し、品質問題の予兆を検知。数値計測では正常でも、作業者のコメントから「いつもと違う」を検出

**先方への問い**:
> 「介護記録のフリーテキスト（年間数百万件）をAIで分析する計画はありますか？ あるなら、テキストデータの取り込み・個人情報マスキング・AI検索用の加工基盤が最初から必要です。**後から追加すると、データパイプライン全体を作り直すことになります**」

---

### 論点B: 「AIに食わせていいデータ」と「ダメなデータ」の線引き

**何が問題か**: 介護データには病歴・障害・介護度といった**要配慮個人情報**が含まれる。AIに学習させるデータと、AIが回答に使うデータの線引きが必要。

**身近な例**: SNSの写真投稿と同じ。「友達に見せていい写真」と「見せたくない写真」がある。AIでも同じで、「AIが学習に使っていいデータ」と「使ってはいけないデータ」がある。

```
SOMPOケアでのデータ分類例:

  AIの学習に使ってOK:
    集計済み施設KPI（稼働率85%、離職率12%等）
    匿名化済みのトレンドデータ

  AIが質問に答える時だけ使ってOK（学習はNG）:
    個人の介護度情報（リアルタイムのケア判断に必要だから）
    施設ごとのスタッフ配置情報

  AI利用NG:
    生の医療記録（診断名、投薬内容）
    マイナンバー、家族の連絡先
    入居者の顔写真

  要検討:
    介護記録のフリーテキスト
    → 「Aさんが転倒」は個人情報。匿名化すれば使える？
    → 匿名化の精度は？日本語の介護用語で固有名詞を正しくマスクできる？
```

**他業界の事例**:
- **GDPR（EU個人情報保護法）対応企業**: データを「学習用」「推論用」「利用禁止」の3段階に分類するのが標準。大手銀行ではAI学習に使うデータセットの承認に毎回法務レビューが入る
- **Apple（プライバシーバイデザイン）**: Siriの音声データは「端末上で処理」と「クラウドで処理」を分離。センシティブなデータは端末から出ない設計

**先方への問い**:
> 「AIモデルの学習に使えるデータと、回答にだけ使えるデータの分類ポリシーは作っていますか？ これが決まらないと**PoCのデータセットすら作れません**」

### 論点C: 1,000施設の「知恵」をどう集約するか

**何が問題か**: 1,000施設にはそれぞれ蓄積されたノウハウがある。全施設共通の知識と、施設ごとの知識をどう分けて管理するか。

**身近な例**: チェーンレストランのレシピ管理。
- 全店共通の基本レシピ（本部管理）→ どの店でも同じ味
- 地域限定メニュー（店舗管理）→ 地域特性に合わせた味
- この2つを**混ぜない**ことが大事

```
全施設共通ナレッジ（本部管理）:
  介護業界の標準指標・ベンチマーク
  法規制・コンプライアンス要件
  SOMPOケア全体の方針・ガイドライン
  ベストプラクティス事例集

施設固有ナレッジ（施設管理）:
  施設ごとの入居者特性（認知症の割合が多い等）
  地域の人材市場情報（介護士の採用難易度）
  施設固有のオペレーション（夜勤体制等）
```

**他業界の事例**:
- **マクドナルド**: 全世界共通のオペレーションマニュアル（本部）+ 各国ローカルメニュー（地域）をナレッジDBで管理。AIによる新人研修にも活用
- **セブン-イレブン**: 全店共通の発注AIモデル（天候・曜日ベース）に、店舗ごとの売上特性をパーソナライズ。共通モデル+ローカル調整のハイブリッド方式

---

### 論点D: テキストデータをAIが使える形にする工場（パイプライン）

**何が問題か**: 介護記録のフリーテキストは「そのまま」ではAIが使えない。AIが検索・分析できる形に加工する「工場」が必要。

```
介護記録テキスト                      AI利用可能な形
  「本日、入居者Aさんが                  {event: "転倒",
   食堂で転倒。右膝を打撲。    →         location: "食堂",
   看護師が確認し湿布対応」               injury: "右膝打撲",
                                         response: "湿布"}
         ↑                                    ↑
    人間は読めるが                     AIが検索・分析できる
    AIには使いにくい                    構造化された形

この変換の途中でやること:
  ① 個人情報の検出と置換（「Aさん」→「入居者001」）
  ② テキストの分割（長い文章を適切な長さに区切る）
  ③ AIが検索しやすい形に変換（ベクトル化）
  ④ 検索用データベースに格納
```

**他業界の事例**:
- **アストラゼネカ（製薬）**: 臨床試験の報告書（年間数万件のテキスト）を自動的に構造化。副作用の報告パターンを検出し、新薬の安全性評価を大幅に効率化
- **楽天（EC）**: 商品レビューのテキストをAIで分析。「壊れやすい」「サイズが小さい」等の品質課題を自動抽出し、出品者にフィードバック

## 3. 【つくる】— AIモデル・エージェントをどう開発・管理するか

### 論点G: 「BIの延長」か「AI開発基盤」か — これが最大の分岐点

**何が問題か**: 今回のプロジェクトの最も根本的な問い。「既製品のAI機能を使うだけ」か「自分たちでAIを開発できる基盤を作る」かで、必要なものが全然違う。

```
Path A: BIの延長（RFI 5社が提案しているもの）
  例えるなら: 「完成品の家電を買う」
  データ → DWH → BIツールのAI機能（Genie/Cortex/Tableau Agent）
  → 稼働率ダッシュボードにAI質問機能がつく程度
  → すぐ使えるが、SOMPOケア独自のAIは作れない

Path B: AI開発基盤（AI OSとして必要なもの）
  例えるなら: 「工房を作って自分で家具を作る」
  データ → DWH → セマンティックレイヤー → 自社開発AIエージェント
  → 転倒予測AI、ケアプラン生成AI、シフト最適化AI 等を自分たちで開発
  → 時間はかかるが、介護ドメイン特化のAIが持てる
```

**他業界の事例**:
- **Path Aの事例（Walmart）**: TableauのAI機能で売上分析を効率化。既製品活用で短期間にROI達成。ただし独自の需要予測AIは別途AWS SageMakerで構築
- **Path Bの事例（Netflix）**: 推薦アルゴリズムを自社開発。独自のML基盤を構築し、全社で200以上のMLモデルを運用。これが競争優位の源泉
- **中間の事例（スターバックス）**: BIはTableau（既製品）、需要予測と在庫最適化は自社MLモデル。基盤はDatabricks上に構築。**最初はBIの延長で始め、徐々にAI開発基盤に進化**させた

**先方への問い**:
> 「3年後にSOMPOケア独自のAIモデル（転倒予測、離職予測等）を開発・運用する計画はありますか？ あるなら、今回のデータ基盤に**AI開発のための基盤**（実験管理、モデル管理、評価の仕組み）を含めるべきです。これは後から追加すると**基盤設計をやり直す**ことになります」

---

### 論点H: 「AIの乗り換え」ができる設計にしておく

**何が問題か**: 今「GPT-4oが最高」でも、半年後に別のAIが出てくる可能性がある。特定のAIに依存する設計は危険。

**身近な例**: スマホの充電器。Lightning専用の充電器を10本買ったら、USB-Cに変わった時に全部無駄になる。最初からUSB-C（＝標準規格）で統一しておけばよかった。

```
AIモデルの世界も同じ:
  2024年: GPT-4が最強 → OpenAIに全部依存した設計
  2025年: Claude 3.5が出た → OpenAIの設計だと切り替えが大変
  2026年: Gemini 2.5やオープンソースモデルが台頭 → また作り直し？

対策: 「モデル切替可能な設計」= LLMゲートウェイ
  アプリ → [LLMゲートウェイ] → GPT-4 / Claude / Gemini / DBRX
                  ↑
            ここで切り替えるだけ。アプリ側は変更不要
```

**他業界の事例**:
- **Stripe**: 決済処理で複数の銀行APIを抽象化。銀行を切り替えても加盟店側のコード変更が不要。同じ発想をLLMに適用するのがLLMゲートウェイ

---

### 論点J: AI開発に具体的に必要なもの（「部品リスト」）

**身近な例**: 料理に例えると、AI開発基盤には以下の「調理器具」が必要:

```
AI開発の「調理器具」:

① レシピノート（実験管理）
   どんな食材で、どんな分量で、どんな調理法を試したか記録
   → 「この組み合わせが一番美味しかった」がわかる
   技術: MLflow / Weights & Biases

② 冷蔵庫（モデルレジストリ）
   完成した料理（学習済みAIモデル）を保管・管理
   → 「昨日作ったカレー」と「今日のカレー」どっちが美味しい？
   → 美味しい方だけお客さんに出す（本番昇格）
   技術: MLflow Model Registry

③ メニューテンプレート集（プロンプト管理）※LLM時代の新要件
   AIへの質問の仕方（プロンプト）の標準テンプレート
   → 「稼働率を聞くときはこのプロンプト」と統一
   → バージョン管理して、どのプロンプトが精度が高いかA/Bテスト

④ 味見のルール（評価基盤）
   AIの回答が正しいか、安全か、偏りがないかを自動チェック
   → 定期的に「10問テスト」を実施して精度を計測
   → 精度が下がったらアラート

⑤ 配膳ルール（エージェント管理）
   どのAIが何のデータにアクセスしていいか
   何をしたか全部記録
   「注文取り消し（=データ削除）」は店長（=人間）の承認が必要
```

### 論点K: 介護記録テキストの「使い道」5段階

SOMPOケア固有の**最大の未活用資産**は介護記録のフリーテキスト。活用のレベルは5段階ある:

```
Level 1: 検索 →「転倒に関する記録を探して」
  必要な基盤: ベクトルDB + RAG
  他社事例: 法律事務所の判例検索（Allen & Overy）

Level 2: 要約 →「入居者Aの3ヶ月の状態変化を要約して」
  必要な基盤: Level 1 + LLMの要約機能
  他社事例: 診療記録の要約（Epic Systems）

Level 3: 構造化 →「テキストから転倒リスクスコアを自動算出」
  必要な基盤: LLMによる情報抽出パイプライン
  他社事例: 保険の事故報告書からの損害自動査定（Lemonade）

Level 4: 予測 →「テキストパターンから3ヶ月後の状態悪化を予測」
  必要な基盤: Level 3 + MLモデル
  他社事例: 電子カルテからの再入院リスク予測（Google Health）

Level 5: 生成 →「ケアプランの自動起案」
  必要な基盤: Level 1-4のすべて + 人間レビューワークフロー
  他社事例: 治療計画の自動生成支援（Tempus）
```

**先方への問い**:
> 「介護記録をどのレベルまで活用する想定ですか？ Level 1-2ならRAGで対応可能です。Level 3以上なら専用のAI開発が必要になり、必要な基盤が大きく変わります」

## 4. 【つかう】— AIをどう安全に展開・運用するか

### 論点L: 「AIが何をしたか」の記録 — これが一番後付けが難しい

**何が問題か**: BIダッシュボードなら「誰がどの画面を見たか」のログだけで済んだ。AIの場合は「AIが何をしたか」の記録が必要で、これは**後から追加するのが極めて難しい**。

**身近な例**: 自動運転車のドライブレコーダー。事故が起きた時に「AIはなぜその判断をしたか」を後から検証できないと困る。介護AIも同じで、「AIはなぜこの入居者を『転倒リスク高』と判定したか」を説明できる必要がある。

```
記録すべきもの（介護の例）:

 AIが「施設Aの稼働率低下リスクあり」と出力した場合:
  ├─ 誰がAIに聞いたか → 本部の○○部長（8:30AM）
  ├─ AIはどのデータを使ったか → 過去3年の入退居データ、地域人口統計
  ├─ AIはどういう推論をしたか → 「近隣に新規施設が開業予定」を検知
  ├─ AIの回答は正しかったか → 3ヶ月後の実績と突き合わせ
  └─ 人間はどう判断したか → 「対策を実施した」/「無視した」
```

**他業界の事例**:
- **EU AI Act（2026年施行）**: 高リスクAI（医療、金融等）には監査ログの保存が法的義務。「AIの判断根拠を説明できない」は法律違反
- **MUFG（三菱UFJ銀行）**: 融資審査AIの全判断を記録。金融庁の検査で「なぜこの企業への融資を拒否したか」を説明できる体制
- **米国FDA**: AIを使った医療機器には「AIの判断プロセスの可視化」が承認要件

**先方への問い**:
> 「AIの判断根拠を行政監査で説明できますか？ 介護領域には実地指導があります。**AI監査ログの設計は初期段階で組み込まないと、本番後に作り直しになります**」

---

### 論点M: 介護 × AI 固有の規制リスク

**何が問題か**: 一般的なAIガバナンスに加え、介護ならではの規制・倫理的課題がある。

```
一般的なAIの心配ごと:
  個人情報漏洩、AIの偏り（バイアス）、説明責任

介護 × AI 固有の心配ごと:
  ├─ 要配慮個人情報: 病歴、障害情報、介護度はAIでどこまで使っていいか
  ├─ 判断の責任: AIが「転倒リスク高」と出して、対応しなかったら誰の責任？
  ├─ 告知義務: 「あなたのデータはAIで分析しています」と利用者に伝える必要があるか
  └─ 自治体差: 地域によって規制が異なる
```

**他業界の事例**:
- **AIによる入退院判断支援（米国）**: ある病院がAIの推奨に従って患者を退院させたが、その後容態が悪化。「AIの判断に従っただけ」は免責にならず、医師の責任が問われた
- **採用AI（Amazon）**: 過去の採用データで学習したAIが女性を差別する結果を出し、利用中止に。介護でも「特定施設の入居者に不利な予測を出す」バイアスの可能性

### 論点N: AIエージェントの「何をしていいか」の管理

**身近な例**: 会社の新入社員に似ている。最初は「資料のコピー」しか任せないが、信頼が積み重なると「顧客対応」も任せるようになる。AIエージェントも同じで、段階的に権限を広げる。

```
AIエージェントの種類と権限:

  分析エージェント（新入社員レベル）:
    できること: データを読んで、質問に答える
    できないこと: データの書き換え、外部への通知
    → 「稼働率を教えて」に答えるだけ

  レポートエージェント（一般社員レベル）:
    できること: 分析 + レポート作成 + メール配信
    できないこと: データの更新、判断の実行
    → 「月次レポートを作ってメールで送って」

  業務エージェント（管理職レベル）:
    できること: 分析 + 提案 + データ更新 ← ★人間の承認が必要
    できないこと: 承認なしでのアクション実行
    → 「ケアプランを起案して」→ 人間が確認 → 承認 → 登録
```
