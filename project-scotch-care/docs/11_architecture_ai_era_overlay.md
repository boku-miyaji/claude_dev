# AI-Ready アーキテクチャへの追加論点 — 「普通のデータ基盤」から「AI OS」へ

> 作成日: 2026-03-04 | 目的: 現行のAI-Ready基盤アーキテクチャ（Databricks構成）に対して、AI開発・運用視点で足りない要素・深掘りすべき論点を整理する。
> 参照: AI-Ready データ基盤アーキテクチャ図（Databricks構成、L0-L7）、AI OS設計最新 v1.1

---

## 0. 現行アーキテクチャの評価

現行の7層アーキテクチャは「AI-Ready」を謳っており、L5にMLflow/Feature Store/RAG等を配置している点で**基盤としての骨格は正しい**。ただし、**データ基盤の延長でAIを載せている設計**であり、**「AIのライフサイクル全体を回す」視点**が不足している。

具体的には:
- **入力側（ためる）は充実**: L0→L1→L3のデータパイプラインは堅実
- **ML基盤（つくる）は部品が並んでいる**: L5にMLflow等があるが、**それらをどう繋いで回すか**の設計が見えない
- **出力側（つかう）が薄い**: L7のTableau+AIチャットだけ。AIの出力管理・フィードバック・監査が不在
- **横断的な統制が不足**: AI固有のガバナンス（コスト制御、LLM監査、エージェント権限）が未整理

**さらに上位の問題**: このアーキテクチャは「AI-Readyなデータ基盤」であり「AI OS」ではない。AI OSとは、**役に立つAIエージェントを、高速に・大量に・安全に作り、効率的に管理・運用し、陳腐化せず進化し続ける状態を実現する技術基盤と運用体制**のこと。現行図はデータの流れは描いているが、**AI自体のライフサイクル（開発→評価→デプロイ→運用→進化）を回す仕組み**が見えない。

---

## 0.5. AI OSの7つの設計軸 — データ基盤設計には無い「OS」の視点

> 参照: AI OS設計最新 v1.1 (Slide 22)

AI OSを「単なるデータ基盤+AI機能」にしないために、**7つの設計軸**で各論点を整理する。一般的なデータ基盤では①連携性と③実効性しか議論されないが、AI OSとして成立するには7つすべてが必要。

### 7つの設計軸と整備フェーズ

```
┌────────────────────────────────────────────────────────────────┐
│  Ph.1（開発初期・PoC）  │  Ph.2（構築・運用初期）  │  Ph.3（運用中期〜）  │
│  ★最優先で設計         │  本格構築               │  成熟・拡張          │
├────────────────────────────────────────────────────────────────┤
│  ① 連携性              │  ③ 実効性               │  ⑥ 運用性            │
│  ② 安全性              │  ④ 速度                 │  ⑦ 進化性            │
│                        │  ⑤ 規模                 │                      │
└────────────────────────────────────────────────────────────────┘
  ※ ⑥⑦は可能ならPh.1で設計だけは実施し、開発で概念を踏襲するのが良い
```

**現行アーキテクチャ図（L0-L7）は③実効性（データ→AI→出力のパイプライン）に偏重**。他の6軸の設計が不十分。以下、各軸で何が足りないかを整理する。

### 軸① 連携性 — マルチクラウド・マルチサービスの中でどう繋ぐか

```
論点:
  ├─ AWS上のDatabricksから、Azure上のSFDC/Agentforceとどう連携するか？
  ├─ マルチクラウド環境でのエージェント間連携（MCP? API Gateway?）
  ├─ 既存の介護システム（ほのぼの、Axist等）との双方向連携
  └─ LLMプロバイダ（Bedrock/Azure OpenAI/直接API）の統一的な呼び出し

現行図で足りないもの:
  → LLMゲートウェイ（後述の新規レイヤー）
  → システム間連携のプロトコル設計（MCP含む）
```

### 軸② 安全性 — 4本柱 + 最優先4機能

**一般的なデータ基盤の「セキュリティ」とは次元が違う**。AIは「自律的に動く」ため、人間のアクセス制御とは別の統制が必要。

#### 安全性の4本柱

```
┌────────────────────────────────────────────────────────────┐
│ 1. ID・権限管理                                            │
│    ├─ AIの独自ID: AIにもIDを持たせ、人間と同等の監査対象に  │
│    ├─ 動的な権限昇格: 必要に応じた権限取得と一時的特権付与  │
│    ├─ 権限の継承: 親エージェント→子、中間データへの継承    │
│    └─ アクセス権の同期: 元データ変更時の派生データへの影響  │
├────────────────────────────────────────────────────────────┤
│ 2. データガバナンス                                        │
│    ├─ データ汚染対策: 学習データへの悪意あるデータ混入検知  │
│    ├─ 推論リスク管理: AIが新たな機密情報を「推論」で生成    │
│    ├─ 動的マスキング: AIによる機微情報の自動マスキング      │
│    └─ メタデータアクセス権: マスク済データへのアクセス管理  │
├────────────────────────────────────────────────────────────┤
│ 3. AIの制御・ガードレール                                  │
│    ├─ 入力ガードレール: プロンプトインジェクション検知拒否  │
│    ├─ 出力ガードレール: ハルシネーション・機密情報ブロック  │
│    └─ 自律性の閾値: AIの「実行」vs「提案」の判断基準       │
├────────────────────────────────────────────────────────────┤
│ 4. 監査・透明性・責任                                      │
│    ├─ 推論プロセスの記録: AIの内部推論経路の保全            │
│    ├─ データリネージ: 出力の根拠ドキュメントの追跡可能性    │
│    └─ 責任分界点の明確化: 損害発生時の責任分担の証跡記録   │
└────────────────────────────────────────────────────────────┘
```

#### 最優先で構築すべき4つの安全基盤機能

| 機能 | 内容 | 現行アーキテクチャの対応 |
|---|---|---|
| **AIの独自ID + 権限設計** | AIにもIDを持たせ、人間の権限をAIに継承。人間の承認による一時的特権付与 | **未対応**。Unity CatalogのRLSは人間向け |
| **入出力ガードレール** | 入力チェック（悪意・誤り）+ 出力チェック（ハルシネーション・機密情報） | **未対応** |
| **トレーサビリティとリネージ** | AI不適切動作時の原因追跡。データソース→コンテキスト→指示→出力の全経路 | **部分対応**（dbt lineageのみ。AI推論経路は未対応） |
| **Human-in-the-Loopワークフロー** | 不可逆アクション（データ削除、メール送信等）は人間承認を強制。承認ログ記録 | **未対応** |

**SOMPOケアでの具体例**:
```
AIが「入居者Aの転倒リスクは高い」と判定 →
  Q: このAIはどの権限で入居者Aのデータにアクセスしたか？ → AI独自ID
  Q: 入力データに不正はなかったか？ → 入力ガードレール
  Q: AIの判断根拠は？ → トレーサビリティ
  Q: この判定に基づいて施設が何かアクションする前に、人間の確認は入るか？ → HITL
```

### 軸③ 実効性 — ExpertAI: 汎用AIではなく「介護のプロ」にする

```
ExpertAI = 専門知識 + 自社ナレッジを融合したAI

汎用LLM: 「介護施設の稼働率とは入居者数÷定員です」（教科書的）
ExpertAI: 「御社の稼働率は冬季に5%下がる傾向があり、
           近隣の○○施設の開設が影響している可能性があります。
           過去3年の同時期データと地域人口動態から...」（自社データに基づく）

ExpertAIの実現に必要なもの:
  ① 暗黙知の形式知化: 現場ノウハウをAIが参照できる形にする
  ② データのAI-Ready化: 非構造データ含めてAIが活用できる品質にする
  ③ エージェント設計: 学習ループ・評価基準・フィードバックの仕組み
```

### 軸④ 速度 — 人間の思考速度に同期する

```
AI OSでは「人間の認知速度」に基づく速度分類が必要:

  即座（<3秒）: チャット応答、検索結果、リアルタイムアラート
  対話（<30秒）: 複雑な分析クエリ、レポート要約
  バッチ（<30分）: 全施設の月次予測、大規模レポート生成
  深考（数時間）: 戦略分析、複数データソースの統合分析

高速化戦略:
  ├─ モデルの使い分け: 軽量モデル(Haiku) vs 重量モデル(Opus)
  ├─ 思考の深さ制御: CoT連鎖回数、reasoning_effortパラメータ
  ├─ 検索範囲最適化: 全件探索 vs top-N vs メタデータ絞り込み
  ├─ 事前の知識体系化: FAQ化、インサイト抽出
  ├─ キャッシュ: 同一質問の結果再利用
  └─ トークン節約: コンテキスト圧縮、不要情報の除外

→ 速度と精度はトレードオフ。「誰がどう判断するか」の設計が必要
→ 現行アーキテクチャにモデルルーティング/速度制御の仕組みが無い
```

### 軸⑤ 規模 — エージェント「量産」の時代

```
AIネイティブ企業では、1社あたり数十〜数百のエージェントが必要になる。
「1エージェントあたり数ヶ月」のペースでは全社展開に数十年かかる。

SOMPOケアの場合:
  ├─ 稼働率予測エージェント
  ├─ 離職予測エージェント
  ├─ 転倒リスク評価エージェント
  ├─ ケアプラン支援エージェント
  ├─ 施設改善レコメンドエージェント
  ├─ 自然言語分析エージェント
  ├─ レポート自動生成エージェント
  ├─ 問い合わせ対応エージェント
  └─ ... 将来的に数十以上

課題:
  ├─ エージェントの「サイロ化」: 各部署がバラバラに作ると共有されない
  ├─ 開発速度: AI開発を「ソフトウェア開発レベル」→「ドキュメント作成レベル」に
  └─ 評価の困難さ: AI挙動は確率的。作るたびに人間が確認は非効率
```

### 軸⑥ 運用性 — 「測れる・変えられる・人が回せる」

**一般的なシステム運用とAIエージェント運用は根本的に違う**。

```
一般的なシステム: 決定論的。同じ入力→同じ出力。障害は明確。
AIエージェント: 確率的。同じ入力→毎回違う出力。「品質劣化」が静かに進行。

AIエージェント運用の難しさ:
  ├─ 故障点の増大: LLM/ツール/データ/ワークフロー/権限が相互依存
  ├─ 品質劣化の原因不明: モデル？プロンプト？データ？ツール？
  ├─ 観測・説明責任: 結果だけでなく「経路・使用データ・途中判断」の追跡が必要
  └─ スケール時の破綻: エージェント増殖で重複・シャドー化・コスト膨張
```

#### 運用性の3軸

| 軸 | 内容 | 現行アーキテクチャの対応 |
|---|---|---|
| **測れる** | トレース（LLM生成/ツールI-O/状態遷移/ガードレール発火/コスト/品質）、ゴールデンセット評価 | **未対応** |
| **変えられる** | prompt/model/tool/datasourceの版管理。回帰テスト。段階リリース。ロールバック基準 | **部分対応**（MLflow/dbtのバージョニングのみ） |
| **人が回せる** | 責任分界（人間 vs AI）、結果レビュー/修正/承認/再実行/停止のUI、HITL設計 | **未対応** |

#### 初手で作るべき運用基盤（最小セット）

```
□ 標準トレース + 監査ログ
  「ただのログ」ではなく、経路再現できるトレース前提
  生成・ツール・handoff・ガードレール・状態遷移を共通フォーマットで

□ ゴールデンセット + 自動評価ハーネス
  タスク別に品質・コスト・速度を常時計測（回帰・劣化検知）
  リスクシナリオ + 自動テスト群（禁止回答・危険操作の検知）

□ Versioning + 回帰テスト + 安全なリリース
  tool/datasource/prompt/modelを版管理し、eval結果と紐付け
  段階リリースとロールバック基準を固定

□ SLO/SLA + エスカレーション + 停止手順
  品質閾値、エスカレーション条件、責任分界点を明文化
  重大時の停止・切替（人手運用への退避）の手順化

□ Agent Ops/PMO + 業務運用UI/UXの型
  横断の評価・監視・改善の常設チーム
  結果レビュー/修正/承認/再実行/停止のUI/UXパターン化

□ エージェントカタログ
  目的・KPI・所有者・権限・コスト・更新状況をメタデータ化
  再利用導線を最初から組み込む（重複開発抑止）
```

### 軸⑦ 進化性 — 「陳腐化せず、進化し続ける」

```
AI OSが直面する進化の課題:
  ├─ 変化の多重化: モデル更新/API変更/プロンプト変更/ナレッジ更新が同時進行
  ├─ ナレッジの毒化: SSOT不在、v混在、鮮度管理不備でRAG品質が継続毀損
  ├─ 評価が本質的に難しい: 多段・多ターン・複数品質軸で単発検証では守れない
  └─ 説明不能=改善不能: 「なぜその結論か」を追跡できないと改善が止まる

初手で作るべきもの:
  ① Trace-first: エージェント作業ログの可視化・監査対応
  ② Eval基盤の標準化: 回帰テスト、ドリフト監視、SLO/SLA、ロールバック基準
  ③ ナレッジSSOS: v混在防止、古い推奨の排除、失敗知見のアーカイブ
  ④ フィードバックループの共通部品: Good/Bad+コメント→バックログ→横展開
  ⑤ 体制・変更プロセス: Agent Ops + 設計審査/本番審査(Go/No-Go)
```

### 非構造データのメダリオンアーキテクチャ

一般的なメダリオン（Bronze→Silver→Gold）は構造化データ向け。非構造データには**別のメダリオン**が必要:

```
非構造データのメダリオン:

Bronze（生ファイル）:
  PPTファイル、PDFファイル、介護記録テキスト、音声ファイル
  → ソースシステムからの同期（定期実行）

Silver（加工処理データ）:
  Markdownデータ、構造化テキスト
  → ドキュメント解析、表・図の解析、テキスト化
  → LLMによる構造化抽出（テキスト→JSON）
  → PII検出・マスキング

Gold（ナレッジデータ）:
  ├─ ベクトルデータ（ベクトルDB）← RAG検索用
  ├─ メタデータ（RDB）← ファイル情報、チャンク情報、カテゴリ
  └─ グラフデータ（将来）← エンティティ間の関係性

メタデータの粒度:
  ファイル単位: ファイル名、著者、カテゴリ、作成日
  ページ単位: ページ数
  チャンク単位: チャンク文章、チャンク数
```

**現行アーキテクチャのL3（dbt Core/PySpark）では非構造データのSilver→Gold変換をカバーできない**。LLMベースの変換パイプラインが別途必要。

## 1. L0（データソース）への追加論点

### 現状: 構造化データソースの列挙

```
現行: 介護記録(340GB) / 眠りSCAN(1.92TB) / ナースコール(7.5GB) / ほのぼの / Axist / LIFE
```

### 追加すべき観点

#### 論点① データの「AI利用可否」分類が未定義

```
L0のデータソースそれぞれに、以下の分類が必要:

  介護記録(340GB):
    ├─ 構造化フィールド → AI学習OK / AI推論OK
    ├─ フリーテキスト → PII検出後にAI利用可（要マスキング）
    └─ 要配慮個人情報 → AI学習NG / 推論は条件付き

  眠りSCAN(1.92TB):
    ├─ バイタルデータ → AI学習OK（匿名化後）
    └─ 個人紐付け情報 → AI推論のみ可

→ データ分類ポリシーが無いと、L5のFeature StoreやRAGに
  「入れていいデータ」と「入れてはいけないデータ」の線引きができない
```

#### 論点② 非構造データの取り込みパスが未設計

```
現行のL0→L1は暗黙的に「構造化データのETL」を想定。

追加が必要:
  介護記録フリーテキスト → PII検出 → マスキング → チャンク分割 → エンベディング → pgvector
  ナースコール音声（将来） → 文字起こし → 構造化抽出 → テーブル格納

これは L3 の dbt/PySpark とは別の「非構造データパイプライン」が必要
→ LLMベースの抽出パイプライン（Bedrock / Databricks Foundation Model APIs）
```

#### 論点③ 2028年CareBase刷新の影響

```
2028年に介護システム刷新 → L0のデータソース構造が変わる
→ L3のETL、L5のFeature定義が全部影響を受ける

対策: L3にデータコントラクト（スキーマ契約）を入れ、
      上流変更時に自動でアラートを出す仕組みが必要
```

## 2. L1（保管庫・計算エンジン）への追加論点

### 現状: Databricks Lakehouse + Iceberg + S3 + pgvector(任意)

### 追加すべき観点

#### 論点④ pgvector は「任意」ではなく「必須」

```
L5にRAG(Vector Search)が必須として入っている以上、
ベクトルDBも必須。pgvectorが最適かは要検討:

  選択肢:
  ├─ pgvector: PostgreSQL拡張。運用が楽。小〜中規模向き
  ├─ Databricks Vector Search: Databricksネイティブ。Unity Catalog統合
  ├─ OpenSearch (AWS): フルマネージド。大規模向き
  └─ Pinecone: SaaS。最も簡単だが外部依存

→ Databricks構成なら Databricks Vector Search が統合性で有利
  pgvectorは「Databricksの外で軽量に使いたい」場合のサブ選択
```

#### 論点⑤ AI Output Store のレイヤーが不在

```
現行L1はBI用のGold Layerまで。
AIの出力（予測結果、分析レポート、LLMクエリログ）の格納先が未設計。

追加: L1に「AI Output Layer」
  ├─ 予測結果テーブル（バッチ推論の出力）
  ├─ LLMクエリ+結果ログ（監査・改善用）
  ├─ フィードバックテーブル（人間の修正・評価）
  └─ エージェント行動ログ（何をしたか）

→ Delta Table上に構築し、Time Travelでバージョン追跡
```

#### 論点⑥ ML用データセットのバージョニング

```
Feature Store + ラベルから作成した学習データセットを、
再現可能な形で保存する必要がある。

  「モデルv1.2はどのデータで学習したか」を後から完全に再現できること
  → Delta Table のバージョン + MLflow Artifact で管理
```

## 3. L3（データ加工・整形）への追加論点

### 現状: dbt Core + PySpark on Databricks + Airbyte

### 追加すべき観点

#### 論点⑦ 「BI用ETL」と「ML用Feature Engineering」は別物

```
現行L3はBI用のETL（データクレンジング、集計）を想定。

ML用の特徴量エンジニアリングは別パイプラインが必要:
  ├─ ウィンドウ関数（直近N日の移動平均、トレンド）
  ├─ ラグ特徴量（N日前の値）
  ├─ Point-in-Time JOIN（時点整合性を保ったデータ結合）
  ├─ 非構造→構造化抽出（LLMによるテキスト→JSON変換）
  └─ ラベルパイプライン（正解データの自動収集）

→ dbt Core でSQL変換は対応可能だが、
  PySpark で書くべき複雑な特徴量もある
→ Feature Engineering 用の dbt プロジェクトを BI 用と分離すべきか？
```

#### 論点⑧ 非構造データの加工パイプライン

```
介護記録テキスト → LLMで構造化抽出:
  「入居者Aが食堂で転倒。右膝打撲。湿布対応」
  → { event: "転倒", location: "食堂", injury: "右膝打撲", severity: "軽度" }

この変換はdbt/PySparkではなく LLMパイプライン:
  Databricks Foundation Model APIs or Amazon Bedrock

→ 現行L3には「LLMベースの変換パイプライン」の位置づけがない
→ L3の中に入れるか、L5とL3の間に「AI前処理レイヤー」として独立させるか
```

#### 論点⑨ データコントラクトの導入

```
上流システム（L0）のスキーマ変更がL3以降に伝播するリスク:
  2028年のCareBase刷新で全パイプラインが壊れる可能性

対策:
  ├─ dbt contracts（スキーマ定義 + バリデーション）
  ├─ Great Expectations（L4に記載あり。連携設計が必要）
  └─ スキーマ変更検知 → Feature Store への影響分析を自動化
```

## 4. L6（セマンティック層）への追加論点

### 現状: dbt MetricFlow / Databricks Metric Views / Timbr.ai (いずれか選択)

### 追加すべき観点

#### 論点⑩ セマンティック層は「BIのため」と「AIのため」で要件が異なる

```
BI用セマンティック層:
  「稼働率とは何か」の定義 → Tableauで正しいKPIを表示

AI用セマンティック層:
  「稼働率とは何か」の定義 → LLMが正しいSQLを生成
  + テーブル間の結合条件
  + カラムの暗黙的フィルタ（is_deleted=false等）
  + NULLの解釈ルール
  + 値のマッピング（status=1は在籍）

→ AI用のほうが情報量が多い。BI用セマンティック層をそのまま使うと
  LLMのSQL生成精度が落ちる
```

#### 論点⑪ Timbr.ai（Ontology代替）は要PoC検証

```
Timbr.aiが「Ontology代替」として記載されているが:
  ├─ Foundryオントロジーのグラフ構造をどこまで再現できるか？
  ├─ dbt MetricFlowとの共存は可能か？
  ├─ LLM（Genie/AIチャット）との連携は？
  └─ ライセンスコスト・サポート体制は？

→ Timbr.aiはニッチな製品。採用実績が少ないリスクがある
→ PoC段階で dbt MetricFlow vs Timbr.ai の比較検証が必要
```

#### 論点⑫ セマンティック層とLLMの接続方法

```
L6 → L7（AIチャット）の間に「LLMがセマンティック層を参照する方法」が未設計:

  パターンA: Databricks Genie（Databricksネイティブ）
    → Unity Catalog のメタデータを自動参照
    → Databricks構成なら最もシームレス

  パターンB: MCP（Model Context Protocol）経由
    → 外部LLM（Claude Desktop等）からセマンティック層にアクセス
    → 将来的なマルチLLM対応に有利

  パターンC: カスタムAPI
    → セマンティック層の定義をAPIで返す → LLMのシステムプロンプトに注入
    → 最も柔軟だが開発・運用コスト高

→ PoCではAを検証しつつ、将来のBへの拡張性を確認
```

## 5. L5（AI・機械学習）への追加論点 — ★最も論点が多い

### 現状: MLflow / Model Serving / Feature Store / RAG / AutoGluon

### 追加すべき観点

#### 論点⑬ 「部品」は揃っているが「繋ぎ方」が未設計

```
現行L5は部品のリスト:
  MLflow ─ Model Serving ─ Feature Store ─ RAG ─ AutoGluon

足りないのは「これらをどう繋いでライフサイクルを回すか」:

  Feature Store → データセット作成 → MLflow実験 → 評価
  → Model Registry → 承認 → Model Serving → モニタリング
  → ドリフト検知 → 再学習トリガー → Feature Store（ループ）

→ この「MLOpsパイプライン」の設計がレイヤー図に表れていない
→ Layer 2（ジョブ管理）でDatabricks Lakeflow/Dagsterが対応するが、
  ML固有のオーケストレーション要件が考慮されているか？
```

#### 論点⑭ プロンプト管理が不在

```
LLM時代の新しい「ソフトウェア資産」:
  ├─ システムプロンプト（AIチャットの振る舞いを定義）
  ├─ 構造化抽出プロンプト（テキスト→JSON変換の指示）
  ├─ 分析プロンプト（特定のKPI分析の指示テンプレート）
  └─ ガードレールプロンプト（AIが出してはいけない回答の制約）

これらは「コード」と同等に管理が必要:
  → バージョン管理（Git）
  → A/Bテスト（どのプロンプトが精度が高いか）
  → 本番昇格フロー（Staging → Production）
  → パフォーマンス監視（プロンプト変更による精度変化）

→ Model RegistryのLLM版 = 「Prompt Registry」が必要
→ 現行アーキテクチャに位置づけが無い
```

#### 論点⑮ エージェントフレームワークが不在

```
L7「AIチャット（カスタム）」は単なるQ&A UIだが、
将来的には「自律的に動くAIエージェント」が必要になる:

  ├─ 分析エージェント: データを自動分析してレポート生成
  ├─ アラートエージェント: 異常値検知→自動通知
  ├─ 予測更新エージェント: 定期的に予測を更新→配信
  └─ 業務エージェント: ケアプラン起案支援

エージェントの管理基盤:
  ├─ 権限管理: エージェントが何にアクセスできるか
  ├─ 行動ログ: エージェントが何をしたか（全行動記録）
  ├─ コスト制御: エージェントごとのLLM/計算コスト上限
  └─ 承認フロー: 書き込みアクションは人間の承認を挟むか

→ L5とL7の間に「Agent Runtime / Agent Management」レイヤーが必要
```

#### 論点⑯ モデルモニタリング・ドリフト検知

```
L5にMLflowはあるが「学習後のモニタリング」が未設計:

  ├─ データドリフト検知: 入力データの分布変化
  ├─ 予測ドリフト検知: 出力の分布変化
  ├─ パフォーマンス追跡: 精度の経時変化
  └─ LLM品質監視: ハルシネーション率、回答精度の推移

技術選択:
  ├─ Databricks Lakehouse Monitoring (ネイティブ)
  ├─ Evidently AI (OSS、ダッシュボード付き)
  └─ Whylogs (OSS、軽量)
```

#### 論点⑰ 自動評価フレームワーク

```
MLモデルとLLMで「自動評価」の方法が異なる:

MLモデル:
  テストデータセットで精度指標（AUC, F1, RMSE等）を自動計算
  → MLflow + 定期バッチジョブで対応可能

LLM:
  「回答が正しいか」の自動評価は難しい
  → LLM-as-a-Judge（別のLLMが評価する）
  → 人間評価のサンプリングワークフロー
  → 定型質問ベンチマーク（10問の正答率を定期チェック）

→ LLM評価基盤の設計が現行アーキテクチャに無い
```

## 7. Layer 4（品質管理・安全管理）への追加論点

### 現状: Great Expectations / dbt tests / Unity Catalog (RLS/監査) / OpenMetadata

### 追加すべき観点

#### 論点⑳ AI固有のガバナンスが不足

```
現行Layer 4は「データの品質管理」。以下のAI固有の統制が不足:

  AI利用ガバナンス:
  ├─ LLM監査ログ: 誰がAIに何を聞いて、何が返ったか
  ├─ プロンプトインジェクション対策: 悪意あるプロンプトの検出・ブロック
  ├─ PII漏洩防止: LLMの出力にPIIが含まれていないかのチェック
  ├─ コスト管理: ユーザー/部門別のLLM利用コスト追跡・制限
  └─ AI倫理チェック: バイアス検出、公平性の定期監査

  → Unity Catalog の RLS/監査 だけでは不十分
  → 「AI Governance Layer」として拡張が必要
```

#### 論点㉑ データ品質がAI精度に直結する「品質のSLA」

```
BI: データ品質が低い → ダッシュボードの数値が間違う → 人間が気づいて修正
AI: データ品質が低い → モデルが間違った予測を出す → 人間は気づかない可能性

→ AI学習に使うデータには、BIより厳しい品質基準が必要:
  ├─ 欠損率: 5%以下（BI用は10%でも可だが、ML用はNG）
  ├─ 鮮度: 24時間以内（日次バッチの遅延がドリフトの原因に）
  ├─ 一貫性: 施設間でデータ定義が統一されているか
  └─ 完全性: ラベル（正解データ）の網羅率
```

```
  ┌─────────────────────────────────┐
  │       LLM Gateway / Router      │
  │                                 │
  │  ├─ 統一API（全コンポーネントはここを経由）  │
  │  ├─ モデルルーティング（質問種別→最適モデル）│

技術選択肢:
  ├─ LiteLLM (OSS): 100+ LLMの統一API。ルーティング・フォールバック対応
  ├─ AWS Bedrock: AWSネイティブ。マルチモデル対応
  ├─ カスタムAPI Gateway: 最も柔軟だが開発コスト高
```

```
現行アーキテクチャにフィードバックの仕組みが無い:
```

## 11. 説明ストーリー — 「この基盤図に、AI OSの思想を注入する」

```
Step 1: 「この基盤アーキテクチャはデータ基盤として正しい」
  → L1-L7のレイヤー構成、Databricks中心の統合管理は適切
  → BI + MLの部品は一通り揃っている

Step 2: 「ただし、これは"AI-Readyなデータ基盤"であって"AI OS"ではない」
  → AI OSとは「AIを高速に・大量に・安全に作り、運用し、進化させる基盤+体制」
  → 現行図はデータの流れは描いているが、
    AIのライフサイクル（開発→評価→デプロイ→運用→進化）を回す設計が薄い

Step 3: 「AI OSには7つの設計軸がある」
  ① 連携性: マルチクラウド・マルチサービスの中でエージェントをどう繋ぐか
  ② 安全性: AIのID・権限・ガードレール・監査（★データ基盤には無い思想）
  ③ 実効性: ExpertAI — 汎用AIではなく自社ドメインの専門家にする
  ④ 速度:   人間の思考速度に同期するモデルルーティング
  ⑤ 規模:   エージェント量産時代のカタログ・再利用設計
  ⑥ 運用性: 「測れる・変えられる・人が回せる」の3軸
  ⑦ 進化性: 陳腐化せず進化し続けるためのeval・SSOT・フィードバック

Step 4: 「現行図は③実効性に偏重。他6軸の設計が不足」
  特に不足しているもの:
  → ②安全性: AIの独自ID、権限委譲、HITL、入出力ガードレール
  → ⑥運用性: トレース、ゴールデンセット評価、版管理、停止手順
  → ⑦進化性: フィードバックループ、ナレッジSSOT、自動eval

Step 5: 「具体的に22の論点 + 新規レイヤーを提案」
  → LLMゲートウェイ、Agent Runtime、Feedback Layer
  → 非構造データのメダリオン、AI Output Store
  → 安全性4本柱の実装設計

Step 6: 「整備のフェーズを分ける」
  → Ph.1（PoC）: 連携性 + 安全性の設計（最低限のID・ガードレール・トレース）
  → Ph.2（構築）: 実効性 + 速度 + 規模の実装
  → Ph.3（運用）: 運用性 + 進化性の成熟

Step 7: 「なぜ今やるべきか」
  → ②安全性と⑥運用性は後付け不能。Ph.1で設計だけは実施すべき
  → 特にAIの独自ID、トレース設計、ゴールデンセット評価は
    初期設計に含めないと本番後に作り直しになる
```
