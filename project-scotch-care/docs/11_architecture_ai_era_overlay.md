# AI-Ready アーキテクチャへの追加論点 — 「普通のデータ基盤」から「AI OS」へ

> 作成日: 2026-03-04 | 目的: 現行のAI-Readyデータ基盤アーキテクチャ（Databricks構成、L0-L7）に対して、AI開発・運用視点で足りないもの・深掘りすべき論点を、わかりやすく整理する。
> 参照: AI-Ready データ基盤アーキテクチャ図（Databricks構成、L0-L7）、AI OS設計最新 v1.1

---

## 0. 現行アーキテクチャの評価 — 「土台は良い。でも屋根がない」

### 建物に例えると

現行の7層アーキテクチャは、**基礎と柱はしっかりしている建物**に似ている。

```
今の状態:
  ✅ 基礎（L0-L1）: データの保管場所はOK
  ✅ 柱（L3）: データの加工・整形の仕組みはOK
  ✅ 部品（L5）: MLflow等のAI用ツールは揃っている
  ⚠️ 屋根（運用）: AIをどう「安全に」「継続的に」運用するかが未設計
  ❌ 空調・電気（横断機能）: AIの監査、コスト管理、フィードバックが不在
```

もう少し具体的に:
- **入力側（データを集める）は充実**: L0→L1→L3のデータパイプラインは堅実
- **ML基盤（AIを作る）は部品が並んでいる**: L5にMLflow等があるが、**それらをどう繋いで動かすか**の設計が見えない
- **出力側（AIを使う）が薄い**: L7のTableau+AIチャットだけ。AIの出力管理・フィードバック・監査が不在
- **横断的な統制が不足**: AI固有のガバナンス（コスト制御、LLM監査、エージェント権限）が未整理

### 核心的な指摘

**このアーキテクチャは「AI-Readyなデータ基盤」であって「AI OS」ではない**。

```
AI-Readyなデータ基盤:
  「AIを動かすためのデータは準備できている」

AI OS:
  「役に立つAIを、高速に・大量に・安全に作り、
   効率的に管理・運用し、進化し続ける状態を実現する
   技術基盤と運用体制の総体」
```

**身近な例**: スマートフォンに例えると:
- 今の状態 = 「部品は全部ある（チップ、カメラ、バッテリー）」
- 必要な状態 = 「iOSのようなOS（アプリ管理、権限管理、アップデート機能）が載っている」

## 0.5. AI OSの7つの設計軸 — データ基盤設計にはない「OS」の視点

> 参照: AI OS設計最新 v1.1 (ACES独自フレームワーク)

### なぜ「7つの設計軸」が必要か

一般的なデータ基盤の設計は「データをどう流すか」（パイプライン設計）が中心。でもAI OSには**それ以外の6つの設計軸**がある。これが「普通のデータ基盤ベンダー」と「AI OS設計ができるコンサルタント」の差になる。

### 7つの設計軸の全体像

```
┌────────────────────────────────────────────────────────────────┐
│  Ph.1（PoC・初期設計）   │  Ph.2（構築・運用初期）  │  Ph.3（運用中期〜）  │
│  ★ 最初に設計すべき     │  本格的に作る           │  育てる・広げる      │
├────────────────────────────────────────────────────────────────┤
│  ① 連携性（つなぐ）     │  ③ 実効性（役に立つ）   │  ⑥ 運用性（回せる）  │
│  ② 安全性（守る）       │  ④ 速度（速い）        │  ⑦ 進化性（育つ）   │
│                          │  ⑤ 規模（広げる）      │                     │
└────────────────────────────────────────────────────────────────┘
※ ⑥⑦はPh.1で設計だけは実施し、Ph.2以降で実装するのが理想
```

**現行アーキテクチャ図（L0-L7）は③実効性に偏重**。他の6軸の設計が不十分。

---

### 軸① 連携性 — いろんなシステムを「つなぐ」

**身近な例**: スマホの例え。iPhone単体では価値が限定的。App Store、iCloud、AirDrop...他のサービスと**つながる**ことで真価を発揮する。

AI OSも同じ。単独のDatabricks環境だけでは不十分で、いろんなシステムと連携する設計が必要。

```
SOMPOケアで「つなぐ」必要があるもの:
  ├─ AWS上のDatabricks ←→ Salesforce/Agentforce（顧客管理AI）
  ├─ 介護システム（ほのぼの、Axist等）←→ AI分析基盤
  ├─ 複数のAIモデル（Bedrock/Azure OpenAI/直接API）を統一的に呼ぶ
  └─ MCP（共通コンセント）で外部AIツールからデータにアクセス

現行図で足りないもの:
  → 「LLMゲートウェイ」（AIモデルの呼び出しを統一する窓口）
  → システム間連携のプロトコル設計
```

**他社の事例**:
- **ユニクロ（ファーストリテイリング）**: 製造・物流・店舗・ECのデータを統合し、需要予測AIが全システムのデータを横断的に参照。「つなぐ」設計がなかったら各システムでバラバラのAIが動くことになる

### 軸② 安全性 — AIを「守る」仕組み

**これが最も重要で、最も見落とされやすい軸**。

一般的なデータ基盤の「セキュリティ」は「人間のアクセス制御」。AI OSの「安全性」は**それとは次元が違う**。AIは「自律的に動く」ので、人間とは別の安全管理が必要。

**身近な例**: 子供の留守番と似ている。
- 「冷蔵庫は開けていいけど、コンロは触らないで」← 権限の設定
- 「知らない人が来たらドアを開けないで」← ガードレール
- 「何かあったらすぐ電話して」← 人間への報告義務

```
安全性の4本柱（建物の例え）:

┌─────────────────────────────────────────────────────────────┐
│ 1本目: 身分証明と鍵の管理（ID・権限管理）                     │
│    AIにも「社員証」を持たせる                                 │
│    「このAIは施設Aのデータだけ見られる」と明確にする          │
│    人間が承認した時だけ、一時的に権限を上げられる              │
│                                                              │
│  事例: Google Cloud AI Platform                               │
│    各AIモデルにサービスアカウント（AI用ID）を付与。           │
│    「このAIはBigQueryのこのデータセットだけアクセス可能」     │
│    と細かく制御。不正アクセスは即座に検知・ブロック           │
├─────────────────────────────────────────────────────────────┤
│ 2本目: 情報の取り扱いルール（データガバナンス）                │
│    AIが「推論」で新たな機密情報を生み出す可能性がある         │
│    例: 「この入居者は○○病の可能性が高い」とAIが推論         │
│    → この推論結果自体が要配慮個人情報になる                   │
│                                                              │
│  事例: JPモルガン                                             │
│    AIが生成したレポートを自動でコンプライアンスチェック。     │
│    機密情報や不適切な推論が含まれていたら自動ブロック         │
├─────────────────────────────────────────────────────────────┤
│ 3本目: 安全装置（AIのガードレール）                           │
│    入力チェック: 悪意ある質問（プロンプトインジェクション）を検知│
│    出力チェック: AIが嘘をついたり、機密情報を漏らすのを防ぐ   │
│    判断基準: 「AIが自分でやっていいこと」と「人間に聞くべきこと」│
│                                                              │
│  事例: Anthropic Claude（自社事例）                           │
│    入出力の安全性チェックを多層的に実装。               │
│    「このトピックには回答しない」等のガードレールを設定可能    │
├─────────────────────────────────────────────────────────────┤
│ 4本目: 記録と説明責任（監査・透明性）                         │
│    AIの「思考過程」を記録する                                 │
│    「なぜこの結論に至ったか」を後から説明できる               │
│    問題が起きた時に「誰の責任か」がわかる                     │
│                                                              │
│  事例: EU AI Act（2026年施行）                                │
│    高リスクAI（医療、金融等）には監査ログの保存が法的義務     │
└─────────────────────────────────────────────────────────────┘
```

#### 最優先で作るべき4つの安全機能

| 機能 | 何をするか | SOMPOケアでの具体例 | 現行アーキテクチャ |
|---|---|---|---|
| **AIの身分証** | AIにもIDを持たせる | 「転倒予測AIはどの権限で入居者データにアクセスしたか」を追跡 | ❌ 未対応 |
| **入出力チェック** | 悪意ある質問と危険な回答をブロック | 「他施設の個人情報を教えて」→ ブロック | ❌ 未対応 |
| **行動記録** | AIの全行動を追跡可能に | 「AIはどのデータを見て『転倒リスク高』と判断したか」 | △ 部分対応 |
| **人間の承認** | 重大な判断は人間が承認 | AIがケアプランを変更する前に施設長の承認を要求 | ❌ 未対応 |

### 軸③ 実効性 — AIが「役に立つ」ようにする

**キーコンセプト: ExpertAI = 汎用AIではなく「介護のプロ」にする**

```
汎用LLM（ChatGPTにそのまま聞く）:
  Q: 「施設Aの稼働率が下がっている理由は？」
  A: 「一般的に稼働率低下の原因は、競合施設の開業、
      スタッフ不足、立地条件の変化などが考えられます」
  → 教科書的な回答。誰でも知っている

ExpertAI（自社データで賢くしたAI）:
  Q: 「施設Aの稼働率が下がっている理由は？」
  A: 「施設Aの稼働率は前月比3%低下（87%→84%）。
      ① 半径2km以内に競合施設が先月オープン（影響度:高）
      ② スタッフ離職率が直近3ヶ月で8%上昇（業界平均+4%）
      ③ 過去3年の季節パターンでは冬季に2-3%低下が通常
      結論: ①が主因。過去の類似ケースでは新施設の影響は
           6ヶ月で安定する傾向」
  → 自社データに基づく具体的な分析
```

**他社の事例**:
- **Bloomberg GPT**: 金融データ特化のLLM。汎用LLMでは答えられない「今日のNYSEの異常な取引パターン」に具体的に回答
- **Harvey AI（法律AI）**: 法律文書に特化。汎用LLMより法律タスクの精度が20-30%高い

ExpertAIの実現に必要なもの:
1. 現場のノウハウを形式知化（ドキュメント化）
2. 自社データをAIが参照できる形に整備（セマンティックレイヤー+RAG）
3. フィードバックループで継続改善

### 軸④ 速度 — 人間の「待てる時間」に合わせる

**身近な例**: レストランの注文。
- 「お水ください」→ すぐ出てくるべき（数秒）
- 「ステーキください」→ 少し待てる（数分）
- 「特注のバースデーケーキ」→ 事前に注文（数時間〜翌日）

AIも同じ。質問の種類によって「待てる時間」が違う。

```
AI OSの速度分類（人間の認知速度に合わせる）:

  即座（3秒以内）: チャットの返答、検索結果、リアルタイムアラート
    → 軽量モデル（Claude Haiku等）を使う
    → 事前計算した結果をキャッシュで返す

  対話（30秒以内）: 複雑な分析、複数テーブルを跨ぐ集計
    → 標準モデル（Claude Sonnet等）を使う
    → SQLを生成して実行

  バッチ（30分以内）: 全施設の月次予測、大規模レポート生成
    → 高精度モデル（Claude Opus等）も可
    → 結果をテーブルに書き込んで後から参照

  深考（数時間）: 戦略分析、複数データソースの統合分析
    → 複数のAIが協調して段階的に分析
    → 人間がレビューしながら進める
```

**ポイント**: 速度と精度はトレードオフ。「3秒で返すが精度80%」と「30秒かかるが精度95%」のどちらを選ぶかは、**タスクの性質**で決まる。

- 「稼働率は？」→ 3秒で返して精度100%（定義に基づく確定値）
- 「なぜ稼働率が下がったか？」→ 30秒かけて精度85%（AIの推論が入る）

**現行アーキテクチャにはこの「速度に応じたモデル切り替え」の仕組みがない**。

### 軸⑤ 規模 — AIエージェントを「量産」する時代

**身近な例**: スマホアプリの世界。
- 初期: 1つのアプリを数ヶ月かけて作る
- 現在: App Storeに数百万のアプリ。個人でもアプリが作れる時代
- 重要なのは「良いアプリ」と「悪いアプリ」を見分けるApp Storeの仕組み

AIエージェントも同じ進化を辿る:

```
SOMPOケアで将来必要になるエージェント:
  ├─ 稼働率予測エージェント
  ├─ 離職予測エージェント
  ├─ 転倒リスク評価エージェント
  ├─ ケアプラン支援エージェント
  ├─ 施設改善レコメンドエージェント
  ├─ 自然言語分析エージェント（「稼働率は？」に答える）
  ├─ レポート自動生成エージェント
  ├─ 問い合わせ対応エージェント（FAQ回答）
  └─ ... 将来的に数十以上

課題:
  各部署がバラバラにAIを作ると「AIのサイロ化」が起きる
  → 同じようなAIが乱立、品質もバラバラ

対策: 「エージェントカタログ」（= App Storeのようなもの）
  ├─ 各エージェントの目的・KPI・所有者を登録
  ├─ 品質スコア（精度・コスト・速度）を表示
  ├─ 再利用可能なエージェントを検索・発見
  └─ 重複開発を防ぐ
```

**他社の事例**:
- **Salesforce Agentforce**: エージェントの「マーケットプレイス」を提供。部門ごとにバラバラだったAIエージェントを統一プラットフォームで管理
- **Microsoft Copilot Studio**: 非エンジニアでもAIエージェントを作れるローコードプラットフォーム。エージェントの「量産」を可能に

### 軸⑥ 運用性 — 「測れる・変えられる・人が回せる」

**これが「データ基盤」と「AI OS」の最大の違い**。

```
普通のシステム運用:
  同じ入力 → 必ず同じ出力（決定的）
  壊れたら明確にエラーが出る
  → 従来の運用監視で対応可能

AIエージェントの運用:
  同じ入力 → 毎回違う出力（確率的）
  壊れても「静かに品質が劣化する」
  → 従来の監視では検知できない

AIの運用が難しい理由:
  ├─ 故障点が多い: LLM、ツール、データ、プロンプト、権限が相互依存
  ├─ 原因がわからない: 精度が落ちた。モデル？データ？プロンプト？
  ├─ 結果だけではダメ: 「なぜその答えになったか」の追跡が必要
  └─ 増えると破綻: エージェントが増殖して重複・コスト膨張
```

#### 運用性の3つの軸

| 軸 | 内容 | 身近な例え |
|---|---|---|
| **測れる** | AIの動作を可視化・記録する | 車のダッシュボード（速度、燃費、エンジン温度が常に見える） |
| **変えられる** | プロンプト/モデル/データをバージョン管理し、問題があれば戻せる | スマホの「元に戻す」ボタン |
| **人が回せる** | AIの結果を人間がレビュー・修正・承認・停止できる | 自動運転車の「手動モード切替」 |

**具体的に初手で作るべきもの**:

```
□ 標準トレース + 監査ログ
  AIが「どのデータを見て」「どう考えて」「何を出力したか」を
  全部記録する（ドライブレコーダーのようなもの）

□ ゴールデンセット + 自動テスト
  「この10問には必ず正しく答えられるはず」というテスト問題集
  毎日自動で実行して、精度が落ちていないかチェック

  事例: Anthropicでは新モデルリリース前に数千件のテストケースを
       自動実行し、品質基準を満たさないとリリースしない

□ バージョン管理 + ロールバック
  プロンプトやモデルを変更したら、いつでも元に戻せる
  → 「昨日まで上手くいっていたのに」→ 昨日の設定に戻す

□ 緊急停止ボタン
  AIが問題のある回答を出し始めたら、すぐ止められる
  → 「人間による手動運用」に切り替える手順を事前に準備
```

### 軸⑦ 進化性 — AIが「陳腐化せず、育ち続ける」

**身近な例**: iPhoneのOS。買った時の状態がピークではなく、アップデートで機能が増えて賢くなっていく。AI OSも同じ。

```
AIが陳腐化する4つのパターン:

① モデルが古くなる: 半年前のLLMは今のLLMに劣る
② ナレッジが古くなる: 去年の業界ガイドラインでRAGしていたら間違った回答を出す
③ 評価ができない: 「AIの品質が良いのか悪いのか」がわからないから改善できない
④ 学習できない: フィードバックを集めていないから、同じ間違いを繰り返す

「進化し続ける」ために必要なもの:

  ① Trace-first（全部記録から始める）
     AIの動作ログを最初から取っておく
     → 問題が起きた時に「なぜか」を追跡できる
     → 改善のデータにもなる

  ② 評価基盤の標準化
     ゴールデンセット（正解がわかっているテスト問題集）を作る
     定期的に自動テストして、精度の推移を追跡
     → 「先月より精度が落ちた」が数字でわかる

  ③ ナレッジの一元管理（SSOT: 唯一の正しい情報源）
     複数のバージョンの情報が混在しないようにする
     古い情報は明示的にアーカイブ
     → 「去年のガイドラインと今年のガイドラインが混ざってAIが混乱」を防ぐ

  ④ フィードバックの仕組み化
     ユーザーの👍/👎 → 改善チケット → 対応 → 精度向上
     → ChatGPTの改善モデル（RLHF）と同じ考え方を社内AIに適用
```

**他社の事例**:
- **Tesla（自動運転の進化）**: 全車両の走行データを収集 → 困難な場面を自動検出 → AIモデルを再学習 → OTAアップデートで全車両に配信。このサイクルが回り続けることで、AIが日々賢くなる
- **Duolingo（語学学習AI）**: ユーザーの正答率・間違いパターンをリアルタイム収集 → 問題の難易度を自動調整 → AIの出題ロジックを継続改善。改善サイクルが回るほど学習効率が上がる

### 非構造データの「料理工程」（メダリオンアーキテクチャ）

一般的なBronze→Silver→Goldは「表形式のデータ」向け。テキストやファイルには**別の料理工程**が必要:

```
非構造データの料理工程:

Bronze（生の食材）:
  PPTファイル、PDFファイル、介護記録テキスト、音声ファイル
  → とりあえず全部冷蔵庫に入れる（S3に保存）

Silver（下ごしらえ済み）:
  Markdownテキスト、構造化JSON
  → ファイルをテキストに変換
  → LLMで「転倒」「場所」「重症度」を抽出
  → 個人情報を自動検出してマスク

Gold（盛り付け済み = AIが使える形）:
  ├─ ベクトルデータ → 「意味で検索」するためのデータ
  ├─ メタデータ → ファイル名、著者、カテゴリ、作成日
  └─ グラフデータ（将来）→ エンティティ間の関係性
```

**現行アーキテクチャのL3（dbt/PySpark）ではこの「非構造データの料理工程」をカバーできない**。LLMベースの変換パイプラインが別途必要。

## 1〜8. レイヤー別の追加論点

### L0（データソース）— 「食材の仕入れ先」

**現状**: 介護記録(340GB) / 眠りSCAN(1.92TB) / ナースコール(7.5GB) / ほのぼの / Axist

#### 論点① AIに使っていいデータの分類が未定義

```
各データソースに「AIで使っていいか」のラベルが必要:

  介護記録:
    数値フィールド → AI学習OK
    フリーテキスト → 個人情報マスク後にOK
    要配慮個人情報 → AI学習NG、推論は条件付き

  眠りSCAN:
    バイタルデータ → 匿名化後にAI学習OK
    個人紐付け情報 → リアルタイム推論のみ

→ この分類がないと「何をFeature StoreやRAGに入れていいか」が決まらない
```

#### 論点② テキストデータの取り込みパスがない

```
現行: 構造化データ（表形式）の取り込みだけ設計されている
追加: テキスト→個人情報マスク→チャンク分割→ベクトル化の工程が必要
→ これはdbt/PySparkとは別の「LLMベースのパイプライン」
```

#### 論点③ 2028年の介護システム刷新

```
2028年にCareBaseが刷新 → L0のデータ構造が変わる
→ L3のETL、L5のFeature定義が全部影響を受ける
→ 対策: 「データコントラクト」（上流の構造変更を自動検知する仕組み）
```

---

### L1（保管庫）— 「冷蔵庫・冷凍庫」

#### 論点④ ベクトルDBは「任意」ではなく「必須」

```
RAG（テキスト検索AI）を使う以上、ベクトルDBは必須:

選択肢と特徴:
  Databricks Vector Search: Databricksと一体。権限管理が楽
  pgvector: PostgreSQL拡張。小規模向き
  OpenSearch: AWSマネージド。大規模向き
  → Databricks構成なら Vector Search が第一候補
```

#### 論点⑤ AI Output Storeがない

```
AIの出力（予測結果、分析レポート、質問への回答）の保存場所がない
→ L1に「AI Output Layer」を追加
→ 予測結果テーブル、LLMクエリログ、フィードバックテーブル
→ Delta Tableで構築し、バージョン管理可能に
```

---

### L3（データ加工）— 「調理場」

#### 論点⑦ 「BI用の調理」と「AI用の調理」は別物

```
BI用: 月次の稼働率を計算してダッシュボードに表示
AI用: 「2025年6月1日時点の直近90日移動平均」を計算
     → ウィンドウ関数、ラグ特徴量、時点整合性JOIN
     → dbtで書けるがBI用とは分離すべき
```

#### 論点⑧ テキストをAIで加工するパイプラインがない

```
介護記録テキスト → LLMで構造化抽出（JSON化）
→ これはdbt/PySparkではなくLLMパイプライン
→ 現行L3には位置づけがない
→ L3の中に「LLMベース加工」を追加するか、独立レイヤーにするか
```

---

### L6（セマンティック層）— 「レシピブック」

#### 論点⑩ AIのためのレシピブックはBIのそれより情報量が多い

```
BIのレシピ: 「稼働率 = 入居者数 ÷ 定員 × 100」→ これだけでダッシュボードは作れる

AIのレシピ: 上記に加えて...
  テーブルの繋ぎ方: facility_metricsとfacility_masterはfacility_idで繋ぐ
  暗黙のフィルタ: is_deleted=false AND date >= '2023-01-01'
  NULLの意味: NULLは「0%」ではなく「データ未入力」
  値のコード: status=1は「在籍」、2は「退職」

→ AIのためのレシピブックはBIの倍以上の情報量が必要
```

#### 論点⑫ レシピブックとAIの接続方法

```
3つのパターン:

パターンA: Databricks Genie（自動で接続）
  → Databricksを使うなら最も楽
  → 内部のAIが自動でレシピブックを参照

パターンB: MCP経由（共通コンセントで接続）
  → 外部AIツール（Claude Desktop等）からもアクセス可能
  → 将来性が高い

パターンC: カスタムAPI（手作りで接続）
  → 最も柔軟だが開発・運用コストが高い

→ PoCではAを試しつつ、Bへの拡張性を確認するのが現実的
```

---

### L5（AI・機械学習）— 「厨房の道具」

**ここが最も論点が多い**。

#### 論点⑬ 「道具は揃っているが、調理手順書がない」

```
現行: MLflow、Model Serving、Feature Store、RAG、AutoGluon
     → 調理道具（フライパン、包丁、オーブン）は並んでいる

足りないのは「調理手順書」（= MLOpsパイプライン）:
  Feature Store → データセット作成 → MLflow実験 → 評価
  → モデル登録 → 承認 → 本番投入 → モニタリング
  → 精度低下検知 → 再学習 → Feature Store（ループ）

→ この「手順を自動化する仕組み」の設計が見えない
```

#### 論点⑭ プロンプトの管理がない

```
LLM時代に新しく必要になった「資産」:
  ├─ システムプロンプト: AIの性格・ルールを定義
  ├─ 分析プロンプト: 特定の分析の指示テンプレート
  ├─ ガードレール: AIが答えてはいけないことの制約
  └─ 構造化抽出: テキスト→JSON変換の指示

これらは「コード」と同じく管理が必要:
  → Gitでバージョン管理
  → どのプロンプトが精度が高いかA/Bテスト
  → 本番昇格フロー（テスト→承認→本番）

→ 「Prompt Registry」= プロンプト版のモデル管理が必要
→ 現行アーキテクチャにこの位置づけがない
```

#### 論点⑮ AIエージェントの管理基盤がない

```
L7「AIチャット」は単なるQ&A画面だが、
将来は「自律的に動くAIエージェント」が必要:

  分析エージェント: データを自動分析してレポート生成
  アラートエージェント: 異常値を検知して自動通知
  予測更新エージェント: 定期的に予測を更新
  業務エージェント: ケアプラン起案の支援

→ L5とL7の間に「Agent Runtime & Management」レイヤーが必要
```

---

### L7（現場が使う画面）— 「お客さんに出す料理」

#### 論点⑱ 「AIチャット」の裏側は複雑なシステム

```
「AIチャット」と一言で書いてあるが、裏側は:

  ユーザー:「先月稼働率が低い施設はどこ？原因は？」
       ↓
  [意図を理解] → 「数値の質問 + 原因分析」と判断
       ↓
  [KPI照会] → セマンティック層を参照 → SQL生成 → 結果取得
       ↓
  [原因分析] → 関連データをRAG検索 → LLMが推論
       ↓
  [回答生成] → 数値 + 分析をまとめて人間にわかる文章に
       ↓
  [権限チェック] → このユーザーに見せていいデータか確認

→ これは「チャットUI」ではなく「Compound AI System」
→ L5〜L7にまたがるシステムアーキテクチャの設計が必要
```

#### 論点⑲ フィードバック収集UIがない

```
AIの回答に対して:
  👍/👎ボタン → 精度改善の入力データ
  「この回答は正しくない」→ 修正内容を記録
  信頼度表示 →「この回答の根拠は3つのデータソースです」

→ フィードバックデータはL1のFeedback Storeに格納
→ AIの改善サイクルの入力になる
```

---

### Layer 4（品質管理）— 「食品衛生検査」

#### 論点⑳ AI固有の品質管理が不足

```
現行: データの品質管理（Great Expectations, dbt tests）
追加: AI固有のガバナンス
  ├─ LLM監査ログ: 誰がAIに何を聞いて、何が返ったか
  ├─ 個人情報漏洩防止: AIの出力に個人情報が含まれていないかチェック
  ├─ コスト管理: 部門・施設別のAI利用コスト追跡
  └─ バイアスチェック: 特定施設・属性への偏りがないか定期監査
```

## 9. 新しく追加すべきレイヤー

### 新規① LLMゲートウェイ — 「AIモデルの交通整理」

**身近な例**: 電話の交換機。社外への電話を全部交換機経由にすることで、通話記録・料金管理・回線切替が一元管理できる。

```
LLMゲートウェイがないと:
  L5のMLflowが直接Claude APIを呼ぶ
  L7のAIチャットが直接GPT-4 APIを呼ぶ
  L3のテキスト処理が直接Bedrock APIを呼ぶ
  → コストがバラバラ、ログがバラバラ、モデル変更が大変

LLMゲートウェイがあると:
  全部ここを経由:
  ├─ 統一API: どのコンポーネントも同じ方法でAIを呼ぶ
  ├─ モデル切替: 「この質問には軽量モデル、この分析には高精度モデル」
  ├─ コスト記録: 全リクエストのコストを自動記録
  ├─ 利用制限: 部門別の月間上限
  ├─ 監査ログ: 全リクエスト/レスポンスを記録
  └─ 障害時: モデルAが落ちたら自動でモデルBに切り替え
```

**他社の事例**:
- **Cloudflare AI Gateway**: LLMへのリクエストをプロキシし、キャッシュ・レート制限・ログ収集を一元管理。導入企業のLLMコストを平均30%削減
- **LiteLLM（OSS）**: 100以上のLLMに対応した統一APIを提供。モデルの切り替えが1行のコード変更で可能

### 新規② Feedback & Evaluation Layer — 「品質改善の仕組み」

```
現行: AIの出力を出すところまで。その後のフィードバックがない

追加:
  ├─ フィードバック収集API（👍/👎、修正内容の記録）
  ├─ 自動テスト（ゴールデンセットで定期的に精度チェック）
  ├─ 予測→実績の答え合わせ（3ヶ月前の予測と実績を自動比較）
  ├─ フィードバック集計ダッシュボード
  └─ 改善チケットの自動生成
```

## 11. 先方への説明ストーリー — 7つのステップ

```
Step 1: 「この基盤アーキテクチャの土台は良い」
  → L1-L7の構成は正しい。Databricks中心の設計は堅実
  → 否定から入らない。まず「良いところ」を認める

Step 2: 「ただし"AI-Readyなデータ基盤"と"AI OS"は違う」
  → 例え: 部品は揃っている(iPhone)。でもOS(iOS)がない
  → データの流れは描いているが、AIのライフサイクルが見えない

Step 3: 「AI OSには7つの設計軸がある」
  ① つなぐ（連携性）: マルチサービスの統合
  ② 守る（安全性）: AIの身分証、ガードレール、人間の承認 ★データ基盤にはない思想
  ③ 役立つ（実効性）: ExpertAI — 介護のプロにする
  ④ 速い（速度）: 人間の待てる時間に合わせたモデル切替
  ⑤ 広げる（規模）: エージェントの量産とカタログ管理
  ⑥ 回せる（運用性）: 測れる・変えられる・人が回せる
  ⑦ 育つ（進化性）: フィードバックで賢くなり続ける

Step 4: 「現行図は③に偏重。他6軸の設計が不足」
  特に②安全性と⑥運用性が不足
  → AIのID管理、ガードレール、トレース、ゴールデンセット評価

Step 5: 「具体的に22の論点 + 新規レイヤーを提案」
  → LLMゲートウェイ、Agent Runtime、Feedback Layer
  → 非構造データのメダリオン、AI Output Store
  → 安全性4本柱の実装

Step 6: 「段階的に整備する」
  Ph.1（PoC）: ①つなぐ + ②守る の設計
  Ph.2（構築）: ③役立つ + ④速い + ⑤広げる の実装
  Ph.3（運用）: ⑥回せる + ⑦育つ の成熟

Step 7: 「なぜ今やるべきか」
  ②安全性と⑥運用性は後付け不能
  → 家の基礎工事と同じ。建てた後から地下室は作れない
  → Ph.1で設計だけでもやっておかないと、本番後にやり直し
```
