# AI-Ready アーキテクチャへの追加論点 — 「普通のデータ基盤」から「AI OS」へ

> 作成日: 2026-03-04 | 目的: 現行のAI-Ready基盤アーキテクチャ（Databricks構成）に対して、AI開発・運用視点で足りない要素・深掘りすべき論点を整理する。
> 参照: AI-Ready データ基盤アーキテクチャ図（Databricks構成、L0-L7）

---

## 0. 現行アーキテクチャの評価

現行の7層アーキテクチャは「AI-Ready」を謳っており、L5にMLflow/Feature Store/RAG等を配置している点で**基盤としての骨格は正しい**。ただし、**データ基盤の延長でAIを載せている設計**であり、**「AIのライフサイクル全体を回す」視点**が不足している。

具体的には:
- **入力側（ためる）は充実**: L0→L1→L3のデータパイプラインは堅実
- **ML基盤（つくる）は部品が並んでいる**: L5にMLflow等があるが、**それらをどう繋いで回すか**の設計が見えない
- **出力側（つかう）が薄い**: L7のTableau+AIチャットだけ。AIの出力管理・フィードバック・監査が不在
- **横断的な統制が不足**: AI固有のガバナンス（コスト制御、LLM監査、エージェント権限）が未整理

以下、各レイヤーに対する追加論点を整理する。

---

## 1. L0（データソース）への追加論点

### 現状: 構造化データソースの列挙

```
現行: 介護記録(340GB) / 眠りSCAN(1.92TB) / ナースコール(7.5GB) / ほのぼの / Axist / LIFE
```

### 追加すべき観点

#### 論点① データの「AI利用可否」分類が未定義

```
L0のデータソースそれぞれに、以下の分類が必要:

  介護記録(340GB):
    ├─ 構造化フィールド → AI学習OK / AI推論OK
    ├─ フリーテキスト → PII検出後にAI利用可（要マスキング）
    └─ 要配慮個人情報 → AI学習NG / 推論は条件付き

  眠りSCAN(1.92TB):
    ├─ バイタルデータ → AI学習OK（匿名化後）
    └─ 個人紐付け情報 → AI推論のみ可

→ データ分類ポリシーが無いと、L5のFeature StoreやRAGに
  「入れていいデータ」と「入れてはいけないデータ」の線引きができない
```

#### 論点② 非構造データの取り込みパスが未設計

```
現行のL0→L1は暗黙的に「構造化データのETL」を想定。

追加が必要:
  介護記録フリーテキスト → PII検出 → マスキング → チャンク分割 → エンベディング → pgvector
  ナースコール音声（将来） → 文字起こし → 構造化抽出 → テーブル格納

これは L3 の dbt/PySpark とは別の「非構造データパイプライン」が必要
→ LLMベースの抽出パイプライン（Bedrock / Databricks Foundation Model APIs）
```

#### 論点③ 2028年CareBase刷新の影響

```
2028年に介護システム刷新 → L0のデータソース構造が変わる
→ L3のETL、L5のFeature定義が全部影響を受ける

対策: L3にデータコントラクト（スキーマ契約）を入れ、
      上流変更時に自動でアラートを出す仕組みが必要
```

## 2. L1（保管庫・計算エンジン）への追加論点

### 現状: Databricks Lakehouse + Iceberg + S3 + pgvector(任意)

### 追加すべき観点

#### 論点④ pgvector は「任意」ではなく「必須」

```
L5にRAG(Vector Search)が必須として入っている以上、
ベクトルDBも必須。pgvectorが最適かは要検討:

  選択肢:
  ├─ pgvector: PostgreSQL拡張。運用が楽。小〜中規模向き
  ├─ Databricks Vector Search: Databricksネイティブ。Unity Catalog統合
  ├─ OpenSearch (AWS): フルマネージド。大規模向き
  └─ Pinecone: SaaS。最も簡単だが外部依存

→ Databricks構成なら Databricks Vector Search が統合性で有利
  pgvectorは「Databricksの外で軽量に使いたい」場合のサブ選択
```

#### 論点⑤ AI Output Store のレイヤーが不在

```
現行L1はBI用のGold Layerまで。
AIの出力（予測結果、分析レポート、LLMクエリログ）の格納先が未設計。

追加: L1に「AI Output Layer」
  ├─ 予測結果テーブル（バッチ推論の出力）
  ├─ LLMクエリ+結果ログ（監査・改善用）
  ├─ フィードバックテーブル（人間の修正・評価）
  └─ エージェント行動ログ（何をしたか）

→ Delta Table上に構築し、Time Travelでバージョン追跡
```

#### 論点⑥ ML用データセットのバージョニング

```
Feature Store + ラベルから作成した学習データセットを、
再現可能な形で保存する必要がある。

  「モデルv1.2はどのデータで学習したか」を後から完全に再現できること
  → Delta Table のバージョン + MLflow Artifact で管理
```

## 3. L3（データ加工・整形）への追加論点

### 現状: dbt Core + PySpark on Databricks + Airbyte

### 追加すべき観点

#### 論点⑦ 「BI用ETL」と「ML用Feature Engineering」は別物

```
現行L3はBI用のETL（データクレンジング、集計）を想定。

ML用の特徴量エンジニアリングは別パイプラインが必要:
  ├─ ウィンドウ関数（直近N日の移動平均、トレンド）
  ├─ ラグ特徴量（N日前の値）
  ├─ Point-in-Time JOIN（時点整合性を保ったデータ結合）
  ├─ 非構造→構造化抽出（LLMによるテキスト→JSON変換）
  └─ ラベルパイプライン（正解データの自動収集）

→ dbt Core でSQL変換は対応可能だが、
  PySpark で書くべき複雑な特徴量もある
→ Feature Engineering 用の dbt プロジェクトを BI 用と分離すべきか？
```

#### 論点⑧ 非構造データの加工パイプライン

```
介護記録テキスト → LLMで構造化抽出:
  「入居者Aが食堂で転倒。右膝打撲。湿布対応」
  → { event: "転倒", location: "食堂", injury: "右膝打撲", severity: "軽度" }

この変換はdbt/PySparkではなく LLMパイプライン:
  Databricks Foundation Model APIs or Amazon Bedrock

→ 現行L3には「LLMベースの変換パイプライン」の位置づけがない
→ L3の中に入れるか、L5とL3の間に「AI前処理レイヤー」として独立させるか
```

#### 論点⑨ データコントラクトの導入

```
上流システム（L0）のスキーマ変更がL3以降に伝播するリスク:
  2028年のCareBase刷新で全パイプラインが壊れる可能性

対策:
  ├─ dbt contracts（スキーマ定義 + バリデーション）
  ├─ Great Expectations（L4に記載あり。連携設計が必要）
  └─ スキーマ変更検知 → Feature Store への影響分析を自動化
```

## 4. L6（セマンティック層）への追加論点

### 現状: dbt MetricFlow / Databricks Metric Views / Timbr.ai (いずれか選択)

### 追加すべき観点

#### 論点⑩ セマンティック層は「BIのため」と「AIのため」で要件が異なる

```
BI用セマンティック層:
  「稼働率とは何か」の定義 → Tableauで正しいKPIを表示

AI用セマンティック層:
  「稼働率とは何か」の定義 → LLMが正しいSQLを生成
  + テーブル間の結合条件
  + カラムの暗黙的フィルタ（is_deleted=false等）
  + NULLの解釈ルール
  + 値のマッピング（status=1は在籍）

→ AI用のほうが情報量が多い。BI用セマンティック層をそのまま使うと
  LLMのSQL生成精度が落ちる
```

#### 論点⑪ Timbr.ai（Ontology代替）は要PoC検証

```
Timbr.aiが「Ontology代替」として記載されているが:
  ├─ Foundryオントロジーのグラフ構造をどこまで再現できるか？
  ├─ dbt MetricFlowとの共存は可能か？
  ├─ LLM（Genie/AIチャット）との連携は？
  └─ ライセンスコスト・サポート体制は？

→ Timbr.aiはニッチな製品。採用実績が少ないリスクがある
→ PoC段階で dbt MetricFlow vs Timbr.ai の比較検証が必要
```

#### 論点⑫ セマンティック層とLLMの接続方法

```
L6 → L7（AIチャット）の間に「LLMがセマンティック層を参照する方法」が未設計:

  パターンA: Databricks Genie（Databricksネイティブ）
    → Unity Catalog のメタデータを自動参照
    → Databricks構成なら最もシームレス

  パターンB: MCP（Model Context Protocol）経由
    → 外部LLM（Claude Desktop等）からセマンティック層にアクセス
    → 将来的なマルチLLM対応に有利

  パターンC: カスタムAPI
    → セマンティック層の定義をAPIで返す → LLMのシステムプロンプトに注入
    → 最も柔軟だが開発・運用コスト高

→ PoCではAを検証しつつ、将来のBへの拡張性を確認
```

## 5. L5（AI・機械学習）への追加論点 — ★最も論点が多い

### 現状: MLflow / Model Serving / Feature Store / RAG / AutoGluon

### 追加すべき観点

#### 論点⑬ 「部品」は揃っているが「繋ぎ方」が未設計

```
現行L5は部品のリスト:
  MLflow ─ Model Serving ─ Feature Store ─ RAG ─ AutoGluon

足りないのは「これらをどう繋いでライフサイクルを回すか」:

  Feature Store → データセット作成 → MLflow実験 → 評価
  → Model Registry → 承認 → Model Serving → モニタリング
  → ドリフト検知 → 再学習トリガー → Feature Store（ループ）

→ この「MLOpsパイプライン」の設計がレイヤー図に表れていない
→ Layer 2（ジョブ管理）でDatabricks Lakeflow/Dagsterが対応するが、
  ML固有のオーケストレーション要件が考慮されているか？
```

#### 論点⑭ プロンプト管理が不在

```
LLM時代の新しい「ソフトウェア資産」:
  ├─ システムプロンプト（AIチャットの振る舞いを定義）
  ├─ 構造化抽出プロンプト（テキスト→JSON変換の指示）
  ├─ 分析プロンプト（特定のKPI分析の指示テンプレート）
  └─ ガードレールプロンプト（AIが出してはいけない回答の制約）

これらは「コード」と同等に管理が必要:
  → バージョン管理（Git）
  → A/Bテスト（どのプロンプトが精度が高いか）
  → 本番昇格フロー（Staging → Production）
  → パフォーマンス監視（プロンプト変更による精度変化）

→ Model RegistryのLLM版 = 「Prompt Registry」が必要
→ 現行アーキテクチャに位置づけが無い
```

#### 論点⑮ エージェントフレームワークが不在

```
L7「AIチャット（カスタム）」は単なるQ&A UIだが、
将来的には「自律的に動くAIエージェント」が必要になる:

  ├─ 分析エージェント: データを自動分析してレポート生成
  ├─ アラートエージェント: 異常値検知→自動通知
  ├─ 予測更新エージェント: 定期的に予測を更新→配信
  └─ 業務エージェント: ケアプラン起案支援

エージェントの管理基盤:
  ├─ 権限管理: エージェントが何にアクセスできるか
  ├─ 行動ログ: エージェントが何をしたか（全行動記録）
  ├─ コスト制御: エージェントごとのLLM/計算コスト上限
  └─ 承認フロー: 書き込みアクションは人間の承認を挟むか

→ L5とL7の間に「Agent Runtime / Agent Management」レイヤーが必要
```

#### 論点⑯ モデルモニタリング・ドリフト検知

```
L5にMLflowはあるが「学習後のモニタリング」が未設計:

  ├─ データドリフト検知: 入力データの分布変化
  ├─ 予測ドリフト検知: 出力の分布変化
  ├─ パフォーマンス追跡: 精度の経時変化
  └─ LLM品質監視: ハルシネーション率、回答精度の推移

技術選択:
  ├─ Databricks Lakehouse Monitoring (ネイティブ)
  ├─ Evidently AI (OSS、ダッシュボード付き)
  └─ Whylogs (OSS、軽量)
```

#### 論点⑰ 自動評価フレームワーク

```
MLモデルとLLMで「自動評価」の方法が異なる:

MLモデル:
  テストデータセットで精度指標（AUC, F1, RMSE等）を自動計算
  → MLflow + 定期バッチジョブで対応可能

LLM:
  「回答が正しいか」の自動評価は難しい
  → LLM-as-a-Judge（別のLLMが評価する）
  → 人間評価のサンプリングワークフロー
  → 定型質問ベンチマーク（10問の正答率を定期チェック）

→ LLM評価基盤の設計が現行アーキテクチャに無い
```

## 6. L7（現場が使う画面・通知）への追加論点

### 現状: Tableau + AIチャット（カスタム）

### 追加すべき観点

#### 論点⑱ AIチャットの内部設計

```
「AIチャット（カスタム）」とあるが、内部は複雑:

  ユーザー入力
       │
       ▼
  [意図分類] → 「KPI質問？」「フリー分析？」「業務操作？」
       │
       ├─ KPI質問 → セマンティック層参照 → SQL生成 → 結果表示
       ├─ フリー分析 → RAG検索 + LLM → 回答生成
       ├─ 予測照会 → Model Serving呼び出し → 予測結果表示
       └─ 業務操作 → エージェント起動 → 承認フロー → 実行

これは「チャットUI」ではなく「Compound AI System」:
  → L5〜L7にまたがるシステムアーキテクチャの設計が必要
```

#### 論点⑲ フィードバック収集UI

```
現行L7にはフィードバック収集の仕組みが無い:

  AIの回答に対して:
  ├─ 👍/👎 ボタン
  ├─ 「この回答は正しくない」→ 修正内容の入力
  ├─ 「この分析を保存する」→ AI Output Storeに永続化
  └─ 信頼度の表示（「この回答の根拠は3つのデータソースです」）

→ フィードバックデータは L1 の Feedback Store に格納
→ 改善サイクルの入力になる
```

## 7. Layer 4（品質管理・安全管理）への追加論点

### 現状: Great Expectations / dbt tests / Unity Catalog (RLS/監査) / OpenMetadata

### 追加すべき観点

#### 論点⑳ AI固有のガバナンスが不足

```
現行Layer 4は「データの品質管理」。以下のAI固有の統制が不足:

  AI利用ガバナンス:
  ├─ LLM監査ログ: 誰がAIに何を聞いて、何が返ったか
  ├─ プロンプトインジェクション対策: 悪意あるプロンプトの検出・ブロック
  ├─ PII漏洩防止: LLMの出力にPIIが含まれていないかのチェック
  ├─ コスト管理: ユーザー/部門別のLLM利用コスト追跡・制限
  └─ AI倫理チェック: バイアス検出、公平性の定期監査

  → Unity Catalog の RLS/監査 だけでは不十分
  → 「AI Governance Layer」として拡張が必要
```

#### 論点㉑ データ品質がAI精度に直結する「品質のSLA」

```
BI: データ品質が低い → ダッシュボードの数値が間違う → 人間が気づいて修正
AI: データ品質が低い → モデルが間違った予測を出す → 人間は気づかない可能性

→ AI学習に使うデータには、BIより厳しい品質基準が必要:
  ├─ 欠損率: 5%以下（BI用は10%でも可だが、ML用はNG）
  ├─ 鮮度: 24時間以内（日次バッチの遅延がドリフトの原因に）
  ├─ 一貫性: 施設間でデータ定義が統一されているか
  └─ 完全性: ラベル（正解データ）の網羅率
```

## 8. Layer 2（ジョブ管理・オーケストレーション）への追加論点

### 現状: Databricks Lakeflow / Dagster (いずれか選択)

### 追加すべき観点

#### 論点㉒ ML固有のオーケストレーション要件

```
BI用ETLのオーケストレーション:
  日次バッチ: Source → Bronze → Silver → Gold → ダッシュボード更新

ML用パイプラインのオーケストレーション:
  ├─ Feature更新: 日次で特徴量を再計算
  ├─ バッチ推論: 週次で全施設の予測を実行
  ├─ 再学習判定: ドリフト検知 → 自動再学習トリガー
  ├─ 評価パイプライン: 再学習後の自動品質チェック
  └─ モデル昇格: 評価パス → Staging → 承認 → Production

→ 単なるジョブスケジューラではなく「ML Pipeline Orchestrator」が必要
→ Databricks Workflows で対応可能だが、
  BI用ジョブとML用ジョブのリソース分離設計が必要
```

## 9. 新規レイヤー: 現行アーキテクチャに存在しないもの

### 新規① LLMゲートウェイ / モデルルーター

```
現行アーキテクチャにLLM呼び出しの統一インターフェースが無い。
L5, L7, L3（非構造データ処理）がバラバラにLLMを呼ぶと:
  ├─ コスト管理が不可能（どこでいくら使っているか不明）
  ├─ モデル切替が困難（GPT→Claudeに変えたい時、全箇所修正）
  ├─ 監査ログが分散（統一的なログ収集ができない）
  └─ レート制限が効かない（全体のAPI呼び出し量を制御不能）

追加: LLMゲートウェイ（Layer 5.5相当）
  ┌─────────────────────────────────┐
  │       LLM Gateway / Router      │
  │                                 │
  │  ├─ 統一API（全コンポーネントはここを経由）  │
  │  ├─ モデルルーティング（質問種別→最適モデル）│
  │  ├─ コスト追跡（リクエストごとにコスト記録） │
  │  ├─ レート制限（ユーザー/部門別クォータ）   │
  │  ├─ 監査ログ（全リクエスト/レスポンス記録） │
  │  └─ フォールバック（モデルA障害→モデルBに切替）│
  └─────────────────────────────────┘

技術選択肢:
  ├─ LiteLLM (OSS): 100+ LLMの統一API。ルーティング・フォールバック対応
  ├─ AWS Bedrock: AWSネイティブ。マルチモデル対応
  ├─ カスタムAPI Gateway: 最も柔軟だが開発コスト高
```

### 新規② Feedback & Evaluation Layer

```
現行アーキテクチャにフィードバックの仕組みが無い:

  AI出力 → ユーザー → フィードバック → ???

追加: Feedback & Evaluation Layer（Layer 4.5相当）
  ┌─────────────────────────────────────┐
  │    Feedback & Evaluation Layer      │
  │                                     │
  │  ├─ フィードバック収集API            │
  │  ├─ 自動評価パイプライン             │
  │  │   ├─ MLモデル: テストデータで定期評価│
  │  │   └─ LLM: ベンチマーク質問で定期評価│
  │  ├─ 予測→実績突き合わせバッチ        │
  │  ├─ フィードバック集計ダッシュボード   │
  │  └─ 改善チケット自動生成              │
  └─────────────────────────────────────┘
```

## 11. 説明ストーリー — 「この基盤図に、AIエンジニアとして何を足すか」

```
Step 1: 「この基盤アーキテクチャは骨格として正しい」
  → L1-L7のレイヤー構成、Databricks中心の統合管理は適切

Step 2: 「ただし、これは"データ基盤にAI機能を載せた"設計」
  → AIの部品（MLflow, Feature Store等）は揃っているが、
    AIのライフサイクル（学習→評価→デプロイ→監視→改善）を回す設計が薄い

Step 3: 「AI OSとして足りない3つの視点」
  ① 入力側: 非構造データパイプライン + データのAI利用可否分類
  ② 統制側: LLMゲートウェイ（コスト・監査・モデル管理の統一）
  ③ 出力側: AI Output Store + Feedback Loop（改善サイクル）

Step 4: 「具体的に足すもの」
  → 22の論点を優先度付きで提示
  → PoCで検証すべき項目を特定

Step 5: 「なぜ今やるべきか」
  → 後付けが困難なもの（LLMゲートウェイ、AI Output Store、データ分類）は
    初期設計に含めないと、本番後に作り直しになる
```
