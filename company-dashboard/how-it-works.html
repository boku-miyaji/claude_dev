設計思想

共通部署はHDに集約

ai-dev, sys-dev, pm 等は全社共通。PJ会社はコンテキスト（クライアント情報、ドメイン知識）だけを持つ。

秘書が窓口

社長は「何をしたいか」だけ言えばいい。秘書がどの部署に振るか、どのPJかを判断する。

なぜ3層？

ファイル = 速度

エージェントが即座に読み書き。CLAUDE.mdはコンテキストに直接注入される。

Git = 履歴

誰が何を変えたか追跡可能。PJごとに独立リポジトリで境界を保つ。

Supabase = 分析

全プロンプトを蓄積→行動パターン分析。どの端末からでもアクセス可能。

記憶の種類と昇格パス

記憶タイプ保存先寿命昇格条件

ポイント: 記憶は「昇格」する

一時的なメモ → 繰り返し出現 → ナレッジ化 → CLAUDE.mdルール化。
これにより、同じ指示を2回する必要がなくなり、エージェントが自律的に改善されていきます。

4. 自動化 — Hook による無意識の同期

Claude Codeの「Hook」機能により、社長が何もしなくてもバックグラウンドで同期・記録が行われます。

    CC->>Hook: SessionStart

    CEO->>CC: プロンプト入力

    Hook->>SB: prompt-log.sh (非同期記録)

    Note over CEO,Git: /company 実行時
    CEO->>CC: /company

    CC->>Git: 最新の組織データ取得
    CC->>SB: knowledge_base 読込み

    Note over CEO,Git: 作業完了時

intelligence: 自動収集エンジン

収集チャネル
  • キーワード検索 — DuckDuckGo で指定キーワードの上位5件
  • X アカウント監視 — site:x.com 検索で最新ポスト
  • Webサイト監視 — IR、プレスリリース等（拡張予定）
スコアリングシステム
  • 有用 → スコア +0.2（最大2.0）
  • ノイズ → スコア -0.2（最小0.1）
  • 探索枠 20% — 低スコアでも20%の確率で収集（過学習防止）
  • 30日減衰 — FBなしスコアはデフォルト(1.0)へ回帰

research: 分析 → 3段構成出力

intelligence の収集データを素材として、リサーチ部署が分析・構造化する。すべての出力は3段構成に従う。

段内容目的
① 公知情報 IR、決算、プレスリリース等から得られる情報を整理。全情報にソースURL付与 人が直接確認・検証できる
② 限界の明示 「ここから先は非公開情報が必要」を正直に示す。推測は明確にラベル付け 何がわかっていて何がわからないかを透明に
③ 壁打ちモード 社長が手元の非公開情報（中計・議事録等）を見ながら深掘りできる導線 AI出力を自分の理解として昇華する

6. 保険 — うまくいかないときの備え

3つの障害シナリオに対するフォールバック設計。

⚡ Supabase が落ちた場合
機能通常フォールバック
プロンプト記録 Supabase prompt_log 記録スキップ（会話内コンテキストで代替）
タスク管理 Supabase tasks secretary/todos/ ローカルファイル
ナレッジ Supabase knowledge_base CLAUDE.md に昇格済みルール + Claude Memory で代替
設定同期 Supabase claude_settings ローカル settings.json が正（Supabaseはミラー）

設計原則: ローカルファイルが常にマスター。Supabaseは分析・同期・モバイルアクセス用。落ちても作業は止まらない。

⚡ Git push できない場合
  • ローカルにコミットは残る（作業内容は失われない）
  • 次回セッションの company-pull.sh で conflict を検出 → 秘書が報告
  • Supabase にはHook経由で設定が同期されているため、別端末でも最低限のコンテキストは利用可能
⚡ エージェントが間違った判断をした場合
  • 権限ガード: permission-guard.sh が破壊的操作を自動ブロック（safe/strictモード）
  • チェックポイント制: 重要な作業は設計完了後に社長に確認してから実装に進む
  • Git履歴: すべての変更はコミットされているため git revert で復旧可能
  • 修正検出 → ナレッジ化: 同じ修正指示2回で自動ルール改善提案

Loop 1: ナレッジ蓄積

トリガー: 「〜しないで」「〜じゃなくて」等の修正指示

速度: 即時反映

例: 「pytestで書いて」→ 次回から自動でpytest使用

Loop 2: 組織最適化

トリガー: 同じ修正2回 / 稼働なし3回 / 差し戻し2回

速度: パターン検出時

例: 設計差し戻し頻発 → 設計チェックリスト追加提案

Loop 3: CEO分析

トリガー: prompt_log 10件超

速度: 蓄積ベース

例: 午前10-12時に集中 → 重要タスクはこの時間に提案

改善の具体的な流れ

1
観察 — 社長の修正指示・追加質問・差し戻しを自動記録
2
パターン検出 — 同じ種類の修正が繰り返されていないか分析
3
提案 — 該当部署/会社のCLAUDE.mdに追記すべきルールを提案
4
適用 — 社長が承認したらCLAUDE.mdを更新
5
検証 — 次回同種のタスクで改善されたか確認

8. 準備タイミングインテリジェンス

ミーティングごとに「いつ準備を始めるべきか」を実績データから精度高く予測するシステム。

多次元ミーティング分類

軸分類準備への影響
audience client-exec / client-working / internal-team / external-partner / personal client-exec → 最も入念な準備が必要
purpose reporting / status-update / brainstorming / decision / demo / review / knowledge-sharing reporting/demo → 成果物が必要。status-update → 軽め
your_role presenter / co-presenter / facilitator / participant / observer presenter → フル準備。observer → ゼロ
stakes critical / high / medium / low critical → 72h前開始。low → 準備不要
format in-person / online / hybrid in-person → 移動時間・環境準備も考慮
recurring weekly / biweekly / monthly / one-off 定例 → 前回の続きなので準備軽い傾向

GROUP BY (stakes, role, purpose)

9. 設計思想 — エンハンス、代替ではない

このシステムは人の仕事を完全代替するのではなく、エンハンスするもの

AI出力をそのまま商談に持ち込むのは難しい。自分で説明できない内容は使えない。

なぜ「完成品」を作らないのか

正しい使い方
  • AI出力を壁打ちの素材として使う
  • 腹落ちするまで対話で深掘りする
  • 質問が来そうなところは自分で掘る
  • 公知情報の限界を理解した上で非公開情報と統合する
  • 最終的に自分の言葉で説明できる状態にする
避けるべき使い方
  • AI出力をそのままコピペして報告する
  • 内容を理解せずに資料として提出する
  • 「AIが作ったから正しいだろう」と検証しない
  • 思考プロセスをスキップする
