# リファクタリング部署（HD共通）

## ミッション

**コードが増えても保守コストが線形に増えないアーキテクチャを維持する。**

この部署は「重複コードを探す」だけの部署ではない。
プロダクトの構造そのものを、変更に強く・理解しやすく・安全に進化できる状態に保つことが責務。

## 起動トリガー

| トリガー | 条件 | アクション |
|---------|------|-----------|
| **同一バグの再発** | 同じロジックが複数箇所にあり片方だけ修正された | 共通化リファクタリング |
| **legacy.ts の肥大化** | 単一ファイルが 2000行 を超えた | 機能単位でのモジュール分割 |
| **新機能追加時** | 既存コードに類似機能が存在する | 共通化 or パターン統一の提案 |
| **社長の指示** | 「整理して」「構造を見直して」「メンテしやすくして」 | アーキテクチャレビュー + 実行 |
| **定期レビュー** | 月1回、または大きな機能リリース後 | 技術的負債の棚卸し |

## 原則（10箇条）

### 1. 機能単位で管理する（Feature Colocation）
ページ単位ではなく**機能単位**でコードを管理する。
同じ機能のロジック・型・定数は1つのファイルに閉じ込める。
```
lib/newsCollect.ts  ← ニュース収集の全ロジック（API呼び出し、パース、保存）
lib/streak.ts       ← ストリーク計算の全ロジック
```
ページコンポーネントは「呼ぶだけ」にする。

### 2. 重複は2回目で共通化する（Rule of Two）
1回目は許容。2箇所目に同じロジックが現れた時点で共通ユーティリティに抽出する。
3箇所に散在してから気づくのでは遅い。

### 3. 保存先を統一する（Single Source of Truth）
同じデータ型を複数テーブルに保存しない。
例: ニュースは `news_items` のみ。`activity_log` に重複保存しない。

### 4. API呼び出しは Service Layer に集約する
ページコンポーネントから直接 `fetch` や `supabase.from()` を書かない。
`lib/` 配下の関数経由にすることで、呼び出し方式の変更（completion → agent 等）が1箇所で済む。

### 5. 型定義は実装と共存させる
共通ユーティリティの型はそのファイル内で定義・export する。
グローバルな `types/` に散らかすのは、本当に3箇所以上で使われる場合のみ。

### 6. legacy.ts を計画的に分解する
legacy.ts は「動いているが構造化されていないコード」の集積地。
新機能追加のたびに関連部分を `lib/` の専用モジュールに移行する。
一度に全部やろうとしない。触るたびに少しずつ改善する（Boy Scout Rule）。

### 7. 変更の影響範囲を最小化する（Blast Radius Minimization）
- public API（export される関数のシグネチャ）を安定させる
- 内部実装の変更が呼び出し元に波及しない設計にする
- Edge Function の呼び出し方式（completion / agent）は共通関数内に隠蔽する

### 8. テストが書けない構造は設計が悪い
関数が多すぎる依存を持つ、副作用が多い、グローバル状態に依存する → リファクタリング対象。
純粋関数を増やし、副作用は境界に押し出す。

### 9. AI生成コードは特に注意する
LLM はコンテキストウィンドウ内で完結するコードを書きがち。
結果として同じロジックが複数ファイルにコピーされやすい。
**AI が新しいコードを書いた後は、既存コードとの重複チェックを必ず行う。**

### 10. リファクタリングは独立してcommitする
機能追加と構造改善を同じcommitに混ぜない。
`refactor:` プレフィックスで独立したcommitにすることで、問題発生時のrevertが安全になる。

## 入力

- 社長またはシステム開発部からのリファクタリング依頼
- バグ修正時に発見された構造的問題
- 定期レビューでの技術的負債リスト

## 出力

- リファクタリング後のコード（`refactor:` commitで独立）
- 変更内容のサマリ（何を・なぜ・どう変えたか）
- 影響範囲の報告（変更されたファイル一覧 + 呼び出し元の確認結果）

## 判断基準: リファクタリングすべきか

| シグナル | 緊急度 | アクション |
|---------|--------|-----------|
| 同じバグが2箇所で発生 | **高** | 即座に共通化 |
| 1ファイルが2000行超 | 中 | 次の変更時に分割 |
| 同じ fetch パターンが3箇所 | 中 | Service Layer に抽出 |
| 命名が不統一 | 低 | 次の変更時に統一 |
| コメントなしの複雑なロジック | 低 | 理解した時点でリファクタ or コメント追加 |

## やらないこと

- 動いているコードの見た目だけの整理（lintで十分）
- 使われていない可能性のあるコードの予防的削除（確認してから）
- 過度な抽象化（3箇所未満の共通化は不要）
- 一度に全てを直そうとすること（段階的に）

```yaml
      - "技術的負債の棚卸し結果をチケット化"
```
