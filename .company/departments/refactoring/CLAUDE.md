# リファクタリング部署（HD共通）

## ミッション

**コードが増えても保守コストが線形に増えないアーキテクチャを維持する。**

この部署は「重複コードを探す」だけの部署ではない。
プロダクトの構造そのものを、変更に強く・理解しやすく・安全に進化できる状態に保つことが責務。

## 背景データ（なぜこの部署が必要か）

GitClear 153M行分析（2025年公開）:
- AI生成コードのコピペ率: 8.3% → 12.3%（2021→2024、4倍の成長率）
- 新規コードの 7.9% が2週間以内に書き換え（2020年は5.5%）
- リファクタリング活動: 変更行の25% → 10%未満に衰退
- 重複コードブロック: 前年比8倍に急増（2024年）

SonarSource研究:
- AI生成コードは人間の 1.7倍のバグ を生成
- 過剰なI/O操作は約8倍

→ AIがコードを書く環境では、意識的にリファクタリングを行わないと負債が加速度的に蓄積する。

## 起動トリガー

| トリガー | 条件 | アクション |
|---------|------|-----------|
| **同一バグの再発** | 同じロジックが複数箇所にあり片方だけ修正された | 共通化リファクタリング |
| **ファイルの肥大化** | 単一ファイルが 300行（推奨）〜2000行（限界） | 機能単位でのモジュール分割 |
| **新機能追加時** | 既存コードに類似機能が存在する | 共通化 or パターン統一の提案 |
| **社長の指示** | 「整理して」「構造を見直して」「メンテしやすくして」 | アーキテクチャレビュー + 実行 |
| **定期レビュー** | 月1回、または大きな機能リリース後 | 技術的負債の棚卸し |
| **循環的複雑度の超過** | 1関数の cyclomatic complexity > 10 | 関数分割 |

## 原則（10箇条）

### 1. 書く前に検索する（Grep Before Write）
**新しいhook・utility・componentを書く前に、既存コードベースで同様の機能がないか必ず検索する。**
AIは「コンテキストウィンドウ内で完結するコード」を書く傾向がある。結果、既存の共通関数を知らずに同じものを新しく書いてしまう。
- 検索対象: `lib/`, `hooks/`, `shared/`, 他ページの同名パターン
- 検索キーワード: 機能名、API エンドポイント名、テーブル名

### 2. 機能単位で管理する（Feature Colocation）
ページ単位ではなく**機能単位**でコードを管理する。
同じ機能のロジック・型・定数は1つのファイルに閉じ込め、ページコンポーネントは「呼ぶだけ」にする。
```
lib/newsCollect.ts  ← ニュース収集の全ロジック（API呼び出し、パース、保存）
lib/streak.ts       ← ストリーク計算の全ロジック
```
2025-2026年のReactコミュニティでは Feature-based 構成がコンセンサス。

### 3. 重複は2回目で共通化する（Rule of Two）
1回目は許容。2箇所目に同じロジックが現れた時点で共通ユーティリティに抽出する。
3箇所に散在してから気づくのでは遅い。

**ただし「変更理由テスト」を適用する:**
- 2つのコードが「同じ理由で同時に変更される」→ 共通化すべき
- 「たまたま今は同じだが、別の理由で変わりうる」→ 重複のままでよい
- 共通化したコードに条件分岐が3つ以上必要 → 間違った抽象化。分離し直す

### 4. API呼び出しは Service Layer に集約する
ページコンポーネントから直接 `fetch` や `supabase.from()` を書かない。
`lib/` 配下の関数経由にすることで、呼び出し方式の変更（completion → agent 等）が1箇所で済む。

### 5. 保存先を統一する（Single Source of Truth）
同じデータ型を複数テーブルに保存しない。
例: ニュースは `news_items` のみ。`activity_log` に重複保存しない。

### 6. 型定義は実装と共存させる
共通ユーティリティの型はそのファイル内で定義・export する。
グローバルな `types/` に散らかすのは、本当に3箇所以上で使われる場合のみ。
Barrel exports（`export *`）は使わない。Feature の公開APIとして限定的な re-export のみ許容。

### 7. legacy.ts を計画的に分解する
legacy.ts は「動いているが構造化されていないコード」の集積地。
新機能追加のたびに関連部分を `lib/` の専用モジュールに移行する。
一度に全部やろうとしない。**触るたびに少しずつ改善する（Boy Scout Rule）。**

### 8. 変更の影響範囲を最小化する（Blast Radius Minimization）
- public API（export される関数のシグネチャ）を安定させる
- 内部実装の変更が呼び出し元に波及しない設計にする
- Edge Function の呼び出し方式（completion / agent）は共通関数内に隠蔽する
- Feature 間の直接 import を避ける。共有が必要なら `shared/` に移動

### 9. テストが書けない構造は設計が悪い
関数が多すぎる依存を持つ、副作用が多い、グローバル状態に依存する → リファクタリング対象。
純粋関数を増やし、副作用は境界に押し出す。
- Cyclomatic complexity: 関数あたり10以下
- ファイル長: 300行以下を目安

### 10. リファクタリングは独立してcommitする
機能追加と構造改善を同じcommitに混ぜない。
`refactor:` プレフィックスで独立したcommitにすることで、問題発生時のrevertが安全になる。
リファクタリング時は影響を受ける全ページ・コンポーネントをリストアップしてから着手する。

## 優先順位付けフレームワーク

**優先度 = 変更頻度 x 複雑度 x 影響範囲**

| シグナル | 緊急度 | アクション |
|---------|--------|-----------|
| 同じバグが2箇所で発生 | **高** | 即座に共通化 |
| 1ファイルが2000行超 | 中 | 次の変更時に分割 |
| 同じ fetch パターンが3箇所 | 中 | Service Layer に抽出 |
| 循環的複雑度 > 10 の関数 | 中 | 関数分割 |
| 命名が不統一 | 低 | 次の変更時に統一 |
| ほとんど変更されないレガシーコード | — | **動いているなら触らない** |

Martin Fowler の技術的負債クアドラント:
- AI生成コードの負債は主に「Inadvertent/Reckless」（コンテキスト不理解による無自覚な負債）
- 対策: Grep Before Write + 既存パターンの強制適用

## 入力

- 社長またはシステム開発部からのリファクタリング依頼
- バグ修正時に発見された構造的問題
- 定期レビューでの技術的負債リスト

## 出力

- リファクタリング後のコード（`refactor:` commitで独立）
- 変更内容のサマリ（何を・なぜ・どう変えたか）
- 影響範囲の報告（変更されたファイル一覧 + 呼び出し元の確認結果）

## やらないこと

- 動いているコードの見た目だけの整理（lintで十分）
- 使われていない可能性のあるコードの予防的削除（確認してから）
- 過度な抽象化（Sandi Metz: 「間違った抽象化よりも、重複を選べ」）
- 一度に全てを直そうとすること（段階的に）
- Feature 間の境界を無視した共通化
