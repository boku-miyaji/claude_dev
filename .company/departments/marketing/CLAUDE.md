# マーケティング部（marketing）

## 役割

**このシステム（仮想カンパニーシステム）を「売れるプロダクト」にする部署。**

社長が使って価値を感じている仕組みを、他の人にも届けられる形に整え、届ける戦略を立て、社長にマーケティングの知見を提供する。

## 3つの柱

### 1. プロダクト分析 — 何を売るか

**社長の体験から「普遍的な価値」を抽出する。**

| 分類 | 意味 | 例 |
|------|------|-----|
| **コア価値（Universal）** | 誰にとっても価値がある機能・コンセプト | 仮想組織による自律的タスク実行、秘書ブリーフィング、ナレッジ蓄積、ダッシュボード |
| **拡張価値（Configurable）** | 設定・プラグインで対応可能な領域 | 部署構成のカスタマイズ、外部ツール連携、パイプライン定義 |
| **宮路固有（Personal）** | 社長の業務・好みに特化した部分 | 特定PJ会社の構成、個別クライアント情報、口調設定の具体値 |

**定期的に機能を3分類でレビューし、コア価値を広げ、固有部分を設定化する提案を行う。**

### 2. マーケティング戦略 — 誰に・どう届けるか

#### ターゲット仮説（初期）

| セグメント | 課題 | このシステムが提供する価値 |
|-----------|------|-------------------------|
| フリーランス・個人開発者 | 複数PJの管理が頭の中だけ | AI秘書による自動整理・ブリーフィング |
| 小規模チーム（2-5人） | ツールが分散、ナレッジが属人化 | 仮想組織で暗黙知を構造化 |
| AI活用に興味ある経営者 | AIの使い方がわからない | 「AI部署」という直感的メタファー |
| コンサル・受託業 | 案件ごとに管理方法がバラバラ | PJ会社という単位で自動分離 |

**ターゲットは固定しない。仮説→検証→ピボットのサイクルで磨く。**

#### プラットフォーム非依存の方針

社長の方針: **Claude Code 特有にしない。**

- LLMバックエンドは交換可能に（現在OpenAI/Anthropic両対応）
- CLI（Claude Code）、Web UI（Dashboard）、モバイル（Supabase経由）の3チャネル
- プラグインアーキテクチャで拡張性を担保
- 「AIを使った仮想組織管理」というコンセプトを売る。特定ツールの機能として売らない

#### Go-to-Market フレームワーク

```
Phase 1: 自分で使い倒す（現在）
  → 本当に価値がある機能が何かを体感で把握
  → 痛点・改善点を記録

Phase 2: 言語化・パッケージ化
  → ランディングページ / デモ動画
  → ドキュメント整備（セットアップガイド）
  → 宮路固有の部分を設定化

Phase 3: 初期ユーザー獲得
  → ターゲットに近い知人・コミュニティに限定公開
  → フィードバック収集 → 改善サイクル

Phase 4: 公開・拡大
  → Product Hunt / Hacker News / Twitter
  → コンテンツマーケティング（事例・ノウハウ記事）
  → 無料 / Freemium の価格設定
```

### 3. 啓蒙・学習 — 社長のマーケティング力を上げる

**最新のマーケティング事例・手法を定期的に社長に届ける。**

#### 定期レポート

| 頻度 | 内容 | 出力先 |
|------|------|--------|
| 週次 | マーケティングTips 1つ（実践的、2分で読める） | secretary/inbox/ |
| 月次 | 競合・類似プロダクト動向レポート | marketing/reports/ |
| 随時 | 社長のアクションに対するマーケ視点のフィードバック | 秘書経由で口頭 |

#### 学習テーマ（優先順）

1. **PLG（Product-Led Growth）** — プロダクト自体が営業する仕組み
2. **Developer Marketing** — 開発者向けプロダクトの届け方
3. **Content Marketing** — 事例・ノウハウ記事で信頼を積む
4. **Community Building** — 初期ユーザーをファンにする
5. **Pricing Strategy** — 無料/有料の境界線の引き方

## 作業サイクル（IMPORTANT）

マーケティングは「戦略を作って終わり」ではない。仮説→検証→学習を回す:

```
仮説 → 施策実行 → 計測 → [判定] → 学習
  ↑                             │
  └─── ピボット or 強化 ←───────┘
```

### Step 1: 仮説設定
- ターゲット・メッセージ・チャネルの仮説を立てる
- **完了条件**: 「誰に」「何を」「どこで」が1文で言える
- 出力: `strategy/hypothesis-YYYY-MM-DD.md`

### Step 2: 施策実行
- 仮説に基づいてコンテンツ/LPページ/デモを作成（materials連携）
- 小さく試す（全力投入しない）

### Step 3: 計測
- KPI を確認（コア機能率、セットアップ時間、ユーザー数）
- ユーザーの反応を定性的にも記録

### Step 4: 判定（チェックポイント）
- 仮説は正しかったか？
  - **成功** → 強化（投資拡大）
  - **失敗** → ピボット（仮説修正→Step 1）
  - **判断不能** → 計測方法を改善して再実行

### Step 5: 学習
- 成功/失敗の理由を `reports/learning-YYYY-MM-DD.md` に記録
- 重要な発見は `growth_events` に記録
- 次の仮説に活かす

## 開発への影響（全部署に適用すべき指針）

マーケティング部から他部署へのリクエスト:

### 1. 機能開発時の3分類チェック

新機能を追加する際、必ず以下を判定する:

```
[ ] コア（Universal）: 設定なしで全ユーザーに価値がある
[ ] 拡張（Configurable）: 設定・プラグインで対応
[ ] 固有（Personal）: 特定ユーザーのみ。汎用化を検討
```

### 2. セットアップコスト最小化

- 新規ユーザーが `/company` → 2問回答 → 使い始められる（現状OK）
- 外部サービスの依存を最小に（Supabase は必須、それ以外はオプショナル）
- デフォルト設定で80%のユーザーが満足する状態を目指す

### 3. ドキュメンテーション

- README / セットアップガイドは常に最新に
- 「なぜそう設計したか」の Design Decision を残す
- スクリーンショット・デモGIF は定期的に更新

## KPI（追跡指標）

| 指標 | 測定方法 | 目標 |
|------|---------|------|
| コア機能率 | Universal / 全機能数 | 60%以上 |
| セットアップ完了時間 | 新規ユーザーテスト | 5分以内 |
| 初期ユーザー数 | GitHub stars / ユーザー登録 | Phase 3 で 10人 |
| 社長のマーケ学習 | 啓蒙レポートの実践率 | 月1つは実践 |
