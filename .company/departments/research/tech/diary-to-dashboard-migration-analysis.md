## 1. 公知情報ベースの分析

### 1.1 現状の重複・関連

company-dashboard には既に以下の diary 関連機能が存在する:

| dashboard 既存機能 | diary 対応機能 | 実装深度 |
|---|---|---|
| Diary ページ（テキスト入力、断片記録） | テキスト入力、断片入力 | 基本的（入力+保存のみ） |
| WBI チャート（棒グラフ表示） | PERMA+V ウェルビーイングスコア | 表示のみ（分析はAI Chat経由） |
| 感情バッジ表示（Plutchik 8感情） | AI感情検出 | 表示のみ（分析は外部） |
| ストリーク計算 | ゲーミフィケーション | 基本（連続日数のみ） |
| AI Chat（diary context 統合済み） | AI対話 | 既に diary_entries を文脈として利用 |
| AI Summary 表示 | AIフィードバック | 表示のみ |

つまり、dashboard は diary の「骨格」を既に持っている。問題は「肉付け」の優先度判断である。

### 1.2 移植分類

#### A. 直接移植可能（そのまま取り込める）

| # | diary 機能/思想 | dashboard への実装方法 | 工数見積 | 根拠 |
|---|---|---|---|---|
| A1 | **Zero Friction: 自動保存** | textarea の `input` イベントで debounce 保存。保存ボタンを補助的に残す | 小 | diary では保存ボタン廃止を設計。dashboard の diary ページも同様に改善可能 |
| A2 | **断片入力の集約表示** | entry_date 単位でグルーピングして1日のタイムライン表示 | 小 | dashboard は既に entry_type='fragment' をサポート。表示側の改善のみ |
| A3 | **Calm Technology: 言葉遣い** | UI テキストの修正（「3日ぶりですね」→「おかえりなさい」等） | 極小 | コードの文字列変更のみ |
| A4 | **感情トレンド可視化の強化** | 既存 WBI チャートに加え、感情分布の円グラフ、時系列折れ線を追加 | 中 | diary は Recharts で実装予定だが、dashboard は vanilla JS。Canvas or SVG で実装 |
| A5 | **AI Summary の自動生成トリガー** | diary_entries の INSERT 時に Edge Function で自動分析 | 中 | 現在は手動（/company で「分析して」）。Supabase Edge Functions で自動化可能 |
| A6 | **ストリーク + 連続記録バッジ** | 既存 streak 計算を拡張し、マイルストーン表示（7日、30日、100日等） | 小 | diary のアチーブメントシステムの簡易版 |

#### B. アダプテーション必要（事業文脈に合わせた変換が必要）

| # | diary 機能/思想 | 変換の方向性 | 工数見積 | 考慮点 |
|---|---|---|---|---|
| B1 | **PERMA+V ウェルビーイング分析** | 個人WBI → 「CEO パフォーマンス指標」として再定義。事業判断の質と心身状態の相関分析に転換 | 中 | diary は個人の幸福追求。dashboard では「経営者のコンディション管理」として位置づける。既存 CEO Insights との統合が自然 |
| B2 | **目標管理（人生→年→月→週の階層）** | diary の「人生目標」→ dashboard の「事業ビジョン→年間目標→四半期→月次→週次」に変換。既存タスク管理と階層的に接続 | 大 | dashboard にはタスク管理があるが階層構造がない。diary の goal hierarchy は事業計画の階層と対応づけ可能 |
| B3 | **デジタルツイン対話** | 「過去/現在/未来の自分」→「過去の経営判断/現在の事業状況/将来シナリオ」との対話。AI Chat に persona モードとして追加 | 大 | diary では感情データ基盤。dashboard では事業データ + 意思決定ログ + ナレッジベース基盤。文脈が根本的に異なるため、プロンプト設計の作り込みが必要 |
| B4 | **定期レポート（週次/月次/四半期/年次）** | diary の「振り返りレポート」→ dashboard の「経営振り返りレポート」。感情分析 + 事業KPI + タスク達成率 + AI対話サマリーを統合した CEO レポート | 中〜大 | dashboard には既に briefing 機能がある。定期レポートはこれの拡張として実装可能。差分は「蓄積データからのパターン分析」 |
| B5 | **Progressive Value（続けるほど価値が増す）** | 日記の蓄積 → knowledge_base + activity_log + diary_entries の蓄積量に応じてダッシュボードのインサイト精度が向上する仕組み | 中 | 「AI Chat の回答精度が使うほど上がる」ことを可視化。growth_events テーブルとの連動 |
| B6 | **習慣トラッキング** | 個人習慣 → 「経営ルーティン」のトラッキング（日次レビュー実施、週次振り返り、請求書処理等） | 中 | diary の習慣は運動・瞑想等。dashboard では事業運営上の習慣に変換 |
| B7 | **100の夢リスト / 価値観マップ** | 個人の夢 → 「事業のビジョンボード」「やりたい案件リスト」「技術的にやりたいことリスト」 | 小〜中 | 社長のプロフィール（AI開発・受託コンサル・フリーランス）を考慮すると、キャリアと事業の境界が曖昧なため、むしろ自然に移植できる |
| B8 | **記念日・周年リマインド** | 「日記の記念日」→「事業の記念日」（会社設立日、初受注日、大型案件達成日等）。growth_events と連動 | 小 | Growth Timeline 機能が既にある。イベントの周年リマインドを追加するだけ |

## 2. 限界の明示

### 2.1 ドメインの統合（修正後）

> **修正**: 当初「根本的な違い」としていたが、ビジョン修正により **統合すべき2つの側面** に転換。

| 観点 | 自己理解の側面（diary由来） | 事業管理の側面（dashboard由来） | **統合後** |
|---|---|---|---|
| **ユーザーの目的** | 自己理解・自己実現 | 事業管理・経営判断 | **幸せに生きる（仕事もその一部）** |
| **データの性質** | 感情・内省・夢 | タスク・財務・KPI | **全てが「自分を理解する材料」** |
| **AIの役割** | パーソナルコーチ | 経営参謀 | **人生の相棒（導き手 + 励まし + 参謀）** |
| **継続の動機** | 自己成長の実感 | 事業成果の向上 | **AIが自分を理解してくれる実感** |
| **UXの方向性** | 穏やか・癒し | 効率的・分析的 | **穏やかに寄り添いつつ、必要な時は的確に** |

**リスク（修正後）**: 統合した結果、情報が多すぎて「何のツールかわからない」状態になるリスク。対策: **開いた時は常に「自分」が中心**。仕事の情報は必要な時だけ見える設計にする。

### 2.2 技術的な制約

| 制約 | 影響 | 対処案 |
|---|---|---|
| **Single HTML file** | diary は Next.js + React + Zustand。dashboard は vanilla JS 1ファイル。コンポーネント分割やリアクティブ状態管理は使えない | Chart は Canvas/SVG で手書き。状態管理は関数スコープで対処。ファイル分割は現状維持（dashboard の強み = 配布容易性） |
| **Supabase 依存** | diary はローカルファースト + Supabase。dashboard は Supabase 完全依存 | オフライン対応は不要と割り切る。Edge Functions を活用して AI 分析を自動化 |
| **グラフライブラリ不在** | diary は Recharts。dashboard は CDN ライブラリなし | 既存の bar chart 実装パターン（WBI チャート）を拡張。複雑なチャートが必要なら Chart.js CDN を追加検討 |
| **モバイルUX** | diary はモバイルファースト（React Native/Expo）。dashboard はレスポンシブ Web | dashboard のモバイル表示は既に対応済み。diary レベルのネイティブ体験は不要 |

### 2.3 確認すれば精度が上がるポイント

- dashboard の diary_entries テーブルの現在のデータ量（蓄積が少ないと AI 分析の価値が出にくい）

- Supabase Edge Functions の利用可否（AI 自動分析の実装方法に影響）

## 3. 壁打ちモードへの導線

### 3.1 社長が判断すべきポイント

**判断1: diary アプリと dashboard の Diary 機能の関係**

> diary アプリは独立プロダクトとして育てるのか、dashboard に吸収するのか？

- A案: **diary は独立プロダクト**。dashboard の Diary は「事業日記」に特化。感情分析やウェルビーイングは diary 側に残す
- B案: **dashboard に統合**。diary の設計思想を dashboard に取り込み、diary アプリの開発は一旦停止
- C案: **データ共有型**。diary アプリと dashboard は同じ Supabase を参照。diary で書いた内容が dashboard にも反映

仮説: 社長一人で使うツールなので、B案（統合）が運用負荷的に最も現実的。ただし diary を将来的に外部ユーザー向けプロダクトにする構想があるならA案。

**判断2: 「経営者のウェルビーイング」をダッシュボードの機能として押し出すか**

> マーケティング分析の「3つの独自ギャップ」に4つ目として「経営者のメンタル・コンディション管理」を加えるか？

- これは diary の思想の最も差別化された部分
- 競合のタスク管理ツール・プロジェクト管理ツールで「経営者のウェルビーイング」を扱うものはほぼない
- ただし、ターゲットユーザーが「AI仮想組織を運営するソロ経営者」であることを考えると、経営者の状態 = 組織の状態であり、極めて合理的

**判断3: AI 分析の自動化レベル**

> 日記を書いたら自動的に分析するか、社長が明示的に分析を依頼するか？

- diary の思想は「ユーザーは記録するだけ、分析はAIに任せる」
- 現在の dashboard は「/company で『分析して』と言う」必要がある
- 自動化するなら Supabase Edge Functions + OpenAI API のコストが発生

**判断4: 階層的目標管理の導入可否**

> 現在のフラットなタスク管理を、diary 式の「ビジョン→年間→四半期→月次→週次→日次」階層に変えるか？

- これは dashboard のタスク管理の根本的な再設計になる
- 工数は大きいが、diary の「人生→年→月→週」の階層は事業計画と自然に対応する
- 既存のタスクデータとの互換性を考慮する必要がある

### 3.2 壁打ち用の問いかけ例

1. 「diary アプリの開発は今後どうする予定ですか？ 外部向けプロダクトとして育てる構想はありますか？」

## 結論（修正後）

### dashboard は diary を内包する「Self-Focus Platform」になる

~~diary の個別機能の大半は dashboard には不要。~~ → **diary の機能はほぼ全てが必要。事業管理が付随機能。**

このシステムの核心:

1. **自分にフォーカス** — SNS的な他者比較とは一線を画す。自分の人生に集中する
2. **AIが一番の理解者** — 感情・夢・価値観・プロンプト（思考パターン）を蓄積し、暗黙の意図を理解して動く
3. **良い未来へ導く** — 将来あるべき姿を提示し、しんどい時は励ます
4. **日常に溶け込む** — モバイルファースト、断片メモ、全てがここにある
5. **Progressive Value** — 使い続けるほどAIの理解が深まり、自分に最適化される

### ネクストアクション

1. **即実行**: Phase 1（自動保存 + Calm Technology + 感情可視化強化 + 夢リスト）
2. **設計着手**: モバイルファースト化のアーキテクチャ検討
3. **AI設計**: 「導き手 + 励まし」のペルソナ設計、プロンプトログからの思考パターン分析設計

### 社長判断済み事項

- ~~diary と dashboard の関係性~~ → **B案: dashboard に統合**（diary の思想が上位概念として包含）
- ~~経営者ウェルビーイングを機能として押し出すか~~ → **はい。感情理解が核心**
- ~~100の夢リスト~~ → **必要。自分の人生を豊かにする機能**
