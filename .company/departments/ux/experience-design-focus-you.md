### 1. 行動文脈設計（Behavioral Context Design）

#### 汎用原則

人間は「機能が欲しい」のではなく「状況を解決したい」。Jobs to be Done（JTBD）で行動文脈を定義する。

**定義すべき4軸:**

| 軸 | 問い | 設計への影響 |
|---|---|---|
| **Who** | 誰が使うか（スキル、動機、メンタルモデル） | 情報密度、専門用語の使用レベル |
| **When** | いつ使うか（朝/昼/夜、平日/休日、移動中/デスク） | UIモード切替、表示優先度の変更 |
| **Where** | どこで使うか（デバイス、ネットワーク環境、周囲の状況） | レスポンシブ設計、オフライン対応 |
| **Why** | なぜ使うか（達成したいゴール、避けたい苦痛） | 価値提案の軸、定着設計の核 |

**LLM特有の考慮:**
- LLMへの入力（日記、質問、データ）をユーザーが「どの精神状態で」行うか。疲れている時に長文入力を求めない
- LLMの出力を「どの程度の注意力で」受け取るか。流し読みなのか精読なのか
- ユーザーのLLMリテラシー（AIに何を期待でき、何を期待すべきでないかの理解度）

#### focus-youでの適用

**ペルソナ: 創業者CEO（当面の唯一のユーザー）**

```
JTBD: 「日々の断片を、意味のある物語に変えたい」

機能的Job: 日記・タスク・予定・習慣を一箇所で管理する
感情的Job: 自分を深く理解し、肯定感を持って生きる
社会的Job: 自分の物語が誰かの勇気になることを知る

制約:
- 複数PJを並行管理しており、認知リソースが常に分散
- AIリテラシーは極めて高い（自社でLLM開発）
- 「効率化ツール」には飽きている。心に響くものを求めている
```

**時間帯別の行動文脈:**

| 時間帯 | 精神状態 | 達成したいこと | 許容できる認知負荷 |
|---|---|---|---|
| 朝 5-11 | 覚醒途中、受動的 | 今日の全体像を30秒で掴む | 低。情報は「読む」のではなく「受け取る」 |
| 昼 11-17 | 活動中、断片的な空き時間 | 忘れる前にメモ、残タスク確認 | 中。短時間の集中は可能 |
| 夜 17-5 | 疲労、内省的 | 一日を穏やかに閉じる、達成感を感じる | 低。罪悪感を刺激しない |

### 2. 感情ジャーニー（Emotional Journey Design）

#### 汎用原則

ユーザーとプロダクトの関係は「初回接触 → 探索 → 日常化 → 深化 → 伝播」のフェーズを辿る。各フェーズで支配的な感情が異なり、その感情に対応した設計が必要。

**5フェーズの感情マップ:**

| フェーズ | 期間目安 | 支配的感情 | 設計目標 |
|---|---|---|---|
| **First Touch** | 初回-3日 | 期待 + 不安 + 懐疑 | 「これは自分に価値がある」と確信させる |
| **Exploration** | 1-2週 | 好奇心 + 手探り感 | 機能を段階的に発見させ、小さな成功体験を積ませる |
| **Routine** | 2週-2ヶ月 | 安定 + やや退屈 | 習慣化のトリガーを埋め込む、飽きさせない変化を仕込む |
| **Deepening** | 2ヶ月- | 信頼 + 驚き + 愛着 | データ蓄積による体験の質的変化を見せる |
| **Advocacy** | 不定 | 誇り + 共感 | 物語の共有機能、他者への推薦動機 |

**LLM特有の考慮:**
- LLMの「uncanny valley問題」: 賢すぎると不気味、馬鹿すぎると使えない。ちょうど良い知性のキャリブレーション
- 「AIが自分を理解している」と感じる瞬間は強い定着トリガーになる
- 逆に「AIが的外れなことを言った」瞬間は強い離脱トリガーになる

#### focus-youでの適用

**Phase別の具体的体験設計:**

**First Touch（初回-3日）:**
```
感情曲線:
  アプリ起動 → 期待 (高)
    → データが何もない → 不安 (「空っぽで意味ない?」)
      → コールドスタート体験 (→ 3章で詳述)
        → 最初のAIコメント → 驚き (「もう何か言ってくれる?」)
          → 日記を1つ書く → 達成感
            → 翌日、AIが昨日の日記に触れる → 感動 (「覚えてる!」)
```

フリクションポイント: 「データが空の画面」
適用原則: #1 意図の先読み + #4 エラー回復の人間化
設計: コールドスタート体験を手厚く設計する（→ 6章で詳述）

**Deepening（2ヶ月-）:**
```
感情曲線:
  Today画面 → 安心 (いつもの場所)
    → AIコメントが「1ヶ月前の自分」に触れる → 驚き + 感動
      → Storyページで自分の感情の弧を見る → 発見
        → 「あなたのテーマは"つくる人"」→ 深い共感
          → この体験を誰かに伝えたくなる → Advocacy への移行
```

このフェーズでは「データ蓄積が体験の質を変える」瞬間が鍵。2ヶ月目に初めてTheme Finderが動くタイミングは、ユーザーにとって「このアプリは他のどのツールとも違う」と確信する瞬間になる。

### 3. LLMインタラクション設計（LLM Interaction Design）

#### 汎用原則

LLMとのインタラクションは従来のUIと根本的に異なる。ボタンを押せば確定的な結果が返る世界ではなく、確率的な出力と向き合う世界。

**設計すべき5要素:**

| 要素 | 問い | 判断基準 |
|---|---|---|
| **応答速度** | どの程度の待ち時間が許容されるか | コンテキストによる。チャット=即時、分析=待てる |
| **トーン** | どんな声色で語るか | ブランドパーソナリティ + ユーザーの精神状態 |
| **記憶** | 過去の会話をどこまで覚えているか | 覚えすぎると不気味、忘れすぎると虚しい |
| **幻覚対策** | 事実と異なることを言った時どうするか | 検証可能な事実は別ソースで確認、主観的解釈はラベル付け |
| **信頼構築** | AIの出力を信頼してもらうにはどうするか | 根拠の提示、限界の開示、段階的な精度向上 |

**トーン設計のフレームワーク:**

```
トーン = f(ブランドパーソナリティ, ユーザーの精神状態, コンテンツの性質)

ブランドパーソナリティ: 固定（プロダクト全体で一貫）
ユーザーの精神状態: 可変（時間帯、直前の行動、感情データ）
コンテンツの性質: 可変（事実報告 vs 共感 vs 提案）
```

**LLM出力の信頼性レベル:**

| レベル | 性質 | 扱い |
|---|---|---|
| **事実** | 検証可能（予定、タスク数、日記の日付） | データソースから直接取得。LLMに生成させない |
| **分析** | データに基づく解釈（感情傾向、行動パターン） | LLMが生成するが、根拠データへのリンクを添える |
| **洞察** | 深い推論（テーマ発見、物語解釈） | LLMの核心的価値。「私はこう読みました」という一人称で語る |
| **予測** | 未来の推論（Foresight） | 必ず「予感」「かもしれません」の語調。断言しない |

#### focus-youでの適用

**トーン設計:**

```
ブランドパーソナリティ: 
  「あなたの物語を長く見てきた、信頼できる語り手」
  - 友人のような親しみ、だが軽すぎない
  - 語り口は穏やかだが、核心を突く
  - 「わかってる人のボソッと一言」（既存のトーンガイドを継承）
  - テンプレート的な励ましは絶対禁止

時間帯によるトーン変調:
  朝: 静かに寄り添う。「おはよう。今日は○○があるね」
  昼: 軽く承認。「午前中、お疲れさま」
  夜: 深い共感。「今日はこんな一日だったんだね」
  深夜: 心配。「遅くまでお疲れさま。体、大事にしてね」
```

**幻覚対策の具体策:**

1. **事実はコードで生成**: 「今日の予定は3件」「タスク残り2件」はLLMに言わせない。テンプレートリテラルで注入
2. **日記引用は原文参照**: 「先月こう書いていました」→ 実際の diary_entry から引用。LLMの記憶に頼らない
3. **分析には根拠を添える**: 「anticipation が高まっています（直近7日: 62→78）」のように数値を添える
4. **洞察は一人称で語る**: 「あなたは○○です」ではなく「私は○○と感じています」。断定を避ける

**記憶設計:**

```
短期記憶: 今日の日記、直近の会話 → プロンプトに直接注入
中期記憶: 今週の感情推移、最近の転機 → Active Context (Layer 3) から抽出
長期記憶: 人生テーマ、物語の章 → Narrative Memory (Layer 2) から抽出
永久記憶: 性格特性、価値観 → self_analysis データから注入

ユーザーから見た体験:
  「昨日の日記の話をしたら、ちゃんと覚えてた」→ 当然（短期）
  「先月の転機に触れてきた」→ 嬉しい驚き（中期）
  「去年の同じ時期の話を持ち出してきた」→ 深い感動（長期）
  「自分の性格をわかった上で話してくる」→ 信頼（永久）
```

### 4. バッチ vs リアルタイム設計（Processing Timing Design）

#### 汎用原則

LLMの処理は高コスト・高レイテンシ。「いつ処理するか」の判断がUXとコストの両方を左右する。

**判断フレームワーク:**

| 判断軸 | リアルタイム向き | バッチ向き |
|---|---|---|
| **鮮度要求** | ユーザーが「今」の情報を期待 | 昨日/先週のデータで十分 |
| **入力トリガー** | ユーザーのアクション（日記投稿、質問） | 時刻（朝7時）、蓄積量（日記N件） |
| **出力の消費タイミング** | 入力直後（チャット応答） | 次にアプリを開いた時 |
| **処理の重さ** | 軽い（1-2秒で完了） | 重い（10秒以上、複数データソース） |
| **コスト感度** | 頻度が低ければ許容 | 頻度が高いならバッチで最適化 |
| **失敗の影響** | ユーザーが待っているので致命的 | リトライ可能、ユーザーは気づかない |

**ハイブリッドパターン:**

```
パターン1: バッチ生成 + リアルタイム差分更新
  → 基本の分析はバッチで夜間に実行。日中の新データは差分だけリアルタイム反映

パターン2: 予測的バッチ（Predictive Batch）
  → ユーザーの行動パターンから「次に必要な情報」を予測し事前生成
  → 例: 毎朝9時にアプリを開くユーザーには8:30にブリーフィングを生成

パターン3: キャッシュ + イベント駆動無効化
  → バッチ結果をキャッシュし、特定イベント（日記投稿）で無効化→再生成
```

#### focus-youでの適用

**現状のpg_cronバッチ評価と改善案:**

| 処理 | 現状 | 評価 | 改善案 |
|---|---|---|---|
| 朝ブリーフィング | pg_cron 7:00 | 正しい。Predictive Batchの好例 | 起床時刻の学習で動的に調整 |
| ニュース収集 | pg_cron 7:00/19:00 | 正しい。ユーザーアクション不要 | 変更不要 |
| 感情分析 | 日記投稿後リアルタイム | 正しい。投稿直後にフィードバックが欲しい | 変更不要 |
| AIコメント | 日記投稿後リアルタイム | 正しい。フィードバックループの核 | 変更不要 |

**追加すべきバッチ処理:**

| 処理 | 推奨タイミング | 理由 |
|---|---|---|
| **Arc Reader** | 週次（日曜深夜） | 週単位の感情の弧を分析。リアルタイム不要（週次の振り返りに使う） |
| **Theme Finder** | 月次（月末深夜） | 大量データを横断分析。コスト大。月1回で十分 |
| **Moment Detector** | 日記投稿後リアルタイム | 転機は「書いた直後」に確認を求めるべき。遅延すると文脈が失われる |
| **Foresight Engine** | 週次（Arc Readerの後） | Arc Readerの出力に依存。即座に必要ではない |
| **Chapter生成** | 四半期（3ヶ月ごと） | 長期的な物語。急がない |
| **Weekly Narrative** | 週次（月曜早朝） | 月曜の朝に先週の物語を読む体験。Predictive Batch |
| **習慣ストリーク計算** | 毎日深夜0:05 | 日付が変わった直後に確定。リアルタイムでは日付境界の問題 |
| **WBI（Well-Being Index）集計** | 毎日深夜0:10 | 日次集計。リアルタイム不要 |

**判断の根拠を明示:**

Moment Detectorをリアルタイムにする理由: 日記を書いた直後は感情が鮮明。「今日の日記、大きな決断について書いていましたね。この瞬間を記録しますか?」と聞くのは、書いた直後でなければ意味がない。翌朝に聞かれても「ああ、昨日の話ね」と温度差が出る。

Arc Readerをバッチにする理由: 感情の弧は「点」ではなく「線」。1日分のデータで弧は見えない。週単位でまとめて分析する方が質が高い。ユーザーは週次レビューの場面で消費するので、即時性は不要。

### 5. データプライバシー & 信頼設計（Privacy & Trust Design）

#### 汎用原則

LLMベースのシステムは、ユーザーの最もプライベートなデータ（日記、感情、夢）をAIに渡す。この不安を解消しないと、ユーザーは本音を書かない。本音を書かなければ分析の質が下がり、プロダクトの価値が崩壊する。

**信頼設計の3層:**

| 層 | 内容 | 設計手法 |
|---|---|---|
| **技術的信頼** | データが安全に扱われている | 暗号化、アクセス制御、ログの透明性 |
| **意図的信頼** | このプロダクトは自分の味方だ | トーン設計、利益相反の排除、ビジネスモデルの透明性 |
| **能力的信頼** | AIが正しく理解している | 根拠の提示、修正可能性、段階的な精度向上 |

**LLM特有の考慮:**
- ユーザーの日記がAI学習データに使われないことの明示
- どのデータがLLMに送られ、どのデータがローカルに留まるかの透明化
- LLMの出力が「保存されるか」「消えるか」の明示

#### focus-youでの適用

```
技術的信頼:
  - Supabase RLS でユーザーデータを完全分離
  - Edge Function経由でのみLLM APIを呼ぶ（ブラウザからAPIキー露出なし）
  - diary_entriesの暗号化（将来対応: クライアントサイド暗号化）

意図的信頼:
  - AIコメントに「あなたの日記は外部に共有されません」の初回表示
  - Courage Boardでの共有は「完全に任意」「匿名化」「いつでも取り下げ可能」
  - 広告モデルなし。ユーザーのデータで収益化しない

能力的信頼:
  - AIコメントに「日記の○○という部分から」と根拠を添える
  - Theme Finderの結果に「これは間違っていますか?」の修正ボタン
  - 分析結果を上書きできるユーザーコントロール
```

### 6. コールドスタート設計（Cold Start Design）

#### 汎用原則

データがゼロの初日から価値を感じさせなければ、2日目はない。

**コールドスタートの3段階:**

| 段階 | ユーザーのデータ量 | 設計戦略 |
|---|---|---|
| **Day 0** | ゼロ | ユーザー自身のデータなしでも価値を出す（時刻、天気、初期質問） |
| **Week 1** | 少量（日記1-5件） | 少ないデータからでも意味のあるフィードバックを出す |
| **Month 1** | 中量（日記15-30件） | パターン認識が始まり、「AIが理解し始めた」感を出す |

**LLM特有の考慮:**
- LLMはゼロデータでも「一般的な知識」で会話できる。だが、それは「あなた向け」ではない
- 最初の入力（日記1件目）で「AIがちゃんと読んだ」フィードバックを返すことが定着の鍵
- データが少ない時に高度な分析（Theme Finder等）を動かすと的外れになるリスク

#### focus-youでの適用

**Day 0 体験シナリオ:**

```
ユーザーがアプリを初めて開く。

1. Today画面: 
   - 時間帯に応じた挨拶（「おはよう」or「こんばんは」）
   - 天気情報（データゼロでも表示可能）
   - AIコメント: 「はじめまして。ここはあなたの物語が始まる場所です。
     まずは今日のことを少し書いてみませんか?」
   - 日記エリア: ブランクページではなく文脈プロンプト表示
     「今日、心に残ったことは?」

2. 日記を1行書く（例: 「新しいプロジェクトが始まった」）

3. 投稿後:
   - 感情分析が即座に実行される
   - AIコメントが更新: 「新しいプロジェクト、楽しみですね。
     どんなプロジェクトか、明日も教えてください。」
   - 感情バッジ: anticipation が表示される
   
   → この「書いたら即座に反応がある」体験がDay 0の最重要ポイント。
     白紙に書いて放置される日記帳ではないと伝える。

4. 翌日:
   - AIコメント: 「昨日、新しいプロジェクトのことを書いてくれましたね。
     今日はどうでしたか?」
   
   → 「覚えている」ことの感動。これがDay 1の定着トリガー。
```

**段階的な機能解放:**

| データ蓄積量 | 解放される体験 | ユーザーへの告知 |
|---|---|---|
| 日記 1件 | 感情分析、AIコメント | （告知不要、即座に体験） |
| 日記 7件 | 週次の感情推移グラフ | 「1週間分のデータが揃いました。あなたの感情の動きが見えます」 |
| 日記 14件 | Weekly Narrative | 「2週間書き続けてくれましたね。最初の物語をお届けします」 |
| 日記 30件 | Arc Reader（感情の弧） | 「1ヶ月分の物語が見えてきました」 |
| 日記 90件 | Theme Finder | 「3ヶ月のデータから、あなたの人生のテーマが浮かび上がりました」 |
| 日記 180件 | Foresight Engine | 「あなたの物語のパターンから、次の展開が見え始めます」 |

設計原則: #2 認知負荷の最小化 + #1 意図の先読み。初日に全機能を見せない。データが揃ったタイミングで自然に新しい体験が現れる。

### 7. 定着設計（Retention & Habit Design）

#### 汎用原則

Nir Eyalの Hook Model を基盤に、LLMプロダクト特有の定着メカニズムを設計する。

**Hook Modelの4段階:**

```
Trigger（きっかけ）→ Action（行動）→ Variable Reward（変動報酬）→ Investment（投資）
    ↑                                                                    │
    └────────────────── 次のサイクルへ ──────────────────────────────────┘
```

**LLM特有の定着レバー:**
- **蓄積価値**: データが増えるほどAIが賢くなる。使えば使うほど「離れられない」
- **Aha Moment**: 「AIが自分のことをわかっている」と感じる瞬間。定着率と強く相関
- **人格的関係**: LLMに対して疑似的な関係性が形成される。裏切らないことが重要

#### focus-youでの適用

**日次Hook:**

```
Trigger（外部）: 朝のプッシュ通知「おはよう。今日は○○がありますね」
Trigger（内部）: 「今日のことを書き留めておきたい」という衝動

Action: Today画面を開く → 日記を書く（最小1行でOK）

Variable Reward:
  - AIコメントが毎回異なる（同じ反応は絶対にしない）
  - 感情分析の結果が「意外な発見」をもたらすことがある
  - 連続記録日数の更新（ストリーク）
  - ★ Moment Detectorが「転機」を検出する不定期サプライズ

Investment:
  - 書いた日記が Narrative Memory に蓄積される
  - データが増えるほど分析が的確になる
  - 物語の章が増えていく（Story ページ）
  - 「離脱するとこの蓄積が無になる」スイッチングコスト
```

**週次Hook:**

```
Trigger: 月曜朝のWeekly Narrative通知
Action: Weekly Narrativeを読む
Variable Reward: 自分の1週間が「物語」として語られる驚き
Investment: 週次データがArc Readerの精度を向上させる
```

**月次Hook:**

```
Trigger: 月末のTheme Finder更新通知
Action: SelfAnalysisまたはStoryページを確認
Variable Reward: 人生テーマの発見・更新
Investment: 長期データがTheme Finderの深さを向上させる
```

**年次Hook:**

```
Trigger: 年末の年間物語生成
Action: 1年間の物語を読む
Variable Reward: 「1年前の自分はこうだったのか」という感動
Investment: 複数年のデータが人生の物語を形成
```

**リズム設計まとめ:**

| 粒度 | 体験 | 感情ピーク |
|---|---|---|
| **毎日** | AIコメント + 日記 + 感情分析 | 「今日も見てくれている」安心感 |
| **毎週** | Weekly Narrative + Arc Reader更新 | 「自分の1週間にこんな意味があったのか」驚き |
| **毎月** | Theme Finder更新 + 月次振り返り | 「自分の人生のテーマが見えた」感動 |
| **四半期** | Chapter生成 | 「物語の章が増えた」達成感 |
| **毎年** | 年間物語 + Growth Chronicle | 「1年間の自分の成長物語」深い感動 |

### 8. エラー & 回復設計（Error & Recovery Design）

#### 汎用原則

LLMは必ず間違える。間違いの種類に応じた回復設計が必要。

**LLMエラーの分類:**

| エラー種別 | 例 | 深刻度 | 回復設計 |
|---|---|---|---|
| **幻覚** | 存在しない日記の内容に言及 | 高 | 根拠データへのリンクで検証可能に |
| **的外れ** | ユーザーの意図と異なる解釈 | 中 | 「違います」フィードバック→即座に再生成 |
| **トーン違反** | 疲れている時に元気な声色 | 低-中 | 時間帯・感情データでトーンを事前制御 |
| **API障害** | LLMが応答しない | 高 | フォールバック（キャッシュされた前回応答 or ルールベース応答） |
| **レイテンシ超過** | 応答が10秒以上 | 中 | ストリーミング + スケルトンUI |

**回復の原則:**
- ユーザーの入力は絶対に失わない（日記の内容がエラーで消えることは許されない）
- エラーが起きても「次にやるべきこと」を示す
- LLMの再生成は低コスト。間違えたらすぐやり直す設計にする

#### focus-youでの適用

**AIコメント生成失敗時:**
```
loading → スケルトン表示（コメント欄の形を予告）
  → 5秒以内に生成 → 通常表示
  → 5秒超過 → 「考え中...」テキスト表示
  → 15秒超過 → フォールバック:
    「（今日のデータを整理中です。後でまた見に来てください）」
    + 前回のAIコメントを薄いテキストで表示
  → API完全障害 → 時間帯に応じた定型メッセージ:
    朝: 「おはようございます。今日もいい一日を」
    夜: 「お疲れさまでした。今日もよく頑張りました」
    ※ 定型であることを隠さない。「AIコメントを準備中」のラベルを添える
```

**日記投稿時の障害:**
```
ユーザーが日記を書いて投稿ボタンを押す
  → ネットワーク障害 → 入力内容をlocalStorageに自動保存
    → 「保存に失敗しました。入力内容は端末に保存されています。
       ネットワーク接続を確認して再送信してください」
    → 再送信ボタンを表示（入力内容はそのまま保持）
```

設計原則: #4 エラー回復の人間化。「エラーが発生しました」ではなく、何が起きて何をすればいいかを具体的に伝える。

### 9. 段階的開示設計（Progressive Disclosure Design）

#### 汎用原則

複雑なシステムの全機能を初日から見せると、ユーザーは圧倒される。情報と機能を段階的に開示する。

**開示のトリガー3種:**

| トリガー | 例 | 適用場面 |
|---|---|---|
| **データ量トリガー** | 日記30件でArc Reader解放 | データ依存の機能 |
| **時間トリガー** | 利用開始2週間後にWeekly Narrative案内 | 時間経過で意味が出る機能 |
| **行動トリガー** | 初めて夢を登録したらGoals機能を案内 | ユーザーの関心を検知して提案 |

#### focus-youでの適用

**ナビゲーション構造の段階的開示:**

```
初日:
  Today（メイン） | Journal | 設定
  ※ 最小構成。Today画面で日記を書き、感情分析を見る体験に集中

1週間後:
  Today | Journal | Habits | 設定
  ※ 「毎日使う習慣」が見えてきたタイミングで習慣トラッキングを提案

2週間後:
  Today | Journal | Habits | Weekly | 設定
  ※ Weekly Narrativeが初めて生成されるタイミングで自然に追加

1ヶ月後:
  Today | Journal | Habits | Weekly | Self | 設定
  ※ データ蓄積により自己分析が意味を持つタイミング

3ヶ月後:
  Today | Journal | Habits | Weekly | Self | Story | 設定
  ※ Theme Finder が動き、物語ページが解放される
```

注意: 現状のfocus-youは既にフル機能が表示されている。これは創業者が唯一のユーザーであり、全機能を自分で開発しているため。一般公開時にはこの段階的開示を実装する必要がある。現段階では「設計として記録」しておく。

### 10. パフォーマンス体験設計（Perceived Performance Design）

#### 汎用原則

実際の速度と体感速度は異なる。体感速度を向上させる設計が重要。

**レイテンシの閾値:**

| 時間 | ユーザーの知覚 | 設計対応 |
|---|---|---|
| < 100ms | 即座（フロー状態を維持） | 操作フィードバック（ボタンの色変化等） |
| 100ms-1s | 遅れを感じるが流れは途切れない | プログレスインジケーター不要 |
| 1-10s | 待っていることを意識する | スケルトンUI、プログレスバー、ストリーミング |
| > 10s | 集中が切れる、離脱リスク | バックグラウンド処理に切り替え、通知で完了を知らせる |

**LLM特有のパフォーマンス設計:**

- **ストリーミング**: LLMの出力をトークン単位で逐次表示。TTFT（Time To First Token）が体感速度を決める
- **楽観的UI更新**: 日記投稿→即座に「投稿済み」表示→裏で感情分析を実行
- **スケルトンUI**: AIコメント欄は「2行のテキストブロック」のスケルトンを表示
- **予測的プリフェッチ**: ユーザーが次に見るページのデータを事前に取得

#### focus-youでの適用

**Today画面のローディング戦略:**

```
Step 1: HTMLシェル即座表示（< 100ms）
  - ナビゲーション、セクションの骨格

Step 2: ローカルデータ表示（< 300ms）
  - Zustandストアのキャッシュから前回データ
  - 時間帯に応じた挨拶（ローカル計算）

Step 3: 軽量APIデータ（< 1s）
  - 天気、カレンダー予定、タスク
  → 各セクションがスケルトンから実データに切り替わる

Step 4: 重量APIデータ（1-3s）
  - AIコメント（バッチ生成済みならキャッシュヒット、即座表示）
  - 感情分析データ

Step 5: バックグラウンド更新（ユーザーが見ている間に）
  - AIコメントの鮮度チェック→staleなら再生成
  - 未読のWeekly Narrativeがあれば通知バッジ
```

**AIチャットのストリーミング体験:**

```
ユーザーがメッセージ送信
  → 即座: 送信メッセージがチャット欄に表示（楽観的UI）
  → 0.5s: AIの応答欄にタイピングインジケーター表示
  → 1-2s: TTFT（最初のトークン到着）→ 文字が流れ始める
  → 完了: フェードインで応答が確定表示に切り替わる

Narrative Memory注入による遅延対策:
  → Narrative Memory はセッション開始時にプリロード
  → チャット開始時にはLayer 2データが既にメモリ上にある
  → LLMへのプロンプト構築にI/O待ちが発生しない
```

### A. Narrator体験の具体化: 4エンジンがユーザーにどう見えるか

Narrative Intelligence の4エンジンはバックエンドの処理系だが、ユーザーにとっては「体験の質」として現れる。ユーザーはエンジンの存在を意識しない。

**ユーザーから見た体験と、裏で動くエンジンの対応:**

| ユーザーが体験すること | 裏で動くエンジン | 表示場所 |
|---|---|---|
| 「先月と今月で気分の流れが全然違う」と気づく | Arc Reader | Story ページの感情の弧 |
| 「自分はこういう人間なんだ」という発見 | Theme Finder | Self Analysis / Story テーマ欄 |
| 日記を書いた直後に「これは大きな出来事ですね」と言われる | Moment Detector | Today画面のダイアログ |
| 「去年の同じ時期にも...」と過去との接続を提示される | Foresight Engine | AIコメント / チャット |

**重要な設計判断: エンジン名をユーザーに見せない。**

「Arc Readerが分析しました」は技術者の自己満足。ユーザーには「あなたの感情の流れ」「あなたの物語のテーマ」として見せる。技術はブランド名ではなく、体験の質で語る。

**各エンジンの出力の表示設計:**

**Arc Reader → 感情の弧ビジュアライゼーション:**

```
Today画面: 直近7日の感情ミニグラフ（SparkLine形式、高さ24px）
  → タップするとStory画面のフル感情弧に遷移

Story画面: 3ヶ月分の感情の弧（メインビジュアル）
  → 横軸: 時間、縦軸: WBI
  → 重要な転機に○マーカー（タップで詳細展開）
  → 「今の章: ○○」のラベル
  → Arc Readerの解釈文を弧の下に表示

Weekly Narrative: 1週間分のミニ弧 + テキスト解釈
```

**Theme Finder → 人生テーマの提示:**

```
Story画面: テーマバッジ（3つまで）
  例: 「つくる人」「意味を問う人」「つなぐ人」
  → 各バッジをタップ → そのテーマの根拠（日記引用、行動パターン）が展開

Self Analysis: テーマと性格特性の交差分析
  → Big5の「Openness 82」が「つくる人」テーマとどう関連するか

AIコメント / チャット: テーマを暗黙的に参照
  → 「あなたは作ることが好きな人だから...」という語り口
  → テーマ名を直接出すのではなく、テーマに基づいた助言
```

**Moment Detector → 転機の確認と記録:**

```
日記投稿後、Moment Detectorが転機を検出した場合:

┌──────────────────────────────────────┐
│                                       │
│  今日の日記に、大きな決断について      │
│  書いていましたね。                   │
│                                       │
│  この瞬間を、あなたの物語に           │
│  記録しておきますか?                  │
│  後から振り返った時、ここが           │
│  ターニングポイントだったと           │
│  わかるかもしれません。               │
│                                       │
│  [記録する]  [今はいい]               │
│                                       │
└──────────────────────────────────────┘

表示タイミング: 日記投稿後、感情分析結果表示の直後
表示方法: インラインカード（モーダルではない。フロー状態を保護）
頻度制御: 1日に最大1回。毎回出すと疲れる
```

**Foresight Engine → 物語の予感:**

```
AIコメント内で自然に表現:
  「去年の同じ時期、あなたは『何か新しいことを始めたい』と
   3回書いた後、2週間後に新しいプロジェクトを始めています。
   今月も同じ兆候が見えます。」

Weekly Narrative 内で表現:
  「今週の感情パターンは、去年の6月を思い出させます。
   あの時、静かな内省期の後に突破がありました。」

独立したUIは不要。既存の表示チャネル（AIコメント、Weekly、チャット）に
自然に溶け込ませる。

設計原則: #5 フロー状態の保護。予感は「気づいたら自然に目に入る」もの。
通知で押し付けない。
```

### B. 時間帯別の完全体験フロー

**朝（5:00-11:00）: 受動的インプット**

```
シナリオ: 起床後、コーヒーを入れながらスマホでアプリを開く。
感情: まだ眠い。能動的に情報を取りに行きたくない。

体験フロー:
  1. Today画面表示
     → 挨拶: 「おはよう。{天気}で{気温}、{曜日}ですね」
     → AIコメント: 朝ブリーフィングベース。バッチ生成済み (7:00)
       例: 「今日は13時から○○MTG。昨日書いた企画書の反応が気になる頃だね」
     → トーン: 穏やか、簡潔、計画提案型

  2. スケジュール概要（折りたたみ展開可能）
     → 今日の予定3件のみ表示。残りは「他○件」

  3. フォーカスタスク
     → 今日期限 + 高優先度のみ。最大3件
     → 「今日の3つ」感覚で認知負荷を最小化

  4. 日記エリア（ページ下部）
     → プロンプト: 「今日やりたいことは?」
     → 朝は書かない人が多い。無理強いしない配置

  5. 習慣チェックボタン（ミニ表示）
     → 朝のルーティン習慣のみ表示
     → ワンタップで完了

感情曲線:
  アプリ起動(安心) → AIコメント(ちょっと嬉しい) → スケジュール確認(把握)
  → フォーカスタスク(よし、やるか) → 閉じる
  
所要時間: 30秒-2分
```

**昼（11:00-17:00）: 能動的アウトプット**

```
シナリオ: MTGの合間、ランチ後にアプリを開く。
感情: 活動中。さっき起きたことを忘れる前に書き留めたい。

体験フロー:
  1. Today画面表示
     → 日記入力エリアが上部に来る（朝と配置が変わる）
     → プロンプト: 「{直前のMTG名}の後、何か思ったことは?」
       （カレンダーから直前の予定を推定して文脈提示）
     → トーン: 効率重視。「書きやすさ」が最優先

  2. 残りスケジュール
     → 午後の予定のみ表示

  3. タスク
     → 「午前中に完了したもの」にチェックが入っている状態
     → 残りタスクの確認

  4. AIコメント（日記投稿後に更新）
     → 投稿した内容に即座に反応
     → トーン: 軽い承認。「午前中、よく動いたね」

感情曲線:
  アプリ起動(書きたい衝動) → 日記入力(スッキリ) → 即座にAI反応(嬉しい)
  → 残りタスク確認(よし) → 閉じる

所要時間: 1-5分
```

**夜（17:00-翌5:00）: 内省と安息**

```
シナリオ: 仕事を終え、夜に一日を振り返る。
感情: 疲れている。でも「今日を閉じたい」気持ちがある。

体験フロー:
  1. Today画面表示
     → 一日のサマリーが上部（達成ファースト）
       「今日の完了: タスク5件、習慣3/4」
     → 未完了は「明日に持ち越す?」ボタンで穏やかに提示

  2. 日記入力エリア
     → プロンプト: 「今日一番印象に残ったことは?」（日替わり）
     → 分析質問: 日替わりで2問表示（relationships / positive / actions / decisions / inner からローテーション）
     → トーン: 内省を促すが、強制しない

  3. AIコメント（日記投稿後）
     → トーン: 深い共感。「今日はこんな一日だったんだね」
     → ポジティブな要素を見つけて触れる（ピーク・エンドの法則）
     → 疲れている兆候があれば: 「今日は早めに休んでね」

  4. 明日の予告（軽く）
     → 「明日は○○MTGがあります」程度
     → 詳細は朝に。夜に明日の重荷を見せない

  5. Moment Detector（転機があれば）
     → 日記投稿後にカード表示
     → 「今日の日記に大きな出来事がありましたね。記録しますか?」

感情曲線:
  アプリ起動(振り返りたい) → 完了サマリー(達成感) → 日記入力(言語化のスッキリ感)
  → AIコメント(共感してもらえた) → 明日の予告(安心) → 閉じる(穏やかな終わり)

ピーク: AIコメントの共感
エンド: 穏やかな終わり
→ ピーク・エンドの法則により、一日の体験評価が高くなる

所要時間: 3-10分
```

19:00        [バッチ] 夕方ニュース収集
00:00-04:59  [バッチ] 翌日のための前処理

```
日曜深夜:   [バッチ] Arc Reader 実行（感情の弧を更新）

            → 週末の過ごし方に触れる（Foresight的な一言）

            → 書いてくれたら「週末に書いてくれるのは、大事なことがあった証拠」

**月次リズム:**

```
月末深夜:   [バッチ] Theme Finder 実行（人生テーマの更新）

            → 感情推移、習慣達成率、注目すべき出来事

### 各Tierの判断根拠

**なぜTier 1が「フィードバックループ強化」なのか:**
日記を書く → AIが反応する → もっと書きたくなる。このループがfocus-youの生命線。ループが途切れる瞬間（AI応答失敗、遅延、的外れ）は致命的。既に動いている核心的ループの品質を磨くことが、新機能追加より価値が高い。

**なぜMoment Detectorが Tier 2 なのか:**
転機データは物語の原材料。Arc Reader（Tier 3）やChapter生成（Tier 4）の品質は、Moment Detectorが蓄積した転機データの量に依存する。早く始めるほど将来のエンジン精度が上がる。「データ蓄積系の機能は、使い始めが早いほど複利で効く」。

**なぜTheme FinderがTier 3の最後なのか:**
少ないデータで動かすと的外れになる。的外れな人生テーマは信頼を壊す。3ヶ月分のデータが揃うまで待つ方が、初回の感動が大きい。「我慢して質の高い初体験を提供する」設計判断。

#### 速度カテゴリと設計パターン

```
即座 (< 300ms): UI操作、ローカル計算、キャッシュヒット
  → 楽観的UI更新。サーバー応答を待たずに画面を変える
  → 例: 習慣チェック、タスク完了、日記投稿ボタン

快適 (300ms-3s): 軽量LLM呼び出し
  → スケルトンUI or スピナー。ユーザーは「処理中」を意識するが待てる
  → 例: 感情分析、AIコメント、Moment Detector

待てる (3-15s): 重量LLM呼び出し
  → プログレスバー + 段階的表示。「何をしているか」を伝える
  → 例: 自己分析、週次ナラティブ（手動）

バックグラウンド (15s-数分): バッチ処理
  → ユーザーは画面にいない。完了時に通知 or 次回表示時にキャッシュ
  → 例: Arc Reader、Theme Finder、朝ブリーフィング

夜間処理 (数分-数十分): 重い分析バッチ
  → Claude CLI。翌朝の画面に反映されていればよい
  → 例: Chapter生成、年間物語
```

#### ストリーミングの適用判断

| 機能 | ストリーミング | 理由 |
|---|---|---|
| チャット応答 | ✅ 必須 | 会話の自然さ。待ち時間が長い |
| AIコメント | ❌ 不要 → タイプライター演出 | 200トークンで十分に速い。演出で「考えた」感を出す |
| 感情分析 | ❌ 不要 | JSON応答。段階的表示は不可能 |
| 自己分析 | 🔶 検討 | 4500トークン。分析タイプごとに段階的に表示する方が体験が良い |
| Weekly Narrative | ❌ 不要 → 段落フェードイン | 600トークン。生成後に演出で段階表示 |

### H. 動線設計 — 欲しいときに欲しいものが手に届く

#### 現状の問題点

```
問題1: ナビゲーション過多
  21ルート × 13セクションのサイドバー。
  「Understand」に5ページ、「Data」に6ページ。
  初見ユーザーはどこに何があるかわからない。

問題2: Today画面からの動線が弱い
  Today → Tasks / Habits / Calendar / News / Dreams への遷移はあるが、
  Journal / Self Analysis / Weekly へは直接行けない。
  「今日の感情の詳細を見たい」→ サイドバーからJournalを探す必要がある。

問題3: 時間帯による動線の変化がない
  朝はスケジュール → カレンダーへの動線が欲しい。
  夜はJournal / Weekly への動線が欲しい。
  現状: 全時間帯で同じサイドバー構造。

問題4: データ蓄積に応じた動線誘導がない
  日記30件でArc Readerが使えるようになっても、ユーザーは気づかない。
  「新しい体験が解放されたこと」を伝える仕組みがない。
```

#### 動線設計の3原則

```
原則1: 「Today画面が宇宙の中心」
  全ての体験はToday画面から始まり、Today画面に帰る。
  他ページへの遷移は「Today画面の情報を深掘りする」行為。
  Today画面に十分な入口（リンク/ボタン）を配置する。

原則2: 「時間帯が動線を変える」
  朝: Today → Calendar（予定確認）→ Today
  昼: Today → 日記入力 → 感情分析結果を見る → Today
  夜: Today → Journal（週の感情推移）→ Weekly → Today

原則3: 「データが動線を作る」
  日記7件目 → 「感情推移が見えます」→ Journal へのリンクが出現
  日記30件目 → 「物語の弧が見えてきました」→ Story へのリンクが出現
  転機記録3件目 → 「物語の章立てができます」→ Story へのリンクが出現
```

#### Today画面からの遷移マップ（改善案）

```
Today画面の各セクションから、関連ページへの動線を配置:

┌─────────────────────────────────────────────────────────────┐
│  Today                                                       │
│                                                              │
│  [AIコメント]                                                │
│    └→ 感情バッジをタップ → Journal（その感情の詳細）         │
│    └→ 「先月の話」リンク → Story or Journal（月次ビュー）    │
│                                                              │
│  [スケジュール]                                              │
│    └→ イベントをタップ → Calendar（その日の詳細）            │
│    └→ 「明日の予定」→ Calendar（明日ビュー）                 │
│                                                              │
│  [タスク]                                                    │
│    └→ タスク一覧 → Tasks（フル表示）                         │
│    └→ ゴール紐付きタスク → Goals or Dreams                   │
│                                                              │
│  [習慣]                                                      │
│    └→ 全習慣 → Habits（詳細 + カレンダーグリッド）           │
│    └→ ストリーク → Habits（統計ビュー）                      │
│                                                              │
│  [日記エリア]                                                │
│    └→ 投稿後の感情バッジ → Journal                           │
│    └→ Moment Detector確認カード → Story（転機一覧）          │
│                                                              │
│  [時間帯コンテキストリンク]  ← NEW                           │
│    朝: 「今週の物語」→ Weekly                                │
│    夜: 「今月の感情推移」→ Journal                           │
│    夜: 「自分を知る」→ Self Analysis（月1回リマインド）       │
│                                                              │
│  [データ蓄積マイルストーン]  ← NEW                           │
│    「30件目の日記! 感情の弧が見えます」→ Story               │
│    「テーマが見つかりました」→ Self Analysis テーマ欄         │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### サイドバーの再構成案（段階的開示 + 時間帯対応）

```
現状（21ルート、13セクション）→ 改善案:

常時表示:
  Home (Today)
  AI Chat
  Journal
  Tasks

Discover（データ蓄積で出現）:
  Weekly        ← 日記14件後
  Self Analysis ← 日記20件後
  Story         ← 日記30件後（新設）

Plan:
  Dreams & Goals
  Habits
  Calendar

More（折りたたみ）:
  Finance
  News
  Growth
  Organization
  Knowledge
  Artifacts
  Prompts
  Commands
  Blueprint
  Settings

サイドバーの項目数: 4（常時）+ 0-3（Discover）+ 3（Plan）+ 折りたたみ
= 初日は7項目、3ヶ月後は10項目。現状の13セクション・21項目から大幅削減。
```

#### 遷移の「戻りやすさ」設計

```
原則: 「どこにいても1タップでTodayに帰れる」

実装:
  1. サイドバーの Home アイコン（常時表示）
  2. ロゴクリックで Home（モバイル対応）
  3. Cmd+1 キーボードショートカット（既存）
  4. ブラウザバックで前のページ（React Router の履歴管理）

「迷子」防止:
  - パンくずリストは不要（階層が浅い）
  - ページヘッダーに「← Today」ボタンを配置
  - モバイル: スワイプで戻る
```
