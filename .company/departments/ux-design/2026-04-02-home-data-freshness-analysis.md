# Home画面 データ鮮度とUX分析レポート

## 1. シナリオ分析

### プライマリシナリオ: 朝の通勤中

```
誰:    社長（複数PJを並行管理するフリーランス / AI開発者）
いつ:  平日 7:30-8:30、電車内
どこ:  モバイルブラウザ（片手操作、電波が不安定な可能性）
何を:  今日の全体像を30秒で把握したい
      - 「急ぎの案件はあるか」
      - 「お金周りに問題はないか」
      - 「昨日の自分はどうだったか（日記/WBI）」
感情:  まだ頭が完全には起きていない。
      「情報を処理したい」のではなく「安心したい」。
      問題がなければ「OK、今日も大丈夫」と思いたい。
      問題があれば「何が問題で、何をすればいいか」だけ知りたい。
```

### セカンダリシナリオ: 夕方の振り返り

```
誰:    同上
いつ:  平日 18:00-19:00、帰宅時
何を:  今日の進捗を確認し、明日の準備を考えたい
感情:  疲れている。達成感がほしい。「今日もちゃんとやった」と思いたい。
```

### テルシャリシナリオ: 週末のざっくり確認

```
誰:    同上
いつ:  土日の隙間時間
何を:  「週明け何かあったっけ」程度の軽い確認
感情:  仕事モードではない。深く読みたくない。
```

### 現状の朝の体験フロー

```

  |
  v

  |                                   フリクション #1: ローディング状態の不透明さ
  v
[画面表示完了]
  |
  +---> [Finance カード] ........... 安心 or 不安（数字を見る）
  |       BUT: この数字はいつ時点？ 昨日の請求書は反映されてる？
  |       フリクション #2: データの鮮度が不明

  |       「5件open」は本当に今の状態？ 昨夜closeしたのに反映されてない？
  |       フリクション #3: リアルタイム性への疑念
  |
  +---> [Insights カード] .......... 混乱
  |       「これいつの分析？」「1週間前の分析が最新？」

  |       最近書いてなければ空状態 → 罪悪感
```

### フリクションポイント一覧

| # | ポイント | 感情 | 深刻度 |
|---|---------|------|--------|
| F1 | ローディング中に何が起きているかわからない | 不安 | 中 |
| F2 | 各データがいつ時点のものかわからない | 不信 | **高** |
| F3 | リアルタイムなデータと古いデータが同じ見た目で並ぶ | 混乱 | **高** |
| F4 | CEO InsightsやKnowledgeの最終更新がClaude Code実行依存 | 疑念 | 高 |
| F5 | 朝の限られた認知リソースに対して情報が多い | 疲弊 | 中 |
| F6 | 今日の予定（カレンダー）がHomeにない | 欠損感 | 高 |
| F7 | 画面を更新する手段がブラウザリロードのみ | 無力感 | 中 |

## 3. 各データソースの鮮度分析

### 質問1への回答: 朝開いた時、各データはいつ時点か？

| データ | 取得タイミング | 鮮度 | リスク |
|--------|--------------|------|--------|
| **Tasks** | ページロード時にSupabaseへSELECT | **リアルタイム**（ロード時点） | 低。ただしロード後に変更があっても反映されない |
| **Finance (invoices/expenses)** | 同上 | **リアルタイム**（ロード時点） | 低。ただしデータ自体は手動登録なので、登録忘れがあれば古い |
| **CEO Insights** | 同上 | **潜在的に古い** | **高**。`/company` を最後に実行した時点の分析。数日前の可能性 |
| **Diary** | 同上 | **最後に書いた日** | 中。最近書いてなければ1週間前のデータ |
| **Comments** | 同上 | **リアルタイム**（ロード時点） | 低 |
| **Knowledge** | 同上 | **潜在的に古い** | **高**。Claude Code使用時に自動蓄積のため、使用頻度依存 |

**重要な発見**: すべてのデータはページロード時にSupabaseへクエリするため、「表示された瞬間のDB状態」は正確。しかし問題は2つ:

1. **ソースデータの鮮度**: Insights/Knowledgeは「DBの最新」≠「現実の最新」。分析自体が古い可能性がある
2. **表示後の変化**: ページロード後にDBが更新されても画面に反映されない（リアルタイム購読なし）

### 質問2への回答: 「いつの情報か」わからないことによる混乱

**はい、混乱が発生する。** 具体的に:

- CEO Insightsの `created_at` は表示されているが、「3月25日」の分析が「4月2日」に最新として表示される状況は、ユーザーに「これは古い情報では？」という疑念を生む
- Financeの年間売上が「昨日の請求書を含んでいるか」がわからない
- Tasksが「今この瞬間のopen数」なのか「昨夜時点」なのかが不明
- **最大の問題**: 全カードが同じ視覚的重みで並んでおり、「リアルタイムのTasks」と「1週間前のInsights」の鮮度差が見た目でわからない

## 4. 設計判断ログ

### 設計判断 D1: データ鮮度インジケーターの導入

**適用原則**: #3 フィードバックループの設計（システムステータスの可視性）

**判断**: 各カードに「最終更新」の時間的距離を表示する。ただし実装は2層に分ける:

- **表示データの取得時刻**（「2分前に取得」）→ すべてのカードに統一表示
- **ソースデータの鮮度**（「分析は3日前」）→ Insights, Knowledgeのみ追加表示

**理由**: Smashing Magazineの2025年記事が推奨する「Data Freshness Indicator」パターンに準拠。ビジネスユーザー向けには「Live / Stale / Paused」のシンプルなステータスが最適。

**見送った案**:
- 各カードに絶対時刻（「07:42取得」）を表示 → 朝の認知負荷が増える。相対時刻（「2分前」）の方が直感的
- リアルタイム購読で自動更新 → 後述のD4で検討

### 設計判断 D2: 情報の優先順位再設計（朝の文脈最適化）

**適用原則**: #1 意図の先読み + #2 認知負荷の最小化

**判断**: 朝の通勤中の社長が最初に見たい情報の優先順位:

```
1. 今日の予定（カレンダー）     ← 現在Homeに存在しない [最重要]
2. 急ぎのタスク（high priority） ← 現在2番目に表示
3. 財務サマリー                 ← 現在1番目に表示
4. 最新コメント                 ← 現在5番目に表示
5. CEO Insights / 日記          ← 現在3-4番目に表示
6. Knowledge                    ← 現在6番目に表示
```

**理由**:
- 朝の最初の関心は「今日何があるか」。MTGの有無で一日の動き方が変わる
- 次に「火事になっていないか」（urgent tasks）
- 財務は「安心確認」であり、毎朝深く見るものではない
- Insights/Diary/Knowledgeは「時間があれば見る」もの

**見送った案**:
- 時間帯による動的並び替え（朝/昼/夜でカード順序変更）→ 一貫性が損なわれる（ヤコブの法則違反）。代わりに固定順序を最適化する
- すべてを1画面に収める → モバイルでは不可能。「Above the fold」に最重要3項目

### 設計判断 D3: Google Calendar のHome統合

**適用原則**: #1 意図の先読み + #5 フロー状態の保護

**判断**: Homeに「今日の予定」カードを追加する。実装済みのGoogle Calendar API連携を活用。

**仕様**:
- Home画面の最上部（挨拶の直下）に配置
- 今日のイベントのみ表示（最大5件）
- 時刻順にソート
- 「次の予定」を視覚的にハイライト（現在時刻に最も近い未来のイベント）
- タップでCalendarページに遷移
- Google未連携時: 「カレンダーを連携する」ボタン（Calendar画面へ誘導）

**理由**:
- 既にCalendarページでGoogle Calendar APIと連携済み（`syncCalendarEvents`関数が存在）
- 朝の通勤中に「今日MTGある？何時？」は最も即時性が高い質問
- Calendarページに遷移せずHomeで完結させることで、コンテキストスイッチを減らす

**見送った案**:
- 全週間の予定を表示 → 認知負荷過多。今日だけで十分
- Supabase `calendar_events` テーブルのみ参照 → 同期タイミング依存で鮮度問題が発生。直接API呼び出しが望ましいが、パフォーマンスとのトレードオフ

### 設計判断 D4: 更新戦略（ポーリング vs 手動リフレッシュ vs リアルタイム）

**適用原則**: #5 フロー状態の保護 + #3 フィードバックループ

**判断**: **手動プルリフレッシュ + 画面遷移時の自動再取得** を採用。ポーリングやリアルタイム購読は見送る。

**理由**:
1. **ユースケースの性質**: 朝の通勤中に1回開いて確認するパターン。画面を長時間開きっぱなしにしない
2. **データ変更頻度**: Tasks/Comments以外は低頻度更新。リアルタイム購読のコスト対効果が悪い
3. **モバイルバッテリー**: WebSocket常時接続はバッテリー消費を増やす
4. **認知負荷**: 画面上のデータが自動で変わると、朝の低覚醒状態では混乱を招く

**具体仕様**:
- **プルリフレッシュ**: モバイルで画面上部からプルダウンでデータ再取得（ネイティブアプリライクな操作感）
- **画面遷移時再取得**: 他のタブからHomeに戻ったとき、前回取得から3分以上経過していれば自動再取得
- **ラストフェッチ表示**: 画面下部に「最終取得: 2分前」を控えめに表示
- **手動リフレッシュボタン**: 挨拶エリア右端に小さな更新アイコン

**見送った案**:
- Supabaseリアルタイム購読 → 1人利用のダッシュボードで常時WebSocket接続は過剰
- 30秒ポーリング → バッテリーと通信量の無駄。変更検知する価値がある頻度ではない
- Service Worker + Background Sync → PWA化が必要で、現時点ではスコープ外

### 設計判断 D5: Staleデータの視覚的区別

**適用原則**: #3 フィードバック + #2 認知負荷の最小化

**判断**: 鮮度に応じてカードの見た目を3段階で変える:

| 鮮度 | 条件 | 視覚表現 |
|------|------|---------|
| Fresh | ソースデータが24時間以内 | 通常表示 |
| Aging | ソースデータが1-3日前 | カードヘッダーにオレンジのドット + 「3日前の分析」テキスト |
| Stale | ソースデータが3日以上前 | カードヘッダーに赤のドット + 「7日前」テキスト + カード全体の opacity: 0.85 |

**適用対象**: CEO Insights と Knowledge のみ。Tasks/Finance/Comments/Diaryはロード時点で常にFreshなため不要。

**理由**: 「リアルタイムのデータ」と「数日前の分析」が同じ見た目で並ぶことが最大の混乱要因。視覚的に差を付けることで、ユーザーは無意識に「これは古い情報だ」と認識できる（認知的流暢性）。

## 5. 質問への回答まとめ

### Q1: 各データはいつ時点の情報か？

上記「3. 各データソースの鮮度分析」参照。要約すると:
- Tasks, Finance, Comments → ページロード時点でリアルタイム
- CEO Insights, Knowledge → `/company` 最終実行時点（数日前の可能性あり）
- Diary → 最終記録日（本人の記録頻度依存）

### Q2: 「いつの情報か」わからない混乱

**発生する。** 特にCEO InsightsとKnowledgeで深刻。対策はD1（鮮度インジケーター）とD5（視覚的区別）。

### Q3: データ鮮度を伝えるUXパターン

D1で提案。3層構造:
1. **ページレベル**: 画面下部に「最終取得: X分前」
2. **カードレベル**: 鮮度ドット（Fresh/Aging/Stale）
3. **ソースレベル**: 「分析日: 3月28日」の明示

### Q4: Google Calendarの統合

**すべき。** D3で提案。朝の通勤中に最も知りたい情報は「今日の予定」であり、現在Homeに欠落している最大のギャップ。

### Q5: 自動更新かリフレッシュか

**手動プルリフレッシュ + 条件付き自動再取得** を推奨。D4で詳述。リアルタイム購読は過剰。

### Q6: 朝の社長が最初に見たい情報の優先順位

D2で提案。再掲:

```
1. 今日の予定（カレンダー）  ← 「今日何がある？」
2. 急ぎのタスク              ← 「火事はないか？」
3. 財務サマリー              ← 「お金は大丈夫？」
4. 最新コメント              ← 「誰かから連絡ある？」
5. Insights / 日記           ← 「時間あれば確認」
6. Knowledge                 ← 「参考情報」
```

## 6. インタラクション仕様

### 6.1 Home画面カード配置（モバイル / 1カラム）

```
[挨拶 + 日付]                    [更新ボタン]
[最終取得: X分前]

--- 今日の予定 ---                 ← NEW
| 09:00  クライアントMTG         |
| 14:00  社内レビュー            |
| → 次の予定: 09:00 クライアントMTG (あと32分) |
+--------------------------------+

--- Tasks --- N件 open             
| [!] High priority: 2件         |
| * タスクA (foundry)            |
| * タスクB (rikyu)              |
+--------------------------------+

--- Finance --- 2026/04            
| 年間売上  年間経費  粗利        |
| ¥X,XXX   ¥X,XXX   ¥X,XXX      |
| 税金積立目安: ¥XX,XXX/月       |
+--------------------------------+

--- Comments --- 最新              
| [foundry] 4/1 コメント内容...  |
| [rikyu]   3/31 コメント内容... |
+--------------------------------+

--- Insights --- [●] 3日前の分析   ← 鮮度表示
| [pattern] 朝型の作業...        |
| [strength] コード品質...       |
+--------------------------------+

--- Diary --- 最近の記録           
| 4/1 WBI 7 今日は...           |
+--------------------------------+

--- Knowledge --- [●] Active       ← 鮮度表示
| [coding] ルール内容...         |
+--------------------------------+
```

### 6.2 状態遷移

| 状態 | 表示 | トリガー |
|------|------|---------|
| **loading** | Skeleton UI（カード形状を模した灰色ブロック。既存の `skeletonRows()` を活用） | ページ遷移時 |
| **loaded** | 全カード表示 + 最終取得時刻 | Promise.all完了 |
| **partial** | 取得済みカードから順次表示（Google Calendar APIが遅い場合、他カードは先に表示） | 個別Promise解決時 |
| **error** | カード単位で `inlineError()` 表示（既存コンポーネント活用）。他カードは正常表示 | 個別API失敗時 |
| **stale** | カードヘッダーに鮮度ドット + テキスト | ソースデータの `created_at` が閾値超過 |
| **empty** | カード単位で `emptyState()` 表示 + 次のアクション誘導 | data.length === 0 |
| **offline** | 画面上部にオフラインバナー「オフラインです。最終取得: X分前のデータを表示中」 | navigator.onLine === false |
| **refreshing** | 更新ボタンが回転アニメーション + 各カードが軽くフェードして再描画 | プルリフレッシュ or 更新ボタン押下 |

### 6.3 プルリフレッシュ仕様

```
操作: 画面上端で下方向にスワイプ（60px以上）
表示: インジケーター（矢印 → 回転アイコン）がページ上部に出現
実行: すべてのデータを再取得（Promise.all）
完了: インジケーター消失 + 最終取得時刻更新 + 変更があったカードに flashRow() 適用
失敗: トースト「更新に失敗しました。後でもう一度お試しください」
```

### 6.4 鮮度ドットの仕様

```css
/* Fresh (24h以内) - 表示なし */

/* Aging (1-3日) */
.freshness-aging::before {
  content: '';
  width: 6px; height: 6px;
  border-radius: 50%;
  background: var(--amber);
  display: inline-block;
  margin-right: 6px;
}

/* Stale (3日超) */
.freshness-stale::before {
  content: '';
  width: 6px; height: 6px;
  border-radius: 50%;
  background: var(--red);
  display: inline-block;
  margin-right: 6px;
}
.freshness-stale { opacity: 0.85; }
```

## 8. 次ステップへの申し送り

### sys-dev部への依頼事項

1. **Google Calendar API呼び出しのHome統合**: 既存の `syncCalendarEvents()` / `loadCalendarFromDB()` をHome用に軽量化したバージョンが必要。Homeでは今日のイベントのみ取得
2. **プルリフレッシュの実装**: touch eventベースのプルリフレッシュ機構。CSS transformでの引き下げアニメーション含む
3. **部分ローディング**: `Promise.all` ではなく `Promise.allSettled` への変更を推奨。1つのAPIが失敗しても他のカードは表示する

### ai-dev部への依頼事項

1. **CEO Insights の鮮度メタデータ**: `ceo_insights` テーブルに「分析対象期間」のカラム追加を検討。`created_at` だけでは「何日分のデータに基づく分析か」がわからない
2. **自動鮮度チェック**: `/company` 実行時に「前回の分析からN日経過」の警告を出す仕組み

### pm部への依頼事項

1. 上記P0-P3の施策をタスク化
2. P1のGoogle Calendar統合は社長へのヒアリングを経てから着手（本当に朝一番に見たい情報か確認）
