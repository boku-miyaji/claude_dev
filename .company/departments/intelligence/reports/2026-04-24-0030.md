## 📄 注目論文（新着・arXiv 2604 series）

### 1. [ByteRover: Agent-Native Memory Through LLM-Curated Hierarchical Context](https://arxiv.org/abs/2604.01599)

**(2604.01599, arXiv:2604.01599, 提出: 2026-04-02)**

**背景・目的**: 従来のエージェント記憶システムは「LLM reasoning」と「external knowledge management」を分離してきた。しかし semantic drift（意味のズレ）と infrastructure complexity（ベクトルDB, graph DB の依存）が問題。ByteRover は「同じ LLM が reasoning と curation の両方を担当」する逆転設計を提案。

**手法**: 
- **Hierarchical Context Tree**: Domain → Topic → Subtopic → Entry の4階層
- **Adaptive Knowledge Lifecycle (AKL)**: importance scoring, maturity tiers, recency decay
- **5-tier progressive retrieval**: ローカル解決を優先、必要時のみ LLM escalation
- **Human-readable markdown**: Vector DB / graph DB 不要。ローカル markdown ファイル構成

**主要結果**:
- LoCoMo, LongMemEval ベンチマークで SOTA
- **Zero external infrastructure** で実現（推論効率と正確性を両立）

**focus-you への直接応用可能性**: 
- Diary entry を単純なリスト構造から「event stream + topic network」に昇格
- Memory Intelligence Agent が「階層的な intent-aware retrieval」を実装可能
- Markdown-based memory は既存の `.company/growth/` ファイル構造と親和性が高い

### 2. [A Survey on the Security of Long-Term Memory in LLM Agents: Toward Mnemonic Sovereignty](https://arxiv.org/abs/2604.16548)

**(2604.16548, arXiv:2604.16548, 掲載: 2026-04月)**

**背景・目的**: エージェントの記憶が「永続・可読・社会的に伝播」する特性を持つにもかかわらず、セキュリティ体系が整備されていない。2023〜2026 のメモリ関連文献を横断的に分析し、ガバナンスフレームワークを提案。

**主要概念**: **Mnemonic Sovereignty**
- 「記憶に対する verifiable で recoverable な governance」
- **6つのライフサイクル段階**:
  1. Write — 何が記録されるか
  2. Store — どこに保持されるか
  3. Retrieve — 誰が読める権限を持つか
  4. Execute — 記憶に基づく行動が許可されるか
  5. Share — エージェント間の記憶伝播ルール
  6. Forget/Rollback — 削除・無効化の方針

**セキュリティ目標**: Integrity, Confidentiality, Availability, Governance

**分析結果**:
- **既存文献の偏り**: Write/Retrieve 時の integrity 攻撃に集中。Confidentiality・Availability の議論が不足
- **ガバナンスギャップ**: 9個の governance primitive を定義したが、既存アーキテクチャが全て対応していない
- **LLM を記憶セキュリティに活用する研究** がほぼ無い（未開拓領域）

**HD運営への示唆**:
- Diary / Memory Intelligence Agent の「誰がいつ何を編集したか」の透明性が research frontier の重要課題
- focus-you の「自分の日記は自分だけ」という設計が、学術的には「mnemonic sovereignty」として位置づけられる

### 3. [Memory for Autonomous LLM Agents: Mechanisms, Evaluation, and Emerging Frontiers](https://arxiv.org/abs/2603.07670)

**(2603.07670, arXiv:2603.07670, 提出: 2026-03月，published as ICLR 2026 MemAgents Workshop)**

**背景・目的**: 2022〜2026 の記憶拡張技術をサーベイ。5つのメカニズムファミリーを分類：

1. **Context-resident compression** — prompt 内での情報圧縮
2. **Retrieval-augmented stores** — external knowledge base + semantic search
3. **Reflective self-improvement** — agent が自身の行動から学習
4. **Hierarchical virtual context** — event tree や topic network（ByteRover の理論基盤）
5. **Policy-learned management** — RL で記憶の acquire/discard を最適化

**結論**: 「Stateless text generator」→「Genuinely adaptive agent」への転換を記憶が可能にする。focus-you の記憶体系はこの5つのうち「hierarchical virtual context + reflective self-improvement」の組み合わせが最適。

## 🔧 セキュリティ・開発実務アップデート

### Version Control & Optimistic Locking（社長プロンプト関連）

**最新実務トレンド**:
- **LLM-powered conflict resolution**: コード intent を理解し、テキスト行単位ではなく semantic レベルで merge conflict を解決
- **Optimistic concurrency control（OCC）**: Version tag に基づく conflict detection（retry + exponential backoff で解決）
- **Rerere（reuse recorded resolution）**: 過去の conflict 解決パターンを再利用

**実装例**:
- git-based version control（focus-you の Diary / Framework update での採用可能）
- Database optimistic locking（row version + CAS pattern）

**focus-you への応用**: 
- Diary の「誰がいつ何を編集したか」を version tag で追跡
- Framework / Pattern / Arc の concurrent update を optimistic locking で安全化

---

### Batch Processing Scheduling（社長プロンプト関連）

**最新実務知見**:
- **Time-based scheduling**: Cron pattern による「朝 6:00 AM のニュース更新」が standard
- **Event-based scheduling**: Diary entry 10件追加 → frameworks auto-recompute（社長リクエスト通りの実装パターン確認）
- **Dependency-based scheduling**: upstream task 完了 → downstream task trigger
- **Morning digest の optimal timing**: 
  - 早すぎる（4:00 AM）→ 最新情報が含まれない
  - 遅すぎる（9:30 AM）→ ユーザー出勤時に未更新
  - **Optimal: 6:00-7:00 AM JST**（朝からの作業に fresh data を提供）

**focus-you への直接活用**: 
- 「morning quote」「Arc/Theme/Chapter auto-update」のバッチを **朝 7:00 AM JST に固定**
- Diary に10件エントリ追加 → framework recompute（社長指示通り）
- Skip guard 1週間（同じ trigger で連続実行防止）
