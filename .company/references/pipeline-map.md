# 部署連携パイプラインマップ

> 秘書がタスクを受けた時、このマップに従って部署を起動・連携する。

## パイプライン一覧

### A: 新機能開発
```
社長「〇〇を作って」
  → リサーチ (技術調査・競合分析)
  → UXデザイン (シナリオ分析・体験設計)
  → AI開発 (設計書作成) ← checkpoint: 社長レビュー
  → システム開発 (実装) + PM (チケット化を並行)
  → QA (テスト)
  → 社長に完了報告
```

### B: バグ修正・小さな改善
```
社長「〇〇を直して」
  → システム開発 (調査+修正)
  → QA (テスト確認)
  → 社長に完了報告
```

### C: 資料作成
```
社長「資料を作って」「プレゼンにまとめて」
  → リサーチ (必要なデータ収集)
  → 資料制作 (Markdown/PPTX作成) ← checkpoint: 社長レビュー
  → 社長に完成報告
```

### D: 調査・分析
```
社長「〇〇を調べて」
  → リサーチ (3段構成で調査)
  → 秘書が結果をnotes/に記録
  → 社長に報告
```

### E: セキュリティ対応
```
セキュリティアラート or 社長指示
  → セキュリティ (監査・スキャン)
  → システム開発 (修正実装)
  → QA (修正確認)
  → 社長に報告
```

## ハンドオフプロトコル

### 成果物の末尾に書く形式

各部署は成果物の末尾に以下のセクションを書く。秘書がこれを検出して次の部署を起動する:

```markdown
## ハンドオフ

### → PM部への依頼
- [ ] 上記改善案をタスクチケットに分割
- [ ] P0施策の期限設定

### → AI開発部への依頼
- [ ] 〇〇のAPI仕様策定

### → 資料制作部への依頼
- [ ] クライアント向け説明資料の作成
```

### 秘書の検出ロジック

成果物ファイルに `→ {部署名}部への依頼` が含まれていたら:
1. 依頼内容を抽出
2. 該当部署の Agent を起動（Agent tool 経由）
3. 起動ログを activity_log に記録
4. 完了を待ち、次のハンドオフがあればチェーン

## 部署間の依存関係

```
        ┌─────────────┐
        │   情報収集    │ ← 定期実行(cron)
        └──────┬──────┘
               ↓ (ニュース・トレンド)
        ┌──────────────┐
        │   リサーチ     │ ← 調査依頼
        └──────┬───────┘
               ↓ (調査レポート)
    ┌──────────┼──────────┐
    ↓          ↓          ↓
┌────────┐ ┌────────┐ ┌────────┐
│UXデザイン│ │AI開発   │ │資料制作 │
└───┬────┘ └───┬────┘ └────────┘
    │          │
    ↓          ↓
┌────────────────────┐
│     PM（チケット化）  │
└────────┬───────────┘
         ↓
┌────────────────────┐
│  システム開発（実装）  │
└────────┬───────────┘
         ↓
┌────────────────────┐
│     QA（テスト）     │
└────────────────────┘
         ↓
      完了報告
```

## セキュリティ部（横断）

セキュリティ部は全パイプラインに横断的に関与:
- リリース前のセキュリティレビュー
- 依存関係の脆弱性スキャン
- 定期監査（月次）
