## 1. 記事の核心メッセージ

everything-claude-code（Anthropicハッカソン優勝者が公開した設定集、90万view超）の本質は **「コンテキストを最適化する」** という考え方。AIに渡す情報を整理し、必要なときに必要なものだけを提供する仕組みを作ること。

---

## 2. 全体像: 6種類のファイル構成

everything-claude-codeは6種類のファイルで構成される。記事では **会社組織へのアナロジー** で説明している。

| ファイル種別 | 会社組織でのたとえ |
|---|---|
| **agents** | 専門部署（設計部、品質管理部、セキュリティ部） |
| **skills** | 業務マニュアル・手順書 |
| **commands** | よく使う操作のショートカット |
| **rules** | 社訓・ポリシー・行動指針 |
| **hooks** | 自動チェック機構・警報装置 |
| **mcp-configs** | 外部取引先との連携設定 |

> 全員がなんでもやるのではなく、**役割分担して効率的に動く仕組み**を作っている。

---

## 3. 5つのポイント

### ポイント1: 専門家チームのように使う（agents）

- 「なんでも屋」より「専門家チーム」を選ぶ
- 具体的なエージェント例:
  - **planner** - 実装計画を立てる設計士
  - **code-reviewer** - コード品質をチェックする品質管理担当
  - **security-reviewer** - 脆弱性を検出するセキュリティ専門家
  - **tdd-guide** - テスト駆動開発を指導するコーチ
- **最重要原則**: 各エージェントに「最小限のツールだけを与える」。50個のツールを持つエージェントより、5個に絞ったエージェントの方が集中して動作する
- 最初から全部作る必要はなく、繰り返し発生するタスクを見つけて徐々に追加する

### ポイント2: マニュアルを整備する（rules / skills）

- rulesとskillsは**暗黙知を明文化する**ためのファイル
- 具体的なルール例:
  - `security.md` - コミット前にチェックすべきセキュリティ項目
  - `coding-style.md` - イミュータビリティ、ファイル構成のルール
  - `testing.md` - TDDワークフロー、80%カバレッジ要件
- 一度整備しておけば「毎回指示する必要がなくなる」
- **注意**: ルールファイルが巨大になりすぎるとコンテキストを圧迫する。関心事ごとにファイルを分割し、モジュラーに管理する
- Claude Codeの「動的ルール読み込み」機能（`.claude/rules/`）で特定のファイル操作時にのみルールを読み込める

### ポイント3: 自動化で確実に（hooks）

- CLAUDE.mdに「ファイル編集後はフォーマットして」と書いても、AIの判断次第で実行されないことがある
- hooksは**特定のイベントで確実に発火する**自動化の仕組み
- 具体的なフック例:
  - TypeScriptファイル編集後にPrettierで自動フォーマット
  - console.logが残っていたら警告を表示
  - git push前にレビュー確認で一時停止
  - 開発サーバーをtmux外で起動しようとしたらブロック
- 設定ミスのリスクがあるため、シンプルなフックから始めて徐々に追加する

### ポイント4: リソースを管理する（MCP）

- MCPを有効にしすぎるとコンテキストウィンドウが大幅に縮小する
- **200kあったコンテキストが70kまで減少する**可能性がある
- 推奨運用:
  - 設定ファイルには20〜30個のMCPを登録
  - プロジェクトごとに**10個以下を有効化**
  - 有効なツールは**80個以下**に抑える
- 著者自身も14個のMCPを設定しながら、有効化しているのは5〜6個だけ

### ポイント5: 設計思想の本質 = コンテキストエンジニアリング

すべてに共通するのは「コンテキストを最適化する」という考え方:

| 仕組み | コンテキストへの効果 |
|---|---|
| エージェント分割 | タスクごとに必要な情報だけを渡す |
| ルール整備 | 毎回説明する情報を事前に与えておく |
| Hooks自動化 | AIの判断に頼らず確実に実行する |
| MCP管理 | 不要なツール情報でコンテキストを浪費しない |

## 4. 当プロジェクト（.company/ 仮想組織）への示唆

### 4.1 組織構造とエージェント設計の一致

記事のアナロジー（agents = 専門部署、rules = 社訓、hooks = 自動チェック機構）は、まさに `.company/` の仮想組織モデルと同じ発想。以下の対応関係を意識して設計を強化すべき:

| .company/ 構造 | everything-claude-code |
|---|---|
| `.company/departments/` | agents（専門部署） |
| `.claude/rules/` | rules（行動指針） |
| `.claude/hooks/` | hooks（自動チェック） |
| skills / commands | skills / commands |
| `.company/secretary/` | 調整・統括エージェント |

### 4.2 コンテキスト最適化の実践

- **ルールの分割管理**: 既に `.claude/rules/` で実践中。CLAUDE.md の肥大化を避ける方針を継続
- **MCP の絞り込み**: 現在有効なMCPが多すぎないか定期的に棚卸しする
- **エージェントへのツール最小化**: サブエージェント（skills）を定義する際は、必要最小限のツールだけを指定する

### 4.3 Hooks の活用余地

- コミット前の自動チェック（リポジトリ判定、機密ファイル検出など）
- セッション終了時の自動サマリー生成
- ファイル編集後の自動フォーマット

### 4.4 段階的な導入

記事の推奨通り、一度に全部作らず **繰り返し発生するタスクから** 段階的にエージェント・フック・ルールを追加していく方針が正しい。
