## 体験設計

### 時間帯適応型UI

ユーザーの意図は時間帯で根本的に変わる。同じ画面を24時間出してはいけない。

| 時間帯 | ユーザーの意図 | UIの方針 |
|--------|-------------|---------|
| 朝 (5-11) | 今日の全体像を受動的に受け取りたい | スケジュール→AIコメント→フォーカスタスク。日記は下部 |
| 昼 (11-17) | 忘れる前にメモ、残タスク確認 | 日記入力→残り予定→タスク。効率重視 |
| 夜 (17-5) | 振り返り、達成感、穏やかに閉じたい | サマリー(達成感)→日記→AIコメント→明日の予定 |

**教訓**: 朝に日記入力をトップに置いたら「まだ書くことがない」。夜に未完了タスクを先に見せたら「罪悪感で日記を書く気が失せる」。

### ブランクページ問題

白紙のテキストエリアは書けない。必ず文脈プロンプトを出す。

- 朝: 「今日やりたいことは?」
- 昼: 「{直前のMTG名}の後、何か思ったことは?」（カレンダーから推定）
- 夜: 「今日一番印象に残ったことは?」（日替わりローテーション）

**根拠**: 構造化プロンプトは白紙より3-5倍の入力完了率（Grid Diary, Reflectlyの知見）

### ポジティブファースト（夜）

夜は「完了したもの」を先に見せる。未完了は「明日に持ち越す?」と穏やかに。

**根拠**: ピーク・エンドの法則。一日の終わりをポジティブにすることで全体の体験評価が向上。

### ファーストビュー3セクション

モバイルでスクロールなしに最重要情報3つ。8セクション全表示はスクロール疲れ。

## AI設計

### AIコメントのパーソナライズ

汎用的な「頑張りましょう」は禁止。10のデータソースから「この人だからこそ」のコメントを生成。

**データソース:**
- 今日の日記 + 直近の感情分析（Plutchik 8感情 + WBI推移）
- 今日/明日のカレンダー
- タスク（完了 + 未完了）
- CEOインサイト（行動パターン・傾向）
- 生活リズム（prompt_log timestamps → 深夜作業率）
- 連続記録日数
- 夢の進捗

**ルール:**
- 最大2-3文、100字以内。長文は絶対NG
- 全データに触れない。**一番響くこと1つ**を選ぶ
- 実行できない約束をしない（「私がやります」禁止）
- 時間帯でトーンを変える（朝=計画提案、昼=軽い承認、夜=深い共感）

**更新タイミング:** 日記投稿後に自動再生成（フィードバックループ）

### LLMへの指示設計

**教訓（HOME一言で3回、NEWS収集で3回失敗して学んだ）:**

1. 「自由に答えて」は不安定。出力形式を厳密に指定する
2. 【絶対禁止】リストを明記する。LLMは「やるな」と言わないとやる
3. 時間帯ルール（深夜は休息メッセージ等）はLLMに判断させず、コードで分岐する
4. 文字数制限はプロンプトとmax_tokensの両方で制御する
5. reasoningモデル（gpt-5-nano等）は`max_completion_tokens`がreasoning+outputの合計。出力が空になるリスクがある

### API呼び出しパターン

**ブラウザからOpenAI APIを直接叩かない。必ずEdge Function経由。**

- 共通ヘルパー `edgeAi.ts` の `aiCompletion()` を使う
- Edge Function `completion` mode で会話管理なしの軽量呼び出し
- APIキーはサーバー側のみ保持
- 自動処理（ping, ブリーフィング, ニュース収集）は `completion` mode（conversationレコードを作らない）

### ハイブリッド分析方式（Self-Analysis）

毎回全データを分析すると日記が増えるほどコスト・レイテンシが増大する。
差分のみだと過去の文脈を失い、直後の再分析で「データなし」エラーになる。

**精神科医モデル**: 毎回カルテを1から読み直すのではなく「前回の診断 + 今回の変化」で判断する。

```
初回: 全データ → 分析結果 + analysis_context(核心引用, 統計スナップショット, 確信度メモ)を保存
更新: 前回結果 + context + 新データのみ → 「前回を更新する」形式で分析 → 新しいcontextを保存
```

**時間の重み付け**: 最新データは生テキスト（重い）、過去データは統計サマリー + 根拠引用のみ（軽い）。
自然と最新データの重みが大きくなるが、前回根拠があるので一貫性も保つ。

### データソース文脈リテラシー

全データソースに「このデータの読み方」を付記してLLMに渡す。

**教訓**: Claude Code への業務指示（「○○して」）を普段の言葉遣いと誤認し、命令的な性格と判定していた。
日記の悩みだけ見て過度にネガティブに分析していた。

| ソース | 性質 | 分析での重み |
|--------|------|-------------|
| 日記 | 本音。ただし悩みバイアスあり | 最重要 |
| AIチャット | 自然な会話・興味・悩み | 重要（日記に次ぐ） |
| Claude Code指示 | 業務指示。命令形はフォーマット | 関心テーマ・時間帯のみ |
| タスク | AI自動完了含む | 内容と種類に注目 |
| 夢・目標 | 価値観の直接証拠 | 重要 |

## データ設計

### キャッシュ戦略

| データ | キャッシュ | 無効化トリガー |
|--------|----------|-------------|
| 天気 | localStorage 1時間 | TTL切れ |
| AIコメント | Zustand in-memory | 日記投稿 / 時間帯変更 / リロード |
| カレンダー予定 | なし（毎回取得） | — |
| タスク/習慣 | Zustand 5分キャッシュ | ミューテーション |

**教訓**: LLM応答のキャッシュは固定TTLではなく「何が変わったら無効化するか」で設計する。時間帯依存のコンテンツに固定1時間キャッシュは不適切。

### タイムゾーン

全レイヤーでJST(Asia/Tokyo)を統一。部分的な修正は新たなバグを生む。

- Google Calendar API: `timeZone: 'Asia/Tokyo'` をURLSearchParamsで指定
- JavaScript Date: `toLocaleString` に `timeZone: 'Asia/Tokyo'` を常に指定
- Supabase: `at time zone 'Asia/Tokyo'` でクエリ

**教訓**: カレンダーUTC/JST地獄で5連続fix。

### RLSポリシー

新テーブルを作ったら必ずRLSポリシーも同時に作る。6回忘れて6回fixした。

## アーキテクチャ設計

### YAGNI原則の徹底

| やったこと | 結果 |
|-----------|------|
| マルチテナント設計 | 即撤回。1ユーザー=1Supabaseプロジェクトで十分 |
| Anthropic API対応 | CORS問題で使えず即削除 |
| 会社セレクターUI | どのページも参照せず。削除 |
| Polaris AI子会社 | 実態なし。circuit会社に統合 |

**ルール**: 「使うかもしれない」機能は作らない。UIは動く機能とセットで追加。

### 1ファイル→分離の成長パターン

最初は1ファイルSPA（Claude Codeで全体を1回で読める）。8000行超で限界 → React + TypeScript に段階的移行。

**教訓**: 最初は1ファイルでいい。育ったら分離する。最初から完璧な構造を作ろうとしない。

### Edge Function設計

- チャット用: agent loop（SSE, 会話管理, ツール使用）
- バックグラウンド処理用: `completion` mode（会話なし、JSON応答）
- ヘルスチェック: `ping` → 即return

**教訓**: 1つのAPIエンドポイントに複数モードを持たせる。用途ごとにエンドポイントを分けない。

## 継続改善の設計（メタ設計）

このシステム自体が改善され続けるための仕組み。「仕組みが仕組みを改善する」構造。

### フィードバックループ

```
ユーザーの行動
  → データ蓄積（diary, prompt_log, tasks, emotion_analysis）
    → AI分析（CEOインサイト, 感情傾向, 生活リズム）
      → AIコメントの質が向上
        → ユーザーが書きたくなる
          → さらにデータが溜まる → ループ
```

**日記ループ**: 書く → 感情分析 → WBI可視化 → パターン発見 → もっと書きたくなる
**AIコメントループ**: データが増える → コメントが的確になる → 信頼が増す → もっと使う

### ナレッジ昇格パイプライン

```
会話内の指摘・修正
  → memory/ に feedback として記録（次セッションから適用）
    → 同じ指摘が2回 → knowledge_base テーブルに昇格
      → confidence ≥ 3 → CLAUDE.md に昇格提案
        → 承認 → 全セッションで自動適用
```

**重要**: 社長が同じことを2回言わなくていい設計。1回目の修正指示がmemoryに残り、2回目は起きない。

### Growth Chronicle（失敗→改善の記録）

```
失敗が起きる
  → growth_events テーブルに記録（what_happened, root_cause, countermeasure, result）
    → Growth ページで可視化（タイムライン）
      → パターンが見える（同じカテゴリの失敗が繰り返されていないか）
        → 対策をCLAUDE.mdやdesign-philosophy.mdに反映
```

**例**: RLSポリシー忘れが6回 → 「テーブル作成時に必ずRLSも作る」ルールを策定 → 再発防止

### 人事部の組織最適化サイクル

```
部署が作業する
  → 社長の追加指示・差し戻しを記録
    → 人事部が5軸で評価（自律完遂率, 一発OK率, 連携効率, 目標寄与度, 稼働率）
      → 低スコアの部署 → CLAUDE.md のルールを改善提案
        → 承認 → 次回から部署の精度が向上
```

**つまり**: AI Agentの能力 = CLAUDE.mdの品質。CLAUDE.mdを継続改善する = システム全体が賢くなる。

### design-philosophy.md 自体の更新トリガー

| トリガー | アクション |
|---------|----------|
| UX設計判断があった | 体験設計セクションに追記 |
| AI/LLMで新しい教訓を得た | AI設計セクションに追記 |
| バグ修正で構造的な問題がわかった | 該当セクションに教訓追記 |
| 新しいアーキテクチャ判断 | アーキテクチャセクションに追記 |
| Growth Chronicleのパターン分析 | 対応するセクションに「教訓」追記 |
| **AI機能の追加・変更** | **HowItWorks AI Featuresタブも更新** |

**memory にルール記録済み** → 次回以降のセッションで自動認識される。

### 知識の3層構造

```
Layer 1: docs/knowledge/     ← 体系的な学習ノート（Claude Code知識等）
  ↓ 実践で効いたルール
Layer 2: knowledge_base       ← 行動ルール（Supabase, ダッシュボードで閲覧）
  ↓ confidence ≥ 3
Layer 3: CLAUDE.md / rules/   ← 全セッション自動適用のルール
```

下に行くほど影響範囲が広く、昇格には実績が必要。

### 自動化されていないが必要なこと（手動チェックリスト）

- [ ] 月1回: design-philosophy.md の棚卸し（古くなった教訓はないか）
- [ ] 月1回: Growth Chronicleのパターン分析（同じ失敗が繰り返されていないか）
- [ ] 月1回: knowledge_base の confidence チェック（昇格候補の確認）
- [ ] 週1回: CLAUDE.md のサイズ確認（200行制限に収まっているか）

## プロダクトの魂: Narrator（語り手）

> 2026-04-05 追加。company-dashboard の根幹思想。

### なぜストーリーなのか

上白石萌歌が「366日」を歌う映像に感動するのは、そこにストーリーがあるから。
スポーツ選手の試合、結婚式の成長ムービー、誰かの克服の話。
ストーリーがあるかないかで、同じ出来事の価値が根本的に変わる。

Mrs. Green Apple のように、才能がストーリーを超越して感動させることもある。
だが、ほとんどの人はそっち側ではない。
**それでも、すべての人の人生には物語がある。**
その物語に光を当てれば、誰の人生にも感動がある。

### 3段階の進化

```
Mirror（鏡）     → データを映すだけ。「今日の感情はこうです」
Compass（羅針盤）→ 提案する。「これが合いそうです」（ルールベース。安直）
Narrator（語り手）→ 物語を読み、語り、導く。「あなたの物語では...」
```

**このシステムは Narrator を目指す。**
データの集計でも、ルールベースの提案でもない。
あなたの日記の行間を読み、感情の波の意味を解釈し、
夢と現実の間にある葛藤を理解し、
**あなた自身がまだ気づいていない自分の物語のテーマを見つける。**

### 設計原則

| 原則 | 意味 | 違反例 |
|------|------|--------|
| **物語で語る** | データの羅列ではなく、弧（arc）として解釈する | 「WBIが5.2です」→ NG。「静かな再構築の時期にいます」→ OK |
| **あなただけの言葉** | テンプレート的な提案は禁止 | 「お疲れなら温泉はどうですか」→ NG。夢リスト×感情×テーマから導く → OK |
| **LLMを安直に使わない、かつケチらない** | ルールベースでできることはルールで。LLMでしかできないことには惜しまず使う | Moment Detector（転機検出）を if文で書く → NG。Theme Finder に gpt-5-nano → NG |
| **過去と今を繋ぐ** | 点ではなく線で理解する | 「今日のWBIは低い」→ 点。「去年の同じ時期にも…」→ 線 |
| **共有は勇気のため** | SNS的な「見せる」ではなく「誰かの勇気になる」ための共有 | いいね数、フォロワー数 → NG。「共感した」ボタンだけ → OK |

### Narrative Intelligence — 4つのエンジン

| エンジン | 役割 | なぜLLMが必要か |
|---------|------|----------------|
| **Arc Reader** | 感情の時系列を「物語の弧」として解釈 | 「この低下は疲労ではなく新しい挑戦への不安」と読み解くのはルールでは無理 |
| **Theme Finder** | 人生の通底テーマを発見 | 数ヶ月の日記×夢×行動から「つくる人」「意味を問う人」を抽出するのは深い推論が必要 |
| **Moment Detector** | 日記から転機をリアルタイム検出 | 「重要な決断」「気づき」「突破」の判定は文脈理解が必要 |
| **Foresight as Question** (旧 Foresight Engine) | 過去の似たパターンを引用し「あの時どうした？」の問いで閉じる | 予言ではない。設計原則 ⑫ で再定義（2026-04-21） |

### Narrative Memory — 3層構造

```
Layer 1: Raw Data       — diary, emotions, dreams, goals, habits（Supabase に蓄積）
Layer 2: Narrative Memory — LLMがデータを「理解」した結果（story_memory に保存、定期更新）
Layer 3: Active Context  — 今の会話に必要な物語文脈（Layer 2 から抽出してプロンプトに注入）
```

Layer 2 が核心。これは SQL クエリの結果ではなく、LLM が数ヶ月分のデータを深く読み解いて生成する「理解」。
identity（人生テーマ）、currentArc（今のフェーズ）、chapters（物語の章）、emotionalDNA（感情パターン）、aspirations（本当の欲求）。

### 既存機能との接続

| 既存機能 | Narrator での役割 |
|---------|------------------|
| 日記 + 感情分析 | Raw Data → Arc Reader の入力。感情の弧の原材料 |
| 自己分析（MBTI/Big5/価値観） | Narrative Memory の identity に統合。「あなたは○○な人」の根拠 |
| 夢リスト | aspirations.surfaceGoals。Theme Finder が深層欲求を推論する土台 |
| 習慣トラッキング | 行動パターンの証拠。「言っていること」と「やっていること」の差分検出 |
| 週次ナラティブ | Chapter の下書き。週次 → 月次 → 四半期で章に昇格 |
| Growth Events | 転機（story_moments）の候補。失敗→対策→進化の弧 |
| AI Partner コメント | Narrative Memory を注入することで「物語を知っている相談役」に進化 |

### 共有機能: Courage Board

- **SNSとは逆の設計**: 評価されない、コメントされない、フォロワーなし
- **「共感した」だけ**: いいねではなく共鳴
- **匿名**: LLMが個人特定情報を自動除去
- **Story Card**: Chapter や転機を美しいカードとして生成・共有
- **目的**: 自分の物語が誰かの勇気になる。結婚式の成長ムービーを日常に

### 差別化の源泉

ChatGPT はあなたを知らない。ジャーナリングアプリは記録するだけ。
このシステムは **記録を物語に変える**。

数ヶ月分の感情 × 行動 × 夢 × 性格の掛け算は、他のどのAIにもできない。

> **キラーフレーズ: 「あなたの人生には、あなたがまだ気づいていない物語がある。」**

詳細設計: `company-dashboard/docs/design/life-companion-evolution.md`

## コーチングを超える — 連続データという固有の強み

> 2026-04-12 追加。「自分に向き合うだけのアプリは虚しいのでは？」という社長の問いから生まれた整理。

### このアプリの存在意義（虚しさへの回答）

人は最終的に誰かと関わらないと意味を感じにくい。だが、人と関わるときに消耗しないためには、まず自分の取扱説明書が要る。シェアハウスがしんどいのも、匿名SNSが流行るのも、「自分の状態が分からないまま他人と密に過ごすのは消耗する」という同じ根。

このアプリは「孤独を完成させる」ものではない。**人と関わるときに消耗しない自分を作る**ための土台。自己理解は他者との関係を切るためではなく、選ぶためにある。

### コーチングとの境界線

| 軸 | コーチング | このアプリ |
|----|-----------|-----------|
| 時間構造 | 点（月1〜週1のセッション） | 線（365日連続） |
| データの粒度 | ユーザーが思い出せる範囲 | ユーザーが忘れた範囲まで |
| 取れる発話 | きれいに話そうとする加工後の言葉 | 加工前のノイズ込みの本音 |
| アクセス | コーチがいる時だけ | 感情のピーク時にいつでも |
| 関係性のコスト | 気を遣う、評価される不安 | 気を遣わない |
| 鋭い問い | コーチの専門性に依存 | 良問いライブラリで再現可能 |
| 沈黙・人格 | コーチが優れる | 原理的にできない |

**コーチが優れるもの**: 鋭い問い、沈黙の圧、非言語の読み取り、人格的な存在感。  
**このアプリが優れるもの**: 連続データ、生のノイズ、24時間性、関係性のしがらみのなさ、過去の自分の参照、ユーザーが忘れた範囲のパターン発見。

### 差別化の核 — 連続データ × 因果の発見

コーチが「最近どう？」と聞いて引き出すのは、ユーザーが**思い出せる範囲**の自己。
このアプリが提示できるのは、**ユーザーが忘れていた範囲**の自己。

> 「3月の第2週、A社の打ち合わせの翌日は必ずスコアが落ちている」

これは人間には観測不可能なパターン。コーチには絶対できない。Narrative Intelligence の Arc Reader / Theme Finder / Foresight Engine はすべてこの方向に位置づく。

### 設計指針: コーチの良問いを日記プロンプトに変換する

ヒアリングが必要なこと（価値観・幸せの瞬間・失敗パターンの掘り下げ）は、すべて**日記の質問例（プロンプト）として設計に組み込む**。

- コーチング本に載っている「価値観を引き出す問い100」を構造化してライブラリ化
- 適切な文脈・タイミングで日記入力プロンプトとして投げる
- ユーザーが書いたものを Theme Finder が横断解釈する

これにより「コーチがいなくても、コーチの良問いを毎日浴びる」状態を作る。1回のセッションより遅いが、深さでは超える可能性がある。

### 価値の3層構造（社長の言葉から）

```
Layer 1: 観察    — 価値観・幸せの瞬間・失敗パターンの生データを集める
Layer 2: 言語化  — それが「腑に落ちる言葉」になる（パターン認識）
Layer 3: 行動    — 言語化を使って自己制御・前向きさに繋げる
```

**多くのジャーナリングアプリは Layer 1 で止まる。コーチングは Layer 2 まで届く。このアプリが目指すのは Layer 3 まで届くこと。**

### 「動けてない日」を責めない設計（必須原則）

ここから導かれる必須原則: **生産性アプリの顔をしてはいけない**。「今日もタスク0件です」と言うアプリは、止まっている人を一番苦しめる相手になる。フリーズの本当の苦しさは、フリーズしていること自体への自己嫌悪。

- 動けていない日も「記録された一日」として扱う
- タスク未達を罪悪感のトリガーにしない
- 「止まっている」を観察対象として外在化する（評価ではなく観察）
- ポジティブファースト原則の延長として、「動けていない日の言語化」も達成として扱う

### 何をしないかの宣言

- ❌ フリーズ検知して詰める
- ❌ タスク未達を可視化して罪悪感を煽る
- ❌ 「もっと頑張りましょう」系の介入
- ❌ 局所的な「気分が上がる工夫」（根本に効かない）
- ✅ 価値観・幸せの瞬間・失敗パターンの連続観察
- ✅ ユーザーが忘れた範囲のパターン提示
- ✅ 言語化を手伝う良問いの継続投入
- ✅ 「自分の取扱説明書」が自分の手で更新されていく感覚

## AI Agent System としての設計原則

> 2026-04-19 追加。Claude Code 設計空間論文（arXiv:2604.14228）から focus-you に取り込む思想。
> Anthropic Claude Code のソース解析から抽出された **5価値・13原則** のうち、個人向け AI プロダクトに効く9項目を focus-you の文脈で再解釈する。

focus-you は AI Agent System の一種であり、Claude Code のような production agent と同じ設計問題を抱える。ただし**デプロイ文脈が違う**（開発ツール vs 個人の自己理解ツール）ため、原則は同じでも実装の答えは変わる。

### ① 長期的な自己理解能力の保全（第6の価値軸）

Claude Code 論文が「evaluative lens」として提案する第6の価値。短期的な能力増幅と引き換えに、長期的な人間能力が萎縮してはいけない、という視点。

**実証データ（論文より）**:
- Cursor 導入 807 リポジトリ: コード複雑度 +40.7%、初期速度 +281% → 3ヶ月でベースライン
- 16-developer RCT: 本人は 20% 速く感じたが実際は **19% 遅く** なった
- EEG 研究: LLM 使用者は AI 除去後も神経接続の弱化が持続
- Shen & Tamkin: AI 支援下で理解度テスト -17%

**focus-you での含意**:

| AI 日記アプリが陥る罠 | focus-you が取るべき逆 |
|---|---|
| AI があなたの代わりに考える | あなたが考えたことを鏡として返す |
| AI が日記を書いてくれる | あなたが書いた一文から感情の構造を見せる |
| AI が意思決定する | AI が判断の根拠を見える化してユーザーに決めさせる |
| 依存が増えるほど "便利" | 依存が減るほど "あなたが育つ" |

**評価指標の二層化**:
- **短期指標**: DAU、書き込み頻度、継続率 ← 既存の KPI
- **長期指標（新設）**: 感情語彙の拡大率、意思決定の後悔率低下、フリーズ頻度の減少、**AI なしでも自分を言語化できる度合い**

**キラーフレーズ候補**: 「このアプリを使うほど、このアプリが要らなくなる」。Narrator が成長させるのは **アプリへの依存** ではなく **ユーザー自身の物語理解能力**。

### ② Approval Fatigue 対策 — 提案を出し惜しむ設計

Claude Code で 93% の permission prompt は慣れで承認される（Hughes 2026）。auto-approve 率は <50 セッションで 20% → 750 セッションで 40%+ と漸増。**通知・確認を増やすほど、ユーザーは機械的に承認するだけになり、安全機構が無力化する**。

既存の「AI PaRTner トーンガイド」（わかってる人のボソッと一言）の **理論的裏付け** として位置付ける。

**実装指針**:
- **Narrator の登場頻度は可変** — 毎日は出さない。感情の揺れ幅・イベント密度で出現判定
- **提案・リマインド系通知は「出さない判断」を classifier で決める** — ルールベースで silent にする方向に倒す
- **確認ダイアログを増やさず、最初に境界を決めて後は黙って動く** — onboarding で境界を固め、日常の確認を最小化
- **通知頻度の上限を設ける** — 1日 N 回まで、同カテゴリは M 時間空ける

**反パターン**: 毎日 Narrator が長文で語りかけてくる → 慣れて読まなくなる → Narrator 無効化と同じ。

### ③ Append-only + Read-time Projection（過去を書き換えない）

Claude Code の session transcript は append-only JSONL、compaction も過去を削除せず境界マーカーと summary event を追記するだけ。Resume/Fork もトランスクリプトを replay して復元。

**なぜ focus-you で重要か**:
- 日記は **ユーザーの履歴そのもの**。過去を書き換えるプロダクトは根本で信頼を失う
- 失敗した日・動けなかった日も「記録された一日」（既存原則「動けてない日を責めない」と一致）
- 「3月の第2週のパターン」を発見するには、過去が**変化していない**ことが大前提

**設計方針**:

| 対象 | 方針 |
|---|---|
| `diary_entries` | edit は許すが、revision は別レコード（`diary_entry_revisions`）として追記。UI で履歴を見られる |
| `emotion_analysis` | 新しい分析が出ても過去の分析は消さない。`superseded_by` で新旧を繋ぐ |
| `story_memory` | `memory_type` ごとに最新を持ちつつ、過去スナップショットを `story_memory_archive` に残す |
| `ceo_insights` | 日次インサイトは追記のみ、上書きしない |
| 週次・月次ダイジェスト | **read-time projection**。元エントリには触らず、view として生成 |

**教訓**: 「要約されたビュー」は計算結果であり、原データではない。原データは **最小単位で不変**、view は **いつでも作り直せる** が設計の核。

### ④ 5層 Context Shapers — 日記の長期蓄積アーキテクチャ

Claude Code は毎モデル呼び出し前に5段の圧縮を順に実行: Budget reduction → Snip → Microcompact → Context collapse → Auto-compact。**RAG/embedding に飛びつく前に、段階的に軽い圧縮から試す**設計。

**focus-you の長期蓄積に適用**:

```
Layer 1: Budget reduction  — 個別エントリのサイズ上限（超過分は reference 化）
Layer 2: Snip             — 古いメタデータ・タグの軽トリム（本文は保持）
Layer 3: Microcompact     — キーワード・感情タグの細粒度抽出
Layer 4: Context collapse — 月単位の read-time projection（月サマリ表示、原エントリは残る）
Layer 5: Auto-compact     — LLM 要約（最終手段、年単位アーカイブ用）
```

**embedding を避ける理由**（論文 §7.2 の思想）:
- ユーザーが「何が検索で引かれたか」見えない
- 感情的な記憶は近傍検索より**時系列・文脈の連続性**が重要
- ファイル粒度（日付・週・月）の LLM スキャンで十分、entry 粒度の embedding はプライバシー・可読性で劣る

**既存の「ハイブリッド分析方式（精神科医モデル）」は Layer 4-5 に相当**。Layer 1-3 は今後の長期蓄積に備えて先行設計しておく。

### ⑤ File-based Transparency over Opaque DB

Claude Code の設計思想で最も強いメッセージ: *stored context should be inspectable and editable by the user*（CLAUDE.md は平文 Markdown、embedding/opaque DB は不採用）。

**focus-you の含意**:
- ユーザーは自分の **全データ（日記・感情分析・Narrator memory）を export/edit/import できる**
- 感情ラベルは編集可能（「AI が怒りと判定したが違う」を上書きできる。ただし③に従い履歴は残す）
- プライバシー設定は「不透明な checkbox」ではなく、**どのデータが何に使われるか** を見える形で提示
- Narrator が生成した `story_memory` も **ユーザーが読めて、間違いを正せる**

**反パターン**: 「AI があなたをこう理解しています」を **変更不可能な読み取り専用** で見せる。これはユーザーを AI の解釈に従属させる。逆に、ユーザーが AI の解釈を書き換えると、その差分自体が重要な自己理解のデータになる。

**実装ヒント**: `/data-export` ページを作り、全テーブルの自分のデータを JSON でダウンロード可能にする。vendor lock-in 回避＋プロダクトとしての信頼獲得。

### ⑦ Deny-first + Values over Rules（外部連携の原則）

Claude Code の安全姿勢: deny-first with human escalation。未知のアクションはデフォルト拒否、人間にエスカレ。`deny` は `allow` を常に上書き。

**focus-you で外部連携（Slack/Calendar/メール送信、Webhook、外部 AI サービス呼び出し）を足す時の原則**:

| ルール | 意味 |
|---|---|
| **デフォルト拒否** | 新規統合は最初はすべて off。ユーザーが明示的に on にする |
| **deny は常に勝つ** | 「全AIコメントを許可」より「感情的に揺れてる時は無効」が強い |
| **values over rules** | 硬直ルールより、文脈判断＋決定論的ガードレール。例: 「夜22時以降は通知なし」は rule、「疲れが見えるときは黙る」は value |
| **reversibility-weighted** | read-only 系（カレンダー閲覧）は軽く、write 系（投稿・送信）は重く。Narrator が SNS に自動投稿は絶対しない |

### ⑧ Pre-trust Initialization Ordering（プラグイン追加時の警戒）

Claude Code で発見された CVE-2025-59536 / CVE-2026-21852 の教訓: **hook/MCP/settings ファイル読込が trust dialog の前に実行される時間窓** があり、ここに悪意あるコードを仕込める。**空間順序（どこで検証するか）だけでなく時間順序（いつ検証するか）を描くべき**。

**focus-you が今後 plugin / integration を増やす時の設計チェックリスト**:

1. 設定ロード → trust dialog → 実行 の **時系列を設計図で描く**
2. 外部コードが `SessionStart` 前に走る経路がないか（auto-pull、hook など）
3. ユーザー同意前に外部 URL を fetch する経路がないか
4. localStorage/env 経由の hijack 経路がないか
5. 新統合の追加は「既存の trust 境界の**内側**」か「新しい trust 境界が必要」か明示する

**現状の focus-you での該当箇所**: Google Calendar 認証、Edge Function 呼び出し、将来的な OpenAI 以外の LLM 統合、将来的な plugin システム。すべて追加時にこのチェックを通す。

### ⑩ Silence over Noise — 価値なきコメントは出さない（社長指示 2026-04-19）

**② Approval Fatigue 対策の先鋭化。** 毎回コメントを生成する姿勢そのものが危険。無難なことを言うくらいなら、何も言わないほうが信頼を壊さない。

**原則**:
- **LLM 自身に「価値あるコメントが出せるか」を判定させる**。出せないと判断したら `SILENT` とだけ返すようプロンプトに指示する
- **fallback を削除する**。「今日も穏やかに始めましょう」のような無難な一言は、ユーザーから見ると AI が何も見ていないのと同じ、むしろ「適当なことを言う存在」として信頼を損なう
- **黙る方が誠実**。既に実装されている `useMomentDetector` の「大半の日記は転機ではない。10件に1件程度」の姿勢を、**全ての AI コメント生成に適用する**

**実装パターン**:

```ts
// プロンプトに追加
"価値ある一言が思いつかない場合は SILENT とだけ返してください。
無難なコメント（「お疲れさまです」「今日も穏やかに」等）を絞り出すくらいなら
黙るほうが誠実です。"

// クライアント側で検出
if (result === 'SILENT' || result.startsWith('[SILENT]')) {
  setMessage(null)  // 表示しない
}

// fallback は null を返す（無難な一言で埋めない）
```

**適用対象**:

| 機能 | 現状 | 変更後 |
|---|---|---|
| AI PaRTner (useMorningBriefing) | fallback で「今日も穏やかに」 | SILENT なら非表示 |
| Moment Detector | 既に 10件に1件の閾値 | ✅ 維持（模範ケース） |
| Theme Finder | 定期更新のみ | SILENT 判定追加（変化なければ更新しない） |
| Arc Reader | 定期更新のみ | SILENT 判定追加 |
| Foresight | 兆候があれば生成 | SILENT をデフォルト姿勢に |

**例外**: ユーザーが明示的に「コメントくれ」ボタンを押した時は、SILENT でも何か返す（明示要求 > 沈黙原則）。自動生成の時だけ沈黙を許す。

**キラーフレーズ**: *「一番親身なパートナーは、毎回何か言う人ではなく、言うべき時を見極める人。」*

### ⑪ Active vs Passive Response Boundary — 能動相談はアドバイスOK、受動生成は想起誘導のみ（社長指示 2026-04-21）

**⑩ Silence over Noise の精緻化。** 沈黙か発話かの二択ではなく、**発話の性格を起点で切り替える**。

focus-you のターゲットは「行動はわかっているが、一歩が出ない層」。答えは本人の記録の中に既にある。外部から処方箋を与える層（=コーチ・メンターが必要な層）はプロダクトのターゲット外。この前提から、AI の振る舞いは起点で二層に分かれる:

| チャネル | 起点 | 許容される返答 | 禁止事項 |
|---|---|---|---|
| **能動チャット（AI Partner 対話）** | ユーザーが質問・相談を投げる | 記録・文脈に基づくアドバイス、提案、助言 | 無難な定型句、「ご自身で考えて」と突き放す |
| **受動生成（Narrator / 未来の自分から / 週次ナラティブ / ブリーフィング等）** | システムが自動で生成 | 問い返し、想起誘導、本人記録の引用、沈黙（SILENT） | 相談されていない助言、解釈の押し付け、汎用アドバイス |

**コア原則**: 本人の記録から答えを呼び起こすのが正。相談されていないのに助言を混ぜるのは、ターゲットの前提（答えは本人の中にある）を裏切る。

**実装指針**:
- 受動生成プロンプトに「アドバイスではなく、問い返し or 記録からの想起誘導のみ」を明示する
- 受動生成で助言を書きたくなったら SILENT を返す（⑩ と組み合わせる）
- 能動チャットのプロンプトには「相談されたらアドバイスしてよい」を明示し、萎縮させない
- ボタン押下・明示コマンド等の「明示的要求」は能動扱い（受動生成の禁止には当たらない）

**反パターン / 望ましい例**:
- ❌ 「未来の自分からの一文」に「○○した方がいいですよ」が混ざる
- ❌ ブリーフィングで相談されていない行動を勝手に提案する
- ❌ AI Partner チャットで「ご自身で答えを見つけてください」と突き放す
- ✅ 「未来の自分からの一文」が「3月の朝、同じ迷いを書いていた」と記録を引いて終わる
- ✅ ブリーフィングが「水曜午後のあなたの集中力、今日はどう使う？」と問いで閉じる
- ✅ AI Partner がチャットで「以前こう書いていたので、こうしてみては？」と明示的に提案する

**updates 履歴**:
- 2026-04-21 社長指示: ターゲット層（一歩が出ない層）は維持。AI Partner 能動チャットではアドバイス可、受動生成は想起誘導のみ。

### ⑫ Positioning Focus — 連続データ × 忘れていた範囲のパターン発見（社長判断 2026-04-21）

リサーチ部の競合調査（StoriedLife / Life Story AI / Autobiographer / Rosebud / Mindsera / Reflection.app / Stoic iOS26 / MIT Future You 等、ナラティブ系・AIジャーナル系・Foresight 特化系 15社超）を踏まえ、表看板を確定した。

**市場事実（2026-04-21 時点）**:
- ナラティブ系（StoriedLife / Life Story AI / Autobiographer）は「過去回顧録限定」ではない。継続サブスク + weekly prompt + cross-session memory で進行中の物語を既に扱う
- 立ち位置が最も近い競合は **Rosebud**。Wrapped 2025 の archetype / arcs / moments が Narrator 4エンジンと機能名レベルで重複
- 「未来予見（Foresight）」は 2026-2027 業界方向の predictive wellness トレンドに吸収されつつあり、表看板としては脆い

**表看板（Main Axis）**:

> **連続データ × 忘れていた範囲のパターン発見**

根拠の構造差（競合には追いつきにくい）:

| 軸 | focus-you | ナラティブ系（音声会話型） | Rosebud/Stoic（単一テキスト） |
|---|---|---|---|
| 入力粒度 | 日記 + 感情分析 + カレンダー + タスク + 夢 + 自己分析（多チャネル） | 音声会話 or 週次テキスト単独 | テキスト日記単独 |
| 観測範囲 | **ユーザーが忘れていた範囲**（連続データから検出） | ユーザーが思い出せる範囲（インタビューで引き出す） | 書き込んだ範囲 |
| 出力目的 | 今日どう動くか（`past → now decision`） | 次世代に残す（`past → legacy`） | 自己成長の可視化 |

**キラーフレーズ（候補）**:
> あなたが忘れていた3月の、あなた自身を覚えています。

**副看板（一緒に運用する）**:
- 「動けてない日も記録された一日」（「コーチングを超える」原則と一体）
- 「進行中で未完成な物語」（ナラティブ系との差別化）
- 「このアプリを使うほど要らなくなる」（業界逆張り、① 長期的自己理解能力の保全と一体）

**Foresight as Question — 予言ではなく兆候の問いへ再定義**:

⑪ Active vs Passive Response Boundary との整合のため、Foresight Engine を次のように再定義する:

- ❌ 「未来予見」「予言」「次はこうなる」（受動生成で予測を宣言するのは⑪違反）
- ✅ 「今日のパターンは2月の第3週と似ています。あの時、あなたはどう動きましたか？」（問いで閉じる）

> 予言するのは占いの仕事。focus-you の仕事は、あなたが忘れていた過去に光を当てて、**あなた自身の答えが浮かび上がる場所を用意すること**。

**業界リスク**:
- 2027 H1 までに Rosebud / Reflect / Mindsera のいずれかが「Chronicle」類似機能を出してくる見込み
- focus-you は 2026-2027 のうちに「連続データの粒度」を明確な構造差として確立する必要がある
- 名称レベルの重複（Rosebud Wrapped と Narrator 4エンジン）は LP/Blueprint で「連続データ粒度」の差で言い訳せず勝負する

- design-philosophy ⑫ を Blueprint Design Philosophy タブにも反映

**updates 履歴**:

### ⑨ Subagent Summary-only Return（秘書組織のハンドオフ仕様）

Claude Code の subagent は **full transcript を親に返さず、最終応答と metadata だけ返す**。生 transcript は sidechain `.jsonl` + `.meta.json` に保存。**context-as-bottleneck 原則の核心実装**。

**秘書組織（HD + PJ 別会社）への適用**:

| 項目 | 現状 | 追加原則 |
|---|---|---|
| ハンドオフフォーマット | YAML（`.claude/rules/handoff.md`）で既に規定済み | ✅ |
| 部署の transcript 保存 | agent_sessions にイベント記録 | ✅（既存の仕組みで代替可） |
| 親に返す情報 | summary（handoff YAML の text） | **full 会話ログを親に注入しない** |
| 生の transcript | 部署成果物ファイル + agent_sessions のイベント | **sidechain 的に別管理、親コンテキストに混入させない** |

**focus-you プロダクトとしての応用**: もし将来「複数視点から日記を分析」（感情視点・行動視点・関係視点）を実装するなら、各視点の full transcript ではなく **summary のみをユーザーに表示**、生 transcript は別テーブルに残す。ユーザーが深堀りしたい時だけ展開する。

### 通底する3つのコミットメント（Claude Code 論文より継承）

1. **Graduated layering over monolithic mechanisms** — 通知・権限・圧縮はすべて段階的な多層にする。単一ルールで解決しようとしない
2. **Append-only designs that favor auditability over query power** — 監査性と再構築可能性を、クエリ性能より優先する
3. **Model judgment within a deterministic harness** — LLM に判断を任せ、周辺を決定論的インフラで支える（1.6% vs 98.4% の比率）

**focus-you の実装配分も同じ目安**: LLM プロンプト・判断ロジックは全体の 2% 程度に留め、残り 98% は **データ整合性・権限・キャッシュ無効化・時間帯判定・文脈組み立て** などの決定論的インフラに投資する。Narrator の質は LLM の賢さではなく、**Narrator に渡すコンテキストの質**で決まる。
